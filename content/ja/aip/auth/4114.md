# 相互TLSによるデバイス証明書認証

相互TLS（別名mTLS）認証は、TLSハンドシェイクにおいてクライアントとサーバーの両方のIDの認証を可能にし、GCPエンタープライズ顧客向けのContext Aware Accessのためのデバイス証明書認証の基盤を提供する。デバイス証明書認証では、サーバーは認可判断を行う際にトランスポート層のクライアント証明書を考慮する。クライアント証明書の使用は、信頼されたデバイスからのアクセスであるというより強力な指標を提供する。これにより、エンタープライズ顧客は信頼されていないデバイスからのアクセスを制限できる。

**注:** このAIPは言語に依存しない形でガイダンスと要件を説明するため、特定の言語や環境では不正確または不適切となる可能性がある一般的な用語を使用している。

## ガイダンス

本セクションでは、mTLSを介したデバイス証明書認証（別名DCA）をサポートするための一般的なガイダンスについて説明する。

### DCAのためのアプリケーション デフォルト クレデンシャル

ユーザーは、クライアントオプションによる手動設定ではなく、[ADC][0]を通じてDCAを有効にする**べきである**（should）。

DCAのためのADCには2つの側面がある。1つ目はデバイス証明書の自動取得である。2つ目はサービスエンドポイントのmTLSバージョンへの自動切り替えである。クライアントは、DCAのためのADCをサポートする際、これらの両方の側面をサポート**しなければならない**（must）。

### 期待される動作

DCAのクライアントサポートは、クライアントオプションで指定されたユーザーオーバーライドを優先**しなければならない**（must）。以下の決定ツリーを使用する**べきである**（should）：

1.  ユーザーがクライアントオプションでデバイス証明書とエンドポイントオーバーライドの両方を指定した場合、そのまま使用する。
1.  ユーザーがデバイス証明書を指定しなかった場合、デフォルトのデバイス証明書を取得して使用しようと試みる。
1.  ユーザーがエンドポイントオーバーライドを指定しなかった場合、デバイス証明書が利用可能であればデフォルトのmTLSエンドポイントを使用し、そうでなければデフォルトの通常エンドポイントを使用する。

上記のロジックから生じる影響：

1.  ユーザーが非mTLSエンドポイントオーバーライドを指定したがデバイス証明書が利用可能な場合、証明書をそのまま渡し、サーバーの判断に任せる。
1.  ユーザーがmTLSエンドポイントオーバーライドを指定したがデバイス証明書が利用できない場合、即座に失敗せず、接続時にサーバーがエラーを返すようにする。

上記の動作により、エンドポイントオーバーライドがmTLS URLかどうかを解析するクライアント側のロジックの導入を回避できる。URLパターンはいつでも変更される可能性があるためである。

### デフォルトのmTLSエンドポイントの取得

サービスのデフォルトのmTLSエンドポイントは、正規表現パターンで生成するのではなく、Discovery Documentのフィールド **"mtlsRootUrl"** から読み取る**べきである**（should）。

### SecureConnectを介したデフォルトのデバイス証明書の取得

デフォルトのデバイス証明書は、[EndpointVerification][1]ワークフローを使用して取得する**べきである**（should）。このワークフローは、ネイティブヘルパーを介してプラットフォーム固有のクレデンシャルストア（macOSのKeyChainなど）から証明書を取得する。

ネイティブヘルパーを介した証明書のエクスポートには、以下の形式のwell-knownなgcloudメタデータファイルで指定された**"cert provider command"**の実行が含まれる：

```json
{
   "version": 1
   "min_cloud_sdk_version": "240.0.0"
   "has_client_cert": true
   "endpoint_verification_error": ""
   "cert_provider_command": "[absolute_path_to_provider_command] --fetch_client_cert"
}
```

LinuxおよびmacOSプラットフォームでは、上記のメタデータファイルは **"~/.secureConnect/context_aware_metadata.json"** に配置される。

cert providerコマンドは証明書をstdoutに出力する。これはX.509証明書の直後に秘密鍵が続く形式である：

  -----BEGIN CERTIFICATE-----
  Common Name: Google Endpoint Verification
  Valid From: November 10, 2019
  Valid To: November 10, 2020
  Serial Number: 4921083229008411918 (0x444b331faf2dbd0e)
  ...
  -----END CERTIFICATE-----
  -----BEGIN PRIVATE KEY-----
  ...
  -----END PRIVATE KEY-----

### ECPを介したデフォルトのデバイス証明書の取得

Enterprise Certificate Proxy（ECP）プロジェクトは、デバイス証明書を取得するための新しく推奨される方法である。従来のSecureConnectメカニズムと比較して、2つの主要な利点がある：

1.  自己署名証明書に依存するのではなく、ネイティブキーストアやTPMに保存されたエンタープライズ証明書と秘密鍵の使用が可能。
1.  署名操作をキーストアに委任するため、秘密鍵がセキュリティレルムから流出することはない。

ECPの設定の詳細については、[ECP公開ドキュメント][2]を参照のこと。

### 環境変数

ADC for DCAの動作を変更する必要がある状況がある（例：統合テストやフェイルセーフ）。これらのシナリオに対応するため、以下の環境変数をサポートする**べきである**（should）。

**GOOGLE_API_USE_MTLS_ENDPOINT**: **"always"** の場合、常にmTLSエンドポイントを使用する。**"never"** の場合、常に通常のエンドポイントを使用する。**"auto"** の場合、デフォルトの動作（デバイス証明書が利用可能な場合はmTLSエンドポイントを使用）を行う。この環境変数のデフォルト値は"auto"である。

**GOOGLE_API_USE_CLIENT_CERTIFICATE**: **"true"** の場合、一般的なガイダンスで説明されているようにデバイス証明書認証がサポートされる。**"false"** の場合、ユーザーが指定した場合でもデバイス証明書を使用しては**ならない**（must not）。デフォルト値は2024年5月3日時点で"true"である**べきである**（should）。DCA機能を無効にしたいユーザーは、この環境変数を明示的に"false"に設定**しなければならない**（must）。

### ファイアウォールとIPアドレスのガイダンス

任意のGCPサービスについて、そのmTLSエンドポイントは非mTLSエンドポイントとは異なるIPアドレスを持つが、同じIP範囲内に収まることが期待される。IP範囲管理に関するGCPのガイダンスはサービスごとに異なる。例については[Compute Engine IP範囲ドキュメント][3]を参照のこと。エンドユーザーがIP範囲ではなく正確なIPアドレスに基づいてファイアウォールルールを設定している場合、"auto" mTLSエンドポイントへのアップグレード動作の影響を受ける可能性がある。この場合のベストプラクティスは、正確なIPアドレスの一致を期待するルールを追加せず、公開GCPドキュメントに従って範囲ベースのIPルールを使用することである。
