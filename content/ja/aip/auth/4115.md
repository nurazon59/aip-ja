# Google Cloud仮想環境向けデフォルトクレデンシャル

クライアントが[Google Compute Engine（GCE）][0]、[Serverless][1]、[Google Kubernetes Engine（GKE）][2]などのGoogle Cloud仮想環境で実行される場合、認証ライブラリはGoogleのデフォルトの相互TLS（mTLS）クレデンシャルを活用し、インスタンスのバインドトークンを取得して**もよい**（may）。認証ライブラリはデフォルトのmTLSクレデンシャルとバインドトークンを使用してGoogle APIにアクセスして**もよい**（may）。

mTLS認証は、TLSハンドシェイクにおいてクライアントとサーバーの両方のIDの認証を可能にする。Google仮想環境で実行されるアプリケーションは、X.509 SPIFFE Verifiable Identity Documents（SVID）を使用してGoogle APIに認証できる。これらのSVIDは、証明書所有者のIDを指定するSPIFFE IDを含むX.509証明書である。

バインドトークンは、mTLS接続の確立に使用されたクレデンシャルの何らかのプロパティにバインドされたアクセストークンである。バインドトークンの利点は、適切なアクセスポリシーが設定されている場合、正しいバインディング情報を持つmTLSクレデンシャルを介して確立されたセキュアチャネル上でのみ使用できることである。したがって、バインドトークンは、盗まれて悪意を持ってリプレイされる可能性があるbearerトークンよりも安全である。

このAIPは以下のフローを説明する：

1.  メタデータサーバー（MDS）エンドポイントを通じて設定を取得する。設定は、GoogleのデフォルトのmTLSクレデンシャルにアクセスする方法を指定する。
2.  バインドトークンを要求する。

**注:** このAIPは言語に依存しない形でガイダンスと要件を説明するため、特定の言語や環境では不正確または不適切となる可能性がある一般的な用語を使用している。

## ガイダンス

### デフォルトのmTLSクレデンシャルへのアクセス

**注:** GoogleのデフォルトのmTLSクレデンシャルを使用しようとする前に、クライアントはリモートのGoogle APIエンドポイントがmTLSをサポートしているかをまず確認**しなければならない**（must）。リモートエンドポイントがmTLSをサポートしていない場合、クライアントはTLSを使用してエンドポイントに接続する**べきである**（should）。エンドポイントがmTLSをサポートしているかどうかの確認方法は、このAIPの範囲外である。リモートエンドポイントがmTLSをサポートしている場合、クライアントはTLSにフォールバックする前に、まずmTLSを使用した接続を試みる**べきである**（should）。リモートAPIのmTLSエンドポイントを見つける方法は、このAIPの範囲外である。ユーザーが[デバイス証明書認証（DCA）][4]を有効にしている場合、クライアントはmTLSクレデンシャルとしてDCAを優先**すべきである**（should）。

GoogleのデフォルトのmTLSクレデンシャルを活用するために、クライアントはMDSから設定を取得する**べきである**（should）。すべての仮想環境（GCE、Serverless、およびGKE）のMDSは、GoogleのデフォルトのmTLSクレデンシャルにアクセスする方法を指定する設定を提供するHTTPエンドポイントを公開している。このエンドポイントはmTLS設定エンドポイントと呼ばれる。

MDSのmTLS設定エンドポイントのURLは以下の通りである：
```
http://metadata.google.internal/computeMetadata/v1/instance/platform-security/auto-mtls-configuration
```

MDSのmTLS設定エンドポイントへのリクエストは、パラメータやペイロードのないHTTP GETリクエストである**べきである**（should）。

MDSのmTLS設定エンドポイントからの応答には、以下の情報が含まれる**べきである**（should）：

* **Secure Session Agent**アドレス：クライアントはmTLSクレデンシャルに直接アクセスできない。Secure Session AgentがデフォルトのmTLSクレデンシャルを管理する。クライアントはSecure Session Agentを通じてのみmTLSクレデンシャルを使用できる。アドレスはIP:portアドレス、またはUnix Domain Socket（UDS）を表すファイルパスの形式を取る。

クライアントは以下の手順に従って、GoogleのデフォルトのmTLSクレデンシャルにアクセス**しなければならない**（must）：

1. リモートエンドポイントがmTLSをサポートしているかを確認する。
   * サポートしている場合、ステップ(2)へ。
   * サポートしていない場合、ステップ(3)へ。
2. MDSのmTLS設定エンドポイントにリクエストを送信する。リクエストが成功し、応答にSecure Session Agentアドレスが含まれている場合、そのアドレスを使用してGoogleのデフォルトのmTLSクレデンシャルにアクセスし、ステップ(4)へ。リクエストが失敗した場合、または応答に空のアドレスが含まれている場合、ステップ(3)へ。
3. TLSにフォールバックする [END]。
4. mTLSハンドシェイク中のクライアント認証にSecure Session Agent（[例][3]）を使用するようTLSライブラリを構成する。

### バインドトークンの要求

バインドトークンを使用してGoogle APIにアクセスするために、クライアントはMDSにトークンを要求する**べきである**（should）。すべての仮想環境（GCE、Serverless、およびGKE）のMDSは、アクセストークンを提供するHTTPエンドポイントを公開している。このエンドポイントはアクセストークンエンドポイントと呼ばれる。

MDSのアクセストークンエンドポイントのURLは以下の通りである：
```
http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
```

MDSのアクセストークンエンドポイントへのリクエストはHTTP GETリクエストである**べきである**（should）。リクエストには、カンマ区切りのスコープリストを持つ"scopes"URLパラメータがあって**もよい**（may）。認証ライブラリは、呼び出し元が必要に応じてカスタムスコープのリストを指定できるようにし、必要なときに"scopes"パラメータをリクエストに追加する**べきである**（should）。ランタイム環境によっては、カスタムスコープのリクエストはサーバーによって透過的に無視されるか、実行されて**もよい**（may）。

MDSのアクセストークンエンドポイントからの応答には、以下のJSON形式のアクセストークンが含まれる**べきである**（should）：

```json
{
      "access_token": "YOUR_ACCESS_TOKEN",
      "expires_in": 3599,
      "token_type": "Bearer"
 }
```

クライアントは、既存のトークンの有効期限が切れた場合、Google APIの新しいアクセストークンを要求するために以下の手順に従わ**なければならない**（must）：

1. MDSアクセストークンエンドポイントにHTTPリクエストを送信し、応答からアクセストークンを取得してステップ(2)へ。
2. ステップ(1)のトークンをGoogle APIへのリクエストに添付する。
