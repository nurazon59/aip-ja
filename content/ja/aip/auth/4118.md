# ワークロードクレデンシャルを使用した相互認証

相互TLS（別名mTLS）認証は、TLSハンドシェイクにおいてクライアントとサーバーの両方のIDの認証を可能にする。ワークロードクレデンシャルを使用すると、Google Cloudで実行されるアプリケーションは[X.509 SPIFFE Verifiable Identity Documents][0]（SVID）を使用してGoogle APIに認証できる。これらのSVIDは、証明書所有者のIDを指定する[SPIFFE ID][1]を含むX.509証明書である。X.509 SVIDを使用したmTLS認証は、クライアントがTLSハンドシェイクを実行する際にX.509 SVIDを使用するときに発生する。

**注:** このAIPは言語に依存しない形でガイダンスと要件を説明するため、特定の言語や環境では不正確または不適切となる可能性がある一般的な用語を使用している。

## ガイダンス

ユーザーがトークンバインディングを有効にする場合、[ADC][2]を介して行う**べきである**（should）。本セクションでは、そのような認証をサポートするための一般的なガイダンスについて説明する。

### Google Cloudでのワークロードクレデンシャルのプロビジョニング

Google Cloudでは、以下のいずれかの方法を使用してワークロードクレデンシャルをプロビジョニングする**べきである**（should）：

- [Envoyを使用したサービスセキュリティの設定][3]。
- [proxyless gRPCを使用したサービスセキュリティの設定][4]。

ワークロードクレデンシャルが適切に使用されるためには、認証ライブラリはサービスエンドポイントのmTLS対応エンドポイントへの自動切り替えをサポート**しなければならない**（must）。

### ワークロードクレデンシャルの使用

ユーザーは、証明書設定のgcloudメタデータファイルを介してワークロードクレデンシャルを使用するよう[ADC][2]を構成する**べきである**（should）。ワークロードクレデンシャルは、以下のように **"cert_configs"** タイプとして追加できる：

```json
{
  "version": 1
  "cert_configs": {
    "workload": {
      "cert_path": "path/to/cert/file"
      "key_path": "path/to/key/file"
      "workload_identity_provider": "..."
      "authenticate_as_identity_type": "gsa/native"
      "service_account_email": "..."
    },
    "keychain": {
      ...
    },
    "pkcs11": {
      ...
    },
    "windows": {
      ...
    },
  },
  "libs": {
   ...
  }
}
```

LinuxおよびmacOSプラットフォームでは、上記のメタデータファイルはwell-knownなgcloud設定ディレクトリの **"~/.config/gcloud/certificate_config.json"** に配置される。このファイルのデフォルトの場所は `GOOGLE_API_CERTIFICATE_CONFIG` 環境変数を使用して変更できることに注意。

以下は、ワークロードクレデンシャルに関連する **"workload"** 証明書情報タイプのフィールドのリストである：

- **"cert_path"**: 指定された値は、ワークロード証明書ファイルを特定するためのフルパスとして使用される。このファイルには、PEMエンコードされたX.509証明書チェーン（リーフからルートの順序）が含まれている**しなければならない**（must）。リーフ証明書は有効なX.509 SVIDである。チェーンはリーフ証明書のみで構成されていて**もよい**（may）。

- **"key_path"**: 指定された値は、ワークロード秘密鍵ファイルを特定するためのフルパスとして使用される。このファイルには、**"cert_path"** で指定されたX.509証明書に関連付けられたPEM形式の秘密鍵が含まれている**しなければならない**（must）。

**"workload_identity_provider"**、**"authenticate_as_identity_type"**、**"service_account_email"** フィールドの説明は[mTLS Token Binding][5]に記載されている。

ワークロードクレデンシャルを使用してGoogle APIへの相互認証を有効にするには、**"workload"** セクションとその **"cert_path"** および **"key_path"** の値が **"~/.config/gcloud/certificate_config.json"** 設定ファイルに存在**しなければならない**（must）。

### 期待される動作

ワークロードクレデンシャルを使用したGoogle APIへのmTLS認証のサポートは、クライアントオプションによるユーザーのmTLSエンドポイントオーバーライドを優先**しなければならない**（must）。認証ライブラリは以下の手順に従わ**なければならない**（must）：

- 上記の設定ファイルを使用してワークロード証明書と秘密鍵ファイルを特定する。これらのファイルのいずれかが存在しない場合、ワークロードクレデンシャルを使用したmTLSは無効になっ**てもよい**（may）。認証ライブラリは、証明書ファイルと鍵ファイルの公開鍵と秘密鍵が一致することを確認してからTLSライブラリに渡す**しなければならない**（must）。

  - 証明書のローテーション中に、別のプロセスがファイルを置き換えている間にクライアントライブラリが2つのファイルを読み取る可能性があるため、不一致が発生することがある。その場合、ライブラリは証明書と秘密鍵ファイルの読み取りと一致状態の確認を最大4回までリトライ**しなければならない**（must）。ライブラリは試行間で5秒間待機する**べきである**（should）。
  - 証明書と秘密鍵ファイルがメモリにロードされている場合（すべてのmTLS接続でディスクから読み取るのではなく）、認証ライブラリは定期的に（少なくとも10分ごと、または証明書の有効期限切れ時）それらをリロード**しなければならない**（must）して、インフラストラクチャがローテーションした後にメモリ内のコピーを更新する。クレデンシャルの更新はバックグラウンドスレッドで行い、使用時に行っては**ならない**（must not）。

- TLSハンドシェイク中のクライアント認証に、見つかり一致した証明書と鍵を使用するようTLSライブラリを構成する。

- ユーザーがクライアントオプションを介してエンドポイントオーバーライドを指定した場合、それをそのまま使用し、指定されたエンドポイントにmTLSを使用して接続する。

- ユーザーがエンドポイントオーバーライドを指定しなかった場合、証明書と鍵ファイルが存在すればデフォルトのmTLSエンドポイントを使用し、そうでなければデフォルトの通常エンドポイントを使用する。

mTLS 1.3は、クライアントIDと証明書の機密性を保持するために、サポートされる唯一のバージョンで**なければならない**（must）。

上記のロジックの1つの影響として、ユーザーがワークロードクレデンシャルを使用したmTLS認証を有効にし、有効な証明書と鍵ファイルを提供し、非mTLSエンドポイントオーバーライドを指定した場合、クライアントライブラリは証明書と鍵をそのまま使用し、サーバーの判断に任せる**べきである**（should）。これにより、エンドポイントオーバーライドがmTLS URLかどうかを解析するクライアント側のロジックの導入を回避できる。URLパターンはいつでも変更される可能性があるためである。

### デフォルトのmTLSエンドポイントの取得

サービスのデフォルトのmTLSエンドポイントは、正規表現パターンで生成するのではなく、Discovery Documentのフィールド **"mtlsRootUrl"** から読み取る**べきである**（should）。
