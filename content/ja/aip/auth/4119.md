# mTLS Token Binding

トークンバインディングにより、mTLSクレデンシャルにバインドされたGoogleアクセストークンを発行できる。このようなmTLSバインドトークンの利点は、バインド先のmTLSクレデンシャルを介して確立されたセキュアチャネル上でのみ使用されることを意図している点にある。したがって、バインドトークンは、盗まれて悪意を持ってリプレイされる可能性があるbearerトークンよりも安全である。

このAIPは、（1）X.509証明書IDにバインドされたアクセストークン（アイデンティティバインドトークンと呼ばれる）を取得し、（2）Google認証ライブラリを使用してそれらをGoogle APIにアクセスするために使用する方法のフローを説明する。

**注:** このAIPは言語に依存しない形でガイダンスと要件を説明するため、特定の言語や環境では不正確または不適切となる可能性がある一般的な用語を使用している。

## ガイダンス

ユーザーがトークンバインディングを有効にする場合、[ADC][0]を介して行う**べきである**（should）。本セクションでは、そのようなトークンをサポートするための一般的なガイダンスについて説明する。

### 前提条件

アイデンティティバインドアクセストークンは、クライアントが[X.509 SPIFFE Verifiable Identity Documents][1]（SVID）を持っていることを必要とする。[ワークロードクレデンシャルを使用した相互認証][2]では、そのようなSVIDがGoogle Cloudでどのようにプロビジョニングされるかを説明している。

さらに、アイデンティティバインドアクセストークンは、Google CloudのIAMでワークロード アイデンティティプールとアイデンティティプロバイダを構成する必要がある。これを行う方法の説明はこのAIPの範囲外である。

### mTLS Token Bindingの使用

認証ライブラリは、**"~/.config/gcloud/certificate_config.json"** 設定ファイル内の以下の値をサポート**しなければならない**（must）。このファイルのデフォルトの場所は `GOOGLE_API_CERTIFICATE_CONFIG` 環境変数を使用して変更できることに注意。

```json
{
  "version": 1
  "cert_configs": {
    "workload": {
      "cert_path": "path/to/cert/file"
      "key_path": "path/to/key/file"
      "workload_identity_provider": "..."
      "authenticate_as_identity_type": "gsa/native"
      "service_account_email": "..."
    },
    "keychain": {
      ...
    },
    "pkcs11": {
      ...
    },
    "windows": {
      ...
    },
  },
  "libs": {
   ...
  }
}
```

以下は、mTLS Token Bindingの設定に関連するフィールドのリストである：

- **"workload_identity_provider"**: 指定された値は、アイデンティティバインドアクセストークンを要求するためにSecurity Token Service（STS）へのリクエストを構築するために使用される。この値は、IAMで構成されたワークロード アイデンティティプールとアイデンティティプロバイダの完全修飾名を指す。指定された値は以下の形式でなければ**しなければならない**（must）。

```
"workload_identity_provider":"//iam.googleapis.com/projects/<project_number>/locations/global/workloadIdentityPools/<pool_identifier>/providers/<provider_identifier>"
```

- **"authenticate_as_identity_type"**: このフィールドは、Google APIへの認証に使用されるIDを指定する。値は `gsa` または `native` に設定できる。`gsa` はワークロードのGCPサービスアカウント（例：GCE VMのGCPサービスアカウント）であり、`native` はネイティブのワークロードID（例：GKEポッドのKubernetesサービスアカウント）である。指定されない場合、デフォルト値は `gsa` である。

- **"service_account_email"**: 設定されている場合、指定された値は、アイデンティティバインドアクセストークンを要求するためにIAM Credentialsサービスへのリクエストを構築するために使用される。この値は、リソースアクセスに使用されるサービスアカウントのメールアドレスを指す。設定されていない場合、サービスアカウントのメールアドレスは、以下のMetadata Serviceエンドポイントに問い合わせることで自動的に決定される：
`http://metadata/computeMetadata/v1/instance/service-accounts/default/email`。このフィールドの値は、**"authenticate_as_identity_type"** が `gsa` に設定されている場合にのみ関連する。

**"cert_path"** および **"key_path"** フィールドの説明は[ワークロードクレデンシャルを使用した相互認証][2]に記載されている。

Google APIと通信する際にトークンバインディングを使用できるようにするには、以下の条件が必要である：

- [ワークロードクレデンシャルを使用した相互認証][2]が有効で**なければならない**（must）。

- **"workload_identity_provider"** が存在**しなければならない**（must）、**"authenticate_as_identity_type"** は設定されて**もよい**（may）、**"service_account_email"** は **"~/.config/gcloud/certificate_config.json"** 設定ファイルの **"workload"** セクションに設定されて**もよい**（may）。

### 期待される動作

アイデンティティバインドアクセストークンの使用をサポートするために、認証ライブラリはGoogle APIにリクエストを送信する際に以下の手順に従わ**なければならない**（must）：

1. [ワークロードクレデンシャルを使用した相互認証][2]で説明されているようにプロビジョニングされたワークロードクレデンシャルを使用して、[STS API][3]のmTLSエンドポイントに接続する。このエンドポイントは `sts.mtls.googleapis.com` で**なければならない**（must）。

1. **"~/.config/gcloud/certificate_config.json"** 設定ファイルの **"workload_identity_provider"** フィールドの情報を使用して、アイデンティティバインドトークンを要求するHTTPリクエストをSTSの[ExchangeToken][5]メソッドに送信する。要求されたトークンのスコープは `https://www.googleapis.com/auth/iam` で**なければならない**（must）。

1. [ワークロードクレデンシャルを使用した相互認証][2]で説明されているようにプロビジョニングされたワークロードクレデンシャルを使用して、[IAM Credentials Service API][4]のmTLSエンドポイントに接続する。このエンドポイントは `iamcredentials.mtls.googleapis.com` で**なければならない**（must）。

1. **"authenticate_as_identity_type"** が `gsa` に設定されている場合、**"~/.config/gcloud/certificate_config.json"** 設定ファイルの **"service_account_email"** フィールドのサービスアカウントメールアドレスを主張するアイデンティティバインドトークンを要求するHTTPリクエストをIAM Credentials Serviceの[GenerateAccessToken][6]メソッドに送信する。このトークンのスコープは、ユーザーが要求されたGoogle APIにアクセスするために定義したものと同じスコープで**なければならない**（must）。

1. ステップ4で返されたトークンをリクエストに添付する。このリクエストは、ステップ1と同じワークロードクレデンシャルを使用してmTLSチャネル経由で送信**しなければならない**（must）。
