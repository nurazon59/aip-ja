# リソースリビジョン

一部のAPIは、ユーザーが時間の経過に伴うリソースの状態を把握できるリビジョン履歴を持つリソースを必要とする。これにはいくつかの理由がある：

- ユーザーが以前のリビジョンにロールバックしたり、以前のリビジョンとの差分を確認したい場合がある。
- APIが特定の時点のリソースから何らかの形で派生したデータを作成する場合がある。そのような場合、後で参照するためにリソースをスナップショットすることが望ましい。

**注記:** 特定のリソースの履歴参照を指すために _revision_ という単語を使用し、API全体のバージョンを指す _version_ という用語は意図的に避けている。

## ガイダンス

APIはリソースのリビジョン履歴を保存 **してもよい**（may）。これが有用な場合の例：

- APIを介して古いバージョンのリソースを公開することが価値がある場合。これにより、顧客がリビジョンを保存および取得するための独自のAPIを作成するオーバーヘッドを回避できる。
- 他のリソースがリソースの異なるリビジョンに依存している場合。
- 時間の経過に伴うリソースの変更を表現する必要がある場合。

リビジョン履歴を持つリソースを実装するAPIは、リソースリビジョンをリソースのネストされたコレクションとして抽象化 **すべきである**（should）。リビジョンコレクションはトップレベルのコレクションになることもあるが、例外として以下が挙げられる：

- リソースリビジョンが親リソースよりも長い寿命を持つことが意図されている場合。つまり、リソース削除後もリソースリビジョンが存在する場合。

```proto
message BookRevision {
  // 書籍リビジョンの名前。
  string name = 1;

  // 書籍のスナップショット
  Book snapshot = 2
    [(google.api.field_behavior) = OUTPUT_ONLY];

  // リビジョンが作成されたタイムスタンプ。
  google.protobuf.Timestamp create_time = 3
    [(google.api.field_behavior) = OUTPUT_ONLY];

  // 同じスナップショットを共有する他のリビジョンID。
  repeated string alternate_ids = 4
    [(google.api.field_behavior) = OUTPUT_ONLY];
}
```

- `message` はリソースとしてアノテーションされ**なければならない**（must）（AIP-123）。
- `message` の名前は `{ResourceType}Revision` と **しなければならない**（must）。
- リソースリビジョンは、親リソースのメッセージ型を持つフィールドを含ま**なければならない**（must）。フィールド名は `snapshot` とする。
  - `snapshot` の値は、リビジョン作成時点での親の設定で**なければならない**（must）。
- リソースリビジョンは `create_time` フィールドを含ま**なければならない**（must）（[AIP-142][] を参照）。
- リソースリビジョンは繰り返しフィールド `alternate_ids` を含んで **してもよい**（may）。これにはリビジョンが別名としても知られているリソースIDのリスト（例：`latest`）が含まれる。

### リビジョンの作成

リソースによって、異なるAPIが異なる戦略を持つ可能性がある：

- 親リソースに変更があるたびに新しいリビジョンを作成する
- 重要なシステム状態が変化したときに新しいリビジョンを作成する
- 明示的に要求されたときに新しいリビジョンを作成する

APIはこれらの戦略のいずれを使用 **してもよい**（may）。APIはリビジョン作成戦略を文書化 **しなければならない**（must）。

### リビジョンのリソース名

リソースの特定のリビジョンを参照する場合、サブコレクション名は `revisions` と **しなければならない**（must）。リソースリビジョンは `{resource_name}/revisions/{revision_id}` という形式の名前を持つ。例えば：
```
publishers/123/books/les-miserables/revisions/c7cfa2a8
```

### サーバー指定のエイリアス

サービスは特定のIDを[エイリアス][alias]（例：`latest`）として予約 **してもよい**（may）。これらは読み取り専用で、サービスによって管理される。

```
GET /v1/publishers/{publisher}/books/{book}/revisions/{revision_id}
```

- `latest` IDが存在する場合、最も最近作成されたリビジョンを表さ**なければならない**（must）。`publishers/{publisher}/books/{book}/revisions/latest` と `publishers/{publisher}/books/{book}` の内容は異なる可能性がある。最新のリビジョンがリソースの現在の状態と異なる場合があるためである。

### ユーザー指定のエイリアス

APIは、カスタムメソッド "alias" を使用して既存のリビジョンに[エイリアス][alias] IDを割り当てるメカニズムを提供 **してもよい**（may）：

```proto
rpc AliasBookRevision(AliasBookRevisionRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*/revisions/*}:alias"
    body: "*"
  };
}
```

```proto
message AliasBookRevisionRequest {
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/BookRevision"
    }];

  // エイリアス先のリビジョンのID。例：`CURRENT` またはセマンティックバージョン。
  string alias_id = 2 [(google.api.field_behavior) = REQUIRED];
}
```

- リクエストメッセージは `name` フィールドを持た**なければならない**（must）：
  - フィールドは[requiredとしてアノテーションされる][aip-203] **しなければならない**（must）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別 **しなければならない**（must）。
- リクエストメッセージは `alias_id` フィールドを持た**なければならない**（must）：
  - フィールドは[requiredとしてアノテーションされる][aip-203] **しなければならない**（must）。
- ユーザーが既存の `alias_id` でメソッドを呼び出した場合、リクエストは成功 **しなければならない**（must）し、エイリアスは指定されたリビジョンを参照するように更新される。これにより、ユーザーは特定のエイリアス（例：`published`）に対してコードを記述でき、コード変更なしでリビジョンを変更できる。

### ロールバック

リビジョン履歴を持つリソースの一般的なユースケースは、特定のリビジョンにロールバックする機能である。APIはこれを `Rollback` カスタム メソッドで処理 **すべきである**（should）：

```proto
rpc RollbackBook(RollbackBookRequest) returns (BookRevision) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*/revisions/*}:rollback"
    body: "*"
  };
}
```

- メソッドは `POST` HTTP動詞を使用 **しなければならない**（must）。
- メソッドはリソースリビジョンを返す **すべきである**（should）。

```proto
message RollbackBookRequest {
  // 書籍のロールバック先のリビジョン。
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/BookRevision"
    }];
}
```

- リクエストメッセージは `name` フィールドを持た**なければならない**（must）。これは、リソースのロールバック先の設定を持つリソースリビジョンを参照する。
  - フィールドは[requiredとしてアノテーションされる][aip-203] **しなければならない**（must）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別 **しなければならない**（must）。

### 子リソース

リビジョン履歴を持つリソースは子リソースを持って **してもよい**（may）。持つ場合、2つのバリアントがある：

- 各子リソースが親リソース全体の子である場合。
- 各子リソースが親リソースの_単一のリビジョンの_子である場合。

APIは複数レベルのリビジョンを持つリソースを含める **すべきではない**（should not）。これはすぐに理解が難しくなるためである。

### 標準メソッド

すべての標準メソッドは、対応するAIP（AIP-131、AIP-132、AIP-133、AIP-134、AIP-135）を実装 **しなければならない**（must）し、以下の追加動作を伴う：

- リストメソッド：デフォルトでは、リストレスポンス内のリビジョンは逆時系列順に並べられ**なければならない**（must）。ユーザーは `order_by` を指定してデフォルトの動作を上書きできる。
- リビジョンがエイリアスをサポートする場合、エイリアスのリソース名（例：`revisions/1.0.2`）を持つ削除メソッドは、リソースを削除する代わりにエイリアスを削除 **しなければならない**（must）。

リビジョンはリソースの下にネストされるため、[カスケード削除][cascading delete]も参照。

## 理由

### リビジョンのネストされたコレクションとしての抽象化

リビジョンをネストされたコレクションの下のリソースにすることで、リビジョンが第一級の市民になる。

- リビジョンは標準的なget、list、deleteメソッドを提供できる。
- リソースメッセージに加えて、リビジョンに新しいフィールドを拡張する柔軟性を保持する。

### エイリアスへのタグ付け

以前は `tag` という概念が存在していた。この概念は[エイリアス][alias]の概念と重複しており、AIPの複雑さを減らすために用語が統合された。

### Output onlyのリソース設定

作成メソッドの一部としてリソース設定をリビジョンが受け取ることもオプションであったが、そうするとユーザーがリソースが実際には取ったことのないリソース設定をリビジョンに送信できてしまう。

`OUTPUT_ONLY` と、作成されたリビジョンが現在の時点でのリソースを表すことを要求することで、その問題が排除される。

## 履歴

### コレクション拡張からサブコレクションへの移行

2023-09に、リビジョンはネストされたリソースコレクションとして抽象化された。これ以前は、リビジョンは `@` シンボルを使用した既存のリソースの拡張のようなものであった。リビジョンのリストと削除はリソースコレクション上のカスタムメソッドであった。単一のGetメソッドがリソースリビジョンまたはリソースのいずれかを取得するために使用されていた。

主な利点は、リソース参照がシームレスにリソースまたはそのリビジョンを参照できることであった。

いくつかの欠点もあった：

- リビジョンのリストはリソースコレクション上のカスタムメソッド（:listRevisions）であった
- リビジョンの削除はリソースコレクション上のカスタムメソッドであった
- API検出ドキュメントに表示されない
- リソースIDが `@` を使用できない

ガイダンスは最終的に、リビジョンがリソースのように振る舞うことを可能にするために修正された。これにより、ユーザーの認知負荷が軽減され、リソース指向のクライアントがリビジョンを簡単にリスト、取得、作成、更新できるようになる。

### タグの代わりにリソースIDを使用

以前の設計では、リビジョンには `tag` として知られるリビジョンの別の識別子が存在し、リビジョン内に配置されていた。

タグは事実上のシャドウリソースIDであり、タグの値に基づいてリビジョンを作成、取得、フィルタリングするためのメソッドが必要であった。

タグの概念をリビジョンIDに統合することで、ユーザーはもはや2番目の取得および識別メソッドのセットに精通する必要がなくなった。
