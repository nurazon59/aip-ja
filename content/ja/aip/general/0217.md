# 到達不能なリソース

ユーザーがリソースの一覧を要求した際に、リスト内の一部のリソースが一時的に利用不可となることがある。最も典型的なユースケースは、[コレクション横断読み取り][aip-159]をサポートする場合である。例えば、ユーザーが複数の親ロケーションにわたってリソースをリストするよう要求したが、そのうちの1つのロケーションが一時的に到達不能である場合である。この状況では、利用可能なすべてのリソースをユーザーに提供しつつ、何かが欠けていることを示すことが望ましい。

## ガイダンス

データ取得メソッドが、1つ以上のリソースが一時的に到達不能であるために部分的に失敗する可能性がある場合、レスポンスメッセージはこれを示すフィールドを**含めなければならない**（must）：

```proto
message ListBooksResponse {
  // The books matching the request.
  repeated Book books = 1;

  // The next page token, if there are more books matching the
  // request.
  string next_page_token = 2;

  // Unreachable resources.
  repeated string unreachable = 3 [
    (google.api.field_behavior) = UNORDERED_LIST
  ];
}
```

- フィールドはrepeated string**でなければならず**（must）、`unreachable`と**命名すべきである**（should）。
- フィールドには、到達不能なリソース、または要求されたコレクションへの到達を妨げるリソース（到達できなかったコレクションの親リソースなど）のリソース名を**含めなければならない**（must）。
  - 例えば、ロケーション全体が到達不能で、要求されたローカライズされたリソースコレクションへのアクセスが妨げられている場合、そのロケーションリソースが含まれる。
- フィールドには*サービス相対*リソース名を**含めなければならず**（must）、完全なリソース名、リソースURI、または単純なリソースIDを含めて**はならない**（must not）。定義については[AIP-122][aip-122]を参照。
  - 例えば、`Book`リソースが到達不能な場合、*完全な*リソース名`"//library.googleapis.com/shelves/scifi1/books/starwars4"`、*親相対*リソース`"books/starwars4"`、リソースID`"starwars4"`、またはリソースURIではなく、*サービス相対*リソース名`"shelves/scifi1/books/starwars4"`が`unreachable`に含まれる。
- レスポンスは、リストされたリソースを到達不能にした問題に関するその他の情報を提供**してはならない**（must not）。
  - 例えば、レスポンスは各`unreachable`エントリのエラー理由を含む追加フィールドを含むことができない。
- サービスは、ユーザーがより具体的なリクエストを行い、追加情報を含むエラーを受け取る方法を提供**しなければならない**（must）。例えば、到達不能なコレクションの親を対象とした標準Getまたは標準Listを介してなど。
  - サービスは、ユーザーが元の呼び出しをより制限的なパラメータで繰り返すことも許可**しなければならない**（must）。
- `unreachable`に現れるリソース名は異種混在で**あってもよい**（may）。
  - `unreachable`フィールド定義は、このフィールドに提供される可能性のあるリソースを文書化し、将来拡張される可能性があることに言及**すべきである**（should）。
  - 例えば、ロケーション全体と、別のロケーションの特定リソースの両方が到達不能な場合、到達不能なロケーションの名前（例：`"projects/example123/locations/us-east1"`）と到達不能なリソースの名前（例：`"projects/example123/locations/europe-west2/instances/example456"`）の両方が`unreachable`に現れる。
- `unreachable`フィールドは、リスト内で意味的に重要な順序や構造を持って**はならない**（must not）。言い換えれば、`unreachable`は順序なしリスト**でなければならない**（must）。
  - そのため、`unreachable`フィールドには`UNORDERED_LIST`フィールド動作のアノテーションを**付けなければならない**（must）（[AIP-203][unordered]参照）。

**重要:** 単一の到達不能なロケーションまたはリソースが定義上データの返却を妨げる場合（例えば、単一のパブリッシャーに対するリストリクエストでそのパブリッシャーが到達不能な場合）、サービスはリクエスト全体をエラーで失敗させ**なければならない**（must）。

### ページネーション

ページ取得RPC（例：[AIP-132][aip-132]標準List呼び出し）を満たすための結果ページを準備中に、サービスが到達不能なリソースまたはコレクションに遭遇した場合、以下を実行**しなければならない**（must）：

- 到達不能なリソースのリソース名を`unreachable`レスポンスフィールドに含める。
  - リソース名は、到達不能なリソースまたはコレクションに対して最も適切なスコープに**しなければならない**（must）。
    - 例えば、リージョン内の特定のゾーンが到達不能な場合、到達不能なリソース名はゾーンLocation（例：`projects/example/locations/us-west1-a`）となるが、リージョン全体が到達不能な場合、リソース名はリージョンLocation（例：`projects/example/locations/us-west1`）となる。
  - リソース名は、`order_by`などの制限的なページングパラメータに関わらず、到達不能と識別された時点で**含めなければならない**（must）。
- 以前は到達不能とみなされていた結果が、可用性が回復し、ページングパラメータがそれらの包含を許可する場合、後続のページに取り込む。
  - ページングパラメータに基づく包含適合性の判断には、リクエストにユーザー指定の順序がない場合の文書化されたデフォルトの順序動作も含まれる。
  - 例えば、リージョン`projects/example/locations/us-west1`が順序付きページング呼び出しの最初のページで利用不可であり、そのリソースを含めると順序付けに違反する場合、それらの順序外れのリソースは次のページに含まれない。
  - 同様に、まったく同じリクエストが行われ、以前は到達不能とみなされていたリソースが再び利用可能になった場合、ページングパラメータの制約内で、それらを取り込ま**なければならない**（must）。
- 到達不能なリソース名をアップスコープした後でも、到達不能なリソース名の数が文書化された最大数を超える場合、特定のレスポンスで返される到達不能なリソース名の数を制限する。
  - この最大数は、`unreachable`フィールドのコメントに直接文書化**しなければならない**（must）。
  - これは呼び出し元が設定した`page_size`とは独立している。

#### 以前の動作の維持

サービスは、変更が[AIP-180][aip-180]に従って互換性のない変更を引き起こす場合、以前に実装された`unreachable`ページネーション動作を**継続してもよい**（may）が、その動作を`unreachable`フィールドに直接文書化**しなければならない**（must）。

### 部分的成功の採用

前述のガイダンスとは*異なる*デフォルト動作（すなわち、API呼び出しが部分的な結果ではなくエラーステータスを返す）を持つ既存APIが`unreachable`パターンを採用するためには、以下を実行**しなければならない**（must）：

- 互換性のない動作変更を避けるため、デフォルト動作は維持**しなければならない**（must）。
  - 例えば、デフォルト動作が任意のロケーションが到達不能な場合にエラーを返すことである場合、そのデフォルト動作は維持**しなければならない**（must）。
- リクエストメッセージは`bool return_partial_success`フィールドを**持たなければならない**（must）。
- レスポンスメッセージは標準の`repeated string unreachable`フィールドを**持たなければならない**（must）。
- 上記2つのフィールドは同時に追加**しなければならない**（must）。

リクエストで`bool return_partial_success`フィールドが`true`に設定された場合、APIは前述のガイダンスに従い`repeated string unreachable`レスポンスフィールドを設定するように動作**しなければならない**（must）。

```proto
message ListBooksRequest {
  // Standard List request fields...

  // Setting this field to `true` will opt the request into returning the
  // resources that are reachable, and into including the names of those that
  // were unreachable in the [ListBooksResponse.unreachable] field. This can
  // only be `true` when reading across collections e.g. when `parent` is set to
  //  `"projects/example/locations/-"`.
  bool return_partial_success = 4;
}

message ListBooksResponse {
  // Standard List Response fields...

  // Unreachable resources. Populated when the request opts into
  // `return_partial_success` and reading across collections e.g. when
  // attempting to list all resources across all supported locations.
  repeated string unreachable = 3 [
    (google.api.field_behavior) = UNORDERED_LIST
  ];
}
```

#### 部分的成功の粒度

`bool return_partial_success`フィールドが`true`に設定されたリクエストが、APIが到達不能リソースを合理的に報告できる粒度を超えている場合、APIは問題を説明する詳細とともに`INVALID_ARGUMENT`エラーを**返すべきである**（should）。例えば、APIが[コレクション横断読み取り][aip-159]の場合にのみ`return_partial_success`をサポートする場合、特定の親リソースコレクションにスコープされたリクエストが与えられたときは`INVALID_ARGUMENT`エラーを返す。サポートされる粒度は`return_partial_success`フィールドに文書化**しなければならない**（must）。

## 根拠

### サービス相対リソース名の使用

一般に、[AIP-122][aip-122]で定義されている相対リソース名は、サービス*内*およびそのサービスが明らかな場合に他のサービスでリソースを名前で参照するためのベストプラクティスである。完全なリソース名形式は消費性が厳密に劣り（例えば、クライアント側での追加の解析が必要）、`unreachable`の用途には過剰指定である。リソースURIはトランスポートに依存せず、gRPCユーザーの標準メソッドでは使用できず、単純なリソースIDは異種混在のリソースリスト内でどのリソースが到達不能であったかについて十分な情報を提供しない。

### レスポンスでの追加エラー詳細の最小化

到達不能なリソースが発見されたコンテキストは機密情報である可能性があり、システムの状態は呼び出し間で変動する。そのため、より具体的なRPCを実行して特定のリソースまたは親の詳細を取得することで、サービスに判断を委ねることが推奨される。これにより、到達不能であった広範なリスト呼び出しに、特権的または古い可能性のある情報を無理やり詰め込むのではなく、リクエスト時に必要なRPCチェックとシステム状態解決を親が処理できるようになる。

### 順序なし`unreachable`内容

`unreachable`の内容が特定の順序や意味構造を持たないことは、広範なAPIの一貫性にとって重要である。各APIが標準フィールドに特定の順序付けを組み込んだ場合、クライアント側でもサーバー側でも、単一の正しい実装は存在しないことになる。

### ページごとの`unreachable`リソース

ページ単位で`unreachable`リソースを設定することで、エンドユーザーはすべての結果をページングし*終えた後*ではなく、ページが不完全であることを即座に識別できる。ページングの完了は保証されないため、特定のページに欠落している到達不能リソースがある場合、できるだけ早期に伝達することが重要である。さらに、これによりユーザーは後続の呼び出しで考慮すべき潜在的な問題を識別できる。最後に、ページング結果の最後まで到達不能リソースを保持することは、サービスが本来独立して完全に分離されるべきAPI呼び出しの状態を保持することを要求する。

### リクエストフィールドによるオプトイン

デフォルト動作を後方互換性を保ちつつユーザーの意図を伝える最良の方法は、部分的な成功動作にオプトインする手段として新しいリクエストフィールドを導入することである。`unreachable`レスポンスフィールドの導入とともにデフォルト動作を変更するという代替手段は、後方互換性のない変更をもたらす。以前は任意のリソースが到達不能な場合に失敗を期待していたユーザーは、成功レスポンスがすべてのリソースがレスポンスに含まれていることを意味すると想定する。

### フィールドの同時導入

リクエストフィールドとレスポンスフィールドを同時に導入するのは、一方のみを追加した場合に生じる無効な中間状態を防ぐためである。`unreachable`のみが追加された場合、それが空であることが、実際にはそうでない場合でもすべてのリソースが返されたことを意味すると想定される可能性がある。`return_partial_success`のみが追加された場合、ユーザーはどのリソースが到達不能であったかを知る手段がなくなる。

### 部分的成功の粒度の制限

リクエストのスコープ粒度があるレベルに達すると、APIは到達不能なリソースを列挙できなくなる。例えば、グローバル専用APIはローカライズされたコレクションレベルでの粒度を提供できない場合がある。そのような場合、`return_partial_success=true`のときに事前にエラーを返すことで、代替手段（問題があれば到達不能リソースを期待するが、何も得られず、したがってすべてが取得されたと誤って想定する）のリスクからユーザーを保護する。これは、満たせないリクエストは事前に失敗させるべきとする本ガイダンスと一致する。

## 履歴

### ページネーションガイダンス

`unreachable`フィールドを設定する方法に関する当初のガイダンスは、その内容をページング結果であるかのように消費することであった。これは、ページングされたリソースと到達不能リソースが同じレスポンス（すなわちページ）で返されることができず、ユーザーが到達不能なものがあるかどうかを確認するためにすべての結果を完全にページングする必要があることを意味した。変更の理由については根拠のセクションを参照。
