# Singletonリソース

APIは、任意の親内に常に1つのインスタンスのみが存在するリソースを表現する必要がある場合がある。一般的なユースケースは設定オブジェクトである。

## ガイダンス

APIは _singletonリソース_ を定義 **してもよい**（may）。singletonリソースは、その親の存在によって常に存在 **しなければならない**（must）し、親ごとに1つだけである。

例えば：

```proto
message Config {
  option (google.api.resource) = {
    type: "api.googleapis.com/Config"
    pattern: "users/{user}/config"
    singular: "config"
    plural: "configs"
  };

  // nameを含む追加フィールド
}
```

`Config` singletonは以下のRPCを持つ：

```proto
rpc GetConfig(GetConfigRequest) returns (Config) {
  option (google.api.http) = {
    get: "/v1/{name=users/*/config}"
  };
}

rpc UpdateConfig(UpdateConfigRequest) returns (Config) {
  option (google.api.http) = {
    patch: "/v1/{config.name=users/*/config}"
    body: "config"
  };
}
```

- Singletonリソースは、ユーザー提供またはシステム生成のIDを持っては**ならない**（must not）。その[リソース名][aip-122]は、親の名前に続いて1つの静的セグメントを含む。
  - 例: `users/1234/config`
- Singletonリソースは常に単数形である。
  - 例: `users/1234/thing`
- Singletonリソース定義は、`singular` フィールドと `plural` フィールドの両方を提供 **しなければならない**（must）（上記の例を参照）。
- Singletonリソースは他のリソースの親に**なってもよい**（may）。
- Singletonリソースは [`Create`][aip-133] または [`Delete`][aip-135] 標準メソッドを定義しては**ならない**（must not）。Singletonは、その親が作成または削除されたときに暗黙的に作成または削除される。
- Singletonリソースは [`Get`][aip-131] および [`Update`][aip-134] メソッドを定義 **すべきである**（should）し、必要に応じてカスタム メソッドを定義 **してもよい**（may）。
  - ただし、リソースのすべてのフィールドが [output only][aip-203] である場合、singletonリソースは [`Update`][aip-134] メソッドを定義しては**ならない**（must not）。
- Singletonリソースは [`List`][aip-132] メソッドを定義**してもよい**（may）が、[AIP-159][aip-159] に従って実装 **しなければならない**（must）。以下の例を参照。
  - 通常はコレクションを表すパスパターンの末尾セグメントは、Singletonリソースの `plural` 形式にする **すべきである**（should）。例: `/v1/{parent=users/*}/configs`
  - AIP-159に従い、ハイフン `-` ではなく親リソースIDが提供された場合、サービスは指定された親リソースに対応する1つのSingletonリソースのコレクションを返す **すべきである**（should）。

```proto
rpc ListConfigs(ListConfigsRequest) returns (ListConfigsResponse) {
  option (google.api.http) = {
    get: "/v1/{parent=users/*}/configs"
  };
}

message ListConfigsRequest {
  // すべてのconfigをリストするには、ユーザーIDとして `-` を使用する。
  // 形式:
  // * `users/-`
  // * `users/{user}`
  //
  // 注: 実際のユーザーIDを指定すると、1つのconfigのコレクションが返される。
  // 代わりにGetConfigを使用すること。
  string parent = 1 [
    (google.api.resource_reference).child_type = "api.googleapis.com/Config"];

  // その他の標準的なページネーションフィールド...
}
```

## 理由

### 標準Listのサポート

Singletonリソースはそれ自体が直接コレクションの一部ではないが、親のコレクションの一部として見なすことができる。親とsingletonの1対1の関係は、1つの親ごとに1つのsingletonインスタンスが存在することを意味し、[コレクション横断読み取り][aip-159]のパターンと組み合わせることで、自然にいくつかのコレクションベースのメソッドを有効にする。Singletonは、親に間接的に基づく1つのコレクションとしてAPIコンシューマーに提示できる。さらに、そのようなメソッドでSingletonリソースを疑似コレクションとして提示することで、将来、Singletonが不十分と判明した場合に実際のコレクションに拡張することが可能になる。

### `plural` 定義の包含

Singletonは定義上単数形であるが、サービスが標準List（ここで定義されている）をサポートする場合など、Singletonリソースが複数形で出現するケースがある。そのため、必要になったときに備えてSingletonリソースタイプの複数形を事前宣言しておく方が、持たないよりは良い。
