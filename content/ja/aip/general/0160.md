# フィルタリング

多くの場合、リソースをリストする際（[AIP-132][] で定義されているリストメソッド、またはそれに類似したものを使用）、コレクションをフィルタリングしてユーザーが関心のある結果のみを返すことが望ましい。

各APIの正確なフィルタリングニーズを処理する構造を定義したくなるかもしれない。しかし、フィルタリング要件は頻繁に変化するため、非技術的なユーザーでもアクセス可能な構造化構文を持つ文字列フィールドを使用することが賢明である。これにより、UIやクライアントの更新を待つことなく、透過的に更新を行うことができる。

**注記:** リストフィルターは、潜在的に非技術的なユーザーを対象としているため、コードで見られる一般的なパターンではなく、口語的なスピーチのパターンを借用することがある。

## ガイダンス

APIは `List` メソッド（または `Search` などのコレクションをクエリする類似メソッド）でフィルタリングをユーザーに提供 **してもよい**（may）。提供する場合、ここで説明するフィルターの共通仕様に従う **すべきである**（should）。構文は [EBNF文法][] で正式に定義されている。

フィルタリングを採用する場合、リクエストメッセージは正確に1つのフィルタリングフィールド `string filter` を持つ **すべきである**（should）。関連オブジェクトのフィルタリングは[トラバーサル](#traversal-operator)または[関数](#functions)を通じて処理される。

**注記:** リストフィルターは曖昧一致特性を持ち、結果のランク付けとスコアリングをサポートする。リストフィルターの決定論的な評価に関心のある開発者は [CEL][] を参照。

### リテラル

ベアリテラル値（例："42"、"Hugo"）は、照合される値である。単独で（フィールドが指定されずに）現れるリテラルは、通常、オブジェクトのフィールド値に現れる可能性がある任意の場所で照合 **すべきである**（should）。

ただし、サービスは特定のフィールドのみを考慮することを選択 **してもよい**（may）。その場合、どのフィールドを考慮するかを文書化 **しなければならない**（must）。サービスは時間の経過とともに新しいフィールドを含める **してもよい**（may）が、慎重に行い、既存のユーザーへの影響を考慮 **すべきである**（should）。

**注記:** 空白で区切られたリテラルは、`AND` の曖昧な変種と見なされる。したがって、`Victor Hugo` はおおよそ `Victor AND Hugo` と同等である。

### 論理演算子

フィルタリング実装は、以下の二項演算子を提供 **すべきである**（should）：

| 演算子   | 例             | 意味                             |
| -------- | -------------- | -------------------------------- |
| `AND`    | `a AND b`      | `a` と `b` が真の場合に真。      |
| `OR`     | `a OR b OR c`  | `a`、`b`、`c` のいずれかが真の場合に真。 |

**注記:** 一般的な話し方のパターンに合わせるため、`OR` 演算子は `AND` よりも優先順位が高い。これはほとんどのプログラミング言語とは異なる。式 `a AND b OR c` は次のように評価される：`a AND (b OR c)`。APIドキュメントと例は、混乱を避けるために明示的な括弧の使用を奨励 **すべきである**（should）が、明示的な括弧を要求 **すべきではない**（should not）。

### 否定演算子

フィルタリング実装は、単項演算子 `NOT` と `-` を提供 **すべきである**（should）。これらは互換的に使用され、否定をサポートするサービスは両方の形式をサポート **しなければならない**（must）。

| 演算子   | 例       | 意味                      |
| -------- | -------- | ------------------------- |
| `NOT`    | `NOT a`  | `a` が真でない場合に真。  |
| `-`      | `-a`     | `a` が真でない場合に真。  |

### 比較演算子

フィルタリング実装は、二項比較演算子 `=`、`!=`、`<`、`>`、`<=`、`>=` を文字列、数値、タイムスタンプ、および期間フィールドに対して提供 **すべきである**（should）（ただし、ブーリアンやenumには提供 **すべきではない**（should not））。

| 演算子   | 例             | 意味                                            |
| -------- | -------------- | ----------------------------------------------- |
| `=`      | `a = true`     | `a` が真の場合に真。                            |
| `!=`     | `a != 42`      | `a` が42に等しくない場合に真。                  |
| `<`      | `a < 42`       | `a` が42未満の数値の場合に真。                  |
| `>`      | `a > "foo"`    | `a` が "foo" より辞書順で後ろの場合に真。       |
| `<=`     | `a <= "foo"`   | `a` が "foo" または辞書順で前の場合に真。       |
| `>=`     | `a >= 42`      | `a` が42以上の数値の場合に真。                   |

**注記:** ほとんどのプログラミング言語とは異なり、フィールド名は比較演算子の左辺に現れ**なければならない**（must）。右辺はリテラルと論理演算子のみを受け付ける。

フィルターはクエリ文字列として受け入れられるため、文字列を適切な強く型付けされた値に変換する型変換が行われる：

- Enumはenumの文字列表現（大文字小文字を区別）を期待する。
- ブーリアンは `true` および `false` リテラル値を期待する。
- 数値は標準の整数または浮動小数点表現を期待する。浮動小数点の場合は指数がサポートされる（例：`2.997e9`）。
- [期間][]は、数値表現の後に `s` 接尾辞（秒）を付ける。例：`20s`、`1.2s`。
- [タイムスタンプ][]は [RFC-3339][] 形式の文字列（例：`2012-04-21T11:30:00-04:00`）を期待する。UTCオフセットがサポートされる。

**警告:** 識別子 `true`、`false`、`null` は、型付けられたフィールド参照のコンテキストで使用された場合にのみ本質的な意味を持つ。

さらに、文字列の等価性を比較する際、サービスは `*` 文字を使用したワイルドカードをサポート **すべきである**（should）。例えば、`a = "*.foo"` は `a` が ".foo" で_終わる_場合に真である。

### トラバーサル演算子

フィルタリング実装は、メッセージ、マップ、または構造体のトラバーサルを示す `.` 演算子を提供 **すべきである**（should）。

| 例               | 意味                                                      |
| --------------- | --------------------------------------------------------- |
| `a.b = true`    | `a` が真であるブーリアンフィールド `b` を持つ場合に真。  |
| `a.b > 42`      | `a` が42を超える数値フィールド `b` を持つ場合に真。      |
| `a.b.c = "foo"` | `a.b` が "foo" である文字列フィールド `c` を持つ場合に真。 |

トラバーサルはリソースのフィールド名を使用して記述 **しなければならない**（must）。サービスが何らかの「暗黙的フィールド」をサポートしたい場合、適切に文書化された[関数](#functions)を通じて行う **しなければならない**（must）。サービスはトラバーサルがサポートされるフィールドのサブセットを指定 **してもよい**（may）。

ユーザーがメッセージに定義されていないフィールドへのトラバーサルを試みた場合、サービスは `INVALID_ARGUMENT` でエラーを返す **すべきである**（should）。サービスはマップや構造体の未定義キーへのトラバーサルを許可 **してもよい**（may）し、この状況での動作を文書化 **すべきである**（should）。

トラバーサルを含む式を評価する際、チェーン内の非プリミティブフィールドが評価対象のエントリに設定されていない場合、そのエントリはスキップされる、つまりフィルター式に一致しないと **すべきである**（should）。これは比較が `!=` であっても適用され、空の値との一致を暗示する。上記の例で、リソースフィールド `a` がリソースインスタンスに設定されていない場合、そのインスタンスは非一致としてスキップされる。

**重要:** `.` 演算子は、`:` 演算子との特定の使用を除き、繰り返しフィールドやリストのトラバーサルに使用しては**ならない**（must not）。

### Has演算子

フィルタリング実装は `:` 演算子を提供 **しなければならない**（must）。これは「has」を意味する。これはコレクション（繰り返しフィールドやマップ）およびメッセージで使用可能で、それぞれの場合で動作が少し異なる。

繰り返しフィールドは、繰り返し構造に一致する要素が含まれているかどうかを問い合わせる：

| 例          | 意味                                                      |
| ----------- | --------------------------------------------------------- |
| `r:42`      | `r` が42を含む場合に真。                                  |
| `r.foo:42`  | `r` が `e.foo = 42` となる要素 `e` を含む場合に真。      |

**重要:** フィルターは繰り返しフィールドの_特定の_要素の値をクエリすることはできない。例えば、`e.0.foo = 42` や `e[0].foo = 42` は有効なフィルターでは**ない**。

マップ、構造体、メッセージは、マップ内のフィールドの存在または特定の値をクエリできる：

| 例          | 意味                                        |
| ----------- | ------------------------------------------- |
| `m:foo`     | `m` がキー "foo" を含む場合に真。            |
| `m.foo:*`   | `m` がキー "foo" を含む場合に真。            |
| `m.foo:42`  | `m.foo` が42の場合に真。                     |

メッセージを解析する際、2つの微妙な区別がある：

- メッセージをトラバースする場合、フィールドは非デフォルト値を持つ場合にのみ存在すると見なされる。
- メッセージをトラバースする場合、フィールド名はスネークケースである。ただし、実装はキャメルケースとスネークケースの自動変換をサポートすることを選択 **してもよい**（may）。

上記のすべての型について、トップレベルのリソースフィールドの存在を単にチェックすることは `*` 値で可能である：

| 例          | 意味                                        |
| ----------- | ------------------------------------------- |
| `r:*`       | `repeated` フィールド `r` が存在する場合に真。  |
| `p:*`       | `map` フィールド `p` が存在する場合に真。       |
| `m:*`       | `message` フィールド `m` が存在する場合に真。   |

**注記:** マップおよび繰り返しフィールドの場合、未設定のフィールドと「空の値で設定」の間に意味的な違いはなく、両方とも「存在しない」と解決される。

### 関数

フィルタリング言語は、API固有の拡張をサポートするために関数呼び出し構文をサポートする。APIは `call(arg...)` 構文を使用して関数を定義 **してもよい**（may）し、サポートする特定の関数を文書化 **しなければならない**（must）。

### 制限

サービスは、ここで定義されているものに加えて、フィルタークエリのさらなる構造や制限を指定 **してもよい**（may）。例えば、サービスは論理演算子をサポートするが、その数を制限して（「死のクエリ」やその他のパフォーマンスの問題を避ける）こともできる。

さらなる構造や制限は明確に文書化**されなければならない**（must）、この文書で設定された要件に違反しては**ならない**（must not）。

### 検証

準拠していない、またはスキーマ的に無効な `filter` 文字列が指定された場合、APIは `INVALID_ARGUMENT` でエラーにする **すべきである**（should）。`filter` の検証が緩和される場合、APIはその違いを文書化 **しなければならない**（must）。

スキーマ的検証とは、以下を含むがこれに限定されない：

- `filter` で参照されるフィールドは、フィルタリングされるスキーマ上に存在 **しなければならない**（must）
- `filter` で提供されるフィールド値は、フィールドの型に適合 **しなければならない**（must）
  - 例えば、`int32 age` フィールドに対して `filter` が `"age=hello"` であるのは無効
- `enum` などの境界付きデータ型のフィールド値は、`filter` で提供される場合、セット内の有効な値で **しなければならない**（must）
- `Timestamp` などの標準化された型のフィールド値は、文書化された標準に準拠 **しなければならない**（must）（そのような型のリストは[比較演算子](#comparison-operators)を参照）
