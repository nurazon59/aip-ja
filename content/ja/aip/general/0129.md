# サーバー変更値とデフォルト値

サービスはリソースフィールドにデフォルト値を提供することが多く、ユーザー入力を正規化してからレスポンスで返すことがある。ここでのガイダンスは、サービスがコンシューマーの利益のためにそのような動作をどのように文書化するかを説明する。

## ガイダンス

### 単一所有者フィールド

フィールドは、クライアントまたはサーバーのいずれかである単一の所有者を持た**なければならない**（must）。サーバー所有のフィールドは、`OUTPUT_ONLY` field_behaviorで示さ**なければならない**（must）。他のすべてのタイプのフィールドは、クライアントが所有するものと見なされ**なければならない**（must）。サーバーは、クライアント所有のすべてのフィールドについて値（またはその欠如）を尊重し、変更**してはならない**（must not）。

### 実効値

クライアントが値を指定しないことを選択した場合に、サービスが値を割り当て、生成、または計算するインスタンスがある。例えば、クライアントが静的なIPアドレスを指定せずに仮想マシンを作成する場合である。そのようなシナリオは、動的IPアドレス割り当てを選択することになる。

これらのタイプのフィールドの例としては、以下のものがある：

* 生成されるもの（UUID）
* 割り当てられるもの（動的IPアドレス）
* 割り当てられるもの（最新のソフトウェアパッケージバージョン）

実効値を持つ属性は、APIで2つのフィールドとして表現***されなければならない***（**must**）：

* ユーザーがオプションで設定でき、サービスによって変更されてはならない可変フィールド
* サービスによって決定された実効値を記録する`OUTPUT_ONLY`フィールド

例：
```proto
message VirtualMachine {
  …
  string ip_address = 4;
  string effective_ip_address = 5 [
    (google.api.field_behavior) = OUTPUT_ONLY
  ];
}
```

#### 命名

実効値は、可変フィールドの名前に`effective_`をプレフィックスとして付けて命名**しなければならない**（must）。

### ユーザー指定フィールド

ユーザー指定フィールドの場合、サービスからのレスポンスの値は、作成または更新リクエストで提供されたものと同じ**でなければならない**（must）。文字列フィールドの場合、これは値を変更せずに返すことを意味する。ただし、1つの例外がある：

* 文字列フィールドにデータ型アノテーションがある場合、指定された値を表す正規化された文字列が返され**てもよい**（may）。

### 正規化

サービスによって正規化されるフィールドは、`google.api.field_info`拡張でアノテーションされ**なければならない**（must）。この拡張の使用に関するガイダンスは（[AIP-202][]）を参照のこと。許可される正規化のセットには、以下の形式が含まれる：

* uuid
* ipv4
* ipv6
* email

フィールドの正規化は、`google.api.field_info`アノテーションを使用して記述**されなければならない**（must）。

## 根拠

サーバー変更値とデフォルト値は、[宣言的クライアント][Declarative clients]の実装をしばしば困難にする。これらのクライアントは、サーバーが値を変更して返すルールが複雑で公開されておらず、再現可能でもないため、望ましい状態が現在の状態と一致するかどうかを判断できないことが多い。

### 単一所有者フィールドの根拠

フィールドに単一の所有者がいない場合、[宣言的クライアント][Declarative clients]に問題を引き起こす可能性がある。これらのクライアントは、サーバー設定値によって上書きされるフィールドに値を設定しようとし、クライアントが変更を修正するために無限ループに陥る可能性がある。

### 命名の根拠

一貫した命名は、API間およびフィールド間で標準的な動作を識別するために重要である。ユーザー指定値と実効値の間のプログラムによる関連付けは、一貫した命名に依存する。

### 正規化の根拠

正規化は、サービスが値を標準的な方法で保存および返却できるようにしながら、クライアントにどの変更が意味的に同一であるかを伝えるために重要である。サービス側で値を正規化することで、サービスは意味的に同一のより広範な入力を受け入れつつ、すべての値を生の文字列として維持する必要がなくなる。適用されている正規化をクライアントに公開することで、クライアント側で送信値と取得値を比較して差異をチェックできるようになる。

例えば、特定のフィールドでメールアドレスを受け入れるリソースでは、クライアントはさまざまな方法でメールアドレスを指定する可能性がある。メール`ada@example.com`に対して、クライアントは`ADA@example.com`、`aDa@example.com`、または`AdA@example.com`を指定することを選択するかもしれない。これらは意味的に同一であり、すべてサービスに受け入れられる*べきである*。サービスはその後、小文字化や正規化を通じて、保存と取得のためにメールアドレスを正規化することを選択してもよい。重要なのは、フィールドの正規化に関してクライアントに公開される情報は、正規化アルゴリズム自体ではなく、2つの値が等しいと見なされるべきかを正確に計算するために使用される比較方法を説明することである。

### フィールド値処理の根拠

許可された正規化を使用しないフィールドの場合、[宣言的クライアント][Declarative clients]はどの変更が意味的に意味があるかを識別できない。[宣言的クライアント][Declarative clients]が特定の値を送信する場合、その値が正しく設定されたことを検証するためにサービスから返却されることを確認する。
