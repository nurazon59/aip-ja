# 標準メソッド: Update

REST APIでは、リソースを更新するためにリソースのURI（例：`/v1/publishers/{publisher}/books/{book}`）に`PATCH`または`PUT`リクエストを行うのが慣例である。

リソース指向設計（[AIP-121][]）は、`Update`メソッド（RESTの`PATCH`動作を反映）を通じてこのパターンを尊重する。これらのRPCはそのリソースを表すURIを受け入れ、リソースを返す。

## ガイダンス

APIは、ユーザーにとって価値がない場合を除き、一般的にリソースに対してupdateメソッドを提供**すべきである**（should）。updateメソッドの目的は、副作用を起こさずにリソースを変更することである。

Updateメソッドは以下のパターンを使用して指定される：

```proto
rpc UpdateBook(UpdateBookRequest) returns (Book) {
  option (google.api.http) = {
    patch: "/v1/{book.name=publishers/*/books/*}"
    body: "book"
  };
  option (google.api.method_signature) = "book,update_mask";
}
```

- RPC名は`Update`という単語で始まら**なければならない**（must）。RPC名の残りの部分は、リソースのメッセージ名の単数形**にすべきである**（should）。
- リクエスト メッセージはRPC名と一致し、`Request`サフィックスが付かな**ければならない**（must）。
- レスポンス メッセージはリソース自体**でなければならない**（must）。（`UpdateBookResponse`は存在しない。）
  - レスポンスは完全に設定されたリソースを含む**べきであり**（should）、送信され更新マスクに含まれたすべてのフィールドを含め**なければならない**（must）。ただし、input only（AIP-203参照）または部分的なレスポンスを返す理由がある場合（[AIP-157][]参照）を除く。
  - update RPCが[long-running](#long-running-update)の場合、レスポンス メッセージはリソース自体に解決される`google.longrunning.Operation`**でなければならない**（must）。
- メソッドは部分的なリソース更新をサポート**すべきであり**（should）、HTTP動詞は`PATCH`**すべきである**（should）。
  - メソッドが完全なリソース置換のみをサポートする場合、HTTP動詞は`PUT`**してもよい**（may）。ただし、これは強く推奨されない。なぜなら、リソースにフィールドを追加すると後方互換性のない変更になるためである。
- リソースの`name`フィールドはURIパスにマッピング**すべきである**（should）。
  - `{resource}.name`フィールドはURIパス内の唯一の変数**すべきである**（should）。
- `google.api.http`アノテーションに`body`キーを**含めなければならない**（must）、リクエスト メッセージのリソースフィールドにマッピング**しなければならない**（must）。
  - 残りのすべてのフィールドはURIクエリパラメータにマッピング**すべきである**（should）。
- `google.api.method_signature`アノテーションは1つだけ存在**すべきであり**（should）、値は`"{resource},update_mask"`とする。
- APIが[管理プレーン][management plane]で動作している場合、操作は[強い一貫性][strong consistency]を持つ**べきである**（should）：update操作の完了は、すべてのユーザー設定可能な値とリソースの存在が定常状態に達し、リソース状態の読み取りが一貫したレスポンスを返すことを意味**しなければならない**（must）。

**注:** 他の4つの標準メソッドとは異なり、ここでのURIパスはネストされたフィールド（例では`book.name`）を参照する。リソースフィールドにワードセパレータがある場合は`snake_case`が使用される。

### リクエスト メッセージ

Updateメソッドは共通のリクエスト メッセージパターンを実装する：

```proto
message UpdateBookRequest {
  // 更新するブック。
  //
  // ブックの `name` フィールドは更新するブックを識別するために使用される。
  // 形式: publishers/{publisher}/books/{book}
  Book book = 1 [(google.api.field_behavior) = REQUIRED];

  // 更新するフィールドのリスト。
  google.protobuf.FieldMask update_mask = 2;
}
```

- リクエスト メッセージにはリソースのフィールドを含め**なければならない**（must）。
  - フィールドは`PATCH`ボディにマッピング**しなければならない**（must）。
  - フィールドは [required としてアノテートされるべきである][aip-203]（should）。
  - リソース メッセージに`name`フィールドを含め**なければならない**（must）。`name`**とすべきである**（should）。
  - フィールドは更新されるリソースの[リソース型][aip-123]を識別**しなければならない**（must）。
- 部分的なリソース更新がサポートされている場合、フィールドマスクを含め**なければならない**（must）。型は`google.protobuf.FieldMask`**でなければならない**（must）、`update_mask`と呼ばれ**なければならない**（must）。
  - フィールドマスクで使用されるフィールドは、更新されるリソース（リクエスト メッセージではない）に対応する。
  - フィールドはoptional**でなければならない**（must）、サービスは省略されたフィールドマスクを、設定されている（空でない値を持つ）すべてのフィールドと同等の暗黙のフィールドマスクとして扱わ**なければならない**（must）。
  - 更新マスクは特別な値`*`をサポート**しなければならない**（must）、これは完全な置換（`PUT`に相当）を意味する。
    - APIプロデューサーは、新しい可変フィールドをリソースに追加した際に、コンシューマーがその新しい可変フィールドを知らずに`*`を使用した場合の扱いに注意する必要がある。同様にコンシューマーは、そのリスクが許容できる場合にのみ`*`を使用する必要がある。一般的には、`*`を使用するよりも、更新するフィールドを明示的に指定するのが最も安全である。
- リクエスト メッセージに他の必須フィールドを含めては**ならない**（must not）、このAIPまたは別のAIPで説明されているものを除き、他のオプショナル フィールドを含める**べきではない**（should not）。

### 副作用

一般に、updateメソッドはリソース内のデータを更新することを目的としている。Updateメソッドは他の副作用をトリガー**すべきではない**（should not）。代わりに、副作用はカスタム メソッドによってトリガー**されるべきである**（should）。

特にこれは、[状態フィールド][state fields]がupdateメソッドで直接書き込み可能であっては**ならない**（must not）ことを意味する。

### PATCHとPUT

**要約:** Google APIは一般に`PATCH` HTTP動詞のみを使用し、`PUT`リクエストはサポートしない。

Googleは安定したAPIを後方互換性のある改善でインプレース更新するため、`PATCH`を標準化する。既存のリソースに新しいフィールドを追加することがしばしば必要になるが、`PUT`を使用するとこれは破壊的変更になる。

これを説明するために、`Book`リソースへの`PUT`リクエストを考えてみる：

    PUT /v1/publishers/123/books/456

    {"title": "Mary Poppins", "author": "P.L. Travers"}

次に、リソースが後で新しいフィールド（ここでは`rating`を追加）で拡張される場合を考える：

```proto
message Book {
  string title = 1;
  string author = 2;

  // 後にv1にインプレース追加...
  int32 rating = 3;
}
```

ブックにレーティングが設定されている状態で既存の`PUT`リクエストを実行すると、ブックのレーティングが消去される。本質的に、`PUT`リクエストは以前のバージョンがそのフィールドを知らなかったために、意図せずデータを消去してしまう。

### Long-running update

一部のリソースは、通常のAPIリクエストとして合理的な時間よりも更新に時間がかかる。この場合、APIは代わりにlong-running operation（[AIP-151][]）を使用**すべきである**（should）：

```proto
rpc UpdateBook(UpdateBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    patch: "/v1/{book.name=publishers/*/books/*}"
    body: "book"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンス型はリソース（RPCがlong-runningでない場合の戻り値型）に設定**しなければならない**（must）。
- `response_type`と`metadata_type`の両方を指定**しなければならない**（must）。

**注:** 宣言的フレンドリーなリソース（[AIP-128][]）はlong-running updateを使用**すべきである**（should）。

### CreateまたはUpdate

サービスがクライアント割り当てのリソース名を使用する場合、`Update`メソッドは`bool allow_missing`フィールドを公開**してもよい**（may）。これにより、ユーザーが存在しないリソースを更新しようとした場合にメソッドが成功する（その過程でリソースを作成する）：

```proto
message UpdateBookRequest {
  // 更新するブック。
  //
  // ブックの `name` フィールドは更新するブックを識別するために使用される。
  // 形式: publishers/{publisher}/books/{book}
  Book book = 1 [(google.api.field_behavior) = REQUIRED];

  // 更新されるフィールドのリスト。
  google.protobuf.FieldMask update_mask = 2;

  // trueに設定され、ブックが見つからない場合、新しいブックが作成される。
  // この場合、`update_mask` は無視される。
  bool allow_missing = 3;
}
```

より具体的には、`allow_missing`フラグは以下の動作をトリガーする：

- メソッド呼び出しが存在しないリソースに対するものである場合、リソースが作成される。提供されたフィールドマスクに関係なく、すべてのフィールドが適用される。
  - ただし、必須フィールドが欠落しているか、フィールドに無効な値がある場合、`INVALID_ARGUMENT`エラーが返される。
- メソッド呼び出しが既に存在するリソースに対するものであり、すべてのフィールドが一致する場合、既存のリソースが変更されずに返される。
- メソッド呼び出しが既に存在するリソースに対するものである場合、フィールドマスクで宣言されたフィールドのみが更新される。

ユーザーは`allow_missing`が`true`に設定されている場合でも、`Update`を呼び出すための更新権限を持ってい**なければならない**（must）。updateメソッドを使用したリソース作成をユーザーに許可したくない場合、IAM条件を使用**すべきである**（should）。

### Etag

APIは、ユーザーが最新データに対して行われることが保証された更新リクエストを送信できるようにする必要がある場合がある（一般的なユースケースは競合状態の検出と回避）。これを有効にする必要があるリソースは、リソースの内容を表す不透明なサーバー計算値を含む`string etag`フィールドを含めることで実現する。

この場合、リソースは`string etag`フィールドを含める**べきである**（should）：

```proto
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
  };

  // ブックのリソース名。
  // 形式: publishers/{publisher}/books/{book}
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // ブックのタイトル。
  // 例: "Mary Poppins"
  string title = 2;

  // ブックの作者。
  // 例: "P.L. Travers"
  string author = 3;

  // このブックのetag。
  // 更新時にこれが指定された場合、サーバーのetagと一致しなければならない。
  string etag = 4;
}
```

`etag`フィールドは[required][required]または[optional][optional]の**いずれでもよい**（may）。設定されている場合、提供されたetagがサーバー計算値と一致する場合にのみリクエストは成功**しなければならない**（must）、そうでない場合は`ABORTED`エラーで失敗**しなければならない**（must）。リクエストの`update_mask`フィールドは`etag`フィールドの動作に影響を与えない。なぜなら、etagは更新**される**フィールドではないためである。

### 高コストなフィールド

APIは、リソース上の一部のフィールドを確実に返すことが高コストまたは不可能な状況に遭遇することがある。

これはいくつかの状況で発生する：

- リソースに、計算に非常にコストがかかり、一般的に更新リクエストでは顧客にとって有用でないフィールドがある場合。
- 単一のリソースが、複数の基盤となる（結果的に一貫性を持つ）データソースからのデータの融合を表すことがある。このような状況では、変更されなかったフィールドに関する信頼できる情報を返すことは不可能である。

この場合、APIは更新されたフィールドのみを返し、残りを省略**してもよく**（may）、その場合はこの動作を文書化**すべきである**（should）。

### エラー

[エラー][errors]を参照のこと、特に[PERMISSION_DENIEDとNOT_FOUNDエラーを使用する場合][permission-denied]。

さらに、ユーザーが適切な権限を持っているが、リクエストされたリソースが存在しない場合、サービスは`allow_missing`が`true`に設定されていない限り、`NOT_FOUND`（HTTP 404）でエラーに**しなければならない**（must）。
