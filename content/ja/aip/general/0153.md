# インポートとエクスポート

多くのユーザーが、APIにデータをロードしたり、既存のデータをAPIから取得したりできることを望んでいる。これは、ベンダーロックインをしばしば懸念するエンタープライズユーザーにとって特に重要である。

## ガイダンス

APIはインポートおよびエクスポート操作をサポートしても**してもよい**（may）。これらは複数の新しいリソースを作成しても**してもよい**（may）し、単一のリソースにデータを設定しても**してもよい**（may）。

### 複数リソース

サービスは、APIへの/からの複数リソースのインポートおよびエクスポートをサポートしても**してもよく**（may）、そのための共通パターンを実装する**すべきである**（should）：

```proto
rpc ImportBooks(ImportBooksRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books:import"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "ImportBooksResponse"
    metadata_type: "ImportBooksMetadata"
  };
}

rpc ExportBooks(ExportBooksRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books:export"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "ExportBooksResponse"
    metadata_type: "ExportBooksMetadata"
  };
}
```

- メソッドは、サービスが完了に数秒以上かかることは決してないと保証できない限り、long-running operationを返す**しなければならない**（must）（[AIP-151][]参照）。
- HTTP動詞は`POST`とし、`body`は`"*"`とする**しなければならない**（must）。
- `parent`フィールドをURIの一部として含める**べきである**（should）。
  - 複数リソースへの/からのインポートまたはエクスポートが必要な場合、APIは`parent`フィールドを維持し、ユーザーが`-`文字を使用して複数の親を指定できるようにする**すべきである**（should）（[AIP-159][]参照）。
  - インポート時に、ユーザーが特定の親を指定した場合、APIは異なる親に追加されるインポートリソースを拒否する**しなければならない**（must）。
- URI接尾辞は`:import`または`:export`と**すべきである**（should）。

### 単一リソースのデータ

サービスは、単一リソースへの/からのデータのインポートおよびエクスポートをサポートしても**してもよく**（may）、そのための共通パターンを実装する**すべきである**（should）：

```proto
rpc ImportPages(ImportPagesRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{book=publishers/*/books/*}:importPages"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "ImportPagesResponse"
    metadata_type: "ImportPagesMetadata"
  };
}

rpc ExportPages(ExportPagesRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{book=publishers/*/books/*}:exportPages"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "ExportPagesResponse"
    metadata_type: "ExportPagesMetadata"
  };
}
```

- メソッドは、サービスが完了に数秒以上かかることは決してないと保証できない限り、long-running operationを返す**しなければならない**（must）（[AIP-151][]参照）。
- HTTP動詞は`POST`とし、`body`は`"*"`とする**しなければならない**（must）。
- データがインポートされるリソースを表すフィールドをURIの一部として含める**べきである**（should）。フィールドはリソースにちなんで命名**すべきであり**（should）（`name`と呼ぶ**べきではない**（should not））。
- URI接尾辞には動詞とデータ自体の名詞の両方を含めるべきである（例：`:importPages`や`:exportPages`）。

### リクエストオブジェクト

インポートとエクスポートでは、多くの場合、2つの根本的に異なるタイプの設定が必要である：

1. ソースまたは宛先に固有の設定。
2. インポートまたはエクスポートされるデータ自体に関する設定。

ソースまたは宛先の設定は、単一のメッセージにグループ化し、oneof内に配置すべきである：

```proto
message ImportBooksRequest {
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];
  oneof source {
    AuthorSource author_source = 2;
    TranslatorSource translator_source = 3;
  }
  string isbn_prefix = 4;
}

message ExportBooksRequest {
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];
  oneof destination {
    PrinterDestination printer_destination = 2;
    TranslatorDestination translator_destination = 3;
  }
  string filter = 4;
}
```

- ソース設定メッセージは、1つしかない場合でも、`oneof source`（インポートの場合）または`oneof destination`（エクスポートの場合）内に配置する**しなければならない**（must）（後で追加できる柔軟性を維持するため）。
- データ自体に関連する（したがってすべてのソースに共通する）設定は、リクエストメッセージのトップレベルに配置する**しなければならない**（must）。

**注:** インポートとエクスポートの設定は互いに異なっていても**してもよい**（may）（例えば、ファイルからインポートし、ディレクトリにエクスポートするのは合理的である）。

### インラインソース

APIは、インポートまたはエクスポートするコンテンツがリクエストまたはレスポンスで提供される「インライン」のインポートおよびエクスポートも許可しても**してもよい**（may）。

```proto
message InlineSource {
  repeated Book books = 1;
}
```

- ソースまたは宛先は`InlineSource`または`InlineDestination`と命名する**すべきである**（should）。
- メッセージはリソースを表すrepeatedフィールドを含む**べきである**（should）。ただし、リソース構造が複雑な場合、APIは別のインライン表現を使用しても**してもよい**（may）。この状況では、インポートとエクスポートの両方で同じ形式を使用**しなければならない**（must）。

### 部分的な失敗

部分的な失敗は通常推奨されないが、インポートおよびエクスポートRPCはメタデータオブジェクトに部分的な失敗情報を含める**べきである**（should）。個々のエラーは、エラーを説明する`google.rpc.Status`オブジェクトと**すべきである**（should）。エラーの詳細については、[AIP-193][]を参照のこと。
