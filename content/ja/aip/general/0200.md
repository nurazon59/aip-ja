# 先例

APIはしばしば、それらのAPIがリリースされた後にこれらの標準に追加された新しいガイダンスに適合しない方法で記述される。さらに、確立されたシステムとの一貫性の維持、厳格なパフォーマンス要件への適合、その他の実用的な懸念など、特定の理由から意図的に標準に違反することが合理的な場合もある。最後に、リリース前にAPIを注意深くレビューしたとしても、ミスが紛れ込むことがある。

過去の間違いを修正したり、あらゆるユースケースに標準を対応させたりすることは多くの場合不可能であるため、APIはかなりの期間、これらの例外を抱えたままになる可能性がある。さらに、新しいAPIは既存のAPIを基に設計（名前、型、構造など）されることが多いため、あるAPIの標準違反が他のAPIに波及する可能性がある。たとえ例外の本来の理由が他のAPIには該当しなくてもである。

この問題の結果として、これらの標準例外が新しいAPIに波及するのを「食い止め」、さらに各例外の理由を文書化して、歴史的な知見が失われないようにすることが重要である。

## ガイダンス

何らかの理由でAPIがAIP標準に違反する場合、その違反をコピーしたり、エラーを「事前に承認されたAPI」の先例として引用したりすることを防ぐために、説明的リンク（[aip.dev/not-precedent]()）を使用してこのドキュメントを参照する内部コメントが存在**しなければならない**（must）。

コメントには、何が標準に違反しているのか、なぜそれが必要なのかの説明も含めるべきである。例えば：

```proto
message DailyMaintenanceWindow {
  // Time within the maintenance window to start the maintenance operations.
  // It must use the format "HH MM", where HH : [00-23] and MM : [00-59] GMT.
  // (-- aip.dev/not-precedent: This was designed for consistency with crontab,
  //     and preceded the AIP standards.
  //     Ordinarily, this type should be `google.type.TimeOfDay`. --)
  string start_time = 2;

  // Output only. Duration of the time window, automatically chosen to be
  // smallest possible in the given scenario.
  // (-- aip.dev/not-precedent: This preceded the AIP standards.
  //     Ordinarily, this type should be `google.protobuf.Duration`. --)
  string duration = 3;
}
```

**Important:** APIはベータ版またはGA版である場合にのみ先例と見なされるべきである。

### ローカルの一貫性

APIが標準に全般的に違反している場合、グローバルな標準に準拠するためだけに既存のパターンを破るのは、ユーザーにとって不統一で不便なものとなる。

例えば、あるAPIの全てのリソースが（[AIP-142][]に記載されている標準フィールド`create_time`の代わりに）`creation_time`を使用している場合、そのAPIの新しいリソースはローカルのパターンに従い続けるべきである。

しかし、そのAPIをコピーしようとする他の開発者は、これが標準に反しており、新しいAPIを立ち上げる際の先例として引用すべきではないことを認識すべきである。

```proto
// ...
message Book {
  // (-- aip.dev/not-precedent: This field was present before there was a
  //     standard field.
  //     Ordinarily, it should be spelled `create_time`. --)
  google.protobuf.Timestamp creation_time = 1;
}

// ...
message Author {
  // (-- aip.dev/not-precedent: `Book` had `creation_time` before there was
  //     a standard field, so we match that here for consistency. Ordinarily,
  //     this would be spelled `create_time`. --)
  google.protobuf.Timestamp creation_time = 1;
}
```

### 既存機能

標準違反はリリース前に見落とされることがあり、その結果、安定したAPIとなり簡単に変更できなくなる。さらに、安定したAPIが標準要件よりも前に存在する場合もある。

これらのシナリオでは、APIを標準に適合させることは困難である。しかし、APIは依然としてその機能が標準に反していることを明記すべきであり、他のAPIがその間違いをコピーし、既存のAPIを設計承認の理由として引用することを防ぐべきである。

### 外部仕様への準拠

特定のリクエストが外部仕様（例えばOAuth）の実装であり、その仕様がAIPガイドラインと矛盾する場合、APIは標準に違反せざるを得ないことがある。この場合、外部仕様に従うことが適切であると考えられる。

### 既存システムへの準拠

上記の外部仕様の例と同様に、既存システムに何らかの形で適合させるために、APIがAIPガイドラインに違反することが適切な場合がある。これは基本的に、顧客の現状に合わせることが賢明であるという同様のケースである。潜在的な例としては、パートナーAPIとの統合や類似性が挙げられる。

### 緊急性

非常に厳しい期限までにAPIサーフェスを必要とし、それが叶わなければ資金が失われるユーザーがいることがある。ほとんどのAPIはビジネス上の目的を果たすため、APIをより良くできるが期限内にユーザーの手に届かない状況が発生することがある。そのような場合、APIレビュー委員会は、時間的・ビジネス的な制約によりガイドラインに違反するAPIの出荷を許可**してもよい**（may）。

### 技術的懸念

内部システムには非常に特殊な実装要件がある場合がある（例えば、UTF-8ではなくUTF-16を扱う操作変換に依存しているなど）。AIPガイドラインに従うと、APIコンシューマーにとって重要な価値を追加しない余分な作業が必要になることがある。将来的にAPIを公開する可能性のあるシステムは、この点を考慮し、AIPガイドラインに従うことを困難にする基盤インフラを構築しないようにすべきである。
