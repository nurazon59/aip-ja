# AIPのスタイルとガイダンス

AIPとは**API Improvement Proposal**の略であり、API設計と開発のための高レベルで簡潔なドキュメントを提供する設計文書である。これらのドキュメントは、GoogleにおけるAPI関連ドキュメントの信頼できる情報源として機能し、APIチームがAPIガイダンスについて議論し合意に達するための方法となることを目的としている。

AIPは、明確かつ簡潔で、単一のトピックまたは問い合わせを適切にカバーするときに最も有用である。AIPがAPIで使用するための一貫したパターンとスタイルを説明するのと同様に、AIP自体も一貫したパターンとスタイルに_従う_。

## ガイダンス

- AIPは単一の個別のトピックをカバーし、明確で実用的なガイダンスを提供**しなければならない**（must）。
- AIPは他のAIPのガイダンスを複製したり矛盾したり**してはならない**（must not）。
- AIPは、すべきでないこと_も_カバー**してもよい**（may）が、_のみ_をカバーする**べきではない**（should not）。
- AIPガイダンスが条件付きである場合（例：Jobsのようなデザインパターン）、ガイダンスはどのような条件下で従うべきかを明確に説明**しなければならない**（must）。

AIPに含まれるガイダンスは、1つ以上のタイプのクライアントまたはその作成者にとって有益**でなければならない**（must）、以下を含むがこれらに限定されない：

- リソースの監査と分析に使用できるアセットインベントリ。
- 探索とシンプルな自動化のためのコマンドラインインターフェース。
- ライブ状態をポーリングし、リソース構成を適宜調整するカスタムコントローラー（例：auto-scaler）。
- オーケストレーションと複数リソースの自動化のための[IaC][]クライアント。
- 特定のユースケースにどのAPIが有用か、およびその使用方法に関するガイダンスを提供するレコメンデーションツール。
- プログラミング言語からAPIと対話するためのSDK。データプレーン操作で頻繁に使用される。
- セキュリティオーケストレーション、自動化、および修復ツール。
- タスクを自動化またはオーケストレーションするシンプルなスクリプト。
- テストフレームワーク。
- 休止状態のリソースデータに対して操作を行うツール。
- 可視化および単発の手動アクションのためのビジュアルユーザーインターフェース。
- ユーザー。

改善の例には以下が含まれる：

- クライアント上でより記述的なインターフェースを可能にする新しいprotoアノテーションの要求（例：`google.api.resource`アノテーションでの`singular`および`plural`の要求）。

AIPガイダンスは、クライアントのユーザビリティや実装の難易度、またはメンテナンスの難易度に重大な悪影響を及ぼし**てはならない**（must not）。

悪影響の例には以下が含まれる：

- 標準メソッドに非統一的なパターンを導入し、すべてのクライアントが十分な利益なしに追加のコードを導入しなければならないようにすること（例：ListがFooで始まるリソースを_除いて_このように動作する）。
- 読みやすさの軽微な改善のために確立されたフィールドを改名すること（例：`expire_time`を`lapse_time`に改名。なぜなら`lapse`が自分のサービスで一般的な用語だから）。

AIPの長さは質問の複雑さに応じて必然的に異なるが、ほとんどのAIPはおおむね印刷ページ2ページ程度で内容をカバーできる**べきである**（should）。

### ファイル構造

AIPはMarkdownで記述**しなければならない**（must）、4桁の番号を使用して命名**しなければならない**（must）（例：`0008.md`）。特定のスコープに属するAIPは、そのスコープのサブディレクトリに配置**しなければならない**（must）。

AIPは適切なfront matterを持つ**しなければならない**（must）。

```yaml
---
id: 8
state: reviewing
created: 2019-05-28
permalink: /8
redirect_from:
  - /08
  - /008
  - /0008
---
```

AIPのfront matterには以下を含める**しなければならない**（must）：

- `aip`キー：
  - `id`: Required。AIPのID（整数）。
  - `state`: Required。AIPの現在の状態（すべて小文字）。有効な状態は[AIP-1][]に記載されており、一般的な状態は`draft`、`reviewing`、`approved`である。
  - `created`: Required。AIPが最初にドラフトされたISO-8601日付（`yyyy-mm-dd`）。引用符不要。
  - `updated`: AIPが最後に改訂されたISO-8601日付（`yyyy-mm-dd`）。
  - `scope`: AIPのスコープ。そのスコープのディレクトリ名と一致**しなければならない**（must）。IDが1000以上のAIPには必須。それ以外では禁止。
- `permalink`キー（required）：これは`/{aip.scope}/{aip.id}`に設定**しなければならない**（must）。スコープがない場合は、代わりに`/{aip.id}`を使用する。
- `redirect_from`キー：読者が入力しそうな`/{aip.id}`の順列のリストを含めるべきである。以下を含む：
  - `/{aip.id}`（パーマリンクにスコープが含まれるAIPの場合）
  - 4桁までの各ゼロパディングレベルでのAIP ID（例：`/08`、`/008`、`/0008`）。

### ドキュメント構造

AIPはAIPのタイトルを持つトップレベルの見出し（`# タイトル`）で始め**なければならない**（must）。タイトルは名詞（命令形ではない）**とすべきである**（should）。例えば、"Bad API precedents"であって"Avoid breaking API precedent"ではない。

AIPはその後、導入文（追加の見出しなし）で始め、続いて`## Guidance`見出しを置く**べきである**（should）。必要に応じて、AIPはガイダンスの後に以下のいずれかを次の順序で含める**してもよい**（may）：

- "Further reading"は、現在のAIPを完全に理解するために有用な他のAIPへのリンクの箇条書きリストである。
- "Appendices"は、同じAIP内でのさらなる説明をカバーする。これらは比較的稀であるが、AIPが決定に対してもっと多くの正当化を必要とする場合に重要である。多くの場合、これはガイダンスを説明するために検討された代替案の説明である。
- "Changelog"は、最初の作成以降にAIPに行われた変更の箇条書きリストである。

ガイダンスセクションは、詳細をさらに掘り下げるサブセクションを含める**してもよい**（may）。サブセクションは自動的に目次にエントリを作成し、引用のためのアンカーを提供する。

以下は、主要な各セクションを使用したAIPシェルの例である：

```md
# AIP title

The introductory text explains the background and reason why the AIP exists. It
lays out the basic question, but does not tell the reader what to do.

## Guidance

The "guidance" section helps the reader know what to do. A common format for
the guidance section is a high-level imperative, followed by an example,
followed by a bulleted list explaining the example.

### Subsection

Individual subsections can be cited individually, and further elaborate
details.

## Rationale

The "rationale" section is optional, and helps the reader understand the
motivation behind specific guidance within the AIP.

Deeper explanations of design justification and tradeoffs **must** be in the
rationale instead of other sections, to ensure the rest of the document acts as
an easily actionable reference.

## History

The "history" section is optional, and documents events and context around a
significant edit to an AIP. For example, explanation of rewrite would be
included in this section

While the changelog is a dotted list of one-line summaries of changes to an AIP,
the history section should elaborate on significant events in a descriptive
format.

The section **must not** be used to exhaustively enumerate all changes. This
is what the changelog provides.

## Further reading

A bulleted list of (usually) other AIPs, in the following format:

- [AIP-1](./0001.md): AIP purpose and guidelines

## Changelog

A bulleted list of changes in reverse chronological order, using the following
format:

- **2020-02-18**: Specified ordering.
- **2019-07-01**: Added a subsection clarifying XYZ.
```

AIPは可能であればこの全体的な形式に従う**べきである**（should）が、必要であれば逸脱**してもよい**（may）（特に、通常の形式でのAIPの読み方にすでに慣れている読者にとってもAIPが理解しにくくなる場合）。

**注:** タイトルを除き、AIPは第2見出しレベル（`##`）以上のみを使用**しなければならない**（must）。AIPは第2および第3見出しレベル（`##`、`###`）のみを使用**すべきである**（should）。

### 要件キーワード

AIPは以下の要件レベルのキーワードを使用する**べきである**（should）："MUST"、"MUST NOT"、"SHOULD"、"SHOULD NOT"、"MAY"。これらは[RFC 2119][]で説明されているように解釈される。

AIPでこれらの用語を使用する場合、小文字で**bold**で**なければならない**（must）。これらの用語は他の方法で使用する**べきではない**（should not）。

"SHOULD"または"SHOULD NOT"を使用する場合、他の懸念事項がガイダンスを上書きする可能性がある有効な例を含め**なければならない**（must）。

**重要:** 根拠を使用する場合、それは背景とより完全な理解を提供するために存在するが、ガイダンスを含ん**ではならない**（must not）（RFC-2119用語を使用し**てはならない**（must not））。

### コード例

AIP内のAPI設計例は[Protocol Buffers][]を使用する**べきである**（should）。例はコンセプトを説明するのに十分な構文のみをカバーする**べきである**（should）。例でRPCを使用する場合、`google.api.http`アノテーションを含める**べきである**（should）。

### AIPの参照

AIPが他のAIPを参照する場合、散文テキストはゼロパディングなしの形式`AIP-XXXX`（例：`AIP-8`、`AIP-0008`ではない）を使用**しなければならない**（must）、関連するAIPにリンク**しなければならない**（must）。AIPリンクは、適切であればAIPの特定のセクションを指す**してもよい**（may）。

**重要:** AIPリンクはリポジトリ内のファイルへの相対パス（コアAIPの場合は`./0008.md`、サブディレクトリ内のAIPの場合は`../0008.md`など）を使用**しなければならない**（must）。これにより、AIPサイト上、GitHubでのMarkdownファイル表示時、ローカル開発サーバーの使用時、またはブランチ上で、リンクが機能することが保証される。

## 根拠

### 幅広いクライアントセット向けの設計

APIガイダンスは、他のソフトウェアと同様に、明確な目的と対象受益者がいる場合に最も有益である。

API設計改善の受益者はユーザーである。これらのユーザーは、上記で列挙したユースケースに応じて、さまざまなクライアントを介してAPIと対話する。

APIガイダンスは、これらのクライアント全体への影響を広く考慮しなければならない。
