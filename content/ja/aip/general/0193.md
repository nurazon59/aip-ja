# エラー

効果的なエラー通信は、シンプルで直感的なAPIを設計する上で重要な部分である。標準化されたエラーレスポンスを返すサービスは、APIクライアントが一元化された共通のエラーハンドリングロジックを構築できるようにする。この共通ロジックはAPIクライアントアプリケーションを簡素化し、面倒なカスタムエラーハンドリングコードの必要性を排除する。

## ガイダンス

サービスは、APIエラーが発生した場合に[`google.rpc.Status`][Status]メッセージを返さ**なければならない**（must）。また、[`google.rpc.Code`][Code]で定義された標準的なエラーコードを使用**しなければならない**（must）。特定のコードに関する詳細情報は、[gRPCステータスコードドキュメント][grpc status code documentation]で入手できる。

エラーメッセージは、合理的に技術的なユーザーが問題を*理解*し*解決*するのに役立つ**べきである**（should）。また、ユーザーが特定のAPIの専門家であると想定**すべきではない**（should not）。さらに、エラーメッセージはユーザーがその基礎となる実装について何か知っていると想定**してはならない**（must not）。

エラーメッセージは簡潔でありながら実行可能**であるべきである**（should）。追加情報は`details`フィールドで提供**すべきである**（should）。さらに多くの情報が必要な場合、読者が詳細情報を得たり質問して問題解決に役立てたりできるリンクを提供**すべきである**（should）。メッセージを書く際には[適切なトーン][writing-tone]を設定することも重要である。

以下のセクションでは`google.rpc.Status`のフィールドについて説明する。

### Status.message

`message`フィールドは開発者向けの人間が読める「デバッグメッセージ」であり、英語である**すべきである**（should）。（ローカライズされたメッセージは`details`フィールド内の`LocalizedMessage`を使用して表現される。詳細については[`LocalizedMessage`](#localizedmessage)を参照。）メッセージの動的な側面は、[`details`](#status-details)に現れる`ErrorInfo`内にメタデータとして含める**しなければならない**（must）。

このメッセージは問題の説明と見なされる。開発者が問題を理解するためのものであり、後述する[`ErrorInfo.reason`][ErrorInfo-reason]よりも詳細である。

メッセージは、エラーの原因となる問題を明確に述べ、それに対する実行可能な解決策を提供するために、理解しやすいシンプルな説明的言語（技術的な専門用語なし）を使用**すべきである**（should）。

機械可読な識別子なしで以前にエラーを返していた既存（brownfield）APIの場合、特定のエラーに対する`message`の値は同じまま**でなければならない**（must）。詳細については、[エラーメッセージの変更](#changing-error-messages)を参照。

### Status.code

`code`フィールドはステータスコードであり、[`google.rpc.Code`][Code] enumの要素の1つの数値**でなければならない**（must）。

例えば、値`5`は`NOT_FOUND` enum要素の数値である。

### Status.details

`details`フィールドにより、追加のエラー情報を含むメッセージをエラーレスポンスに含めることができ、それぞれ`google.protobuf.Any`メッセージにパックされる。

Googleは、APIエラーの最も一般的なニーズをカバーする、エラー詳細用の[標準的な詳細ペイロード][details]セットを定義している。サービスは、可能な場合これらの標準的な詳細ペイロードを使用**すべきである**（should）。

各タイプの詳細ペイロードは最大1回まで含める**しなければならない**（must）。例えば、`details`内に複数の[`BadRequest`][BadRequest]メッセージがあっては**してはならない**（must not）が、`BadRequest`と[`PreconditionFailure`][PreconditionFailure]があっても**してもよい**（may）。

すべてのエラーレスポンスは、`details`内に`ErrorInfo`を含める**しなければならない**（must）。これにより機械可読な識別子が提供され、ユーザーはエラーの特定の側面に対してコードを書くことができる。

以下のセクションでは、最も一般的な標準詳細ペイロードについて説明する。

#### ErrorInfo

[`ErrorInfo`][ErrorInfo]メッセージは、機械可読な識別子を送信する主要な方法である。コンテキスト情報は`ErrorInfo`の`metadata`に含める**べきである**（should）。また、エラーメッセージ内に現れる場合は含める**しなければならない**（must）。

`reason`フィールドは、エラーの原因を表す短いsnake_caseの説明である。エラーの理由は、特定のエラードメイン内で一意である。理由は最長63文字**でなければならない**（must）、正規表現`[A-Z][A-Z0-9_]+[A-Z0-9]`に一致**しなければならない**（must）。（これは先頭または末尾のアンダースコアがなく、先頭に数字がないUPPER_SNAKE_CASEである。）

理由は簡潔である**すべきである**（should）が、人間の読者が言及している内容を理解できる程度に意味がある**すべきである**（should）。

良い例：

- `CPU_AVAILABILITY`
- `NO_STOCK`
- `CHECKED_OUT`
- `AVAILABILITY_ERROR`

悪い例：

- `THE_BOOK_YOU_WANT_IS_NOT_AVAILABLE`（冗長すぎる）
- `ERROR`（一般的すぎる）

`domain`フィールドは、`reason`が所属する論理的なグループである。ドメインはグローバルに一意な値**でなければならない**（must）、通常はエラーを生成したサービスの名前（例：`pubsub.googleapis.com`）である。

(reason, domain)のペアは、特定のエラーを識別する機械可読な方法を形成する。サービスは同じエラーに対して同じ(reason, domain)ペアを使用**しなければならない**（must）、論理的に異なるエラーに対して同じ(reason, domain)ペアを使用しては**してはならない**（must not）。2つのエラーが「同じ」かどうかの判断は常に明確ではないが、一般的にはクライアントがそれらを解決するために取るであろう期待されるアクションの観点から考える**すべきである**（should）。

`metadata`フィールドは、コンテキストとして追加の動的情報を提供するキー/値ペアのマップである。`metadata`内の各キーは最長64文字**でなければならない**（must）、正規表現`[a-z][a-zA-Z0-9-_]+`に準拠**しなければならない**（must）。

`Status.message`または`LocalizedMessage.message`メッセージに寄与するリクエスト固有の情報は、`metadata`内で表現**しなければならない**（must）。このプラクティスは、機械的なアクターが情報を抽出するためにエラーメッセージをパースする必要がないようにするために重要である。

例えば以下のメッセージを考える：

> An &lt;e2-medium&gt; VM instance with &lt;local-ssd=3,nvidia-t4=2&gt; is currently unavailable
> in the &lt;us-east1-a&gt; zone. Consider trying your request in the &lt;us-central1-f,us-central1-c&gt;
> zone(s), which currently has/have capacity to accommodate your request. Alternatively,
> you can try your request again with a different VM hardware configuration
> or at a later time. For more information, see the troubleshooting documentation.

同じエラーの`ErrorInfo.metadata`マップは以下のようになる：

- `"zone": "us-east1-a"`
- `"vmType": "e2-medium"`
- `"attachment": "local-ssd=3,nvidia-t4=2"`
- `"zonesWithCapacity": "us-central1-f,us-central1-c"`

エラーメッセージに現れない追加のコンテキスト情報も、クライアントによるプログラム的な使用を可能にするために`metadata`に含めて**もよい**（may）。

特定の(reason,domain)ペアに含まれるメタデータは時間とともに進化できる：

- 新しいキーを含めて**もよい**（may）
- これまでに含まれていたすべてのキーは含め続け**なければならない**（must）（ただし値が空でもよい）

言い換えれば、ユーザーが(reason, domain)ペアの特定のキーを一度観測した場合、サービスはそのキーが将来も存在し続けることに依存することを許可**しなければならない**（must）。

各(reason, domain)ペアで提供されるキーのセットは他のペアから独立しているが、サービスは一貫したキー命名を目指す**べきである**（should）。例えば、同じドメイン内の2つのエラー理由が`vmType`と`virtualMachineType`というメタデータキーを使用すべきではない。

#### LocalizedMessage

[`google.rpc.LocalizedMessage`][LocalizedMessage]は、可能な場合はユーザー指定のロケールにローカライズされるべきエラーメッセージを提供するために使用される。

[`Status.message`](#status-message)フィールドが[エラーメッセージの変更](#changing-error-messages)セクションの制約により変更できない最適でない値を持つ場合、`LocalizedMessage`を使用して、ユーザー指定のロケールが利用できない場合でもより良いエラーメッセージを提供**してもよい**（may）。

メッセージのロケールがどのように決定されたかに関係なく、`locale`と`message`の両方のフィールドを設定**しなければならない**（must）。

`locale`フィールドはメッセージのロケールを指定し、[IETF bcp47](https://www.rfc-editor.org/rfc/bcp/bcp47.txt)（言語識別タグ）に従う。値の例：`"en-US"`、`"fr-CH"`、`"es-MX"`。

`message`フィールドにはローカライズされたテキスト自体が含まれる。これにはエラーの簡単な説明とエラーを解決するためのアクションの呼び出しを含める**べきである**（should）。メッセージは可能な限り具体的にするためにコンテキスト情報を含める**べきである**（should）。メッセージ内のコンテキスト情報は`ErrorInfo.metadata`に含める**しなければならない**（must）。コンテキスト情報をメッセージと対応するメタデータに含める方法の詳細については、[`ErrorInfo`](#errorinfo)を参照。

`LocalizedMessage`ペイロードにはエラーに対する完全な解決策を含める**べきである**（should）。このペイロードに合理的に収まらないほど多くの情報が必要な場合、追加の解決情報は`Help`ペイロードで提供**しなければならない**（must）。ガイダンスについては[Help](#help)セクションを参照。

#### Help

他のテキストエラーメッセージ（`Status.message`または`LocalizedMessage.message`）がユーザーに十分なコンテキストや実行可能な次のステップを提供しない場合、またはトラブルシューティングで考慮すべき複数の障害点がある場合、補足的なトラブルシューティングドキュメントへのリンクを`Help`ペイロードで提供**しなければならない**（must）。

この情報は、明確な問題定義と実行可能な解決策に加えて提供すること。それらの代わりとしてはならない。リンク先のドキュメントはエラーに明確に関連してい**なければならない**（must）。単一のページに複数のエラーに関する情報が含まれている場合、[`ErrorInfo.reason`](#errorinfo)値を使用して関連情報を絞り込ま**なければならない**（must）。

`description`フィールドはリンク先情報のテキストによる説明である。これはハイパーリンクのテキストとしてユーザーに表示するのに適してい**なければならない**（must）。これはプレーンテキスト（HTML、Markdownなどではない）で**しなければならない**（must）。

`description`値の例：`"STOCKOUTエラーのトラブルシューティングドキュメント"`

`url`フィールドはリンク先のURLである。これはスキームを含む絶対URL**でなければならない**（must）。

`url`値の例：
`"https://cloud.google.com/compute/docs/resource-error"`

公開ドキュメント化されたサービスの場合、実際の使用にアクセス制御があるものでも、リンク先のコンテンツは認証なしでアクセス可能**でなければならない**（must）。

非公開ドキュメント化されたサービスの場合、リンク先のコンテンツは認証を要求**してもよい**（may）。

### エラーメッセージ

テキストのエラーメッセージは`Status.message`と`LocalizedMessage.message`フィールドの両方に存在できる。メッセージは簡潔でありながら実行可能**であるべきである**（should）。リクエスト固有の情報（リソース名やリージョンなど）は、適切な場合に正確な詳細を提供する。リクエスト固有の詳細は[`ErrorInfo.metadata`](#errorinfo)に存在**しなければならない**（must）。

#### エラーメッセージの変更

`Status.message`の内容を時間とともに変更する場合は、以前からすべての情報をメッセージに依存せざるを得なかったクライアントを壊さないように注意して行う必要がある。詳細については[根拠セクション](#updating-statusmessage)を参照。

特定のRPCについて：

- RPCが*常に*機械可読な情報を含む`ErrorInfo`を返してきた場合、`Status.message`の内容は時間とともに変更**してもよい**（may）。（例えば、APIプロデューサーがより明確な説明やより多くのリクエスト固有の情報を提供する場合がある。）
- それ以外の場合、`Status.message`の内容は安定してい**なければならず**（must）、同じリクエスト固有の情報で同じテキストを提供する。`Status.message`を変更する代わりに、APIは`Status.details`内に[`LocalizedMessage`](#localizedmessage)を含める**べきである**（should）。

RPCが常に`ErrorInfo`を返してきた場合でも、APIは既存の`Status.message`を安定に保ち、`Status.details`内に[`LocalizedMessage`](#localizedmessage)を追加**してもよい**（may）。

`LocalizedMessage.details`の内容は時間とともに変更**してもよい**（may）。

### 部分エラー

APIは部分エラーをサポート**すべきではない**（should not）。部分エラーはユーザーに大きな複雑さをもたらす。なぜなら、通常はエラーコードの使用を回避し、それらのエラーコードをレスポンスメッセージに移動させ、ユーザーは問題に対処するために特殊なエラーハンドリングロジックを書か**なければならない**（must）からである。

ただし、部分エラーが必要な場合もある。特にバルク操作において、単一のエントリの問題のために大きなリクエスト全体を失敗させることはユーザーにとって敵対的である場合である。

部分エラーが必要なメソッドは、[long-running operations][]を使用**すべきである**（should）。メソッドは部分的な障害情報をメタデータメッセージに入れる**べきである**（should）。エラー自体は依然として[`google.rpc.Status`][Status]オブジェクトで表現**しなければならない**（must）。

### Permission Denied

ユーザーがリソースまたは親へのアクセス権限を持っていない場合（存在有無にかかわらず）、サービスは`PERMISSION_DENIED`（HTTP 403）でエラーに**しなければならない**（must）。権限はリソースまたは親の存在を確認する前にチェック**しなければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースまたは親が存在しない場合、サービスは`NOT_FOUND`（HTTP 404）でエラーに**しなければならない**（must）。

## HTTP/1.1+JSON表現

クライアントが[AIP-127](./0127.md)に従ってHTTP/1.1を使用する場合、エラー情報はレスポンスのボディにJSONオブジェクトとして返される。後方互換性の理由から、これは`google.rpc.Status`に正確にマッピングされるわけではないが、同じコア情報を含む。スキーマは以下のprotoで定義されている：

```proto
message Error {
  message Status {
    // `google.rpc.Status.code`に対応するHTTPステータスコード。
    int32 code = 1;
    // これは`google.rpc.Status.message`に対応する。
    string message = 2;
    // これは`google.rpc.Status.code`のenumバージョンである。
    google.rpc.Code status = 4;
    // これは`google.rpc.Status.details`に対応する。
    repeated google.protobuf.Any details = 5;
  }

  Status error = 1;
}
```

最も重要な違いは、JSONの`code`フィールドがHTTPステータスコードであり、`google.rpc.Status.code`の直接の値ではないことである。例えば、`code`値が5の`google.rpc.Status`メッセージは、以下のコード関連フィールド（およびmessage、detailsなど）を含むオブジェクトにマッピングされる：

```json
{
  "error": {
    "code": 404,          // "not found"のHTTPステータスコード
    "status": "NOT_FOUND" // 値5に対するgoogle.rpc.Codeの名前
  }
}
```

以下のJSONは、エラーレスポンスの完全に設定されたHTTP/1.1+JSON表現を示している。

```json
{
  "error": {
    "code": 429,
    "message": "The zone 'us-east1-a' does not have enough resources available to fulfill the request. Try a different zone, or try again later.",
    "status": "RESOURCE_EXHAUSTED",
    "details": [
      {
        "@type": "type.googleapis.com/google.rpc.ErrorInfo",
        "reason": "RESOURCE_AVAILABILITY",
        "domain": "compute.googleapis.com",
        "metadata": {
          "zone": "us-east1-a",
          "vmType": "e2-medium",
          "attachment": "local-ssd=3,nvidia-t4=2",
          "zonesWithCapacity": "us-central1-f,us-central1-c"
        }
      },
      {
        "@type": "type.googleapis.com/google.rpc.LocalizedMessage",
        "locale": "en-US",
        "message": "An <e2-medium> VM instance with <local-ssd=3,nvidia-t4=2> is currently unavailable in the <us-east1-a> zone. Consider trying your request in the <us-central1-f,us-central1-c> zone(s), which currently has/have capacity to accommodate your request. Alternatively, you can try your request again with a different VM hardware configuration or at a later time. For more information, see the troubleshooting documentation."
      },
      {
        "@type": "type.googleapis.com/google.rpc.Help",
        "links": [
          {
            "description": "Additional information on this error",
            "url": "https://cloud.google.com/compute/docs/resource-error"
          }
        ]
      }
    ]
  }
}
```

## 根拠

### ErrorInfoの必須化

`ErrorInfo`が必須なのは、エラーをさらに識別するためである。`Status.status`の[利用可能な値][Code]が約20個しかないため、[APIサービス][API Service]全体でエラーを区別することは困難である。

また、エラーメッセージには変数情報を表現する動的なセグメントが含まれることが多いため、クライアントがそのような情報をプログラム的に使用できるように、*すべての*エラーレスポンスに機械可読なコンポーネントが必要である。

### LocalizedMessageを含めること

`LocalizedMessage`は、代替のエラーメッセージを提示する場所として選択された。`LocalizedMessage`はリクエストで指定されたロケールを使用**してもよい**（may）が、サービスはユーザー指定のロケールがなくても`LocalizedMessage`を提供**してもよい**（may）。通常は[`Status.message`を変更できない状況](#updating-statusmessage)でより良いエラーメッセージを提供するためである。ロケールがユーザーによって指定されていない場合、`en-US`（米国英語）に**すべきである**（should）。

サービスは、同じメッセージが`Status.message`で提供され、ユーザー指定のロケールへのローカライズがサポートされていない場合でも、`LocalizedMessage`を含めて**もよい**（may）。その理由は以下の通り：

- 近い将来にユーザー指定のローカライズをサポートする意図があり、クライアントが一貫して`LocalizedMessage`を使用し、機能が導入されてもエラー報告コードを変更しないようにする。
- サービス内のすべてのRPC間での一貫性：一部のRPCが`LocalizedMessage`を含み、一部が`Status.message`のみをエラーメッセージに使用する場合、クライアントはどのRPCが何を行うかを認識するか、フォールバックメカニズムを実装する必要がある。すべてのRPCに`LocalizedMessage`を提供することで、シンプルで一貫性のあるクライアントコードを書くことができる。

### Status.messageの更新

クライアントが`Status.message`が設定されている（常に設定されている）が`ErrorInfo`がないエラーを観測したことがある場合、そのクライアントの開発者は、`Status.code`が伝える以上の情報を見つけるために`Status.message`をパースせざるを得なかった可能性が高い。その情報は、特定のテキストのマッチング（例：「原因不明の接続断」）やメッセージのパースによるメタデータ値（例：リソース不足のリージョン）の発見によって見つけられる。その時点で、`Status.message`は暗黙的にAPI契約の一部となるため、更新しては**してはならない**（must not）。それは破壊的変更となる。これが`Status.details`に`LocalizedMessage`を導入する理由の1つである。

*常に*`ErrorInfo`を含んできたRPCはより良い立場にある：契約は特定のエラーに対する`ErrorInfo`の安定性に関するものとなる。理由とドメインは時間とともに一貫している必要があり、特定の(reason,domain)に対して提供されるメタデータは拡張のみ可能である。クライアントが`ErrorInfo`を使用する代わりに`Status.message`をパースしている可能性は依然としてあるが、常により堅牢なオプションが利用可能であった。
