# ベータブロッカーな変更

APIは、顧客から早期のフィードバックを得るためにAlphaバージョンをリリースすることがよくある。このAPIは暫定的なものであり、重要なフィードバックが組み込まれ、APIがBeta向けに安定するまでに何度も変更される可能性がある。

Alphaの目的はフィードバックを収集することであるため、APIはまだ完璧である必要はなく、API作成者が全てのユーザビリティの問題に対処したり、API標準の全てのポイントに対応したりする必要は厳密にはない。多くの場合、API作成者とAPIレビュアーは最適な設計に同意せず、最善の方法はユーザーにAPIを試用してもらうことである。

しかし、フィードバックが収集され、APIがBetaに昇格される際には、ユーザビリティの問題やスタイルの問題に対処する必要がある。これらの問題が忘れられないように、API内に明示的に文書化すべきである。

## ガイダンス

APIにユーザビリティの問題があるかAPI標準に違反しており、現在の設計がBetaバージョンに持ち越される前に追加の精査を受けるべきである場合、説明的リンク（[aip.dev/beta-blocker]()）を使用してこのドキュメントを参照する内部コメントが**しなければならない**（must）存在し、APIがBetaにリリースされる前に設計が修正されることを保証する。

コメントはまた、Betaに向けてどのような変更を行うべきかを示す**しなければならない**（must）。例えば：

```proto
message InputConfig {
  // Parameters for input.
  // (-- aip.dev/beta-blocker: Convert well-known parameters into explicit
  //     fields before the Beta launch. --)
  map<string, string> parameters = 1;
}
```

BetaやGAに持ち越さなければならないAPI標準の例外がある場合は、[AIP-200][]を参照のこと。
