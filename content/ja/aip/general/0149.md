# 未設定フィールド値

多くのメッセージでは、多くのフィールドがオプションである：ユーザーはそれらを提供する必要はなく、出力フィールドの場合、サービスがフィールドを設定しないことがある。

ほとんどの場合、フィールドをデフォルト値（`0`など）に設定することと、まったく設定しないことの間に意味のある違いはない。しかし、この区別が意味を持つ場合もある。

## ガイダンス

Protocol Buffersで定義されるサービスは、フィールドをデフォルト値（`0`、`false`、または空文字列）に設定することとまったく設定しないことを区別する必要がある場合にのみ、プリミティブに`optional`キーワードを使用**すべきである**（should）：

```proto
// A representation of a book in a library.
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
  };

  // The name of the book.
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // The rating for the book, from 0 to 5.
  // 0 is distinct from no rating.
  optional int32 rating = 2;
}
```

**重要:** サービスは、ほとんどの場合、デフォルト値と未設定を区別**すべきではない**（should not）。もしそのような区別を必要としない代替設計があるならば、通常はそちらが優先される。実際には、これは`optional`は整数と浮動小数点にのみ使用**すべきである**（should）ことを意味する。

**重要:** フィールドの存在を追跡することは、[AIP-203][]で定義されているAPIフィールド動作を文書化することとは異なる。例えば、存在追跡のために`optional`とラベル付けされたフィールドは、フィールドが設定されなければならない場合、`google.api.field_behavior = REQUIRED`としてアノテーションされても**してもよい**（may）。フィールドのサーバーが認識する動作のみを文書化したい場合は、[AIP-203][]を参照のこと。

### 後方互換性

既存のフィールドに`optional`修飾子を追加または削除することは、後方互換性のない変更である。これは、コンパイルされたsrc APIが（一部の言語で）変更されるためである。例えば、Golangでは、`optional`を追加するとプリミティブのフィールド型が元の型のポインタバリアントに変わる。例えば、以前は`string`型だったフィールドが`*string`になるなど。したがって、この変更はクライアントとサーバーの両方が変更されたフィールドの使用法を同時に更新する必要があり、これはリスクが高くエラーが発生しやすい。追加情報は[Protobufによって文書化されている][proto docs]。

## 根拠

### field behaviorと`optional`

フィールド動作アノテーションと`optional`ラベルは、異なる問題に対処するため、相互排他的ではない。前者の`google.api.field_behavior`は、API内でのフィールドに対するサーバーの認識（必須かどうか、immutableかどうかなど）を伝えることに焦点を当てている。後者のproto3の`optional`は、ワイヤフォーマットとコード生成オプションであり、厳密にフィールド存在追跡を切り替えるためのものである。フィールドが`google.api.field_behavior = REQUIRED`と同時にアノテーションされ、`optional`とラベル付けされることは混乱を招くかもしれないが、実際にはこれらは無関係であり、合理的に一緒に使用できる。
