# フィールドマスク

多くの場合、リソースを更新する際（AIP-134で定義されている更新メソッド、またはそれに類似したものを使用）、どのフィールドが更新されているかを正確に指定し、ユーザーが新しい値を送信した場合でもサービスが残りを無視できるようにすることが望ましい。

各APIの正確なニーズを処理するマスク形式を定義したくなるかもしれない。しかし、マスク要件は進化するため、構造化構文を使用することが賢明である。これにより、UIやクライアントの更新を待つことなく、透過的に更新を行うことができる。

## ガイダンス

これらのフィールド名のマスクは「フィールドマスク」と呼ばれる。フィールドマスクを表すフィールドは `google.protobuf.FieldMask` 型を使用 **しなければならない**（must）。フィールドマスクは更新リクエスト（AIP-134）で最も一般的である。

フィールドマスクは常にリソースを基準と**しなければならない**（must）：

**警告:** リクエストメッセージ上の単一のフィールドとしての読み取りマスク、例えば `google.protobuf.FieldMask read_mask` は**非推奨**である。代わりにAIP-157を参照。

```proto
message UpdateBookRequest {
  // 更新する書籍。
  //
  // 書籍の `name` フィールドは更新する書籍を識別するために使用される。
  // 形式: publishers/{publisher}/books/{book}
  Book book = 1 [(google.api.field_behavior) = REQUIRED];

  // 更新するフィールドのリスト。
  // フィールドは書籍を基準として指定される
  // （例：`title`、`rating`。`book.title` や `book.rating` ではない）。
  google.protobuf.FieldMask update_mask = 2;
}
```

### 読み取り-書き込みの一貫性

フィールドマスクの読み取りと書き込みの動作は、マスクが存在する場合に自己整合性を **しなければならない**（must）：

- ユーザーが特定のマスクでリソースを更新し、その後同じマスクで同じリソースを読み取った場合、まったく同じデータが返される **しなければならない**（must）。
  - 例外: [Output onlyフィールド](#output-only-fields)。
- 同様に、特定のマスクでリソースを読み取り、返されたデータと同じマスクでリソースを更新することは、no-opで **しなければならない**（must）。

**注記:** これは、読み取りまたは書き込みのいずれかに有効なマスクは、両方に有効で**なければならない**（must）ことを意味する。

### 特定フィールドの指定

フィールドマスクは、定義された構造体内の特定のフィールドの指定を許可 **しなければならない**（must）。トラバーサルには `.` 文字を使用する。

フィールドマスクは常にリソースを基準とするため、リソース上の直接フィールドにはトラバーサルは不要である（例：`title`、`rating`）。トラバーサルは、リソースがメッセージを含む場合に使用される（例：`author.given_name`）。

**注記:** ユーザーは、フィールド全体またはそのサブフィールドのいずれかを指定でき**なければならない**（must）：`author` と `author.given_name` は両方とも有効である。

### マップフィールド

フィールドマスクは、マップのキーが文字列または整数である場合に限り、マップ内の特定のフィールドの指定を許可 **してもよい**（may）。トラバーサルには `.` 文字を使用する。

フィールドマスクは、フィールドマスク構文に問題のある文字を含む文字列キーを、バックティック文字を使用してサポート **すべきである**（should）。

```proto
message Book {
  // 書籍の名前。
  // 形式: publishers/{publisher}/books/{book}
  string name = 1;

  // 裏表紙のレビュー。キーはレビューの著者、値はレビューのテキスト。
  //
  // 有効なフィールドマスク: reviews, reviews.smith, reviews.`John Smith`
  map<string, string> reviews = 2;
}
```

### ワイルドカード

フィールドマスクは、繰り返しフィールドまたはマップ上の `*` 文字の使用を許可 **してもよい**（may）し、コレクション内の特定のサブフィールドの指定を示す：

```proto
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
  };

  // 書籍の名前。
  // 形式: publishers/{publisher}/books/{book}
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // 書籍の著者。
  // 有効なフィールドマスク: authors, authors.*.given_name, authors.*.family_name
  // 無効なフィールドマスク: authors.0, authors.0.given_name
  repeated Author authors = 2;
}

message Author {
  // 著者の名。
  string given_name = 1;

  // 著者の姓。
  string family_name = 2;
}
```

**注記:** フィールドマスクは、インデックスによる繰り返しフィールドの特定の要素へのアクセスを許可しては**ならない**（must not）。試みられた場合、`INVALID_ARGUMENT` エラーを返さ**なければならない**（must）。

### Output onlyフィールド

ユーザーがワイルドカードを使用したり、output onlyのサブフィールドを含むメッセージ全体を指定することで、更新マスクにoutput onlyフィールドを間接的に含めた場合、サービスは入力として提供されたoutput onlyフィールドを無視 **しなければならない**（must）。それらがクリアまたは変更された場合でも同様である。

ユーザーが更新マスクでoutput onlyフィールドを直接指定した場合、サービスは入力として提供されたoutput onlyフィールドを無視 **しなければならない**（must）。それらがクリアまたは変更された場合でも、同じフィールドマスクを入力と出力の両方に使用できるようにするためである。

### 無効なフィールドマスクエントリ

データ読み取り時、フィールドマスクは存在し得ない値（存在しないフィールド、またはサービスが無効と見なすマップキー）を指すエントリを無視 **してもよい**（may）。

データ書き込み時、フィールドマスクは存在し得ない値を指すエントリに対して `INVALID_ARGUMENT` エラーを返す **すべきである**（should）。ただし、サービスは削除を許可 **してもよい**（may）。
