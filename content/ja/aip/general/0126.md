# 列挙型

フィールドが離散的で限られた値のセットのみを受け入れるか提供することは一般的である。そのような場合、許可される値のセットを明確に伝えるために列挙型（通常「enum」と略される）を使用することが有用である。

## ガイダンス

APIは、変更頻度が低いと予想される値のセットに対してenumオブジェクトを公開**してもよい**（may）：

```proto
// A representation of a book.
message Book {
  // Other fields...

  // Possible formats in which the book may be published.
  enum Format {
    // Default value. This value is unused.
    FORMAT_UNSPECIFIED = 0;

    // The printed format, in hardback.
    HARDBACK = 1;

    // The printed format, in paperback.
    PAPERBACK = 2;

    // An electronic book format.
    EBOOK = 3;

    // An audio recording.
    AUDIOBOOK = 4;
  }

  // The format of the book.
  Format format = 99;

  // Other fields...
}
```

- すべてのenum値は`UPPER_SNAKE_CASE`を使用**しなければならない**（must）。
- enumの最初の値は、enum自体の名前の後にサフィックス`_UNSPECIFIED`を付けたもの**であるべきである**（should）。
  - このルールの例外は、明確に有用なゼロ値が存在する場合である。特に、enumが`UNKNOWN`を提示する必要がある場合、通常は両方を持つよりも、それをゼロ値にする方が明確で有用である。`UNKNOWN`値には、enum値名の衝突を避けるための典型的な方法として、enum名をプレフィックスとして付け**てもよい**（may）。
- 単一のメッセージでのみ使用されるenumは、そのメッセージ内にネスト**すべきである**（should）。その場合、enumは使用される直前に宣言**すべきである**（should）。
  - そのようなネストされたenum定義の非ゼロ値は、enum自体の名前をプレフィックスとして付ける**べきではない**（should not）。これにより、通常、ユーザーはコードで`MyState.MYSTATE_ACTIVE`と書く必要がなくなり、不必要に冗長になるのを防ぐ。
- 複数のメッセージで使用されるenumは、パッケージレベルで定義**すべきであり**（should）、protoファイルの下部で定義**すべきである**（should）（[AIP-191][]参照）。
  - 一部の言語（C++を含む）はenum値を親の名前空間に巻き上げるため、同じprotoパッケージ内で同じ値を持つenum間の競合が発生する可能性がある。値の共有を避けるため、APIはパッケージレベルのenum値にenumの名前をプレフィックスとして付ける**べきである**（should）。
- enumは、凍結されているか、将来値を追加する予定かを文書化**すべきである**（should）。

### enumを使用するタイミング

enumは多くの場合、文字列やブール値よりもアクセスしやすく読みやすいが、変更時にはオーバーヘッドが発生する。したがって、enumは新しい値が追加される頻度が低い**べきである**（should）。「頻度が低い」の定義は個々のユースケースによって異なる可能性があるが、目安としては年に1回以下である。頻繁に変更されるenumについては、APIは文字列を使用し、その形式を文書化**すべきである**（should）。

さらに、enumは、競合する広く採用された標準表現が存在する場合（[言語コード][bcp-47]や[メディアタイプ][media types]など）には使用**すべきではない**（should not）。

**注:** 列挙値をAPI間で共有する必要がある場合、enumを使用**してもよい**（may）が、enum値と対応する整数の間の割り当ては一致**しなければならない**（must）。

### 代替案

許可される値のセットが頻繁に変更される列挙値の場合、APIは代わりに`string`フィールドを使用**すべきであり**（should）、許可される値を文書化**しなければならない**（must）。列挙値を持つ文字列フィールドは、その値に`kebab-case`を使用**すべきである**（should）。

競合する広く採用された標準表現（一般的に、必ずしもではないが、文字列）が存在する列挙値の場合、その標準表現を使用**すべきである**（should）。許可される値のごく一部のみが使用される場合でも同様である。なぜなら、この状況でenumを使用すると、複数のAPIを一緒に使用する際にイライラするルックアップテーブルが頻繁に発生するためである。

ブール値フィールドは、さらなる柔軟性が不要であることが明らかな状況で使用**してもよい**（may）。デフォルト値は`false`**でなければならない**（must）。

**注:** Protocol Buffersを使用する場合、`false`と未設定を区別することは不可能である。これが必要な場合、enumの方がより良い設計選択となることがある（`google.protobuf.BoolValue`も利用可能である）。
