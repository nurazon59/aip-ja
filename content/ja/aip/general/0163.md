# 変更検証

ユーザーは、実際に変更を加える前に、意図した変更を検証して結果を確認したい場合がある。例えば、フリートに新しいサーバーをプロビジョニングするリクエストは、全体的なフリートサイズとコストに影響を与え、予期しないダウンストリーム効果をもたらす可能性がある。

## ガイダンス

APIは、リクエストを検証するが実際には実行せず、リクエストが実際に実行された場合に返されるのと同じレスポンス（ステータスコード、ヘッダー、レスポンス本文）を返すオプションを提供 **してもよい**（may）。

このオプションを提供するために、メソッドはリクエストメッセージに `bool validate_only` フィールドを含める **べきである**（should）：

```proto
message ReviewBookRequest {
  string name = 1 [(google.api.resource_reference) = {
    type: "library.googleapis.com/Book"
  }];
  int32 rating = 2;
  string comment = 3;

  // If set, validate the request and preview the review, but do not actually
  // post it.
  bool validate_only = 4;
}
```

APIは、実際のリクエスト（"live" request）で実行されるであろう権限チェックおよびその他の検証を実行 **しなければならない**（must）。`validate_only` を使用するリクエストは、実際のリクエストが失敗すると判断した場合に失敗 **しなければならない**（must）。

**注記:** 完全な出力を提供することが実行不可能な場合もある。例えば、リソースを作成すると自動生成IDが作成される場合、検証時にこれを行うのは意味がない。APIはこの状況ではそのようなフィールドを検証リクエストで省略 **すべきである**（should）。

### 宣言的フレンドリーなリソース

宣言的フレンドリー（AIP-128）なリソースは、リソースを変更するメソッドに `validate_only` フィールドを含め **なければならない**（must）。
