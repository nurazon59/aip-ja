# 標準メソッド: List

多くのAPIでは、リソースのリストを取得するためにコレクションのURI（例：`/v1/publishers/1/books`）に`GET`リクエストを行うのが慣例であり、各リソースはそのコレクション内に存在する。

リソース指向設計（[AIP-121][]）は、`List`メソッドを通じてこのパターンを尊重する。これらのRPCは親コレクション（および必要に応じて他のパラメータ）を受け入れ、その入力に一致するレスポンスのリストを返す。

## ガイダンス

APIは、リソースが[シングルトン][singleton]でない限り、`List`メソッドを提供**しなければならない**（must）。`List`メソッドの目的は、有限のコレクション（一般的には単一だが、操作が[コレクションをまたいだ読み取り][reading across collections]をサポートする場合は単一とは限らない）からデータを返すことである。

Listメソッドは以下のパターンを使用して指定される：

```proto
rpc ListBooks(ListBooksRequest) returns (ListBooksResponse) {
  option (google.api.http) = {
    get: "/v1/{parent=publishers/*}/books"
  };
  option (google.api.method_signature) = "parent";
}
```

- RPC名は`List`という単語で始まら**なければならない**（must）。RPC名の残りの部分は、リストするリソースの複数形**とすべきである**（should）。
- リクエスト メッセージとレスポンス メッセージはRPC名と一致し、`Request`および`Response`サフィックスが付かな**ければならない**（must）。
- HTTP動詞は`GET`**でなければならない**（must）。
- リソースがリストされているコレクションはURIパスにマッピング**すべきである**（should）。
  - コレクションの親リソースは`parent`**とすべきである**（should）、URIパス内の唯一の変数**とすべきである**（should）。残りのパラメータはすべてURIクエリパラメータにマッピング**すべきである**（should）。
  - コレクション識別子（上記の例では`books`）はリテラル文字列**でなければならない**（must）。
- `google.api.http`アノテーションの`body`キーは省略**しなければならない**（must）。
- リストするリソースがトップレベルリソースでない場合、`google.api.method_signature`アノテーションは1つだけ存在**すべきであり**（should）、値は`"parent"`とする。リストするリソースがトップレベルリソースである場合、`google.api.method_signature`アノテーションは存在しないか、1つだけ存在**すべきであり**（should）、値は`""`とする。

### リクエスト メッセージ

Listメソッドは共通のリクエスト メッセージパターンを実装する：

```proto
message ListBooksRequest {
  // このブックのコレクションを所有する親。
  // 形式: publishers/{publisher}
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // 返すブックの最大数。サービスはこの値より少ない数を返すことがある。
  // 指定しない場合、最大50件のブックが返される。
  // 最大値は1000。1000を超える値は1000に切り捨てられる。
  int32 page_size = 2;

  // 以前の `ListBooks` 呼び出しから受け取ったページトークン。
  // これを指定して後続のページを取得する。
  //
  // ページネーション時、`ListBooks` に指定された他のすべてのパラメータは
  // ページトークンを提供した呼び出しと一致しなければならない。
  string page_token = 3;
}
```

- リストするリソースがトップレベルリソースでない限り、`parent`フィールドを含め**なければならない**（must）。`parent`**とすべきである**（should）。
  - フィールドは [required としてアノテートされる][aip-203]**すべきである**（should）。
  - フィールドはリストされるリソースの[リソース型][aip-123]を識別**しなければならない**（must）。
- ページネーションをサポートする`page_size`および`page_token`フィールドは、すべてのリストリクエスト メッセージに指定**しなければならない**（must）。詳細は[AIP-158][]を参照のこと。
  - `page_size`フィールドの上のコメントは、最大許容値、およびフィールドが省略された（または`0`に設定された）場合のデフォルト値を文書化**すべきである**（should）。必要に応じて、APIはサーバーが適切なデフォルトを使用することを記載**してもよい**（may）。このデフォルトは時間とともに変更**してもよい**（may）。
  - ユーザーが最大許容値を超える値を指定した場合、APIは値を最大許容値に切り捨てる**べきである**（should）。
  - ユーザーが負の値やその他の無効な値を指定した場合、APIは`INVALID_ARGUMENT`エラーを送信**しなければならない**（must）。
- `page_token`フィールドはすべてのリストリクエスト メッセージに含め**なければならない**（must）。
- リクエスト メッセージには、リストメソッドに関連する共通のデザインパターン（`string filter`や`string order_by`など）のフィールドを含め**てもよい**（may）。
- リクエスト メッセージに他の必須フィールドを含めては**ならない**（must not）、このAIPまたは別のAIPで説明されているものを除き、他のオプショナル フィールドを含める**べきではない**（should not）。

**注:** Listメソッドは、コレクションに対して正常なListリクエストを行う権限を持つすべてのユーザーに対して同じ結果を返す**すべきである**（should）。Searchメソッドはこれに関してより緩やかである。

### レスポンス メッセージ

Listメソッドは共通のレスポンス メッセージパターンを実装する：

```proto
message ListBooksResponse {
  // 指定された出版社からのブック。
  repeated Book books = 1;

  // `page_token` として送信して次のページを取得できるトークン。
  // このフィールドが省略されている場合、後続のページはない。
  string next_page_token = 2;
}
```

- レスポンス メッセージには、返されるリソースに対応する1つのrepeatedフィールドを含め**なければならず**（must）、別のAIP（例：AIP-217）で説明されている場合を除き、他のrepeatedフィールドを含める**べきではない**（should not）。
  - レスポンスは通常、完全に設定されたリソースを含む**べきである**（should）。ただし、部分的なレスポンスを返す理由がある場合を除く（AIP-157参照）。
- ページネーションをサポートする`next_page_token`フィールドは、すべてのリストレスポンス メッセージに含め**なければならない**（must）。後続のページがある場合は設定**しなければならない**（must）、レスポンスが最終ページを表す場合は設定しては**ならない**（must not）。詳細はAIP-158を参照のこと。
- メッセージには、コレクション内のアイテム数を持つ`int32 total_size`（または`int64 total_size`）フィールドを含め**てもよい**（may）。
  - 値は推定値**でもよい**（may）（その場合、フィールドはその旨を明確に文書化**すべきである**（should））。
  - フィルタリングが使用されている場合、`total_size`フィールドはフィルタ適用**後**のコレクションのサイズを反映**すべきである**（should）。

### 順序付け

`List`メソッドは、クライアントがソート順を指定できるように**してもよい**（may）。その場合、リクエスト メッセージに`string order_by`フィールドを含める**べきである**（should）。

- 値はカンマ区切りのフィールドリスト**とすべきである**（should）。例：`"foo,bar"`。
- デフォルトのソート順は昇順。フィールドに降順を指定する場合、ユーザーは`" desc"`サフィックスを追加する。例：`"foo desc, bar"`。
- 構文内の冗長なスペース文字は意味を持たない。`"foo, bar desc"`、`" foo , bar desc "`、`"foo,bar desc"`はすべて同等である。
- サブフィールドは`.`文字で指定する。例：`foo.bar`、`address.street`。
- 結果のリスト順は、フィールド型の自然なコンパレータに基づく**すべきである**（should）。例：数値は数値順、文字列は辞書順など。ただし、APIは異なる順序を選択**してもよい**（may）。その場合は`order_by`の定義に文書化**しなければならない**（must）。
  - さらに、`Timestamp`や`Duration`のような[well-known][]型は、その代表的な型として比較される。`Timestamp`は時間として（例：前か後か）、`Duration`は量として（例：多いか少ないか）比較される。

### フィルタリング

Listメソッドは、クライアントがフィルタを指定できるように**してもよい**（may）。その場合、リクエスト メッセージに`string filter`フィールドを含める**べきである**（should）。フィルタリングの詳細はAIP-160で説明される。

**注:** 確立された必要性がある場合のみフィルタリングを含めること。後でフィルタリングを追加することは常に可能だが、削除は破壊的変更となる。

### ソフト削除されたリソース

一部のAPIはリソースを"[ソフト削除][soft delete]"し、削除済みまたは削除待ちとしてマークし（オプションで後でパージする）必要がある。

これを行うAPIは、デフォルトで削除されたリソースをリストリクエストに含める**べきではない**（should not）。リソースのソフト削除があるAPIは、リストリクエストに`bool show_deleted`フィールドを含める**べきであり**（should）、これが設定されるとソフト削除されたリソースが含まれるようになる。
