# APIバージョニング

このトピックでは、Google APIで使用されるバージョニング戦略について説明する。一般的に、これらの戦略はすべてのGoogle管理サービスに適用される。

## ガイダンス

すべてのGoogle APIインターフェースは_メジャーバージョン番号_を提供**しなければならない**（must）。これはprotobufパッケージの末尾にエンコードされ、REST APIのURIパスの最初の部分として含まれる。APIに互換性のない変更が必要な場合、該当サーフェスの安定性レベルに基づいて必要な手順について[AIP-180][]および[AIP-181][]を参照すること。

注：上記の「メジャーバージョン番号」という用語は[セマンティック バージョニング][semantic versioning]から引用したものである。ただし、従来のセマンティック バージョニングとは異なり、Google APIはマイナーバージョンやパッチバージョン番号を公開**してはならない**（must not）。例えば、Google APIは`v1`を使用し、`v1.0`、`v1.1`、`v1.4.2`は使用しない。ユーザーの観点からは、メジャーバージョンはマイナー/パッチ相当の変更でその場で更新され、ユーザーは移行なしで新機能を受け取る。

APIの新しいメジャーバージョンは、同じAPIの以前のメジャーバージョンに依存**してはならない**（must not）。APIサーフェスは、[AIP-213][]および[AIP-215][]で概説されている場合を除き、他のAPIに依存しては**してはならない**（must not）。

同じAPIの異なるバージョンは、合理的な移行期間中、単一のクライアントアプリケーション内で同時に動作でき**なければならない**（must）。この期間により、クライアントは新しいバージョンにスムーズに移行できる。古いバージョンは、シャットダウンされる前に合理的で十分に伝達された非推奨期間を経**なければならない**（must）。

[alphaまたはbetaの安定性][AIP-181]を持つリリースの場合、APIは以下の戦略のいずれかを使用して、protobufパッケージとURIパスにメジャーバージョン番号の後に安定性レベルを追加**しなければならない**（must）：

- チャネルベースバージョニング（推奨）
- リリースベースバージョニング
- 可視性ベースバージョニング

### チャネルベースバージョニング

_安定性チャネル_は、所定の安定性レベルにおける長期存続リリースであり、その場で更新を受ける。メジャーバージョンにおいて、安定性レベルごとに最大1つのチャネルが存在する。この戦略では、alpha、beta、stableの最大3つのチャネルが利用可能である。

alphaチャネルとbetaチャネルはバージョンに安定性レベルを追加**しなければならない**（must）が、stableチャネルは安定性レベルを追加**してはならない**（must not）。例えば、`v1`はstableチャネルでは許容されるバージョンであるが、`v1beta`や`v1alpha`は許容されない。同様に、`v1beta`や`v1alpha`はそれぞれbetaおよびalphaチャネルでは許容されるバージョンであるが、`v1`はどちらにとっても許容されない。これらの各チャネルは、新機能と更新を「その場で」受け取る。

betaチャネルの機能はstableチャネルの機能のスーパーセットで**なければならず**（must）、alphaチャネルの機能はbetaチャネルの機能のスーパーセットで**なければならない**（must）。

#### API機能の非推奨化

API要素（フィールド、メッセージ、RPC）は、どのチャネルでも非推奨としてマーク**してもよい**（may）。これにより、それらを今後使用すべきでないことを示す：

```proto
// 巻物を表す。巻物より書籍が推奨される。
message Scroll {
  option deprecated = true;

  // ...
}
```

非推奨のAPI機能は、alphaからbetaへ、またbetaからstableへ昇格**してはならない**（must not）。言い換えれば、どのチャネルでも「事前非推奨」の状態で機能が到着**してはならない**（must not）。

betaチャネルの機能は、十分な期間非推奨にされた後に削除**してもよい**（may）。推奨期間は180日である。alphaチャネルにのみ存在する機能については、非推奨はオプションであり、機能は予告なく削除**してもよい**（may）。機能が削除前にAPIのalphaチャネルで非推奨にされる場合、APIは同じアノテーションを適用する**すべきであり**（should）、任意の期間を使用**してもよい**（may）。

### リリースベースバージョニング

**重要：** このパターンは新しいサービスでは一般的に使用されない。これに従う既存のサービスもあるが、チャネルベースバージョニングが推奨メカニズムである。

_個別リリース_は、限られた期間利用可能であることが期待されるalphaまたはbetaリリースであり、その機能がstableチャネルに組み込まれた後、個別リリースはシャットダウンされる。リリースベースバージョニング戦略を使用する場合、APIは各安定性レベルで任意の数の個別リリースを持つことができる。

注：チャネルベースとリリースベースの両方の戦略で、_stable_バージョンはその場で更新される。個別のstableリリースではなく、単一のstableチャネルが存在する（リリースベース戦略を使用する場合でも）。

Alphaおよびbetaリリースは、バージョンに安定性レベルを追加**しなければならない**（must）、その後にインクリメントするリリース番号を続ける。例えば、`v1beta1`や`v1alpha5`など。APIはこれらのバージョンの時系列順序をドキュメント（コメントなど）に文書化する**すべきである**（should）。

各alphaまたはbetaリリースは、後方互換性のある変更でその場で更新**してもよい**（may）。betaリリースの場合、後方互換性のない更新はリリース番号をインクリメントし、変更を含む新しいリリースを公開することによって行う**すべきである**（should）。例えば、現在のバージョンが`v1beta1`の場合、次に`v1beta2`がリリースされる。

Alphaおよびbetaリリースは、その機能がstableチャネルに到達した後にシャットダウンする**すべきである**（should）。alphaリリースはいつでもシャットダウン**してもよい**（may）が、betaリリースはユーザーに合理的な移行期間を許可する**すべきである**（should）。推奨期間は180日である。

### 可視性ベースバージョニング

[API可視性][api visibility]は、Google APIインフラストラクチャが提供する高度な機能である。これにより、APIプロデューサーは1つの内部APIサーフェスから複数の外部APIビューを公開でき、各ビューはAPI_可視性ラベル_に関連付けられる。例えば：

```proto
import "google/api/visibility.proto";

message Resource {
  string name = 1;

  // プレビュー。本番環境ではこの機能を使用しないでください。
  string display_name = 2
    [(google.api.field_visibility).restriction = "PREVIEW"];
}
```

可視性ラベルは大文字小文字を区別する文字列で、任意のAPI要素にタグ付けするために使用できる。慣例により、可視性ラベルは常に大文字（UPPER CASE）を使用する**すべきである**（should）。上記の例のように明示的な可視性ラベルが適用されない限り、暗黙の`PUBLIC`ラベルがすべてのAPI要素に適用される。

各可視性ラベルは許可リストである。APIプロデューサーは、APIコンシューマーがラベルに関連付けられたAPI機能を使用するために、可視性ラベルを付与する必要がある。言い換えれば、API可視性ラベルはACL付きAPIバージョンのようなものである。

複数の可視性ラベルは、カンマ区切りの文字列（例：`"PREVIEW,TRUSTED_TESTER"`）を使用して要素に適用**してもよい**（may）。複数の可視性ラベルが使用される場合、クライアントはそれらのうちの_1つ_のみを必要とする（論理`OR`）。

デフォルトでは、APIコンシューマーに付与された可視性ラベルが受信リクエストの検証に使用される。ただし、クライアントは以下のように明示的な可視性ラベルでリクエストを送信できる：

```
GET /v1/projects/my-project/topics HTTP/1.1
Host: pubsub.googleapis.com
Authorization: Bearer y29....
X-Goog-Visibilities: PREVIEW
```

単一のAPIリクエストは最大1つの可視性ラベルを指定できる。

APIプロデューサーは、APIバージョニングにAPI可視性を使用できる（`INTERNAL`や`PREVIEW`など）。新しいAPI機能は`INTERNAL`ラベルで始まり、その後`PREVIEW`ラベルに移行する。機能が安定して一般提供可能になると、すべてのAPI可視性ラベルがAPI定義から削除される。

一般的に、API可視性は段階的な変更に対してAPIバージョニングよりも実装が容易であるが、洗練されたAPIインフラストラクチャのサポートに依存する。Google Cloud APIは、プレビュー機能にAPI可視性をよく使用する。
