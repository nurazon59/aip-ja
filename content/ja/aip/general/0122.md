# リソース名

ほとんどのAPIは、ユーザーが作成、取得、操作できる_リソース_（主要な名詞）を公開する。さらに、リソースには_名前_が付けられている：各リソースは、ユーザーがそのリソースを参照するために使用する一意の識別子を持ち、これらの名前がユーザーがリソースの正規名として_保存_すべきものである。

## ガイダンス

APIによって定義されるすべてのリソース名は、そのAPI内で一意**でなければならない**（must）。（API間でのリソースの参照方法の詳細については、以下の[完全リソース名](#full-resource-names)のセクションを参照のこと。）

リソース名は[URIパススキーマ][]に従ってフォーマットされるが、先頭のスラッシュは含まない：

    publishers/123/books/les-miserables
    users/vhugo1802

- リソース名の構成要素は、通常、コレクション識別子（例：`publishers`、`books`、`users`）とリソースID（例：`123`、`les-miserables`、`vhugo1802`）を交互に配置**すべきである**（should）。
- リソース名は、`/`文字を使用して個々のセグメントを区切ら**なければならない**（must）。
  - リソース名の非終端セグメントは、`/`文字を含ん**ではならない**（must not）。
  - リソース名の終端セグメントは、`/`文字を含む**べきではない**（should not）。
- リソース名は、[RFC-1123](https://tools.ietf.org/html/rfc1123)で定義されているDNS名で使用可能な文字のみを使用**すべきである**（should）。
  - さらに、リソースIDは大文字を使用**すべきではない**（should not）。
  - 追加の文字が必要な場合、リソース名はURLエスケープが必要な文字やASCII範囲外の文字を使用**すべきではない**（should not）。
  - Unicode文字を避けられない場合、リソース名はNormalization Form Cで保存**しなければならない**（must）（[AIP-210][]参照）。
- リソースは、そのリソース名を含む`name`フィールドを公開**しなければならない**（must）。
  - リソースは、リソースIDを別のフィールド（例：`book_id`）として提供**してもよい**（may）。このフィールドは[`OUTPUT_ONLY`](./0203.md#output-only)フィールドビヘイビア分類を適用**しなければならない**（must）。
  - リソースは、別のシステム生成の一意IDフィールド[（`uid`）](./0148.md#uid)を公開**してもよい**（may）。このフィールドは[`OUTPUT_ONLY`](./0203.md#output-only)フィールドビヘイビア分類を適用**しなければならない**（must）。
  - リソースは、タプル、セルフリンク、またはその他の形式のリソース識別を公開**してはならない**（must not）。
  - すべてのIDフィールドは文字列**であるべきである**（should）。

**注:** ここで説明するリソース名は、単一のAPIのスコープ内（または所有APIがコンテキストから明らかな状況）で使用され、そのスコープ内でのみ一意であることが要求される。このため、これらは_相対リソース名_と呼ばれることもあり、_完全リソース名_（後述）と区別される。

[aip-210]: ./0210.md
[uri path schema]: https://datatracker.ietf.org/doc/html/rfc3986#appendix-A

### コレクション識別子

リソース名のコレクション識別子セグメントは、リソースに使用される名詞の複数形**でなければならない**（must）。（例えば、`Publisher`リソースのコレクションは、リソース名では`publishers`と呼ばれる。）

- コレクション識別子は、簡潔なアメリカ英語の用語**でなければならない**（must）。
- コレクション識別子は`camelCase`**でなければならない**（must）。
- コレクション識別子は小文字で始まり、ASCIIの文字と数字のみを含ま**なければならない**（must）（`/[a-z][a-zA-Z0-9]*/`）。
- コレクション識別子は複数形**でなければならない**（must）。
  - 複数形の単語がない場合（"info"）、または単数形と複数形が同じ場合（"moose"）、非複数形（単数形）が正しい。そのような場合、コレクションセグメントは"s"を追加して単語を作っ**てはならない**（must not）（例："infos"は避ける）。
- 任意の単一のリソース名内で、コレクション識別子は一意**でなければならない**（must）（例：`people/xyz/people/abc`は無効）。


#### ネストされたコレクション

リソース名に複数レベルの階層が含まれ、親コレクションの名前が子リソースの名前のプレフィックスとして使用される場合、子コレクションの名前はプレフィックスを省略**してもよい**（may）。例えば、通常`users`の下にネストされる`UserEvent`リソースのコレクションがある場合：

```
users/vhugo1802/userEvents/birthday-dinner-226
```

APIは、より冗長でない形式を使用**すべきである**（should）：

```
users/vhugo1802/events/birthday-dinner-226
```

この状況では、_メッセージ_と_リソースタイプ_は依然として`UserEvent`と呼ばれる。パターン内のコレクションおよびリソース識別子のみが短縮される。_リソースタイプ_は短縮されないため、`singular`と`plural`も同様に_短縮されない_。

```
message UserEvent {
  option (google.api.resource) = {
    type: "example.googleapis.com/UserEvent"
    // Only the collection & resource identfiers in the `pattern` are shortened.
    pattern: "projects/{project}/users/{user}/events/{event}"
    singular: "userEvent"
    plural: "userEvents"
  };

  string name = 1;
}
```

**注:** これを行うAPIは、定義されたすべての`pattern`エントリおよびAPI内でリソースが参照される他のすべての場所でこの形式を一貫して守ら**なければならず**（must）、そうでなければまったく使用してはならない。

### リソースIDセグメント

リソースIDセグメントは、その親コレクション内のリソースを識別する。リソース名`publishers/123/books/les-miserables`では、`123`は出版社のリソースIDであり、`les-miserables`は書籍のリソースIDである。

- リソースIDがユーザー指定の場合、APIは許可される形式を文書化**しなければならない**（must）。ユーザー指定のリソースIDは[RFC-1034][]に準拠**すべきである**（should）。RFC-1034は、文字、数字、ハイフンに制限し、最初の文字は文字、最後は文字または数字、最大63文字とする。
  - さらに、ユーザー指定のリソースIDは小文字に制限**すべきである**（should）（`^[a-z]([a-z0-9-]{0,61}[a-z0-9])?$`）。
  - ASCII範囲外の文字は許可**すべきではない**（should not）。ただし、Unicode文字が必要な場合、APIは[AIP-210][]のガイダンスに従わ**なければならない**（must）。
- リソースIDがユーザー設定可能でない場合、APIは基本形式と上限（例：「最大63文字」）を文書化**すべきである**（should）。
- 詳細については、[create][]標準メソッドを参照のこと。

[create]: ./0133.md#user-specified-ids
[rfc-1034]: https://tools.ietf.org/html/rfc1034

### リソースIDエイリアス

リソースIDの一般的な検索パターンにエイリアスを提供することが有用な場合がある。例えば、リソース階層の最上位に`users`を持つAPIは、認証されたユーザーの情報を取得するためのショートカットとして`users/me`を提供したい場合がある。

APIは、一般的な検索パターンにプログラムによるエイリアスを提供**してもよい**（may）。ただし、APIから返されるすべてのデータは、正規のリソース名を使用**しなければならない**（must）。

### 完全リソース名

ほとんどの場合、リソース名は単一のAPI内でのみ使用されるか、所有APIが明らかなコンテキストで使用される（例：`string pubsub_topic`）。

ただし、サービスが任意のAPIのリソースを参照する必要がある場合がある。この状況では、サービスは_完全リソース名_を使用**すべきである**（should）。完全リソース名は、所有する[APIのサービス名](./0009.md#api-service-name)に相対リソース名を続けた、スキームなしのURIである：

```
//library.googleapis.com/publishers/123/books/les-miserables
//calendar.googleapis.com/users/vhugo1802
```

**注:** 完全リソース名は、所有APIが明らかな場合のクロスAPI参照には使用**すべきではない**（should not）。曖昧さが生じる可能性がある場合に、複数のAPIのリソースをフィールドが参照する場合にのみ使用される。

### リソースURI

完全リソース名はスキームなしのURIであるが、リソースにアクセスするために使用する完全URIとは若干異なる。後者には、プロトコル（HTTPS）、APIバージョン、および対象の特定の[サービスエンドポイント](./0009.md#api-service-endpoint）が含まれる：

```
https://library.googleapis.com/v1/publishers/123/books/les-miserables
https://calendar.googleapis.com/v3/users/vhugo1802
```

バージョンは完全リソース名に含まれない。完全リソース名はバージョン間で永続することが期待されるためである。APIサーフェスはメジャーバージョン間で変更される可能性があるが、同じAPIの複数のメジャーバージョンは同じ基盤データを使用することが期待される。

**注:** 完全リソース名とサービスのエンドポイントの間の相関は慣例によるものである。特に、1つのサービスが複数のエンドポイントを持つことができ（ユースケース例：リージョナリゼーション、MTLS、プライベートアクセス）、完全リソース名はこれらの間で変更されない。

### リソース名を表すフィールド

リソースを定義する際、最初のフィールドはリソース名**とすべきであり**（should）、リソース名の場合は型`string`で名前`name`**でなければならない**（must）。メッセージには、型を宣言する`google.api.resource`アノテーションを含める**べきである**（should）（詳細は[AIP-123][]を参照）。

```proto
// A representation of a book in the library.
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
  };

  // The resource name of the book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // Other fields...
}
```

既存のリソースを取得または操作するメソッド（`GetBook`や`ArchiveBook`など）を定義する場合、リクエストメッセージの最初のフィールドはリソース名**とすべきであり**（should）、リソース名の場合は型`string`で名前`name`**でなければならない**（must）。フィールドは、リソースタイプを参照する`google.api.resource_reference`アノテーションも付与**すべきである**（should）（[AIP-123][]）。

```proto
// Request message for ArchiveBook
message ArchiveBookRequest {
  // The book to archive.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/Book"
    }];

  // Other fields...
}
```

**注:** フィールドはこの目的以外で`name`と呼ん**ではならない**（must not）。他のユースケースでは、別の用語を使用するか、形容詞を前置する（例：`display_name`）。

[aip-123]: ./0123.md

### リソースの親を表すフィールド

コレクションからリソースを取得したり、コレクションに新しいリソースを追加したりするメソッド（`ListBooks`や`CreateBook`など）を定義する場合、リクエストメッセージの最初のフィールドは型`string`で、コレクションのリソース名に対して`parent`と呼ぶ**べきである**（should）。`parent`フィールドは、親のリソースタイプを参照する`google.api.resource_reference`アノテーションも付与**すべきである**（should）（[AIP-123][]）。

```proto
// Request message for ListBooks.
message ListBooksRequest {
  // The publisher to list books from.
  // Format: publishers/{publisher_id}
  string parent = 1 [(google.api.resource_reference) = {
    type: "library.googleapis.com/Publisher"
  }];

  // Other fields (e.g. page_size, page_token, filter, etc.)...
}
```

親タイプが複数存在する可能性がある場合、`parent`フィールドは代わりに`google.api.resource_reference`の`child_type`キーでアノテーション**すべきである**（should）：

```proto
// Request message for ListBooks.
message ListBooksRequest {
  // The parent to list books from.
  // Format:
  //   - publishers/{publisher_id}
  //   - authors/{author_id}
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // Other fields (e.g. page_size, page_token, filter, etc.)...
}
```

**注:** フィールドはこの目的以外で`parent`と呼ぶ**べきではない**（should not）。他のユースケースでは、可能であれば同義語を使用する。

### 別のリソースを表すフィールド

フィールドが別のリソースを表す場合、そのフィールドは型`string`で、対象リソースのリソース名を受け入れる**べきである**（should）。フィールド名は、対応するメッセージの名前をsnake_caseにしたものと同等**であるべきである**（should）。

- フィールド名には、適切な場合に先頭の形容詞を含め**てもよい**（may）（例：`string dusty_book`）。
- フィールド名は、曖昧さを避けるために必要な場合を除き（例：`crypto_key_name`）、`_name`サフィックスを使用**すべきではない**（should not）。
- 別のリソースを表すフィールドは、参照されるリソースタイプを指定した`google.api.resource_reference`アノテーションを提供**すべきである**（should）。
- リソース名の使用が不可能で、IDコンポーネントのみの使用が厳密に必要な場合、フィールドは`_id`サフィックスを使用**すべきである**（should）（例：`shelf_id`）。

フィールドは、リソースを実装する`message`を使用した型`message`**すべきではない**（should not）。ただし、以下の条件のいずれかに該当する場合は除く：

- APIが内部専用で、ライフサイクル関係が緊密であり、埋め込まれたリソースへの継承アクセスを可能にするパーミッションモデルを持つ場合。
- リソースの埋め込みがAIP-162 revisionsパターンの一部として行われる場合。

リソース参照の例：

```proto
// A representation of a book in a library.
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
  };

  // Name of the book.
  // Format is `publishers/{publisher}/books/{book}`
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // The shelf where the book currently sits.
  // Format is `shelves/{shelf}`.
  string shelf = 2 [(google.api.resource_reference) = {
    type: "library.googleapis.com/Shelf"
  }];

  // Other fields...
}
```

## 根拠

### IDの代わりに名前を使用する

大規模なシステムでは、多くの種類のリソースが存在する。単純なリソースIDを使用してリソースを識別するには、実際にはリソース固有のタプルを使用して確実に識別する必要がある（例：`(bucket, object)`や`(user, album, photo)`）。これにより、いくつかの問題が発生する：

* 開発者はそのような匿名タプルを理解し記憶する必要がある。
* タプルの受け渡しは、通常文字列の受け渡しよりも難しい。
* ロギングやアクセス制御システムなどの集中インフラストラクチャは、特殊なタプルを理解しない。
* 特殊なタプルは、再利用可能なAPIインターフェースの提供など、API設計の柔軟性を制限する。例えば、[Long Running Operations][aip-151]は、柔軟なリソース名を使用するため、他の多くのAPIインターフェースと連携できる。

[aip-151]: ./0151.md

### `name`への標準化

リソース名の概念は新しいものではなく、Uniform Resource Name（URN）としてUniform Resource Identifier（URI）およびUniform Resource Locator（URL）と共に形式化されている。「name」という用語が一般に非常に過負荷であることを考慮すると、非常によく定義された意味の外での使用は開発者にとって混乱を招く。したがって、フィールド名`name`はAIP準拠のAPIのコンテキストで予約され、リソース名との混乱を排除し、他の「name」フィールド候補にはより具体的なフィールド名を強制する。

### リソースの埋め込みの禁止

別のリソース内のフィールドの型としてリソースメッセージを直接使用することは、いくつかの理由で問題がある：

* リソースのライフサイクルが複雑化する：依存リソースが削除された場合、依存側リソースの埋め込み参照はどうなるのか？データ保持とクリーンアップ操作が大幅に複雑化する。
* パーミッションをバイパスする：すべてのリソースが独自のパーミッションセットを持つ場合、依存リソースへの読み取り権限を持つユーザーが依存対象リソースに同じ権限を持たない場合、突然完全なリソースを見られなくなる。
* すべての側面でリソースを強く結合する：どちらかのリソースのスキーマ、パーミッション、またはその他の要件を変更すると、もう一方のリソースに影響を与え、ロールアウトの複雑さが大幅に増大する。

推奨されるように名前で参照することで、リソースデータの重複を防止し、参照の解決（標準メソッド経由）に所有サービスを関与させることを強制することで、リソースごとの論理的関心の分離を保証し、これらすべての複雑さを排除する。

## 履歴

### ユーザー指定IDでのUUIDの禁止

APIをより宣言的クライアントフレンドリーにする取り組みの一環として、APIは作成時にユーザー指定のリソースIDを許可することが要求された。これに伴い、UUIDを含む補助フィールド`string uid`も推奨された。その後、_ユーザー指定のリソースID_がUUIDまたはそれに類似したものであることを避けるためのガイダンスが追加された。

表向きは、`uid`がサービス生成のUUIDを含むという事実が、ユーザー指定のリソースIDに対するこの制限を動機付け、ユーザー指定のリソースIDをより「意味のある」値に強制していた。しかし、この要件にはもはや価値が見いだせなくなったため、削除された。
