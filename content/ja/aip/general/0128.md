# 宣言的クライアントフレンドリーなインターフェース

多くのサービスは、一般的なDevOpsツール、特にネットワークアドレス可能なリソース（仮想マシン、ロードバランサー、データベースインスタンスなど）を作成および管理するツールと対話する必要がある。これらのツールは「構成としてのコード」の原則に基づいている：ユーザーは完全な意図する構成を指定し、ツールはユーザーの仕様を達成するために必要な変更を行う責任を負う。

これらのツールは**宣言的**である：具体的な_アクション_を指定するのではなく、望ましい_結果_を指定し、アクションは現在の状況と意図する状況の差分に基づいて導出される。

さらに、多数の人気のあるDevOpsツールが存在し、毎年新しいものが導入されている。数百のリソースタイプを複数のツールと統合するには、統合を自動化できるように統一性が必要である。

## ガイダンス

### リソース

宣言的クライアントフレンドリーなリソースは、リソースライフサイクル管理に強い整合性を持つ標準メソッドのみを使用**しなければならない**（must）。これにより、ツールはこれらのリソースを汎用的にサポートできるとともに、他の宣言的クライアントフレンドリーなガイダンス（[参考資料](#further-reading)参照）に準拠できる。

宣言的クライアントフレンドリーなリソースは、宣言的クライアントフレンドリースタイルに従うことを指定**すべきである**（should）：

```proto
message Book {
  option (google.api.resource) = {
    type: "library.googleapis.com/Book"
    pattern: "publishers/{publisher}/books/{book}"
    style: DECLARATIVE_FRIENDLY
  };

  // Name and other fields...
}
```

### 調整

更新が反映されるまでに時間がかかる（数秒以上）場合、リソースは変更が進行中であることを開示するために`bool reconciling`フィールドを含める**べきである**（should）。このフィールドはoutput only**でなければならない**（must）。

リソースの現在の状態がユーザーの意図する状態と一致せず、システムがそれらを調整するために動作している場合、リソースは`reconciling`フィールドを`true`に設定**しなければならない**（must）。これは、調整に入った根本原因がユーザーのアクションかシステムのアクションかに関わらない。

**注:** `GET`リクエストに応答するサービスは、リソースの現在の状態（意図する状態ではない）を返却**しなければならない**（must）。

## 参考資料

宣言的クライアントフレンドリーなインターフェースでは、これらのリソース上の自動化に焦点を当てているため、多くのガイダンスがより厳格になる。このリストは、他のAIPにおける宣言的クライアントフレンドリーなガイダンスの包括的なリファレンスである：

- リソースはカスタム メソッドを採用**すべきではない**（should not）：AIP-136を参照。
- リソースは繰り返しフィールドに`Update`メソッドを使用**しなければならない**（must）：AIP-144を参照。
- リソースは特定の標準フィールドを含ま**なければならない**（must）：AIP-148を参照。
- リソースは`etag`フィールドを持た**なければならない**（must）：AIP-154を参照。
- リソースは変更検証を提供**すべきである**（should）：AIP-163を参照。
- リソースはソフトデリートを実装**すべきではない**（should not）。IDが再利用できない場合、リソースはソフトデリートとundelete RPCを実装**しなければならない**（must）：AIP-164を参照。
