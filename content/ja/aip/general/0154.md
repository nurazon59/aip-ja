# リソースのフレッシュネス検証

APIはしばしば、リソースに対して何らかのアクションを実行する前に、クライアントとサーバーがそのリソースの現在の状態について一致していることを検証する必要がある。例えば、2つのプロセスが同じリソースを並行して更新すると、後続のプロセスが前者の更新を「上書き」してしまう競合状態が発生する可能性がある。

ETagは、サーバーがリソースの現在のコンテンツに基づいてチェックサムを送信し、クライアントがそのチェックサムを返送することで、サーバーがリクエストを処理する前にチェックサムが一致することを確認できるようにする機能を提供する。

## ガイダンス

リソースは、特定のリクエストを処理する前にクライアントが最新のリソースを保持していることを確認することが重要な場合に、任意のリソースに `etag` フィールドを含め**てもよい**（may）：

```proto
// 書籍の表現。
message Book {
  // その他のフィールド...

  // このチェックサムは他のフィールドの値に基づいてサーバーが計算し、
  // 更新や削除リクエストで送信され、クライアントが処理を進める前に
  // 最新の値を持っていることを確認するために使用される。
  string etag = 99;
}
```

- etagフィールドは文字列**でなければならない**（must）、名前は `etag` で**なければならない**（must）。
- _リソース上の_ etagフィールドには、動作アノテーションを付与 **すべきではない**（should not）。
- etagフィールドは出力時にサーバーから提供され**なければならず**（must）、その値は [RFC 7232][] に準拠**すべきである**（should）。
- ユーザーが現在のetag値と一致するetagを送信した場合、サービスはリクエストを許可 **しなければならない**（must）（他にエラーとなる理由がない限り）。
- ユーザーが現在のetag値と一致しないetagを送信した場合、サービスは `ABORTED` エラーレスポンスを送信 **しなければならない**（must）（`PERMISSION_DENIED` など、他のエラーが優先される場合を除く）。
- ユーザーがetag値をまったく送信しない場合、サービスはリクエストを許可 **すべきである**（should）。ただし、強い整合性や並行性の要件があるサービスは、ユーザーに常にetagの送信を要求 **してもよい**（may）し、その場合 `INVALID_ARGUMENT` エラーでリクエストを拒否する。

**注記:** ETagの値には、[RFC 7232][] に記載されているように引用符を含める **すべきである**（should）。例えば、有効なetagは `"foo"` であり、`foo` ではない。

### 宣言的フレンドリーなリソース

宣言的フレンドリー（AIP-128）なリソースは、`etag` フィールドを含め**なければならない**（must）。

### リクエストメソッドにおけるEtag

状況によっては、etagはリソース自体ではなくリクエストメッセージに属する必要がある。例えば、`Update` 標準メソッドはリソース上の `etag` フィールドを「利用」できるが、`Delete` 標準メソッドはできない：

```proto
message DeleteBookRequest {
  // 書籍の名前。
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/Book"
    }];

  // 書籍の現在のetag。
  // etagが提供され、書籍の現在のetagと一致しない場合、
  // 削除はブロックされ、ABORTEDエラーが返される。
  string etag = 2 [(google.api.field_behavior) = OPTIONAL];
}
```

リクエストメッセージ上の `etag` フィールドには、`REQUIRED` または `OPTIONAL` のいずれかの動作アノテーションを付与 **すべきである**（should）。詳細はAIP-203を参照。

`etag` フィールドは、上記の例と同様に、カスタム メソッドでも使用 **してもよい**（may）。

### 強力なEtagと弱いEtag

ETagには「強検証」と「弱検証」の2種類がある：

- 強検証されたetagは、同じetagを持つ2つのリソースがバイト単位で同一であることを意味する。
- 弱検証されたetagは、同じetagを持つ2つのリソースが等価であるが、サービスが重要とみなさない点で異なる可能性があることを意味する。

リソースは、必要に応じて強または弱のetagを使用**してもよい**（may）が、その動作を文書化**すべきである**（should）。さらに、弱いetagは [RFC 7232][] で必須とされている `W/` プレフィックスを持た**なければならない**（must）。

[rfc 7232]: https://tools.ietf.org/html/rfc7232#section-2.3
