# ドキュメンテーション

ドキュメンテーションはAPI設計の最も重要な側面の1つである。APIのユーザーは実装を掘り下げてAPIをより理解することができない。多くの場合、APIサーフェス定義とそれに対応するドキュメントだけがユーザーが持つものになる。したがって、ドキュメントが可能な限り明確、完全、かつ曖昧でないことが重要である。

## ガイダンス

protocol buffersで定義されるAPIでは、すべてのコンポーネント（service、method、message、field、enum、enum値）にprotocol buffersのコメント形式を使用してパブリックコメントを含め**なければならない**（must）。これはコメントが簡潔で面白みがない場合でも重要である。なぜなら、多くのツールがこれらのコメントを読み、使用するからである。

特にサービスは、サービスが何であり、ユーザーがそれで何ができるかを説明する説明的なコメントを持つ**べきである**（should）。

**注：** 多くの読者は英語を母国語としない。コメントは専門用語、スラング、複雑な比喩、ポップカルチャーの参照、または簡単に翻訳できないその他のものを避ける**べきである**（should）。さらに、多くの読者は異なる背景や視点を持つ。人に関する例を書く場合、コメントは論争の的にならず、すでに故人である人物を使用**すべきである**（should）。

### スタイル

コメントは文法的に正しいアメリカ英語である**すべきである**（should）。ただし、各コメントの最初の文は主語を省略し、三人称現在形に**すべきである**（should）：

```proto
// 指定されたパブリッシャーの下にブックを作成する。
rpc CreateBook(CreateBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
  };
}
```

### 説明

メッセージやフィールドの説明は簡潔でありながら完全**であるべきである**（should）。コメントが形骸化せざるを得ない場合もある。言うべきことがほとんどないからである。ただし、その結論に飛びつく前に、以下の質問のいくつかが関連するかどうかを検討すること：

- それは何か？
- どのように使用するか？
- 成功するとどうなるか？失敗するとどうなるか？
- それは冪等か？
- 単位は何か？（例：メートル、度、ピクセル）
- 副作用は何か？
- それを壊す可能性のある一般的なエラーは何か？
  - 期待される入力形式は何か？
  - 受け入れる値の範囲は何か？（例：`[0.0, 1.0)`、`[1, 10]`）
    - 範囲は包括的か、排他的か？
  - 文字列の場合、最小長と最大長は何か、どの文字が許可されるか？
    - 値が最大長を超える場合、切り詰めるか、エラーを送信するか？
- それは常に存在するか？（例：「投票情報のコンテナ。投票情報が記録されている場合にのみ存在する。」）
- デフォルト設定はあるか？（例：「`page_size`が省略された場合、デフォルトは50です。」）

### 書式設定

コメント内の書式設定は[CommonMark][]で**しなければならない**（must）。見出しやテーブルは使用**してはならない**（must not）。これらはいくつかのツールで問題を引き起こし、クライアントライブラリのリファレンスドキュメントには不適切であるためである。

コメントは、フィールド名やメソッド名、リテラル（`true`など）に`コードフォント`を使用**すべきである**（should）。

生のHTMLは使用**してはならない**（must not）。

proto内で図を表現しようとする「ASCIIアート」は使用**してはならない**（must not）。proto内のMarkdownは多数のレンダラーで消費され、ASCIIアートがそれらすべてで適切に表示される可能性は非常に低い。APIを理解するために図が有用な場合は、図を画像として含むドキュメントページへのリンクを含めること。

### 相互参照

コメントは、Markdown参照リンクとして別のコンポーネント（service、method、message、field、enum、enum値）に「リンク」できる。参照は以下の形式のいずれかで**しなければならない**（must）：

- 要素の完全修飾名。例：`[Book][google.example.v1.Book]`
- スコープ相対参照。例：`[Sci-Fi genre][Genre.GENRE_SCI_FI]`
- 暗黙の参照。例：`[Book][]` は `[Book][Book]` と同等

これらの参照は、[名前解決][name resolution]ルールに従って解決される。

包含フィールド名は参照に使用**してはならない**（must not）。解決されない。代わりに元の定義を参照**しなければならない**（must）。例えば、`[author][Book.author.family_name]`（`author`は`Book`のフィールド）は解決されないが、`[author][Author.family_name]`は解決される。

[name resolution]: https://protobuf.dev/programming-guides/proto3/#name-resolution

### 外部リンク

コメントは、パブリックコメント自体で説明されている内容を超えた背景情報を提供するために、外部ページにリンク**してもよい**（may）。外部リンクは、プロトコル（通常は`https`）を含む絶対URL（相対URLではなく）を使用**しなければならない**（must）。ドキュメントが特定のホストにあると想定**すべきではない**（should not）。例えば：
`[Spanner Documentation](https://cloud.google.com/spanner/docs)`

### 商標名

コメントで企業や製品の正式な商標名を参照する場合、頭字語は使用**すべきではない**（should not）。ただし、その頭字語が支配的な口語的使用となっており、避けると参照が不明瞭になる場合を除く（例：[IBM][]）。

コメントは、商標所有者の現在のブランディングと一致するように商標名を綴り、大文字小文字を区別する**すべきである**（should）。

### 非推奨

コンポーネント（service、method、message、field、enum、enum値）を非推奨にするには、`deprecated`[オプション](https://developers.google.com/protocol-buffers/docs/proto#options)を`true`に設定**しなければならない**（must）。それぞれのコメントの最初の行は`"Deprecated: "`で始め**なければならず**（must）、開発者に代替ソリューションを提供**しなければならない**（must）。代替ソリューションがない場合は、非推奨の理由を記載**しなければならない**（must）。

コメントは、内部コンテンツを`(--`と`--)`で囲むことにより、明示的に内部としてマーク**してもよい**（may）。

非公開リンク、内部実装ノート（`TODO`や`FIXME`ディレクティブなど）、およびその他のそのような資料は、内部としてマーク**しなければならない**（must）。

**注：** コメントは先頭コメントのみを使用**すべきである**（should）（末尾コメントや分離コメントは使用しない）。特に、コンポーネントを説明するために先頭コメントと末尾コメントの両方を使用**してはならない**（must not）。これは内部コンテンツアノテーションの不注意による省略の一般的な原因であるためである。

[commonmark]: https://commonmark.org/
[ibm]: https://en.wikipedia.org/wiki/IBM
