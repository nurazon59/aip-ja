# ステート

多くのAPIリソースは「ステート」の概念を持つ。これは通常、リソースのライフサイクルにおける位置を表す。例えば、仮想マシンはプロビジョニング中、使用可能、停止中、あるいは他のいくつかの状態にある可能性がある。ジョブやクエリは実行準備中、実行中、完了済みなどであり得る。

## ガイダンス

ステートを伝達する必要があるリソースは、enumを使用**すべきである**（should）し、そのenumは`State`と命名**すべきである**（should）（または、より具体的にする必要がある場合は、`State`で終わる名前とする）。このenumは、そのメッセージ内のフィールドとしてのみ使用される場合、それが記述するメッセージ内にネスト**すべきである**（should）。

**重要:** `Status`ではなく`State`という用語を使用する（`Status`はHTTPおよびgRPCステータス用に予約されている）。

### Enum値

理想的には、Google APIは同じ意味概念を表現する際に、全体を通して同じ用語を使用する。特定の状態を表現するために利用可能な言葉は通常多数あるが、顧客は複数のAPIを併用することが多く、用語が一貫している方が顧客にとって理解しやすい。

大まかな方針：

- 使用可能なリソースは`ACTIVE`（"ready"や"available"よりも推奨）。
- （通常は終端となる）要求されたアクションを完了したリソースは、`SUCCEEDED`（"successful"ではない）、`FAILED`（"failure"ではない）、`DELETED`、`SUSPENDED`などの過去分詞（通常`-ED`で終わる）を使用。
- 現在状態遷移中のリソースは、`RUNNING`、`CREATING`、`DELETING`などの現在分詞（通常`-ING`で終わる）を使用。この場合、ステートは一時的であり、ユーザーの追加アクションなしに自動的に別のステートに遷移することが期待される。

**注:** 顧客にとって有用なステートのみを追加することを忘れないこと。内部システムに存在するという理由だけで多数のステートを公開することは不必要であり、顧客に混乱をもたらす。各ステートには、それがなぜ必要かのユースケースが伴わなければならない。

### Output only

リソース内の`State` enumを参照するフィールドは、[AIP-203](./0203.md)に従い、Output onlyとして動作し、文書化**すべきである**（should）。

APIは、`State` enumが「update」メソッドを通じて直接更新されたり（または「create」メソッドを通じて直接設定されたり）することを認める**べきではない**（should not）。代わりに、カスタムの状態遷移メソッドを使用**すべきである**（should）。

これは、updateメソッドは一般的に副作用を持たないことが期待されるためであり、またステートを直接更新することは、任意の利用可能な値にステートを設定できることを暗に意味するが、実際にはステートは通常、リソースのライフサイクルを通じた進行を反映するためである。

### 状態遷移メソッド

状態遷移メソッドは、ステートフィールドをあるenum値から別のenum値に遷移させる役割を担う、特別な種類のカスタム メソッドである。遷移の一部として、他のフィールド（例：`update_time`フィールド）も変更される可能性がある。メソッド定義は以下のようになる**べきである**（should）：

```proto
// Publishes a book.
// The `state` of the book after publishing is `PUBLISHED`.
// `PublishBook` can be called on Books in the state `DRAFT`; Books in a
// different state (including `PUBLISHED`) returns an error.
rpc PublishBook(PublishBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:publish"
    body: "*"
  };
}
```

- メソッド名は、動詞の後にリソースのメッセージ名の単数形を続けたものにに**すべきである**（should）。
- リクエストメッセージはRPC名に`Request`サフィックスを付けたものに一致**しなければならない**（must）。
- レスポンスメッセージはリソース自体にに**すべきである**（should）。
  - RPCが[long-running][]の場合、レスポンスメッセージはリソース自体に解決される`google.longrunning.Operation`にに**すべきである**（should）。
- HTTP動詞は`POST`**しなければならない**（must）。
- HTTP URIは`:`文字の後にカスタム動詞（上記の例では`:publish`）を使用**しなければならない**（must）。URI内の動詞はRPC名の動詞と一致**しなければならない**（must）。
  - 単語区切りが必要な場合は、`camelCase`を使用**しなければならない**（must）。
- `google.api.http`アノテーションの`body`句は`"*"`**でなければならない**（must）。
- リソース名を受け取るリクエストメッセージフィールドはURIパスにマップ**すべきである**（should）。
  - このフィールドは`name`と命名**すべきである**（should）。
  - `name`フィールドはURIパス内の唯一の変数にに**すべきである**（should）。残りのパラメータはすべてURIクエリパラメータにマップ**すべきである**（should）。
- 状態遷移が許可されない場合、サービスは`FAILED_PRECONDITION`（HTTP 400）でエラーを返さ**なければならない**（must）。

リクエストメッセージは次のようになる**べきである**（should）：

```proto
message PublishBookRequest {
  // The name of the book to publish.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/Book"
    }];
}
```

- リソース名フィールドを含める**しなければならない**（must）。`name`と命名**すべきである**（should）。
- フィールドのコメントはリソースパターンを文書化**すべきである**（should）。
- 他のフィールドを含めても**してもよい**（may）。

## 追加ガイダンス

### デフォルト値

各ステートenumのゼロ値は、以下の規則に従う**べきである**（should）：

```proto
enum State {
  // The default value. This value is used if the state is omitted.
  STATE_UNSPECIFIED = 0;

  // Other values...
}
```

リソースは未指定のステートをユーザーに提供すべきでなく、この値は実際に使用される**べきではない**（should not）。

### 値の一意性

同じパッケージ内の複数のトップレベルenumは、同じ値を共有**してはならない**（must not）。これは、C++ protocコードジェネレータがトップレベルenumの値を単一の名前空間にフラット化するためである。

ステートenumはリソース定義内に置く**すべきである**（should）。

### プレフィックス

すべてのenum値に`STATE_`プレフィックスを使用する必要はない。ステートenumの値には、デフォルト値`STATE_UNSPECIFIED`を除き、enum名をプレフィックスとして付ける**べきではない**（should not）。

### 破壊的変更

**要約:** ステートenumは将来新しい値を受け取る可能性があることをユーザーに明確に伝え、既存のenumへのステート追加については慎重に行うこと。

既存のステートenumにステートを追加することは既存のユーザーコードを*壊す可能性*があるが、ステートの追加は破壊的変更とはみなされない。`ACTIVE`と`DELETED`の2つの値のみを持つステートを考えてみる。ユーザーが`if state == ACTIVE`とチェックし、elseの場合に単にリソースは削除されたとみなすコードを追加するかもしれない。APIが後で別の目的で新しいステートを追加すると、そのコードは壊れる。

最終的にこの動作を制御することはできないが、APIのドキュメントは、ステートenumに対して将来新しい値を受け取る可能性があることを期待してコードを記述するようユーザーに積極的に促す**べきである**（should）。

APIは、適切な場合に既存のState enumに新しいステートを追加**してもよい**（may）し、新しいステートの追加は破壊的変更とは*みなされない*。

### ステートを避けるべき場合

特に、ステートの取り得る値がごく少数である場合や、ステートが相互排他的でない場合には、`State` enumがAPIに最適でないことがある。

上記で議論した`ACTIVE`と`DELETED`のみのステートの例を考える。この状況では、APIは`google.protobuf.Timestamp delete_time`を公開し、その設定有無に基づいて削除を判断するようユーザーに指示する方が良い場合がある。

### 一般的なステート

以下は一般的に使用されるステートのリストである。APIはステート名を決定する際に先行事例を考慮**すべきである**（should）し、矛盾する先行事例がある場合は、グローバルな一貫性よりもローカルな一貫性を重視**すべきである**（should）。

#### 静止ステート

「静止ステート」とは、ユーザーのアクションがない限り、無期限に継続することが期待されるライフサイクルステートである。ただし、ユーザーはアクションを開始して、静止ステートにあるリソースを特定の他のステート（静止またはアクティブ）に遷移させることができる。

- `ACCEPTED`
- `ACTIVE`
- `CANCELLED`
- `DELETED`
- `FAILED`
- `SUCCEEDED`
- `SUSPENDED`
- `VERIFIED`

#### アクティブステート

「アクティブステート」とは、通常、単一の期待される静止ステートに自動的に解決されるライフサイクルステートである。

**注:** 顧客にとって有用なステートのみを公開することを忘れないこと。アクティブステートは、リソースがそのステートに十分な期間とどまる場合にのみ価値がある。ステート変更が即時に行われる場合、アクティブステートは不要である。

- `CREATING`（通常`ACTIVE`になる）
- `DELETING`（通常`DELETED`になる）
- `PENDING`（通常`RUNNING`になる）
- `REPAIRING`（通常`ACTIVE`になる）
- `RUNNING`（通常`SUCCEEDED`になる）
- `SUSPENDING`（通常`SUSPENDED`になる）
