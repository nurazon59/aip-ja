# カスタム メソッド

リソース指向設計（AIP-121）は、標準メソッドのみではモデル化が難しい任意のアクションを表現する手段を提供するためにカスタム メソッドを使用する。カスタム メソッドは、APIの語彙がユーザーの意図に従う手段を提供するため重要である。

## ガイダンス

カスタム メソッドは、標準メソッドでは簡単に表現できない機能にのみ使用**すべきである**（should）。可能な場合は、一貫したセマンティクスを持つ標準メソッドを優先すること。（もちろん、これは該当する機能が実際に通常のセマンティクスに準拠している場合にのみ適用される。標準メソッドを「なんとか機能させる」ために無理に適合させることは良い考えではない。）

カスタム メソッドは設計方法が大きく異なるが、多くの原則が一貫して適用される：

```proto
// 指定されたブックをアーカイブする。
rpc ArchiveBook(ArchiveBookRequest) returns (ArchiveBookResponse) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:archive"
    body: "*"
  };
}
```

**注:** 上記のパターンは、特定のリソースに対して操作するカスタム メソッドを示している。カスタム メソッドはリソース、コレクション、またはサービスに関連付けることができる。以下の箇条書きは3つのケースすべてに適用される。

- メソッドの名前は動詞に続いて名詞**すべきである**（should）。
  - 名前に前置詞（"for"、"with"など）を含めては**ならない**（must not）。
  - 名前の動詞には標準メソッドの動詞（[Get][get]、[List][list]、[Create][create]、[Update][update]、[Delete][delete]）を含める**べきではない**（should not）。
  - 名前に`Async`という用語を含めては**ならない**（must not）。代わりに、即時RPCとlong-running RPCを区別する目的では、サフィックス`LongRunning`をこの目的に使用**してもよい**（may）。例えば、long-runningなブック作成RPCを作成する場合（標準の`CreateBook`メソッドがlong-runningの側面を考慮する前に設計されていた場合）、カスタムの`CreateBookLongRunning`メソッドを導入できる。
- HTTPメソッドは`GET`または`POST`**でなければならない**（must）：
  - `GET`はデータまたはリソース状態を取得するメソッドに使用**しなければならない**（must）。
  - `POST`はメソッドに副作用がある場合、またはリソースやデータを変更する場合に使用**しなければならない**（must）。
- HTTP URIは`:`文字の後にカスタム動詞（上記の例では`:archive`）を使用**しなければならない**（must）、URIの動詞はRPC名の動詞と一致**しなければならない**（must）。
  - 単語の区切りが必要な場合、`camelCase`を使用**しなければならない**（must）。
- `google.api.http`アノテーションの`body`句は`"*"`**すべきである**（should）。
  - 詳細は[HTTPとgRPCのTranscoding][transcoding]を参照。
- カスタム メソッドはRPC名に一致するリクエスト メッセージを取る**すべきであり**（should）、`Request`サフィックスを付ける。
- カスタム メソッドはRPC名に一致するレスポンス メッセージを返す**べきであり**（should）、`Response`サフィックスを付ける。
  - 特定のリソースに対して操作する場合、カスタム メソッドはリソース自体を返して**してもよい**（may）。

### リソースベースのカスタム メソッド

カスタム メソッドは、APIがそのようにモデル化できる場合はリソースに対して操作**しなければならない**（must）：

```proto
// 指定されたブックをアーカイブする。
rpc ArchiveBook(ArchiveBookRequest) returns (ArchiveBookResponse) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:archive"
    body: "*"
  };
}
```

- リソース名のパラメータは`name`**と呼ばれなければならず**（must）、URIパス内の唯一の変数でなければならない。

### コレクションベースのカスタム メソッド

ほとんどのカスタム メソッドは単一のリソースに対して操作するが、一部のカスタム メソッドは代わりにコレクションに対して操作**してもよい**（may）：

```proto
// この出版社のブックをソートする。
rpc SortBooks(SortBooksRequest) returns (SortBooksResponse) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books:sort"
    body: "*"
  };
}
```

- コレクションのリソースに親がある場合、そのリソースは`parent`**と呼ばれなければならず**（must）、URIパス内の唯一の変数でなければならない。
- コレクションキー（上記の例では`books`）はリテラル**でなければならない**（must）。

### ステートレス メソッド

一部のカスタム メソッドはリソースにまったく関連付けられない。これらのメソッドは一般的に*ステートレス*であり、リクエストを受け入れてレスポンスを返し、API内のデータに永続的な影響を与えない。

```proto
// 提供されたテキストをある言語から別の言語に翻訳する。
rpc TranslateText(TranslateTextRequest) returns (TranslateTextResponse) {
  option (google.api.http) = {
    post: "/v1/{project=projects/*}:translateText"
    body: "*"
  };
}
```

- メソッドが特定のスコープ（上記の例のようにプロジェクトなど）で実行される場合、リクエスト メッセージのフィールド名はスコープリソースの名前**すべきである**（should）。単語区切りが必要な場合、`snake_case`を使用**しなければならない**（must）。
- URIは動詞と名詞の両方を`:`セパレータの後に配置**すべきである**（should）（この場合、コレクションがないため、URIに「偽のコレクションキー」を避ける）。例えば、`text:translate`よりも`:translateText`が推奨される。
- ステートレス メソッドは、課金が発生する場合`POST`を使用**しなければならない**（must）。

### 宣言的フレンドリーなリソース

宣言的フレンドリーなリソースは通常、カスタム メソッドを使用**すべきではない**（should not）（他のAIPで議論されている特定の宣言的フレンドリーなカスタム メソッドを除く）。なぜなら、宣言的フレンドリーなツールはそれらをどう扱うかを自動的に判断できないためである。

例外は、`Move`や`Rename`操作など、めったに使用されない本質的に命令的な操作であり、宣言的サポートの期待はない。

## 理由

### HTTPパス

標準メソッドと同様に、リソースまたはコレクションに対して操作するカスタム メソッドは、操作するリソースを示すために`name`または`parent`パラメータを必要とする。この規則により、クライアントはカスタム メソッドを適切なリソースにマッピングできる。

### HTTPメソッド

`GET`と`POST`の両方のHTTP動詞を許可することで、どのメソッドがデータを変更せず、どのメソッドが変更するかを明確に区別できる。データを読み取るだけのメソッドは、一部のクライアント（TerraformのDataSource）で第一級の概念を持ち、実行時の影響のリスクなしにどのメソッドを呼び出せるかをユーザーに明確に示す。

### 前置詞の禁止

一般に、前置詞を含むメソッド名は、既存のメソッドにフィールドを追加するか、メソッドが異なる動詞を使用すべき箇所で新しいメソッドが使用されていることを示す。例えば、`CreateBook`メッセージが既に存在し、`CreateBookFromDictation`の追加を検討している場合、代わりに`TranscribeBook`メソッドを検討する。同様に、プロパティ固有の検索メソッドが必要な場合、`GetBookByAuthor`の代わりに`author`フィールドを検索次元として持つ`SearchBooks`を検討する。これにより、APIとクライアントのサーフェスを肥大化させ、APIの管理と消費の両方に複雑さを追加する、過度に焦点を絞ったメソッドの爆発を防ぐことができる。

### RPC名

「async」という用語は、特定のメソッド呼び出しが同期か非同期かを示すためにプログラミング言語で一般的に使用され、RPCの作成も含まれる。この同期/非同期の側面は、RPC自体がlong-running operationを開始することを意図しているかどうかとは異なる抽象化レベルにある。RPC名自体に「async」を使用すると混乱を引き起こし、一部の言語ではRPCを呼び出すための同期メソッドと非同期メソッドの両方を生成するクライアントライブラリに問題を引き起こすことさえある。
