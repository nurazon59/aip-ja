# バッチメソッド: Update

単一トランザクションでリソースのセットを変更できるようにする必要があるAPIがある。バッチUpdateメソッドはこの機能を提供する。

## ガイダンス

APIは以下の2つのパターンを使用してBatch Updateをサポート**してもよい**（may）：

同期的にレスポンスを返す

```proto
rpc BatchUpdateBooks(BatchUpdateBooksRequest) returns (BatchUpdateBooksResponse) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books:batchUpdate"
    body: "*"
  };
}
```

非同期的にレスポンスに解決されるOperationを返す

```proto
rpc BatchUpdateBooks(BatchUpdateBooksRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books:batchUpdate"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "BatchUpdateBooksResponse"
    metadata_type: "BatchUpdateBooksOperationMetadata"
  };
}
```

- RPC名は`BatchUpdate`で始まる**しなければならない**（must）。RPC名の残りの部分は、更新されるリソースの複数形にする**すべきである**（should）。
- リクエストメッセージとレスポンスメッセージは、RPC名に`Request`および`Response`サフィックスを付けたものに一致**しなければならない**（must）。
- バッチメソッドが`google.longrunning.Operation`を返す場合、`response_type`フィールドと`metadata_type`フィールドの両方を指定**しなければならない**（must）。
- HTTP動詞は`POST`**しなければならない**（must）。
- HTTP URIは`:batchUpdate`で終わる**しなければならない**（must）。
- URIパスは、単純なCRUD操作で使用されるコレクションと一致するリソースのコレクションを表す**すべきである**（should）。操作が複数の親にまたがる場合、ダッシュ（`-`）をワイルドカードとして受け入れても**してもよい**（may）。
- `google.api.http`アノテーションのbody句は`"*"`にする**すべきである**（should）。

### アトミック vs 部分的成功

- バッチUpdateメソッドは、アトミック（すべてのリソースが更新されるか、いずれも更新されない）または部分的な成功動作をサポート**してもよい**（may）。選択するには、以下の要素を考慮すること：
  - **アトミック性確保の複雑さ:** 単純なデータベーストランザクションのパススルーである操作はアトミック操作を使用**すべきである**（should）し、複雑なリソースを管理する操作は部分的な成功操作を使用**すべきである**（should）。
  - **エンドユーザーエクスペリエンス:** APIコンシューマーの視点を考慮する。特定のユースケースにおいて、単一または少数のエントリの問題で大きなバッチ全体が失敗する可能性があっても、アトミックな動作が好ましいか？
- 同期バッチUpdateはアトミックで**しなければならない**（must）。
- 非同期バッチUpdateはアトミックまたは部分的な成功をサポート**してもよい**（may）。
  - 部分的な成功をサポートする場合、[Operation metadata message](#operation-metadata-message)の要件を参照。

### リクエストメッセージ

バッチUpdateメソッドのリクエストは、以下のパターンで指定**すべきである**（should）：

```proto
message BatchUpdateBooksRequest {
  // The parent resource shared by all books being updated.
  // Format: publishers/{publisher}
  // If this is set, the parent field in the UpdateBookRequest messages
  // must either be empty or match this field.
  string parent = 1 [
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // The request message specifying the resources to update.
  // A maximum of 1000 books can be modified in a batch.
  repeated UpdateBookRequest requests = 2
    [(google.api.field_behavior) = REQUIRED];
}
```

- 更新されるリソースがトップレベルリソースでない限り、`parent`フィールドを含める**すべきである**（should）。呼び出し元がこのフィールドを設定し、更新されるリソースの名前の親コレクションが一致しない場合、リクエストは失敗**しなければならない**（must）。
  - このフィールドは、リクエストごとに1つの親のみが許可される場合、requiredにする**すべきである**（should）。
  - フィールドは、それが参照する[リソースタイプ][aip-122-parent]を識別**すべきである**（should）。
  - フィールドのコメントはリソースパターンを文書化**すべきである**（should）。
- リクエストメッセージには、[標準Updateメソッド][request-message]で指定されているように、更新するリソースを指定するリクエストメッセージを受け入れるrepeatedフィールドを含める**しなければならない**（must）。フィールドは`requests`と命名**すべきである**（should）。
  - フィールドはrequiredにする**すべきである**（should）。
- 他のフィールドは[標準Updateリクエスト][request-message]から「hoist」しても**してもよい**（may）。これは、フィールドがバッチレベルまたは子リクエストレベルのいずれかで設定できることを意味する。`parent`と同様に、バッチレベルと子リクエストレベルの両方で同じフィールドに値が設定された場合、値は一致**しなければならない**（must）。
  - `update_mask`フィールドはhoistingの良い候補である。
- リクエストメッセージには、その他のrequiredフィールドを含めて**してはならず**（must not）、本AIPまたは他のAIPで説明されているものを除き、他のoptionalフィールドを含める**すべきではない**（should not）。
- `requests`フィールドの上のコメントは、許可されるリクエストの最大数を文書化**すべきである**（should）。

### レスポンスメッセージ

バッチUpdateメソッドのレスポンスは、以下のパターンで指定**すべきである**（should）：

```proto
message BatchUpdateBooksResponse {
  // Books updated.
  repeated Book books = 1;
}
```

- レスポンスメッセージには、更新されたリソースに対応する1つのrepeatedフィールドを含める**しなければならない**（must）。

### Operation metadata message

- `metadata_type`メッセージは、RPC名に`OperationMetadata`サフィックスを付けたものと一致するか、タイプが複数のBatchメソッドで共有される場合は`Batch`プレフィックスと`OperationMetadata`サフィックスで命名**しなければならない**（must）。
- バッチUpdateメソッドが部分的な成功をサポートする場合、メタデータメッセージは部分的な失敗を伝達するために`map<int32, google.rpc.Status> failed_requests`フィールドを含める**しなければならない**（must）。
  - このマップのキーは、バッチリクエスト内の`requests`フィールドにおけるリクエストのインデックスである。
  - 各マップエントリの値は、単一の標準Updateメソッドによって通常返されるエラーをミラーリング**しなければならない**（must）。
  - サーバー側のリトライにより失敗したリクエストが最終的に成功する可能性がある場合、そのような一時的なエラーは`failed_requests`を使用して伝達して**してはならない**（must not）。
  - バッチ内のすべてのリクエストが失敗した場合、`Operation.error`は`code = google.rpc.Code.Aborted`、`message = "None of the requests succeeded, refer to the BatchUpdateBooksOperationMetadata.failed_requests for individual error details"`で設定**しなければならない**（must）。
- メタデータメッセージは、操作の進捗を伝達する他のフィールドを含めても**してもよい**（may）。

### 部分的な成功の採用

既存のBatch APIが部分的な成功パターンを採用するためには、以下を実行する必要がある：

- 互換性のない動作変更を避けるため、デフォルト動作は維持されなければならない。
- APIがOperationを返す場合：
  - リクエストメッセージは`bool return_partial_success`フィールドを持た**しなければならない**（must）。
  - Operationの`metadata_type`は`map<int32, google.rpc.Status> failed_requests`フィールドを含める**しなければならない**（must）。
  - リクエストで`bool return_partial_success`フィールドがtrueに設定された場合、APIは部分的な成功動作を許可するべきであり、それ以外の場合はデフォルトとしてアトミックな動作を継続すべきである。
- APIが同期的に直接レスポンスを返す場合：
  - 既存のクライアントは成功レスポンスをアトミック操作として扱うため、既存バージョンのAPIは部分的な成功パターンを採用して**してはならない**（must not）。
  - 代わりに、Operationを返し、本AIPで説明されている部分的な成功パターンに従う新しいバージョンを作成**しなければならない**（must）。

## 根拠

### 同期バッチメソッドのアトミック制限

同期バッチメソッドがアトミックでなければならないという制限は、以下の考慮事項の結果である。

本AIPの以前のイテレーションでは、バッチメソッドはアトミックでなければならないと推奨していた。同期レスポンスのステータスコードで部分的な失敗を伝達する明確な方法はない。なぜなら、OKはすべてが機能したことを意味するからである。したがって、部分的な失敗を示す新しいフィールドをレスポンスに追加することは破壊的変更となる。なぜなら、既存のクライアントはOKレスポンスをすべてのリソースが更新されたと解釈するからである。

一方、[AIP-193](https://aip.dev/193)で説明されているように、Operationsは部分的な状態を提示する能力が高い。Operationのレスポンスステータスコードは、基になる操作の結果について何も伝えず、クライアントはレスポンス本文を確認して操作が成功したかどうかを判断する必要がある。

### 部分的な失敗の伝達

本AIPは、部分的な失敗を伝達するために`map<int32, google.rpc.Status> failed_requests`フィールドを使用することを推奨する。ここで、キーは元のバッチリクエストにおける失敗したリクエストのインデックスである。検討された他のオプションは以下の通り：

- `repeated google.rpc.Status`フィールド。どのエントリがどのリクエストに対応するかが明確でないため却下された。
- `map<string, google.rpc.Status>`フィールド（キーは失敗したリクエストのリクエストID）。以下の理由で却下された：
  - クライアントは部分的な成功レスポンスを使用するために、request_id -> requestのマップを維持する必要がある。
  - エラー伝達目的でリクエストIDを設定することは、サービスが複数のバッチリクエスト間で個々のリクエストの冪等性を保証できない場合、[AIP-155](https://aip.dev/155)と競合する可能性がある。
- `repeated FailedRequest`フィールド（FailedRequestは個々の更新リクエストと`google.rpc.Status`を含む）。ユーザーデータの機密性に関する追加の課題があるため、レスポンスでリクエストペイロードをエコーバックすることは推奨されないため却下された。
