# 用語集

簡潔さのために、このAIPは各AIP個別ではなく、ここでいくつかの一般的な用語を定義する。

## ガイダンス

以下の用語はAIP全体で一貫して使用される**べきである**（should）。

### API

アプリケーションプログラミングインターフェース。ローカルインターフェース（クライアントライブラリなど）またはネットワークAPI（後述）のいずれかである。

### API backend

APIサービスのビジネスロジックを実装するサーバーと関連インフラストラクチャのセット。個々のAPIバックエンドサーバーは、多くの場合APIサーバーと呼ばれる。

### API consumer

APIサービスを利用するエンティティ。Google APIの場合、通常はクライアントアプリケーションまたはサーバーリソースを所有するGoogleプロジェクトである。

### API definition

APIの定義。通常はProtocol Bufferサービスで定義される。API定義は任意の数のAPIサービスによって実装され得る。

### API frontend

ロードバランシングや認証など、APIサービス間で共通の機能を提供するサーバーと関連インフラストラクチャのセット。個々のAPIフロントエンドサーバーは、多くの場合APIプロキシと呼ばれる。

**注:** APIフロントエンドとAPIバックエンドは互いに隣接して実行されることも、遠く離れて実行されることもある。場合によっては、単一のアプリケーションバイナリにコンパイルされ、単一のプロセス内で実行されることもある。

### API interface

APIメソッドをグループ化するAPI仕様IDLの要素。例えば、Protocol Buffersの`service`定義など。これは通常、ほとんどのプログラミング言語における`class`や`interface`のような類似の高レベルのグループ化メカニズムにマッピングされる。

### API method

API内の個々の操作。通常はProtocol Buffersで`rpc`定義によって表現され、ほとんどのプログラミング言語ではAPI内の関数にマッピングされる。

### API producer

APIサービスを生成するエンティティ。Google APIの場合、通常はAPIサービスに責任を持つGoogleチームである。

### API product

APIサービスとその関連コンポーネント（利用規約、ドキュメント、クライアントライブラリ、サービスサポートなど）は、まとめて顧客にAPIプロダクトとして提供される。例：Google Calendar API。

**注:** 人々はAPIプロダクトを単にAPIと呼ぶことがある。

### API service

1つ以上のAPIのデプロイされた実装であり、Cloud Pub/Sub APIなど、1つ以上のネットワークアドレスで公開される。

### API service definition

APIサービスの定義に使用されるAPI定義（`.proto`ファイル）とAPIサービス構成（`.yaml`ファイル）の組み合わせ。Google APIサービス定義のスキーマは`google.api.Service`である。

### API service endpoint

APIサービスが着信APIリクエストを処理するために使用するネットワークアドレスを指す。1つのAPIサービスに複数のAPIサービスエンドポイントがある場合がある。例：`https://pubsub.googleapis.com`や`https://content-pubsub.googleapis.com`。

### API service name

APIサービスの論理識別子を指す。Google APIはRFC 1035 DNS互換名をAPIサービス名として使用する。例：`pubsub.googleapis.com`。

### API title

APIサービスのユーザー向けプロダクトタイトルを指す。例："Cloud Pub/Sub API"。

### API request

APIメソッドの単一の呼び出し。多くの場合、課金、ログ記録、監視、レート制限の単位として使用される。

### API version

APIまたはAPIグループ（一緒に定義されている場合）のバージョン。APIバージョンは多くの場合、文字列（"v1"など）で表現され、APIリクエストやProtocol Buffersのパッケージ名に現れる。

### Client

クライアントは、APIを呼び出して特定のタスクを実行するプログラム、またはCLIのような、ユーザーがアクセス可能な形でAPIを公開したり休止状態のリソースデータを操作したりする汎用ツールである。

クライアントの例には以下が含まれる：

- コマンドラインインターフェース
- 特定のプログラミング言語のSDKなどのライブラリ
- APIからリソースのJSON表現を読み取った後にそれを操作するスクリプト
- [宣言的クライアント][Declarative clients]などのツール
- WebアプリケーションなどのビジュアルUI

### Google API

Googleサービスによって公開されるネットワークAPI。これらのほとんどは`googleapis.com`ドメインでホストされている。クライアントライブラリやSDKなどの他のタイプのAPIは含まれない。

### Declarative Clients

宣言的クライアント（Declarative Clients）は、Infrastructure as Code（IaC）とも呼ばれ、APIによって公開されるリソースを表現するマークアップ言語またはコードを利用し、リソースをその望ましい状態に導くために適切な命令型アクションを実行するクライアントのカテゴリを指す。どのような変更を行うか、および一連の更新が成功したかどうかを判断するために、宣言的クライアントはサーバー側のリソース属性とクライアント定義の値を比較する。この比較機能により、作成または更新の正確性が保証されるが、サービスはクライアントが設定したフィールドをread-onlyとして扱い、それらの値を忠実に保持することが必要となる。

宣言的クライアントが抽象化する複雑さの例には以下が含まれる：

- 望ましい状態を達成するための適切な命令型アクション（作成/更新/削除）の決定
- これらの命令型アクションの順序付け

[Terraform][]はそのようなクライアントの例である。

### User

cURLなどを使用してAPIを直接利用している人間。この用語は、AIP内での人間の*ユーザー*とプログラムによる*クライアント*の使い分けを明確にするために定義されている。

### Network API

コンピュータのネットワークを介して動作するAPI。ネットワークAPIはHTTPを含むネットワークプロトコルを使用して通信し、多くの場合、それらを利用する組織とは別の組織によって生成される。
