# HTTPとgRPCのトランスコーディング

[リソース指向設計][aip-121]に従うAPIは[RPC][rpc]を使用して定義されるが、リソース指向設計フレームワークにより、REST/JSONの慣習に概ね従うAPIとしても提示できる。これは、開発者が既存の知識を活用できるようにするために重要である：利用可能なパブリックAPIの80%以上がほとんどのREST慣習に従っており、開発者はそのパターンに慣れている。

## ガイダンス

APIは、定義する各RPCに対してHTTP定義を提供**しなければならない**（must）。ただし、双方向ストリーミングRPCは例外であり、HTTP/1.1ではネイティブにサポートできない。双方向ストリーミングメソッドを提供する場合、APIは双方向ストリーミングに依存しない代替メソッドも提供**すべきである**（should）。

### HTTPメソッドとパス

Protocol Buffersを使用する場合、各RPCは`google.api.http`アノテーションを使用してHTTPメソッドとパスを定義**しなければならない**（must）：

```proto
rpc CreateBook(CreateBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
  };
}

message CreateBookRequest {
  // The publisher who will publish this book.
  // When using HTTP/JSON, this field is automatically populated based
  // on the URI, because of the `{parent=publishers/*}` syntax.
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // The book to create.
  // When using HTTP/JSON, this field is populated based on the HTTP body,
  // because of the `body: "book"` syntax.
  Book book = 2 [(google.api.field_behavior) = REQUIRED];

  // The user-specified ID for the book.
  // When using HTTP/JSON, this field is populated based on a query string
  // argument, such as `?bookId=foo`. This is the fallback for fields that
  // are not included in either the URI or the body.
  // Note that clients use camelCase format to communicate the field names
  // to the service.
  string book_id = 3;
}
```

- 最初のキー（この例では`post`）はHTTPメソッドに対応する。RPCは`get`、`post`、`patch`、または`delete`を使用**してもよい**（may）。
  - RPCは、[AIP-131][]、[AIP-132][]、[AIP-133][]、[AIP-134][]、および[AIP-135][]で説明されているように、各標準メソッドに規定されたHTTP動詞を使用**しなければならない**（must）。
  - RPCは、[AIP-136][]で説明されているように、カスタム メソッドに規定されたHTTP動詞を使用**すべきである**（should）。
  - RPCは`put`または`custom`を使用**すべきではない**（should not）。
- 対応する値はURIを表す。
  - URIは、リクエストprotoに入力されるべき変数を表すために`{foo=bar/*}`構文を使用**しなければならない**（must）。[リソース名][aip-122]を抽出する場合、変数はIDコンポーネントだけでなく、リソース名全体を含ま**なければならない**（must）。
  - URIは、変数名にネストされたフィールドを使用**してもよい**（may）。（さらに、AIP-134は`Update`リクエストにこれを義務付けている。）
  - URIは、`/`を除くすべてのURIセーフ文字にマッチするIDコンポーネントを表すために`*`文字を使用**しなければならない**（must）。URIは、`/`のマッチングが必要な場合、URIの最終セグメントとして`**`を使用**してもよい**（may）。
- `body`キーは、リクエスト内のどの単一のトップレベルフィールドがHTTPボディとして送信されるかを定義する。ボディが`*`の場合、リクエストオブジェクト自体がHTTPボディであることを示す。リクエストボディは、Protocol Buffersの標準的な[JSONエンコーディング][json encoding]で定義されるようにJSONとしてエンコードされる。
  - RPCは、`GET`または`DELETE` HTTP動詞を使用するRPCに対しては`body`をまったく定義**してはならない**（must not）。
  - RPCは、Create（[AIP-133][]）およびUpdate（[AIP-134][]）リクエストに対して規定された`body`を使用**しなければならない**（must）。
  - RPCは、カスタム メソッド（[AIP-136][]）に対して規定された`body`を使用**すべきである**（should）。
  - `body`はネストされたフィールドを含ん**ではならない**（must not）（または`.`文字を使用してはならない）、
  - `body`はURIパラメータと同じであっ**てはならない**（must not）。
  - `body`は`repeated`フィールドであっ**てはならない**（must not）。
  - フィールドは、後方互換性の理由でない限り、JSONのフィールド名を変更するために`json_name`アノテーションを使用**すべきではない**（should not）。

**注:** 双方向ストリーミングRPCには`google.api.http`アノテーションをまったく含めるべきではない。可能であれば、サービスは非ストリーミングの同等のRPCを提供**すべきである**（should）。

### 複数のURIバインディング

RPCが複数のURIに対応する必要がある場合がある：

```proto
rpc CreateBook(CreateBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
    additional_bindings: {
      post: "/v1/{parent=authors/*}/books"
      body: "book"
    }
    additional_bindings: {
      post: "/v1/books"
      body: "book"
    }
  };
}
```

- RPCは任意の数の追加バインディングを定義**してもよい**（may）。構造は`google.api.http`アノテーションと同一である（実際、これは再帰的な参照である）。
- RPCは追加バインディング内に追加バインディングを定義**してはならない**（must not）。
- `body`節は、トップレベルのアノテーションと各追加バインディングで同一**でなければならない**（must）。
