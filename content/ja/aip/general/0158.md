# ページネーション

APIはしばしばデータのコレクションを提供する必要があり、最も一般的には[List][]標準メソッドで使用される。しかし、コレクションは任意のサイズになることがあり、時間の経過とともに成長してルックアップ時間とネットワーク経由で送信されるレスポンスのサイズが増加する。したがって、コレクションをページネーションすることが重要である。

## ガイダンス

コレクションのデータを返すRPCは、_最初から_ ページネーションを提供 **しなければならない**（must）。なぜなら、既存のメソッドにページネーションを追加することは[後方互換性のない変更](#backwards-compatibility)だからである。

```proto
// 書籍をリストするためのリクエスト構造。
message ListBooksRequest {
  // この書籍コレクションを所有する親。
  // 形式: publishers/{publisher}
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // 返す書籍の最大数。サービスはこの値より少なく返すことがある。
  // 指定がない場合、最大50冊の書籍が返される。
  // 最大値は1000。1000を超える値は1000に調整される。
  int32 page_size = 2;

  // 以前の `ListBooks` 呼び出しから受け取ったページトークン。
  // これにより後続のページを取得できる。
  //
  // ページネーション時、`ListBooks` に提供された他のすべてのパラメータは、
  // ページトークンを提供した呼び出しと一致しなければならない。
  string page_token = 3;
}

// 書籍をリストした結果のレスポンス構造。
message ListBooksResponse {
  // 指定された出版社からの書籍。
  repeated Book books = 1;

  // `page_token` として送信して次のページを取得できるトークン。
  // このフィールドが省略されている場合、後続のページはない。
  string next_page_token = 2;
}
```

- コレクションのリクエストメッセージは、`int32 page_size` フィールドを定義 **すべきである**（should）し、ユーザーが返す結果の最大数を指定できるようにする。
  - `page_size` フィールドはrequiredであっては **してはならない**（must not）。
  - ユーザーが `page_size` を指定しない（または `0` を指定する）場合、APIは適切なデフォルトを選択し、APIはそれを文書化 **すべきである**（should）。APIはエラーを返しては **してはならない**（must not）。
  - ユーザーがAPIの許可する最大値を超える `page_size` を指定した場合、APIは許可された最大ページサイズに切り下げ **すべきである**（should）。
  - ユーザーが `page_size` に負の値を指定した場合、APIは `INVALID_ARGUMENT` エラーを送信 **しなければならない**（must）。
  - APIは、要求された数より少ない結果（0件を含む）を返す **してもよい**（may）（コレクションの末尾でない場合でも）。
- コレクションのリクエストメッセージは、`string page_token` フィールドを定義 **すべきである**（should）し、ユーザーがコレクション内の次のページに進むことができるようにする。
  - `page_token` フィールドはrequiredであっては **してはならない**（must not）。
  - ユーザーが後続のページのリクエストで `page_size` を変更した場合、サービスは新しいページサイズを尊重 **しなければならない**（must）。
  - ユーザーはRPCの他のすべての引数を同じに保つことが期待される。引数が異なる場合、APIは `INVALID_ARGUMENT` エラーを送信 **すべきである**（should）。
- レスポンスはストリーミングレスポンスであっては **してはならない**（must not）。
- コレクションのレスポンスメッセージは、`string next_page_token` フィールドを定義 **すべきである**（should）し、ユーザーに次のページを取得するために使用できるページトークンを提供する。
  - ページネーション結果を含むフィールドは、メッセージの最初のフィールドでフィールド番号 `1` と **すべきである**（should）。これは、1ページ分の結果を構成するリソースのリストを含む繰り返しフィールドと **すべきである**（should）。
  - コレクションの末尾に達した場合、`next_page_token` フィールドは空で **しなければならない**（must）。これがユーザーに「コレクションの末尾」を伝える_唯一の_方法である。
  - コレクションの末尾に達していない（またはAPIが時間内に判断できない）場合、APIは `next_page_token` を提供 **しなければならない**（must）。
- コレクションのレスポンスメッセージは、ユーザーにリスト内の総アイテム数を提供する `int32 total_size` フィールドを提供 **してもよい**（may）。
  - この合計は推定値でも **してもよい**（may）（ただし、APIはその旨を明示的に文書化 **すべきである**（should））。

### 結果のスキップ

ページネーション操作のリクエスト定義は、ユーザーが結果をスキップできるように `int32 skip` フィールドを定義 **してもよい**（may）。

`skip` 値は、スキップする個々のリソースの数を指さ**なければならず**（must）、ページ数ではない。

例えば：

- ページトークンがなく `skip` 値が `30` のリクエストは、31番目の結果から始まる1ページ分の結果を返す。
- 51番目の結果に対応するページトークン（最初の50件の結果が最初のページで返されたため）と `skip` 値が `30` のリクエストは、81番目の結果から始まる1ページ分の結果を返す。

巨大なデータセットのクエリによるレイテンシなどの理由で `skip` 値を満たせない場合、レスポンスは空の結果セットで `200 OK` で**なければならない**（must）。カーソルがコレクションの総サイズを超えることが_既知_である場合、レスポンスは `next_page_token` を含めては **してはならない**（must not）。

### 不透明性

APIが提供するページトークンは、不透明な（ただしURLセーフな）文字列で**なければならない**（must）あり、ユーザーが解析可能であっては **してはならない**（must not）。これは、ユーザーがこれらを分解できる場合、_実際にそうする_ためである。これにより、APIのページネーションの実装詳細がAPIサーフェスの一部となり、ユーザーを壊さずにそれらの詳細を更新することが不可能になる。

**警告:** それ以外は透過的なページトークンをBase-64エンコードすることは、十分な難読化メカニズムでは**ない**。

データベースに保存する必要がなく、機密データを含まないページトークンの場合、APIは必要なデータを持つ内部プロトコルバッファメッセージを定義し、シリアライズされたprotoをbase-64エンコードして送信することで、ページトークンを難読化 **してもよい**（may）。

ページトークンは、ページネーションプロセスを続行する場所を示すことのみに限定 **しなければならない**（must）。基盤となるリソースに対するいかなる形式の認可も提供しては **してはならない**（must not）。認可は、ページトークンの有無にかかわらず、他のリクエストと同様にリクエストに対して実行 **しなければならない**（must）。

### ページトークンの有効期限

多くのAPIはページトークンを内部的にデータベースに保存する。この状況では、APIは送信後妥当な時間でページトークンを期限切れに **してもよい**（may）。使用される可能性が低い大量のデータを不必要に保存しないためである。この動作を文書化する必要はない。

**注記:** 妥当な時間はAPIによって異なるが、目安として3日が推奨される。

## 後方互換性

既存のRPCにページネーションを追加することは、後方互換性のない変更である。これは奇妙に思えるかもしれない。protoメッセージにフィールドを追加することは一般的に後方互換性がある。しかし、この変更は_動作的に_互換性がない。

75のリソースを持つコレクションを持ち、すでにコードを作成してデプロイしているユーザーを考えてみよう。APIが後でページネーションフィールドを追加し、デフォルトを50に設定した場合、そのユーザーのコードは壊れる。すべてのリソースを取得していたのに、今は最初の50件しか取得できず（ページネーションを進める方法を知らない）。APIがより高いデフォルト制限（100など）を設定したとしても、ユーザーのコレクションが成長し、_その時点で_コードが壊れる可能性がある。

さらに、[クライアントライブラリは自動ページネーションを実装](https://google.aip.dev/client-libraries/4233)しており、通常はページネーションありとなしのRPCを異なるメソッドシグネチャで表現する。つまり、以前はページネーションがなかったメソッドにページネーションを追加すると、それらのライブラリに破壊的変更が生じる。

このため、コレクションを返すRPCには常に_最初から_ページネーションを追加することが重要である。これらは一貫して重要であり、既存のユーザーに問題を引き起こさずに後から追加することはできない。

**警告:** これは、ページネーションフィールドを提示するだけでなく、非無限のデフォルト値で_実際に実装_され**なければならない**（must）ことも意味する。最初は小さなコレクションの場合、メモリ内バージョン（すべてを取得してからページネーションする）を実装することは合理的である。

## 理由

### `skip` 応答の低下

大規模なコレクション、複雑なクエリ、グローバルに分散されたデータはすべて、ページネーションメソッドが特定の `skip` リクエストを迅速または確実に満たせなくなる原因となり得る。バックエンドクエリがタイムアウトしたり、データ照合に時間がかかったりする可能性があり、エンドユーザーエクスペリエンスがその影響を受ける必要はない。そのような場合、`next_page_token` を提供することでクライアントのエンゲージメントを維持しながら、サービスが適切な結果を収集するためにページネーションインターフェースを活用できる。サービスが要求された `skip` が利用可能な結果を超えると最終的に判断した場合、再びページネーションインターフェースが適用され、`next_page_token` が省略されて結果の終了が示される。
