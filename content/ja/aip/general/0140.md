# フィールド名

ユーザーにとって直感的な方法でフィールドに名前を付けることは、API設計の最も困難な側面の1つであることが多い。これには多くの理由があり、作者にとって完全に直感的に思えるフィールド名が、読者を困惑させることは珍しくない。

さらに、ユーザーが1つのAPIだけを使用することは稀であり、多くのAPIを一緒に使用する。その結果、同じ名前が異なる意味で（または異なる名前が同じ意味で）使用されると、ユーザーはあるAPIで学んだことを別のAPIに適用できなくなるため、不必要な混乱を引き起こすことが多い。

要するに、フィールド名がシンプルで直感的で、相互に一貫性がある場合、APIは最も理解しやすい。

## ガイダンス

フィールド名は正しいアメリカ英語**であるべきである**（should）。

フィールド名は、提示される概念を明確かつ正確に伝え、曖昧な過度に一般的な名前を避ける**すべきである**（should）。とはいえ、フィールド名は不必要な単語を含めることを避ける**すべきである**（should）。特に、常に適用され認知的な価値をほとんど追加しない形容詞を含めることは避ける。例えば、共有されていないバリアントが存在しない場合、`proxy_settings`フィールドは`shared_proxy_settings`と同じくらい有用かもしれない。

**重要:** フィールド名は生成されたクライアントサーフェスに頻繁に現れる。適切に説明的で、適切な長さであることを確認すること。

### ケース

protoファイルのフィールド定義は`lower_snake_case`名を使用**しなければならない**（must）。これらの名前はJSONおよび生成コード内の適切な命名規則にマッピングされる。

さらに、フィールド内の各単語は数字で始まっては**ならない**（must not）。これはスネークケースとキャメルケースの変換時に曖昧さを生み出すためである。同様に、フィールドには先頭、末尾、または隣接するアンダースコアを含めては**ならない**（must not）。

### 統一性

APIは、可能な限り同じ概念には同じ名前を、異なる概念には異なる名前を使用するよう努める**すべきである**（should）。これには複数のAPIにわたる名前も含まれ、特にそれらのAPIが一緒に使用される可能性が高い場合が該当する。

### Repeatedフィールド

Repeatedフィールドは`books`や`authors`などの適切な複数形を使用**しなければならない**（must）。一方、非repeatedフィールドは`book`や`author`などの単数形を使用**すべきである**（should）。これは、フィールド名がリソース名に従うべきであるため、リソース名も単数形を使用**すべきである**（should）ことを意味する（例：`repeated Book books`を使用し、`Books books = 1`は使用しない）。

### 前置詞

フィールド名には前置詞（"with"、"for"、"at"、"by"など）を含める**べきではない**（should not）。例：

- `error_reason`（`reason_for_error`ではない）
- `author`（`written_by`ではない）

この規則に従うことで、フィールド名がより頻繁に一致しやすくなる。さらに、フィールド名の前置詞は、過度に制限的なフィールドや最適でないデータ型など、設計上の問題を示している可能性もある。これは特に"with"に関して当てはまる。`book_with_publisher`というフィールド名は、ブックリソースが不適切に構造化されており、再設計する価値があることを示している可能性が高い。

**注:** "per"という単語はこの規則の例外である。単位の一部として"per"が使用される場合のガイダンスについては[AIP-141][]を参照のこと（例："miles per hour"）。

### 略語

ソフトウェア開発者の間でよく知られている名前の略語（"config"や"spec"など）については、API定義では完全なスペルではなく略語を使用**すべきである**（should）。これにより、ソースコードの読み書きが容易になる。例：

- `config`（`configuration`ではない）
- `id`（`identifier`ではない）
- `info`（`information`ではない）
- `spec`（`specification`ではない）
- `stats`（`statistics`ではない）

さらに、単位のよく知られた略語はフィールド名で使用**すべきである**（should）。数量を表すフィールドの詳細なガイダンスについては[AIP-141][]を参照のこと。例：

- `distance_km`（`distance_kilometers`ではない）
- `width_px`（`width_pixels`ではない）

### 形容詞

統一性のため、名詞と形容詞の両方を含むフィールド名は、形容詞を名詞の**前に**配置**すべきである**（should）。例：

- `collected_items`（`items_collected`ではない）
- `imported_objects`（`objects_imported`ではない）

### 動詞

フィールド名は意図やアクションを反映するように命名しては**ならない**（must not）。動詞であっては**ならない**（must not）。むしろ、フィールドは変更（例：CreateやUpdate）のための*望ましい値*と、読み取り（例：GetやList）のための*現在の値*を定義するため、名前は名詞**でなければならない**（must）。何をすべきかではなく、何がそうであるかを定義する。

- `collected_items`（`collect_items`ではない）
- `disabled`（`disable`ではない）

対照的に、標準メソッドとカスタム メソッドの両方のメソッド名は、リソースの側面を変更し、動詞として命名される。

### ブール値

ブール値フィールドは接頭辞"is"を省略**すべきである**（should）。例：

- `disabled`（`is_disabled`ではない）
- `required`（`is_required`ではない）

**注:** そうでなければ[予約語](#reserved-words)になるフィールド名はこの規則の例外である。例：`is_new`（`new`ではない）。

### String vs bytes

`bytes`を使用する場合、JSONをワイヤーで使用する際にフィールドの内容はbase64エンコードされる。サービスは、バイナリコンテンツをワイヤーで送信する必要がある場合に`bytes`を使用**すべきであり**（should）、ユーザーに手動でフィールドをbase64エンコードさせて`string`フィールドに入力させるべき**ではない**（should not）。1つの例外は、APIが保存時にbase64エンコードされることを意図したデータを扱い、`bytes`を使用した結果として二重のbase64エンコードが望ましくない場合である。この場合、サービスは`string`を使用**してもよい**（may）。

### URI

任意のURIを表すフィールド名は`uri`を使用**すべきである**（should）。特に、URLはURIであるが、すべてのURIがURLであるとは限らないことに注意。

フィールド名は、必要に応じて`uri`の前に接頭辞を使用**してもよい**（may）。

URLのみを表すことができるフィールド名は`url`を使用**すべきである**（should）。

```proto
message Book {
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // ブックの画像を指すURL。
  string image_url = 2;

  // ブックを識別するURI。
  // ISBNまたはURLの可能性がある。
  string uri = 3;
}
```

**注:** 以前にURLフィールドに`uri`を使用していたAPIは、不必要なAPI変更を避け、ローカルの一貫性を維持するために継続して使用してもよい。

### 予約語

フィールド名は、一般的なプログラミング言語のキーワード（`new`、`class`、`function`、`import`など）と競合する可能性のある名前を避ける**すべきである**（should）。予約キーワードは、その言語でAPIを使用する開発者に困難をもたらす可能性がある。

### 競合

メッセージは、それを囲むメッセージと同じ名前（大文字小文字変換は無視）のフィールドを含める**べきではない**（should not）。これは一部の言語でコード生成時に競合を引き起こす。

### 表示名

多くのリソースは人間が読める名前を持ち、UIでの表示によく使用される。このフィールドは`display_name`**とすべきである**（should）、一意性要件を持つ**べきではない**（should not）。

エンティティに正式な名称（会社名や書籍のタイトルなど）がある場合、APIは代わりに`title`をフィールド名として使用**してもよい**（may）。`title`フィールドは一意性要件を持つ**べきではない**（should not）。

## 理由

### URI vs URL

ガイダンス自体で、すべてのURLはURIであるが、すべてのURIがURLであるとは限らないと述べている。URIに合わせる目的は、より汎用化可能なフィールドを可能にし、多様なユースケースを扱えるようにするとともに、API全体の標準化を推進し、より一貫性のあるサーフェスを生み出すことである。同時に、要件が**すべきである**（should）であるため、本当に価値がある場合にはより具体的な用語を許可している。
