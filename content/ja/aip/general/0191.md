# ファイルとディレクトリ構造

一貫したファイルとディレクトリ構造は、技術的には最小限の違いしか生まないが、APIサーフェス定義をユーザーやレビューアにとって読みやすくする。

## ガイダンス

**注：** 以下のガイダンスは、Google全体で使用されているものなど、protocol buffersで定義されたAPIに適用される。このガイダンスの精神は他の仕様言語や形式で定義されたAPIにも適用されるが、特定の推奨事項の一部は無関係である可能性がある。

### 構文

protocol buffersで定義されるAPIは`proto3`構文を使用**しなければならない**（must）。

### 単一パッケージ

protocol buffersで定義されるAPIは、各APIを単一のパッケージで定義**しなければならない**（must）。このパッケージはバージョンコンポーネントで終了**しなければならない**（must）。例えば：

```proto
syntax = "proto3";

package google.cloud.translation.v3;
```

Google APIは、protocol bufferの`package`ディレクティブと一致するディレクトリに存在**しなければならない**（must）。例えば、上記のパッケージはディレクトリが`google/cloud/translation/v3`であることを規定する。

### ファイル名

API定義を複数のファイルに分割することは有用であることが多い。ファイル名は`snake_case`を使用**しなければならない**（must）。

APIは明白な「エントリ」ファイルを持つ**べきである**（should）。通常はAPI自体にちなんで命名される。少数の個別サービス（Google Cloud Pub/Subの`Publisher`と`Subscriber`は良い例）を持つAPIは、サービスごとに個別のエントリファイルを持って**もよい**（may）。

ファイルが1つだけのAPIは、APIの名前に対応するファイル名を使用**すべきである**（should）。

APIの`service`定義と関連するRPCリクエストおよびレスポンスの`message`定義は、同じファイルで定義**すべきである**（should）。

ファイル名はクライアントライブラリのモジュール名になることが多く、顧客は`import`文や`use`文でそれらを使用することに留意する。したがって、説明的で言語キーワードを含まないファイル名を選ぶことは重要である。例えば、`import.proto`というファイルはPythonで問題を引き起こす可能性がある。

**注：** バージョンはファイル名として使用**してはならない**（must not）。これはクライアントライブラリで奇妙なインポートを生み出すためである。`v3.proto`や`v1beta1.proto`のようなファイル名は禁止されている。

### ファイルレイアウト

個々のファイルは、より高レベルで重要な定義を、より低レベルで重要でない定義の前に配置**すべきである**（should）。

protoファイルでは、コンポーネントは以下の順序で配置**すべきである**（should）。これらの各要素は空行で区切る**すべきである**（should）：

- 著作権およびライセンス表示（該当する場合）。
- protoの`syntax`ステートメント。
- protoの`package`ステートメント。
- `import`ステートメント（アルファベット順）。
- ファイルレベルの`option`ステートメント。
- `service`定義。
  - メソッドは影響を与えるリソースごとにグループ化**すべきである**（should）。標準メソッドはカスタム メソッドの前に配置**すべきである**（should）。
- リソースの`message`定義。親リソースは子リソースの前に定義**しなければならない**（must）。
- RPCリクエストおよびレスポンスの`message`定義。対応するメソッドと同じ順序で配置する。各リクエストメッセージは対応するレスポンスメッセージの前に配置**しなければならない**（must）（存在する場合）。
- 残りの`message`定義。
- トップレベルの`enum`定義。

### パッケージングアノテーション

Protocol Buffersには、生成されたファイルのパッケージまたは名前空間（ターゲット言語の語彙に依存）を宣言するためのアノテーションが用意されている。例えば、`go_package`や`csharp_namespace`を設定すると、推測されるパッケージ名が上書きされる。

APIを定義する際は、以下のルールが適用される：

- Java
  - `java_package`アノテーションを設定**しなければならない**（must）。正しい値は通常、適切なTLDを前置したprotoパッケージである。例：`com.google.example.v1`。
  - `java_multiple_files`アノテーションを`true`に設定**しなければならない**（must）。
  - `java_outer_classname`アノテーションを設定**しなければならない**（must）。protoファイル名を`PascalCase`にし、`Proto`を追加した値に設定する**すべきである**（should）。例：`LibraryProto`。
- その他の言語
  - 他の言語のパッケージまたは名前空間ディレクティブは、protoパッケージ内のすべてのファイルで設定するか、まったく設定しないかのいずれかで**しなければならない**（must）。設定する場合、値はすべてのファイルで同一で**しなければならない**（must）。
  - protobufパッケージの一部が複合名（accessapprovalなど）である場合、C#、Ruby、PHPのオプションはPascalCase（UpperCamelCase）を使用して単語の区切りを考慮するために指定**しなければならない**（must）。例：
    ```proto
    option csharp_namespace = "Google.Cloud.AccessApproval.V1";
    option php_namespace = "Google\\Cloud\\AccessApproval\\V1";
    option ruby_package = "Google::Cloud::AccessApproval::V1";
    ```
  - `go_package`の値は、Goコードがどのように管理されるかに直接依存する（つまり、モジュール名がVCSプロバイダーに基づくか、リモートインポートパスを使用するか）。ただし、多くの場合一貫した構造を持つ。
    - モジュールは製品領域に基づいて異なる**してもよい**（may）。例えば、`google.cloud.accessapproval.v1`はモジュール`cloud.google.com/go/accessapproval`に配置される。
    - パッケージインポートパスはprotoパッケージから導出する**すべきである**（should）。
    - protoパッケージ内のAPIバージョンには`api`を前置する**すべきである**（should）。例えば、protoパッケージセグメント`v1`は`apiv1`になる。
    - 末端のインポートパスセグメントはprotoパッケージ内の製品名に基づく**すべきである**（should）。`pb`をサフィックスとして付加**しなければならない**（must）。例えば、`accessapproval`は`accessapprovalpb`になる。
    - この値は生成コードを所有するチームに委ねる**すべきである**（should）。

すべてのパッケージングアノテーションは、名前のアルファベット順に指定**すべきである**（should）。言語パッケージオプションの詳細については、[Protobufドキュメント][package docs]を参照すること。

**重要：** Java以外の言語では複合名を含まないAPIに対して適切なデフォルトがあるが、このアノテーションを（デフォルトと等しくない値で）_追加_することは、その言語における破壊的変更を構成することに注意すること。protoをリリースする際は、省略が意図的であることを確認すること。

## 根拠

### Javaパッケージングオプション

`java_multiple_files`オプションをtrueに設定すると、よりクリーンなファイル構造が得られる。これにより`protoc`はProtobufタイプごとに1つの出力ファイルを作成し、より細かい粒度のインポートが可能になる。`java_outer_classname`オプションは`java_multiple_files`と組み合わせて必須である。これにより`protoc`はコンパイルされた各Protobufタイプを、オプションの値を名前とするJavaクラスにラップする。これにより、生成されたタイプ間の潜在的な名前の衝突を防ぐ。

### Goパッケージングオプション

Goパッケージングオプションは、生成コードを所有するチームが決定する必要がある。これはチームのソースコード管理プラクティスに直接結びついているためである。すべてのprotoパッケージが独自のGoパッケージを決定できるようにすると、コード管理に不整合と摩擦が生じる。所有チーム内で、Goパッケージ命名の一貫した構造は、一貫したエンドユーザー体験にとって重要である。
