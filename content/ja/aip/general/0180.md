# 後方互換性

APIは基本的にユーザーとの契約であり、ユーザーはしばしばAPIに対してコードを書き、それを本番サービスにデプロイして、それが継続して動作することを期待する（APIに別のことを示す[安定性レベル][aip-181]がある場合を除く）。したがって、何が後方互換性のある変更で、何が後方互換性のない変更かを理解することは重要である。

## ガイダンス

既存のクライアントコードは、新しいマイナーまたはパッチリリースに更新するサービスによって壊されては**してはならない**（must not）。古いクライアントは、新しいサーバー（同じメジャーバージョン番号）に対して動作でき**しなければならない**（must）。

**重要：** 変更が互換性があるかどうかは常に明確であるとは限らない。ここでのガイダンスは、考えられるすべての変更の包括的なリストではなく、指標として扱う**すべきである**（should）。

考慮すべき3つの異なるタイプの互換性がある：

1. ソース互換性：以前のバージョンに対して書かれたコードは、新しいバージョンでコンパイルでき**なければならず**（must）、新しいバージョンのクライアントライブラリで正常に実行でき**なければならない**（must）。
2. ワイヤー互換性：以前のバージョンに対して書かれたコードは、新しいサーバーと正しく通信でき**なければならない**（must）。つまり、入力と出力が互換性があるだけでなく、シリアライズとデシリアライズの期待も一致し続ける。
3. セマンティック互換性：以前のバージョンに対して書かれたコードは、ほとんどの合理的な開発者が期待するものを引き続き受け取ら**なければならない**（must）。（ただし、これは実際には難しい場合があり、ユーザーが何を期待するかを判断するには状況判断が必要な場合がある。）

**注：** 一般的に、ここでの具体的なガイダンスは、トランスポート形式としてprotocol buffersとJSONを使用することを前提としている。他のトランスポート形式では、若干異なるルールがある場合がある。

**注：** このガイダンスは、APIが複数の言語で書かれたさまざまなコンシューマーから呼び出されることを意図しており、コンシューマーがいつ更新するかを制御できないことを前提としている。より限定されたスコープを持つAPI（例えば、APIプロデューサーと同じチームが書いたクライアントコードからのみ呼び出されるAPI、または更新を強制できる方法でデプロイされているAPI）は、自身の互換性要件を慎重に検討する**すべきである**（should）。

### コンポーネントの追加

一般的に、新しいコンポーネント（インターフェース、メソッド、メッセージ、フィールド、enum、またはenum値）は、同じメジャーバージョン内の既存のAPIに追加**してもよい**（may）。

ただし、これを行う際は以下のガイドラインに留意すること：

- 以前のサーフェスに対して書かれたコード（したがって新しいコンポーネントを認識しない）は、以前と同じように扱われ続け**なければならない**（must）。
  - 新しい必須フィールドは、既存のリクエストメッセージやリソースに追加**してはならない**（must not）。
  - クライアントが設定するフィールドは、フィールドが導入される前の動作と一致するデフォルトの動作を持た**なければならない**（must）。
    - これは場合によっては難しい場合がある。例えば、以前はすべてのアイテムが返されていた（つまり`page_size`が無限であり、これは推奨されない）場合に、後からページネーションを追加する。新しい`page_size`フィールドのデフォルトが以前返されていたものより_小さい_場合、古いクライアントはすべての結果が返されたと誤って想定する。
  - 以前にサーバーが設定していたフィールドは、冗長性が生じても設定し続け**なければならない**（must）。
- 特にenum値については、ユーザーコードが新しい値を適切に処理しない可能性があることに注意する。
  - enum値は、リクエストメッセージでのみ使用されるenumに自由に追加**してもよい**（may）。
  - レスポンスメッセージやリソースで使用され、新しい値を受け取ることが予想されるenumは、これを文書化する**すべきである**（should）。この状況でもenum値の追加は**してもよい**（may）。ただし、適切な注意を払う**すべきである**（should）。

**注：** 既存のコンポーネントに密接に関連するコンポーネント（例えば、`string foo`がすでに存在する場合の`string foo_value`）を追加すると、生成されたコードが競合する状況に陥る可能性がある。サービス所有者は、自分自身やユーザーが使用する可能性が高いツールの微妙な点を認識する**すべきである**（should）（ツール作成者は、可能であればそのような微妙な点を避けるよう努める**すべきである**（should））。

### コンポーネントの削除または名称変更

既存のコンポーネント（インターフェース、メソッド、メッセージ、フィールド、enum、またはenum値）は、同じメジャーバージョン内の既存APIから削除**してはならない**（must not）。コンポーネントの削除は後方互換性のない変更である。

**重要：** コンポーネントの名称変更は、意味的に「削除して追加」と同等である。この種の変更が望ましい場合、サービスは新しいコンポーネントを追加**してもよい**（may）が、既存のものを削除**してはならない**（must not）。これによりユーザーが同じ意味的概念に対して競合する値を指定できる状況では、動作を明確に指定**しなければならない**（must）。

### ファイル間のコンポーネント移動

既存のコンポーネントはファイル間で移動**してはならない**（must not）。

コンポーネントを同じパッケージ内の別のprotoファイルに移動することはワイヤー互換性があるが、C++やPythonなどの言語で生成されるコードは、`import`や`#include`が正しいコード位置を指さなくなるため、破壊的な変更となる。

### oneofへの移動

既存のフィールドはoneofに移動したり、oneofから移動したり**してはならない**（must not）。これはGo protobufスタブにおける後方互換性のない変更である。

### フィールドの型変更

既存のフィールドとメッセージは、新しい型がワイヤー互換であっても、型を変更**してはならない**（must not）。型変更は生成コードを破壊的な方法で変更するためである。

### 文字列長の変更

APIは、`string`フィールドのサイズまたは上限（入力として受け入れられる場合）の上限を増やすことを避ける**すべきである**（should）。APIは、想定されるサイズ上限の増加を互換性のない変更として扱う**すべきである**（should）（例として[リソース名の変更](#changing-resource-names)を参照）。一貫したサイズを確保する必要がある場合、APIはフィラー文字で値を埋める**してもよい**（may）が、これを行う場合は文書化**しなければならない**（must）。

### リソース名の変更

リソースはその[名前][aip-122]を変更**してはならない**（must not）。

ほとんどの破壊的変更とは異なり、これはメジャーバージョンにも影響する：クライアントがv2.0を使用してv1.0で作成されたリソースにアクセスすることを期待する場合、またはその逆の場合、同じリソース名が両方のバージョンで使用**しなければならない**（must）。

より微妙な点として、有効なリソース名のセットも変更**すべきではない**（should not）。理由は以下の通り：

- リソース名の形式がより制限的になると、以前は成功していたリクエストが失敗するようになる。
- リソース名の形式が以前のドキュメントよりも制限的でなくなると、以前のドキュメントに基づいて仮定を行っていたコードが壊れる可能性がある。ユーザーはリソース名を他の場所に保存する可能性が非常に高く、許可される文字のセットや名前の長さに敏感な方法で保存する場合がある。あるいは、ユーザーがドキュメントに従って独自のリソース名検証を実行する可能性がある。
  - 例えば、Amazonはより長いEC2リソースIDを許可し始めたとき、顧客に[十分な警告][ec2]を与え、移行期間を設けた。

### セマンティックの変更

コードは、_明示的にサポートまたは文書化されていない場合でも_、APIの動作やセマンティクスに依存することが多い。したがって、APIは合理的なユーザーコードを壊す可能性が高い方法で、可視的な動作やセマンティクスを変更**してはならない**（must not）。そのような変更は、それらのユーザーにとって破壊的と見なされるためである。

**注：** これにはある程度の判断が含まれる。提案された変更がユーザーを壊す可能性が高いかどうかは必ずしも明確ではなく、このガイダンスを広く解釈すると、事実上_あらゆる_変更を防ぐ可能性がある（これは意図ではない）。

#### 値の形式または構築方法の変更

APIは、APIバージョン内で、既存のフィールドの値の期待される形式または構築に使用されるアルゴリズムを変更**してはならない**（must not） - たとえそのフィールドが`OUTPUT_ONLY`でAPIサービスによって設定される場合でも。これを行うには新しいAPIバージョンが必要である。

例えば、IPv4形式に準拠する`ip_address`フィールドの形式を、代わりにIPv6値を含むように変更することは破壊的変更である。

#### デフォルト値は変更してはならない

デフォルト値とは、クライアントによって指定されなかった場合にリソースに対してサーバーが設定する値である。このセクションはリソース上のフィールドの静的なデフォルト値にのみ適用され、リソースのデフォルトIPアドレスのような動的なデフォルトには適用されない。

デフォルト値の変更は破壊的と見なされ、行っては**ならない**（must not）。リソースのデフォルト動作はそのデフォルト値によって決定され、これはマイナーバージョン間で変更**してはならない**（must not）。

例えば：

```proto
message Book {
    // google.api.resource およびその他のアノテーションとフィールド

    // 本のジャンル
    // 作成時にこれが設定されていない場合、フィールドにはFICTIONの値が与えられる。
    enum Genre {
      UNSPECIFIED = 0;
      FICTION = 1;
      NONFICTION = 2;
    }
}
```

次のように変更する：

```proto
message Book {
    // google.api.resource およびその他のアノテーションとフィールド

    // 本のジャンル
    // 作成時にこれが設定されていない場合、フィールドにはNONFICTIONの値が与えられる。
    enum Genre {
      UNSPECIFIED = 0;
      FICTION = 1;
      NONFICTION = 2;
    }
}
```

これは破壊的変更となる。

#### デフォルトのシリアライズ

APIは、デフォルト値を持つフィールドのシリアライズ方法を変更**してはならない**（must not）。例えば、値がデフォルトと等しい場合にフィールドがレスポンスに現れない場合、シリアライズを変更してデフォルト値のフィールドを含めるように**してはならない**（must not）。クライアントはリソース内のフィールドの有無を意味的に重要であると依存する可能性があるため、存在しない値のシリアライズ方法の変更はマイナーバージョン内で発生しては**しなければならない**（must not）。

以下のprotoを考える。ここで`wheels`のデフォルト値は`2`である：

```proto
// 自動車の表現
message Automobile {
    // google.api.resource およびその他のアノテーションとフィールド

    // 自動車の車輪の数。
    // デフォルト値は2、クライアントから値が送信されていない場合。
    int wheels = 2;
}
```

まず、`wheels`の値が`2`の場合のprotoのJSONへのシリアライズは以下の通り：

```json
{
    "name": "my-car"
}
```

その後、APIサービスが値がデフォルト値`2`と等しい場合でも`wheel`を含めるようにシリアライズを変更する：

```json
{
    "name": "my-car",
    "wheels": 2
}
```

これはメジャーバージョン内で後方互換性のない変更を構成する。

## 根拠

### 文字列長変更のリスク

エンドユーザーは、リソースのプロパティ（`name`など）を、長さが制限された専用のデータベースカラムに保存する場合がある。サービスが、元々文書化/観測された長さの2倍の`name`の値を返し始めると、顧客のデータベースが予期せず壊れる可能性がある。さらに、URL（クエリパラメータを含む）に現れる文字列プロパティはクライアント側の制限がある可能性が特に高く、長さの変更に対してより敏感である。

### 値の形式または構築方法変更のリスク

顧客は、クライアント側のパース、ハッシュ、またはデータベーステーブル構築のためにフィールドの形式やアルゴリズムによる構築に依存することが多い。既存のフィールドでそれを変更すると、そのクライアント側での利用が壊れる可能性がある。
