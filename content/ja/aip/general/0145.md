# 範囲

サービスは、離散値または連続値の範囲を表現する必要があることがよくある。これらは意味が大きく異なり、整数、浮動小数点、タイムスタンプなど多くの型があり、範囲の期待される意味は、扱う範囲の種類によって微妙に異なることがある。

## ガイダンス

範囲を表すリソースまたはメッセージは、通常、同じ型の2つの別々のフィールドに`start_`および`end_`プレフィックスを付けて使用**すべきである**（should）：

```proto
// A representation of a chapter in a book.
message Chapter {
  string title = 1;

  // The page where this chapter begins.
  int32 start_page = 2;

  // The page where the next chapter or section begins.
  int32 end_page = 3;
}
```

### 包含範囲と排他範囲

範囲を表すフィールドは、ほとんどの状況で包含的開始値と排他的終了値（半開区間）を使用**すべきである**（should）。区間表記では：`[start_xxx, end_xxx)`。

排他的終了値が推奨される理由は以下の通り：

- 特にタイムスタンプなどの連続値に対するユーザーの期待に適合し、不正確な「限界値」（例：`2012-04-20T23:59:59`）を表現する必要性を回避する。
- C++、Java、Python、Goをはじめとする一般的なプログラミング言語の多くと一貫性がある。
- 隣接する範囲についての推論が容易：`[0, x), [x, y), [y, z)`。値が1つの範囲から次の範囲へ連鎖可能。

### タイムスタンプ区間

以下のセクションでは、[AIP-213][]で説明されている共通プロトに含まれる[google.type.Interval][interval]型の使用について説明する。この型は2つのタイムスタンプ間の範囲を表し、包含的開始値と排他的終了値を持つ。

`Interval`メッセージの期待に適合する2つのタイムスタンプ間の範囲は、個別の開始フィールドと終了フィールドを持つのではなく、これを使用**すべきである**（should）。これにより、クライアントコードを`Interval`メッセージに対して記述し（特定のタイムスタンプが区間内にあるかのチェックなど）、同一API内の複数の区間、さらには複数のAPIにわたって再利用できる。

APIは代わりに開始および終了タイムスタンプフィールドを使用**してもよい**（may）。特に、API内のメッセージが本質的に、その区間に関する追加情報を含めて区間を記述している場合、`Interval`メッセージを使用することで生じる追加のネストのレベルが望ましくないことがある。

### 例外

場合によっては、包含的開始値と終了値（閉区間）について重要な慣習的な先例があり、排他的終了値を使用すると、排他的終了値に慣れている人々にとっても混乱を招く可能性がある。

例えば、日付（タイムスタンプと混同しないこと）について議論する場合、ほとんどの人は包含的終了を使用する：「April 21-23」という日程の会議は、April 21、April 22、April 23の3日間開催されることが期待される。これは曜日についても同様で、「Monday through Friday」に営業している事業者は金曜日も営業しており、閉店してはいない。

このような状況では、代わりに`first`および`last`プレフィックスを使用**すべきである**（should）：

```proto
// A representation of a chapter in a book.
message Chapter {
  string title = 1;

  // The first page of the chapter.
  int32 first_page = 2;

  // The last page of the chapter.
  int32 last_page = 3;
}
```

包含的開始値と終了値について重要な慣習的先例がある範囲を表すフィールドは、それらの範囲に限り、`first_`および`last_`プレフィックスを付けた包含的終了値を使用**すべきである**（should）。サービスは、これが適用されない他の範囲には依然として排他的終了値を使用**すべきであり**（should）、各範囲が包含的か排他的かを明確に文書化**しなければならない**（must）。
