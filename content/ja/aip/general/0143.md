# 標準化コード

言語、国、通貨など、多くの一般的な概念には、データ通信や処理で使用される共通のコード（通常は[国際標準化機構][iso]によって標準化されている）が存在する。これらのコードは、同じ概念を文章で表現する方法が複数存在する（例えば「United States」と「USA」、あるいは「Español」と「Spanish」）という問題に対処する。

## ガイダンス

標準化されたコードが存在し一般的に使用されている概念については、それらの概念を表すフィールドは、入力と出力の両方で標準化されたコードを使用**すべきである**（should）。

```proto
// A message representing a book.
message Book {
  // Other fields...

  // The IETF BCP-47 language code representing the language in which
  // the book was originally written.
  // https://en.wikipedia.org/wiki/IETF_language_tag
  string language_code = 99;
}
```

- 標準化された概念を表すフィールドは、標準コードに適したデータ型（通常は`string`）を使用**しなければならない**（must）。
  - 標準化された概念を表すフィールドは、たとえ値の小さなサブセットのみを許可する場合でも、enumを使用**すべきではない**（should not）。この状況でenumを使用すると、複数のAPIを一緒に使用する際に、しばしばイライラさせるルックアップテーブルにつながる。
  - 標準化された概念を表すフィールドは、どの標準に従っているかを示さ**なければならない**（must）。できればリンク（標準自体、Wikipediaの説明、または類似のものへのリンク）を付けること。
- フィールド名は、その概念に明らかにより明確な接尾辞がない限り、`_code`または`_type`で終わる**べきである**（should）。
- ユーザーが提供した値を受け付ける場合、バリデーションは、曖昧さが生じない限り（例えば、`en-gb`と`en-GB`の両方を受け入れる）、大文字小文字を区別**すべきではない**（should）。ユーザーに値を提供する場合、APIは正規の大文字小文字を使用**すべきである**（should）（上記の例では`en-GB`）。

### コンテンツタイプ

コンテンツまたはメディアタイプを表すフィールドは、[IANAメディアタイプ][iana media types]を使用**しなければならない**（must）。レガシーな理由から、フィールド名は`mime_type`と**すべきである**（should）。

### 国と地域

個々の国または国家を表すフィールドは、[Unicode CLDRリージョンコード][cldr]（[一覧][list]）（`US`や`CH`など）を使用**しなければならない**（must）、フィールド名は`region_code`と**しなければならない**（must）。

**重要:** この要件の[根拠](#countryregion-field-naming)を参照のこと。

### 通貨

通貨を表すフィールドは、[ISO-4217通貨コード][iso-4217]（`USD`や`CHF`など）を使用**しなければならない**（must）、フィールド名は`currency_code`と**しなければならない**（must）。

**注:** 通貨コードそのものではなく、特定の通貨での金額を表すには、[`google.type.Money`][money]を使用すること。

### 言語

話言葉を表すフィールドは、[IETF BCP-47言語コード][bcp-47]（[一覧][list]）（`en-US`や`de-CH`など）を使用**しなければならない**（must）、フィールド名は`language_code`と**しなければならない**（must）。

### タイムゾーン

タイムゾーンを表すフィールドは、[IANA TZ][iana tz]コードを使用**すべきであり**（should）、フィールド名は`time_zone`と**しなければならない**（must）。

フィールドは、タイムゾーンではなくUTCオフセットを表すことも**してもよい**（may）（これらは微妙に異なることに注意）。この場合、フィールドはUTCオフセットを表すために[ISO-8601形式][iso-8601 format]を使用**しなければならない**（must）、フィールド名は`utc_offset`と**しなければならない**（must）。

## 根拠

### 国/地域フィールドの命名

`country_code`ではなく`region_code`を使用することは、いかなる国とも異なる地域を伝達できるようにし、また当該地域の主権や帰属に関する政治的な紛争を避けるために極めて重要である。Googleおよび多くの企業はUnicode CLDRのサポーターであり、製品の国際化をUnicode CLDRに標準化している。APIもこれと同様である。さらに、Unicode CLDRがサポートする値の多くは単独の国ではないため、より汎用的な名前を使用することは実際には仕様との互換性が高まる。
