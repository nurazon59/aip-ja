# リソースタイプ

ほとんどのAPIは、ユーザーが作成、取得、操作できる_リソース_（主要な名詞）を公開する。APIは、このAIPの要件の範囲内で、リソースタイプにかなり自由に名前を付けることができ、そのAPI内での一意性のみを確保する必要がある。これは、異なるAPIが同じタイプ名を使用することが可能であり（そしてしばしば望ましい）であることを意味する。例えば、Memcache APIとRedis APIはどちらも`Instance`をタイプ名として使用したいと考えるだろう。

ただし、APIとそのリソース間の関係をマッピングする場合、単一のグローバルに一意なタイプ名を持つことが重要になる。さらに、KubernetesやGraphQLなどのツールは、複数のプロバイダーのAPIと対話する。

## 用語

以下のガイダンスでは、次の用語を使用する：

- **サービス名:** これは[サービス設定][service configuration]で定義される名前である。これは通常（必須ではないが）ユーザーがサービスを呼び出すために使用するホスト名と一致する。例：`pubsub.googleapis.com`。これはKubernetesの[API Group][]に相当する。
- **タイプ:** これはAPI内でタイプに使用される名前であり、例えばProtobufの`message`の名前である。これはKubernetesの[Object][]に相当する。

## ガイダンス

APIは、API内の各リソースに対して、次のパターンに従ってリソースタイプを定義**しなければならない**（must）：`{Service Name}/{Type}`。タイプ名は以下の条件を満たさ**なければならない**（must）：

- 含まれるAPIタイプの名前と一致する。
- 大文字で始まる。
- 英数字のみを含む。
- 名詞の単数形である。
- PascalCase（UpperCamelCase）を使用する。

### 例

リソースタイプの例：

- `pubsub.googleapis.com/Topic`
- `pubsub.googleapis.com/Subscription`
- `spanner.googleapis.com/Database`
- `spanner.googleapis.com/Instance`
- `networking.istio.io/Instance`

### リソースタイプのアノテーション

APIは、[`google.api.resource`][resource]アノテーションを使用して、API内の各リソースのリソースタイプをアノテーション**すべきである**（should）：

```proto
// A representation of a Pub/Sub topic.
message Topic {
  option (google.api.resource) = {
    type: "pubsub.googleapis.com/Topic"
    pattern: "projects/{project}/topics/{topic}"
    singular: "topic"
    plural: "topics"
  };

  // Name and other fields...
}
```

- パターンは[リソース名][aip-122]に対応**しなければならない**（must）。
- パターン変数（中括弧内のセグメント）は`snake_case`を使用**しなければならない**（must）、`_id`サフィックスを使用**してはならない**（must not）。
- パターン変数は形式`[a-z][_a-z0-9]*[a-z0-9]`に準拠**しなければならない**（must）。
- パターン変数は任意のパターン内で一意**でなければならない**（must）（例：`projects/{abc}/topics/{abc}`は無効。これは通常、コレクション識別子がパターン内で一意であることの自然な帰結である）。
- [複数パターン][multi-pattern-resources]を持つリソースは順序を維持**しなければならない**（must）：新しいパターンはリストの最後に追加され、既存のパターンは削除または再順序付け**されてはならない**（must not）。これはクライアントライブラリの後方互換性を壊すためである。
- Singular（単数形）はタイプのlower camel case**でなければならない**（must）。
  - パターン変数はリソースタイプの単数形**でなければならない**（must）。例：`Topic`リソースIDを表すパターン変数は`{topic}`という名前である。
- Plural（複数形）はsingularのlower camel case複数形**でなければならない**（must）。
  - パターンコレクション識別子セグメントは、[ネストされたコレクション][nested collections]の場合を除き、リソースの複数形と一致**しなければならない**（must）。

#### パターンの一意性

リソース内で複数のパターンが定義されている場合、これらのパターンは相互に一意**でなければならない**（must）。ここでの一意性とは、すべてのリソースIDパスセグメントを削除し、すべての`/`セパレータを残した後に、文字単位で同一であることと定義される。

したがって、以下の2つのパターンは同じリソース内で定義**されてはならない**（must not）：

- `user/{user}`
- `user/{user_part_1}~{user_part_2}`

## 根拠

### タイプとメッセージ名の一致

単純なスキーマとリソースの一貫性と整合性に加えて、多くのコンシューマーは`{Type}`と`message`名が一致することで利益を得る。コンシューマーはより単純なルックアップが可能になり、クライアントライブラリはそれに加えてリソース指向コードが生成された`message`コードと命名が一致するという統一されたユーザー体験を得られ、生成されるリファレンスドキュメントはリソースと`message`ドキュメントを一致させるなど、さまざまな利点がある。

### SingularとPlural

明確に定義されたリソースの単数形と複数形により、クライアントはコードとドキュメントで使用する適切な名前を決定できる。

lowerCamelCaseは、UpperCamelCaseやsnake_caseなど、リソース名の他の一般的な形式に変換可能である。
