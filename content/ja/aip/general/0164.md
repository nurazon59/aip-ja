# ソフトデリート

クライアントがソフトデリートと復元機能を望む理由はいくつかあるが、最も重要なのはミスからの回復である。復元をサポートするサービスは、ユーザーが誤って削除したリソースを復旧することを可能にする。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し復旧する必要がある状況に対応するために、「復元」機能をサポート**してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete`メソッドはリソースを削除済みとしてマークするにとどめ、システムから完全に削除しないことを**しなければならない**（must）。このメソッドがこのように動作する場合、`google.protobuf.Empty`ではなく更新されたリソースを返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148で説明されているように、`delete_time`と`purge_time`フィールドの両方を持つ**べきである**（should）。さらに、リソースに`state`フィールドがある場合（AIP-216）、リソースは`DELETED`状態値を含める**べきである**（should）。

### 復元

ソフトデリートをサポートするリソースは、`Undelete`メソッドを提供する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTPメソッドは`POST`で**なければならない**（must）。
- `body`句は`"*"`で**なければならない**（must）。
- レスポンスメッセージはリソースそのもので**なければならない**（must）。`UndeleteBookResponse`は存在しない。
  - レスポンスには、実行不可能でない限り、完全にデータが設定されたリソースを含める**べきである**（should）。
  - 復元RPCが[長時間実行](#long-running-undelete)の場合、レスポンスメッセージはリソースそのものに解決される`google.longrunning.Operation`で**なければならない**（must）。

### 復元リクエストメッセージ

復元メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name`フィールドを含め**なければならない**（must）。フィールド名は`name`とす**べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]する**べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する**べきである**（should）。
  - フィールドのコメントにはリソースパターンを記載する**べきである**（should）。
- リクエストメッセージには他の必須フィールドを含めては**ならない**（must not）。また、このAIPまたは他のAIPで説明されているものを除き、他のオプショナルフィールドを含める**べきではない**（should not）。

### 長時間実行の復元

一部のリソースの復元には、通常のAPIリクエストに妥当な時間よりも長い時間がかかる。このような状況では、APIは代わりに長時間実行オペレーション（AIP-151）を使用する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPCが長時間実行でなかった場合の戻り値の型）に設定され**なければならない**（must）。
- `response_type`と`metadata_type`フィールドの両方を指定し**なければならない**（must）。

### 強制削除

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除をトリガーできるようにする`Expunge`カスタムメソッドを提供**してもよい**（may）。このメソッドは、現在`CREATING`、`READY`、または`SOFT_DELETED`状態にあるリソース（例：`delete_time`が設定されている）に対して動作することができる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URIは`:expunge`サフィックスを持つカスタムメソッドを使用しなければならない。
- HTTPメソッドは`POST`でなければならず、`body`句は`"*"`でなければならない。
- レスポンスメッセージは`google.protobuf.Empty`または`google.longrunning.Operation`でなければならない。

### 長時間実行の強制削除

強制削除プロセスにかなりの時間がかかる場合、メソッドは`google.longrunning.Operation`（AIP-151）に**してもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### 強制削除リクエストメッセージ

強制削除メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、強制削除するリソースを名前で参照し**なければならない**（must）。
- 他のリクエストフィールドは存在す**べきではない**（should not）。

### ListとGet

ソフトデリートされたリソースは、デフォルトでは`List`（AIP-132）レスポンスで返される**べきではない**（should not）（`bool show_deleted`がtrueの場合を除く）。ソフトデリートされたリソースに対する`Get`（AIP-131）リクエストは、リソースを返す**べきである**（should）（`NOT_FOUND`エラーではなく）。

ソフトデリートするAPIは、リソースを完全に削除するための合理的な戦略を選択**してもよい**（may）。妥当な期間（30日など）後の自動削除、ユーザーによる有効期限の設定（AIP-214）、またはリソースの無期限保持が含まれる。どの戦略を選択する場合でも、APIはソフトデリートされたリソースがいつ完全に削除されるかを文書化する**べきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいてハードデリートされたリソースよりも体験が劣る。ソフトデリートされたリソースのIDは、カスタムメソッド（復元）が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタムメソッドの使用を組み込むための手書きコードが必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持っていない場合、リソースの存在有無に関わらず、サービスは`PERMISSION_DENIED`（HTTP 403）エラーを返さ**なければならない**（must）。権限のチェックは、リソースの存在確認よりも前に行われ**なければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合（作成されたことがないか、すでに強制削除されている）、サービスは`NOT_FOUND`（HTTP 404）エラーを返さ**なければならない**（must）。

ソフト`Delete`を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、サービスは`allow_missing`が`true`であれば成功し**なければならない**（must）。`allow_missing`が`false`であれば`NOT_FOUND`（HTTP 404）エラーを返す**べきである**（should）。

`Undelete`を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは`ALREADY_EXISTS`（HTTP 409）を返さ**なければならない**（must）。

`Expunge`を呼び出すユーザーが、存在しないリソース（作成されたことがないか、すでに強制削除されている）を要求した場合、メソッドは`NOT_FOUND`（HTTP 404）を返さ**なければならない**（must）。

リソースが存在するが準備完了（ready）状態またはソフトデリート状態でない場合、メソッドは`FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別に、明示的な強制削除権限（例：`<service>.<resource>.expunge`）を要求し**なければならない**（must）。
