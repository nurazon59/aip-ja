# ソフトデリート

クライアントがソフトデリートおよび復元（undelete）機能を望む理由はいくつかあるが、最も重要な理由の一つは「ミスからの回復」である。復元をサポートするサービスは、ユーザーが誤って削除したリソースを回復することを可能にする。

## ガイダンス

API は、ユーザーが誤ってリソースを削除し、回復する必要がある状況に対応するために、復元（"undelete"）機能をサポート **してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドは、リソースを削除済みとしてマークするだけで、システムから完全には削除 **してはならない**（must not）。メソッドがこのように動作する場合、`google.protobuf.Empty` の代わりに更新後のリソースを返す **べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されているように、`delete_time` フィールドと `purge_time` フィールドの両方を持つ **べきである**（should）。さらに、リソースに `state` フィールドが含まれている場合（AIP-216）、リソースは `DELETED` 状態値を含める **べきである**（should）。

### 復元（Undelete）

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP メソッドは `POST` で **なければならない**（must）。
- `body` 句は `"*"` で **なければならない**（must）。
- レスポンスメッセージはリソースそのもので **なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - レスポンスには、実行不可能でない限り、完全に設定されたリソースを含める **べきである**（should）。
  - 復元 RPC が[長時間実行](#long-running-undelete)の場合、レスポンスメッセージは `google.longrunning.Operation` で **なければならず**（must）、その結果はリソースそのものに解決される。

### 復元リクエストメッセージ

復元メソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め **なければならない**（must）。`name` という名前にする **べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]する **べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する **べきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する **べきである**（should）。
- リクエストメッセージには、他の必須フィールドを含めては **ならず**（must not）、この AIP や他の AIP で説明されているものを除き、他のオプショナルフィールドを含める **べきではない**（should not）。

### 長時間実行の復元

一部のリソースでは、通常の API リクエストに適切な時間よりも長く復元に時間がかかる場合がある。この状況では、API は代わりに長時間実行オペレーション（AIP-151）を使用する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソースに設定し **なければならない**（must）（RPC が長時間実行でない場合の戻り値の型）。
- `response_type` と `metadata_type` の両方のフィールドを指定し **なければならない**（must）。

### 完全削除（Expunge）

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除をトリガーできるようにする `Expunge` カスタムメソッドを提供 **してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態（例：`delete_time` が設定されている）にあるリソースに対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックス付きのカスタムメソッドを使用しなければならない。
- HTTP メソッドは `POST`、`body` 句は `"*"` でなければならない。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` でなければならない。

### 長時間実行の完全削除

完全削除プロセスにかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）でも **よい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### 完全削除リクエストメッセージ

完全削除メソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、完全削除するリソースを名前で参照し **なければならない**（must）。
- 他のリクエストフィールドは存在 **すべきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは `List`（AIP-132）のレスポンスに返される **べきではない**（should not）（`bool show_deleted` が true の場合を除く）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、リソースを返す **べきである**（should）（`NOT_FOUND` エラーではなく）。

ソフトデリートを行う API は、それらのリソースを完全に削除するための合理的な戦略を選択 **してもよい**（may）。これには、妥当な期間（例：30日）経過後の自動完全削除、ユーザーによる有効期限の設定許可（AIP-214）、またはリソースの無期限保持が含まれる。選択された戦略に関わらず、API はソフトデリートされたリソースがいつ完全に削除されるかを文書化する **べきである**（should）。

### 宣言的フレンドリーなリソース

ソフトデリート可能なリソースは、ハードデリートされたリソースと比較して、宣言的クライアントでの体験が劣る。ソフトデリートされたリソースの ID は、カスタムメソッド（復元）が呼び出されない限り再利用できないため、命令的クライアントを導入するか、カスタムメソッドの使用を組み込むために手書きのコードが必要になる。

### エラー

リソースにアクセスする権限がユーザーにない場合、リソースの存在の有無に関わらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーとし **なければならない**（must）。権限は、リソースの存在確認よりも先にチェックし **なければならない**（must）。

ユーザーが適切な権限を持っているが、リクエストされたリソースが存在しない場合（一度も作成されていないか、すでに完全削除されている場合）、サービスは `NOT_FOUND`（HTTP 404）でエラーとし **なければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、リクエストされたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功し **なければならず**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーとする **べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、リクエストされたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答し **なければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（一度も作成されていないか、すでに完全削除されている）をリクエストした場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ **なければならない**（must）。

リソースが存在するが Ready 状態でもソフトデリート状態でもない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ **なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的な完全削除権限（例：`<service>.<resource>.expunge`）を要求し **なければならない**（must）。
