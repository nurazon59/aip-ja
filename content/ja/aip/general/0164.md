# ソフトデリート

クライアントがソフトデリートとアンデリート機能を望む理由はいくつかあるが、その中でも際立っているのがミスからの回復という理由である。アンデリートをサポートするサービスは、ユーザーが誤って削除したリソースを復旧することを可能にする。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し、復旧する必要がある状況に対応するため、「アンデリート」機能をサポート**してもよい**（may）。

リソースがアンデリートをサポートする必要がある場合、`Delete` メソッドはリソースを削除済みとしてマーク**しなければならない**（must）が、システムから完全に削除**してはならない**（must not）。メソッドがこのように動作する場合、`google.protobuf.Empty` ではなく更新されたリソースを返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されているように `delete_time` と `purge_time` フィールドの両方を持つ**べきである**（should）。さらに、リソースに `state` フィールド（AIP-216）が含まれている場合、`DELETED` 状態値を含める**べきである**（should）。

### アンデリート

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージはリソース自体で**なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - 実行不可能でない限り、レスポンスは完全に埋められたリソースを含める**べきである**（should）。
  - アンデリート RPC が [long-running](#long-running-undelete) である場合、レスポンスメッセージはリソース自体に解決される `google.longrunning.Operation` で**なければならない**（must）。

### アンデリートリクエストメッセージ

アンデリートメソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。これは `name` と呼ばれる**べきである**（should）。
  - フィールドは [required としてアノテーション][aip-203] される**べきである**（should）。
  - フィールドは参照する [リソースタイプ][aip-123] を識別する**べきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する**べきである**（should）。
- リクエストメッセージは他の必須フィールドを含め**てはならない**（must not）、またこの AIP または他の AIP で説明されているものを除き、他のオプショナルフィールドを含める**べきではない**（should not）。

### long-running アンデリート

一部のリソースは、通常の API リクエストに適切な時間よりもアンデリートに長い時間がかかる。この状況では、API は代わりに long-running operation（AIP-151）を使用する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPC が long-running でない場合の戻り値の型）に設定**しなければならない**（must）。
- `response_type` と `metadata_type` フィールドの両方を指定**しなければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時の完全な削除をトリガーできるようにする `Expunge` カスタムメソッドを提供**してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態（例：`delete_time` が設定されている）にあるリソースに対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は :expunge サフィックスを持つカスタムメソッドを使用し**なければならない**（must）。
- HTTP 動詞は POST で**なければならない**（must）、`body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` で**なければならない**（must）。

### long-running expunge

expunge 処理にかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）**にしてもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは expunge 対象のリソースを名前で参照**しなければならない**（must）。
- 他のリクエストフィールドは存在**すべきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは `List`（AIP-132）レスポンスで返される**べきではない**（should not）（`bool show_deleted` が true の場合を除く）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、リソースを返す**べきである**（should）（`NOT_FOUND` エラーではなく）。

リソースをソフトデリートする API は、それらのリソースを完全削除するための合理的な戦略を選択**してもよい**（may）。例えば、合理的な期間（30日など）後の自動完全削除、ユーザーによる有効期限（AIP-214）の設定、またはリソースの無期限保持が考えられる。どの戦略が選択されるかにかかわらず、API はソフトデリートされたリソースがいつ完全に削除されるかを文書化する**べきである**（should）。

### 宣言的フレンドリーなリソース

ソフト削除可能なリソースは、ハードデリートのリソースと比較して宣言的クライアントでのエクスペリエンスが劣る。ソフトデリートされたリソースの ID はカスタムメソッド（アンデリート）が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタムメソッドの使用を組み込むための手書きコードが必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持っていない場合、そのリソースが存在するかどうかにかかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラー**しなければならない**（must）。権限はリソースの存在を確認する前にチェック**しなければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合（作成されたことがないか、すでに expunge されている）、サービスは `NOT_FOUND`（HTTP 404）でエラー**しなければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功**しなければならない**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーする**べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答**しなければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（作成されたことがないか、すでに expunge されている）を要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ**なければならない**（must）。

リソースが存在するが ready またはソフトデリートされた状態にない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準的な削除権限とは別の明示的な expunge 権限（例：`<service>.<resource>.expunge`）を要求**しなければならない**（must）。
