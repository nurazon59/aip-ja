# ソフト削除

クライアントがソフト削除と復元機能を望む理由はいくつかあるが、最も包括的な理由はミスからの回復である。復元をサポートするサービスは、ユーザーが誤って削除したリソースを回復することを可能にする。

## ガイダンス

APIは、ユーザーがリソースを誤って削除し、回復する必要がある状況に対応するため、「復元」機能をサポート **してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドは、リソースをシステムから完全に削除するのではなく、単に削除済みとしてマーク **しなければならない**（must）。メソッドがこのように動作する場合、`google.protobuf.Empty` ではなく更新されたリソースを返す **べきである**（should）。

ソフト削除をサポートするリソースは、AIP-148 で説明されているように、`delete_time` と `purge_time` の両方のフィールドを持つ **べきである**（should）。さらに、リソースが `state` フィールドを含む場合（AIP-216）、リソースは `DELETED` 状態値を含める **べきである**（should）。

### 復元

ソフト削除をサポートするリソースは、`Undelete` メソッドを提供 **すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で **なければならない**（must）。
- `body` 句は `"*"` で **なければならない**（must）。
- レスポンスメッセージはリソースそのもので **なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - 実行不可能でない限り、レスポンスには完全に設定されたリソースを含める **べきである**（should）。
  - 復元 RPC が[長時間実行](#long-running-undelete)の場合、レスポンスメッセージは `google.longrunning.Operation` で **なければならず**（must）、これはリソースそのものに解決される。

### 復元リクエストメッセージ

Undelete メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め **なければならない**（must）。これは `name` という名前にする **べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]する **べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する **べきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する **べきである**（should）。
- リクエストメッセージには他の必須フィールドを含め **てはならず**（must not）、この AIP または他の AIP で説明されているものを除き、他のオプショナルフィールドを含める **べきではない**（should not）。

### 長時間実行復元

一部のリソースは、通常の API リクエストに適した時間よりも復元に時間がかかる。この状況では、API は代わりに長時間実行オペレーション（AIP-151）を使用 **すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPC が長時間実行でなかった場合の戻り値の型）に設定 **しなければならない**（must）。
- `response_type` と `metadata_type` の両方のフィールドを指定 **しなければならない**（must）。

### Expunge

ソフト削除をサポートするリソースは、ユーザーがリソースの即時の完全削除をトリガーできるようにする `Expunge` カスタムメソッドを提供 **してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態（例：`delete_time` が設定されている）にあるリソースに対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタムメソッドを使用 **しなければならない**（must）。
- HTTP 動詞は `POST`、`body` 句は `"*"` で **なければならない**（must）。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` で **なければならない**（must）。

### 長時間実行 Expunge

Expunge プロセスにかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）に **してもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、Expunge 対象のリソースを名前で参照 **しなければならない**（must）。
- 他のリクエストフィールドは存在 **すべきではない**（should not）。

### List と Get

ソフト削除されたリソースは、デフォルトでは（`bool show_deleted` が true でない限り）`List`（AIP-132）レスポンスで返される **べきではない**（should not）。ソフト削除されたリソースに対する `Get`（AIP-131）リクエストは、（`NOT_FOUND` エラーではなく）リソースを返す **べきである**（should）。

リソースをソフト削除する API は、合理的な期間（30日など）後の自動パージ、ユーザーが有効期限（AIP-214）を設定できるようにすること、またはリソースを無期限に保持することなど、それらのリソースをパージするための合理的な戦略を選択 **してもよい**（may）。どの戦略を選択するかにかかわらず、API はソフト削除されたリソースがいつ完全に削除されるかを文書化 **すべきである**（should）。

### 宣言的フレンドリーなリソース

ソフト削除可能なリソースは、宣言的クライアントにおいてハード削除されたリソースよりも劣るエクスペリエンスを持つ。ソフト削除されたリソースの ID は、カスタムメソッド（復元）が呼び出されない限り再利用できないため、命令的クライアントを導入するか、カスタムメソッドの使用を組み込むための手書きコードが必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持たない場合、そのリソースが存在するかどうかにかかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーに **しなければならない**（must）。権限は、リソースが存在するかどうかを確認する前にチェック **しなければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない（一度も作成されていないか、すでに Expunge されている）場合、サービスは `NOT_FOUND`（HTTP 404）でエラーに **しなければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功 **しなければならず**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーに **すべきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答 **しなければならない**（must）。

`Expunge` を呼び出すユーザーが存在しない（一度も作成されていないか、すでに Expunge されている）リソースを要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ **なければならない**（must）。

リソースが存在するが、ready またはソフト削除状態でない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ **なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的な Expunge 権限（例：`<service>.<resource>.expunge`）を要求 **しなければならない**（must）。
