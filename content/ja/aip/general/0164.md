# Soft delete

クライアントがソフトデリートおよびアンデリート機能を望む理由はいくつかあるが、最も重要な理由はただ一つ：ミスからの回復である。アンデリートをサポートするサービスは、ユーザーが誤って削除したリソースを回復することを可能にする。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し回復する必要がある状況に対応するため、アンデリート機能をサポート **してもよい**（may）。

リソースがアンデリートをサポートする必要がある場合、`Delete`メソッドはリソースを削除済みとしてマークするのみで、システムから完全に削除**してはならない**（must not）。このメソッドがそのように動作する場合、`google.protobuf.Empty`ではなく更新されたリソースを返す**すべきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148に記載されているように、`delete_time`と`purge_time`の両方のフィールドを持つ**すべきである**（should）。さらに、リソースが`state`フィールド（AIP-216）を含む場合、`DELETED`状態値を含める**すべきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete`メソッドを提供する**すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP動詞は`POST`で**しなければならない**（must）。
- `body`句は`"*"`で**しなければならない**（must）。
- レスポンスメッセージはリソース自体で**しなければならない**（must）。`UndeleteBookResponse`は存在しない。
  - レスポンスは完全に設定されたリソースを含める**すべきである**（should）（それが不可能でない限り）。
  - Undelete RPCが[long-running](#long-running-undelete)の場合、レスポンスメッセージは`google.longrunning.Operation`で**しなければならない**（must）。これはリソース自体に解決される。

### Undeleteリクエストメッセージ

Undeleteメソッドは共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // 削除されたbookの名前。
  // 形式: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name`フィールドを含め**しなければならない**（must）。`name`という名前にする**すべきである**（should）。
  - フィールドは[requiredとしてアノテーションされる][aip-203]**すべきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する**すべきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する**すべきである**（should）。
- リクエストメッセージは他の必須フィールドを含んでは**してはならず**（must not）、また本AIPまたは別のAIPに記載されているものを除き、他のオプショナルフィールドを含める**すべきではない**（should not）。

### Long-running undelete

一部のリソースでは、リソースのアンデリートに通常のAPIリクエストとして妥当な時間以上かかる場合がある。この状況では、APIは代わりにlong-running operation（AIP-151）を使用する**すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPCがlong-runningでない場合の戻り値の型）に設定**しなければならない**（must）。
- `response_type`と`metadata_type`の両方のフィールドを指定**しなければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがreadyまたはsoft-deleteされたリソースの即時完全削除をトリガーできるようにする`Expunge`カスタム メソッドを提供**してもよい**（may）。このメソッドは現在readyまたはsoft-delete状態（例：`delete_time`が設定されている）のリソースに対して動作できる。

```proto
// ソフトデリートされたBookを完全に削除する。
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URIは`:expunge`サフィックス付きのカスタム メソッドを使用しなければならない。
- HTTP動詞はPOST、body句は"*"でなければならない。
- レスポンスメッセージは`google.protobuf.Empty`または`google.longrunning.Operation`でなければならない。

### Long-running expunge

expunge処理に時間がかかる場合、メソッドは代わりに`google.longrunning.Operation`（AIP-151）**してもよい**（may）：

```proto
// ソフトデリートされたBookを完全に削除する。
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expungeリクエストメッセージ

Expungeメソッドは共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // 削除（expunge）するソフトデリートされたbookの名前。
  // 形式: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは名前によって削除対象のリソースを参照**しなければならない**（must）。
- 他のリクエストフィールドは存在**すべきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは`List`（AIP-132）レスポンスに返される**すべきではない**（should not）（`bool show_deleted`がtrueの場合を除く）。ソフトデリートされたリソースへの`Get`（AIP-131）リクエストは、リソースを返す**すべきである**（should）（`NOT_FOUND`エラーではなく）。

ソフトデリートを実装するAPIは、リソースをパージするための合理的な戦略を選択**してもよい**（may）。これには、合理的な期間（30日など）後の自動パージ、ユーザーが有効期限を設定できるようにする（AIP-214）、またはリソースを無期限に保持することが含まれる。選択した戦略に関係なく、APIはソフトデリートされたリソースがいつ完全に削除されるかを文書化する**すべきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいてハードデリートされたリソースよりも劣った体験を提供する：ソフトデリートされたリソースのIDは、カスタム メソッド（undelete）が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタム メソッドの使用を組み込むための手書きコードが必要となる。

### エラー

ユーザーがリソースへのアクセス権限を持っていない場合（リソースの存在有無にかかわらず）、サービスは`PERMISSION_DENIED`（HTTP 403）でエラーに**しなければならない**（must）。権限はリソースの存在を確認する前にチェック**しなければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合（作成されたことがない、またはすでにexpungeされた）、サービスは`NOT_FOUND`（HTTP 404）でエラーに**しなければならない**（must）。

ソフト`Delete`を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、サービスは`allow_missing`が`true`の場合は成功**しなければならない**（must）、`allow_missing`が`false`の場合は`NOT_FOUND`（HTTP 404）でエラーにする**すべきである**（should）。

`Undelete`を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは`ALREADY_EXISTS`（HTTP 409）で応答**しなければならない**（must）。

`Expunge`を呼び出すユーザーが存在しないリソース（作成されたことがない、またはすでにexpungeされた）を要求した場合、メソッドは`NOT_FOUND`（HTTP 404）を返す**しなければならない**（must）。

リソースが存在するがreadyまたはsoft-delete状態でない場合、メソッドは`FAILED_PRECONDITION`（HTTP 400）を返す**しなければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的なexpunge権限（例：`<service>.<resource>.expunge`）を要求**しなければならない**（must）。
