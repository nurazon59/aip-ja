# ソフトデリート

ソフトデリートと復元（undelete）機能を必要とする理由はいくつかあるが、
最も重要な理由はミスからの回復である。復元をサポートするサービスでは、
ユーザーが誤って削除したリソースを復旧できる。

## ガイダンス

API は、ユーザーが誤ってリソースを削除した場合の回復を可能にするために、
「復元（undelete）」機能をサポート**してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドは単にリソースを
削除済みとしてマークし、システムから完全に削除してはならない。このように動作する場合、
`google.protobuf.Empty` ではなく、更新後のリソースを返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されているように、
`delete_time` と `purge_time` の両方のフィールドを持つ**べきである**（should）。
さらに、リソースに `state` フィールド（AIP-216）が含まれる場合、`DELETED` 状態値を
含める**べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する**べきである**（should）:

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージはリソース自体で**なければならない**（must）。
  `UndeleteBookResponse` は存在しない。
  - レスポンスには、実行不可能でない限り、完全に設定されたリソースを含める
    **べきである**（should）。
  - undelete RPC が[長時間実行](#long-running-undelete)の場合、レスポンスメッセージは
    `google.longrunning.Operation` で**なければならず**（must）、リソース自体に解決される。

### Undelete リクエストメッセージ

Undelete メソッドは共通のリクエストメッセージパターンを実装する:

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。`name` と呼ぶ**べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]する**べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する**べきである**（should）。
  - フィールドのコメントにはリソースパターンをドキュメント化する**べきである**（should）。
- リクエストメッセージには他の必須フィールドを含めて**はならず**（must not）、
  この AIP または他の AIP で説明されているものを除き、他のオプションフィールドを
  含める**べきではない**（should not）。

### 長時間実行 Undelete

一部のリソースでは、undelete に通常の API リクエストとしては妥当でない時間が
かかる場合がある。このような状況では、API は代わりに長時間実行オペレーション
（AIP-151）を使用する**べきである**（should）:

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPC が長時間実行でない場合の戻り値型）に設定
  し**なければならない**（must）。
- `response_type` と `metadata_type` の両方のフィールドを指定し**なければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースを即座に完全削除できるようにする
`Expunge` カスタムメソッドを提供**してもよい**（may）。このメソッドは、現在 `CREATING`、
`READY` または `SOFT_DELETED` 状態（例: `delete_time` が設定されている）のリソースに
対して動作する。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタムメソッドを使用しなければならない。
- HTTP 動詞は POST でなければならず、body 句は `"*"` でなければならない。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` でなければならない。

### 長時間実行 Expunge

expunge プロセスにかなりの時間がかかる場合、代わりに `google.longrunning.Operation`
（AIP-151）を**してもよい**（may）:

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは共通のリクエストメッセージパターンを実装する:

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expunge 対象のリソースを名前で参照し**なければならない**（must）。
- 他のリクエストフィールドがある**べきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは `List`（AIP-132）レスポンスに
含め**るべきではない**（should not）（`bool show_deleted` が true の場合を除く）。
ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、リソースを
返す**べきである**（should）（`NOT_FOUND` エラーではなく）。

ソフトデリートを行う API は、それらのリソースを削除するための合理的な戦略を
選択**してもよい**（may）。例えば、合理的な期間（30 日など）後の自動削除、
ユーザーによる有効期限の設定（AIP-214）、またはリソースの無期限保持などがある。
どの戦略を選択する場合でも、API はソフトデリートされたリソースがいつ完全に
削除されるかをドキュメント化する**べきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、ハードデリートされたリソースよりも宣言的クライアント
にとって体験が劣る。ソフトデリートされたリソースの ID は、カスタムメソッド（undelete）
が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタムメソッドの
使用を組み込むための手書きコードが必要になる。

### エラー

リソースの存在有無にかかわらず、ユーザーがリソースにアクセスする権限を持たない場合、
サービスは `PERMISSION_DENIED`（HTTP 403）でエラーにし**なければならない**（must）。
権限チェックはリソースの存在確認よりも先に行わ**なければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合（作成されたことがない、
または既に expunge された場合）、サービスは `NOT_FOUND`（HTTP 404）でエラーに
し**なければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが既に
削除されている場合、`allow_missing` が `true` であればサービスは成功し**なければならず**（must）、
`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーにする**べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されて
いない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答し**なければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（作成されたことがない、または既に
expunge された）を要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ
**なければならない**（must）。

リソースが存在するが ready または soft-deleted 状態でない場合、メソッドは
`FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の delete
権限とは別の明示的な expunge 権限（例: `<service>.<resource>.expunge`）を要求
し**なければならない**（must）。
