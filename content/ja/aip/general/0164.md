# ソフトデリート

クライアントがソフトデリートと復元機能を望む理由はいくつかあるが、その中でも最も重要な理由は、ミスからの回復である。復元をサポートするサービスでは、ユーザーが誤って削除したリソースを回復することが可能になる。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し、回復する必要がある状況に対応するために、「復元」機能をサポート**してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドはリソースをシステムから完全に削除するのではなく、単に削除済みとしてマークし**なければならない**（must）。メソッドがこのように動作する場合、`google.protobuf.Empty` ではなく更新されたリソースを返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されているように `delete_time` フィールドと `purge_time` フィールドの両方を持つ**べきである**（should）。さらに、リソースが `state` フィールドを含む場合（AIP-216）、`DELETED` 状態値も含む**べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは `Undelete` メソッドを提供する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンス メッセージはリソースそのもので**なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - レスポンスには、実現不可能でない限り、完全に設定されたリソースを含める**べきである**（should）。
  - 復元 RPC が [long-running](#long-running-undelete) の場合、レスポンス メッセージはリソースそのものに解決される `google.longrunning.Operation` で**なければならない**（must）。

### Undelete リクエスト メッセージ

Undelete メソッドは共通のリクエスト メッセージ パターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。`name` という名前にする**べきである**（should）。
  - フィールドは [required としてアノテーション][aip-203]する**べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する**べきである**（should）。
  - フィールドのコメントにはリソース パターンを記載する**べきである**（should）。
- リクエスト メッセージには、これ以外の必須フィールドを含め**てはならない**（must not）。また、この AIP または他の AIP で説明されているものを除き、他のオプション フィールドを含める**べきではない**（should not）。

### Long-running undelete

一部のリソースは、通常の API リクエストに適した時間よりも長い時間を復元に要する。このような状況では、API は代わりに long-running operation（AIP-151）を使用する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンス タイプはリソースに設定し**なければならない**（must）（RPC が long-running でなかった場合の戻り値の型となるもの）。
- `response_type` と `metadata_type` の両方のフィールドを指定し**なければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除を実行できるようにするための `Expunge` カスタム メソッドを提供**してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態にあるリソース（例：`delete_time` が設定されている）に対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタム メソッドを使用しなければならない。
- HTTP 動詞は POST、body 句は `"*"` でなければならない。
- レスポンス メッセージは `google.protobuf.Empty` または `google.longrunning.Operation` でなければならない。

### Long-running expunge

Expunge プロセスにかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）に**してもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエスト メッセージ

Expunge メソッドは共通のリクエスト メッセージ パターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエスト メッセージは、削除対象のリソースを名前で参照し**なければならない**（must）。
- 他のリクエスト フィールドを含める**べきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは（`bool show_deleted` が true の場合を除き）`List`（AIP-132）のレスポンスで返す**べきではない**（should not）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、（`NOT_FOUND` エラーではなく）リソースを返す**べきである**（should）。

ソフトデリートを行う API は、これらのリソースをパージするための合理的な戦略を選択**してもよい**（may）。これには、適切な期間（例：30日）後の自動パージ、ユーザーによる有効期限の設定（AIP-214）、またはリソースの無期限保持が含まれる。どの戦略を選択する場合でも、API はソフトデリートされたリソースがいつ完全に削除されるかを文書化する**べきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいてハードデリートされたリソースよりも劣る体験をもたらす。ソフトデリートされたリソースの ID は、カスタム メソッド（undelete）が呼び出されない限り再利用できないため、宣言的クライアントに命令型クライアントを導入するか、カスタム メソッドの使用を組み込むための手書きコードが必要になる。

### エラー

ユーザーがリソースへのアクセス権限を持たない場合、リソースの存在有無にかかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーとし**なければならない**（must）。権限チェックは、リソースの存在確認より先に実行し**なければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない（一度も作成されていないか、すでに完全削除されている）場合、サービスは `NOT_FOUND`（HTTP 404）でエラーとし**なければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功し**なければならない**（must）。また、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーとする**べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答し**なければならない**（must）。

`Expunge` を呼び出すユーザーが存在しない（一度も作成されていないか、すでに完全削除されている）リソースを要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ**なければならない**（must）。

リソースが存在するが `CREATING`、`READY`、または `SOFT_DELETED` 状態でない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的な完全削除権限（例：`<service>.<resource>.expunge`）を要求し**なければならない**（must）。
