# ソフト削除

クライアントがソフト削除および削除取消し機能を望む理由はいくつかあるが、
最も重要な理由はミスからの回復である。削除取消しをサポートするサービスは、
ユーザーが誤って削除したリソースを回復することを可能にする。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し回復する必要がある状況に対応するために、
「削除取消し」機能をサポート**してもよい**（may）。

リソースが削除取消しをサポートする必要がある場合、`Delete`メソッドはリソースを
削除済みとしてマークするだけで、システムから完全に削除し**なければならない**（must）。
メソッドがこのように動作する場合、`google.protobuf.Empty`ではなく更新された
リソースを返す**べきである**（should）。

ソフト削除をサポートするリソースは、AIP-148で説明されている`delete_time`フィールドと
`purge_time`フィールドの両方を持つ**べきである**（should）。さらに、リソースに
`state`フィールド（AIP-216）が含まれている場合、`DELETED`状態値を含める
**べきである**（should）。

### 削除取消し

ソフト削除をサポートするリソースは、`Undelete`メソッドを提供**すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTPメソッドは`POST`で**なければならない**（must）。
- `body`句は`"*"`で**なければならない**（must）。
- レスポンスメッセージはリソースそのもので**なければならない**（must）。
  `UndeleteBookResponse`は存在しない。
  - レスポンスは、それが実行不可能でない限り、完全に設定されたリソースを含む
    **べきである**（should）。
  - 削除取消しRPCが[長時間実行](#long-running-undelete)の場合、レスポンス
    メッセージはリソース自体に解決される`google.longrunning.Operation`で
    **なければならない**（must）。

### 削除取消しリクエストメッセージ

削除取消しメソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name`フィールドを含め**なければならない**（must）。これは`name`と呼ばれる
  **べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]される**べきである**（should）。
  - フィールドは参照する[リソース型][aip-123]を識別する**べきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する**べきである**（should）。
- リクエストメッセージには他の必須フィールドを含め**てはならず**（must not）、
  このAIPまたは別のAIPで説明されているものを除き、他のオプショナルフィールドを
  含める**べきではない**（should not）。

### 長時間実行の削除取消し

一部のリソースは、通常のAPIリクエストに合理的な時間よりも長い削除取消し時間を
要する。この状況では、APIは代わりにlong-running operation（AIP-151）を使用
**すべきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンス型はリソースに設定**しなければならない**（must）（RPCが長時間実行
  でない場合の戻り値の型）。
- `response_type`と`metadata_type`の両方のフィールドを指定**しなければならない**
  （must）。

### Expunge

ソフト削除をサポートするリソースは、ユーザーがリソースの即時恒久削除を
トリガーできるようにする`Expunge`カスタムメソッドを提供**してもよい**（may）。
このメソッドは、現在`CREATING`、`READY`、または`SOFT_DELETED`状態にある
リソース（例：`delete_time`が設定されている）に対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URIは:expungeサフィックスを伴うカスタムメソッドを使用し**なければならない**（must）。
- HTTPメソッドはPOSTで、body句は"*"で**なければならない**（must）。
- レスポンスメッセージは`google.protobuf.Empty`または
  `google.longrunning.Operation`で**なければならない**（must）。

### 長時間実行のexpunge

expungeプロセスにかなりの時間がかかる場合、メソッドは代わりに
`google.longrunning.Operation`（AIP-151）であっ**てもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expungeリクエストメッセージ

Expungeメソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expungeするリソースを名前で参照
  **しなければならない**（must）。
- 他のリクエストフィールドは存在**すべきではない**（should not）。

### ListとGet

ソフト削除されたリソースは、デフォルトでは（`bool show_deleted`がtrueの場合を
除き）`List`（AIP-132）レスポンスで返される**べきではない**（should not）。
ソフト削除されたリソースに対する`Get`（AIP-131）リクエストは、
（`NOT_FOUND`エラーではなく）リソースを返す**べきである**（should）。

リソースをソフト削除するAPIは、合理的な期間（30日など）後の自動パージ、
ユーザーによる有効期限（AIP-214）の設定、またはリソースの無期限保持を含む、
それらのリソースをパージするための合理的な戦略を選択**してもよい**（may）。
どの戦略を選択するかにかかわらず、APIはソフト削除されたリソースがいつ完全に
削除されるかを文書化**すべきである**（should）。

### 宣言的フレンドリーなリソース

宣言的クライアントにおいて、ソフト削除可能なリソースはハード削除されたリソースより
も体験が劣る。ソフト削除されたリソースのIDはカスタムメソッド（削除取消し）が
呼び出されない限り再利用できないため、命令的クライアントを導入するか、カスタム
メソッドの使用を組み込むために手書きのコードが必要になる。

### エラー

ユーザーがリソースへのアクセス権限を持たない場合、リソースが存在するかどうかに
かかわらず、サービスは`PERMISSION_DENIED`（HTTP 403）でエラーに
**しなければならない**（must）。権限はリソースの存在確認よりも先にチェック
**しなければならない**（must）。

ユーザーが適切な権限を持っていても、リクエストされたリソースが存在しない
（作成されたことがないか既にexpungeされている）場合、サービスは
`NOT_FOUND`（HTTP 404）でエラーに**しなければならない**（must）。

ソフト`Delete`を呼び出すユーザーが適切な権限を持っているが、リクエストされた
リソースが既に削除されている場合、`allow_missing`が`true`であればサービスは
成功**しなければならず**（must）、`allow_missing`が`false`であれば
`NOT_FOUND`（HTTP 404）でエラーに**すべきである**（should）。

`Undelete`を呼び出すユーザーが適切な権限を持っているが、リクエストされた
リソースが削除されていない場合、サービスは`ALREADY_EXISTS`（HTTP 409）で
応答**しなければならない**（must）。

`Expunge`を呼び出すユーザーが、存在しない（作成されたことがないか既にexpunge
されている）リソースをリクエストした場合、メソッドは`NOT_FOUND`（HTTP 404）を
返さ**なければならない**（must）。

リソースが存在するがready状態でもソフト削除状態でもない場合、メソッドは
`FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除
権限とは別の明示的なexpunge権限（例：`<service>.<resource>.expunge`）を要求
**しなければならない**（must）。
