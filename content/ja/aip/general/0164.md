# ソフトデリート

クライアントがソフトデリートおよび復元機能を必要とする理由はいくつかあるが、その中でも最も重要なのはミスからの回復である。復元をサポートするサービスは、誤って削除されたリソースをユーザーが回復することを可能にする。

## ガイダンス

API は、ユーザーが誤ってリソースを削除した場合に回復する能力が必要な状況に対応するため、「復元」機能を **してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドはリソースを削除済みとしてマークするだけで、システムから完全に削除してはならない。このメソッドがこのように動作する場合、`google.protobuf.Empty` ではなく、更新されたリソースを返す **べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されている `delete_time` フィールドと `purge_time` フィールドの両方を持つ **べきである**（should）。さらに、リソースに `state` フィールド（AIP-216）が含まれている場合、リソースは `DELETED` の状態値を含める **べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で **なければならない**（must）。
- `body` 句は `"*"` で **なければならない**（must）。
- レスポンスメッセージはリソースそのもので **なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - レスポンスには、実行不可能でない限り、完全に設定されたリソースを含める **べきである**（should）。
  - Undelete RPC が[長時間実行](#long-running-undelete)の場合、レスポンスメッセージはリソースそのものに解決される `google.longrunning.Operation` で **なければならない**（must）。

### Undelete リクエストメッセージ

Undelete メソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め **なければならない**（must）。これは `name` と呼ばれる **べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]される **べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別する **べきである**（should）。
  - フィールドのコメントにはリソースパターンを文書化する **べきである**（should）。
- リクエストメッセージには他の必須フィールドを含めては **ならず**（must not）、この AIP または他の AIP で説明されているものを除き、他のオプションフィールドを含める **べきではない**（should not）。

### 長時間実行 Undelete

一部のリソースでは、復元に通常の API リクエストよりも妥当な時間がかかる。この状況では、API は代わりに長時間実行オペレーション（AIP-151）を使用する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPC が長時間実行でない場合の戻り値の型）に設定され **なければならない**（must）。
- `response_type` と `metadata_type` の両方のフィールドを指定し **なければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除をトリガーできるようにする `Expunge` カスタムメソッドを提供して **してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態にあるリソース（例：`delete_time` が設定されている）に対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを使用するカスタムメソッドを使用しなければならない。
- HTTP 動詞は `POST` で、`body` 句は `"*"` でなければならない。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` でなければならない。

### 長時間実行 Expunge

Expunge プロセスにかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）であって **してもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは、共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expunge するリソースを名前で参照し **なければならない**（must）。
- 他のリクエストフィールドは **存在すべきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは（`bool show_deleted` が true でない限り）`List`（AIP-132）レスポンスで返される **べきではない**（should not）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、（`NOT_FOUND` エラーではなく）リソースを返す **べきである**（should）。

ソフトデリートを行う API は、それらのリソースを削除（purge）するための合理的な戦略を選択して **してもよい**（may）。これには、妥当な期間（例：30日）後の自動削除、ユーザーによる有効期限の設定（AIP-214）、またはリソースの無期限保持が含まれる。どの戦略を選択する場合でも、API はソフトデリートされたリソースがいつ完全に削除されるかを文書化する **べきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、ハードデリートされたリソースよりも宣言的クライアントでのエクスペリエンスが劣る。ソフトデリートされたリソースの ID は、カスタムメソッド（undelete）が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタムメソッドの使用を組み込むための手書きコードが必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持たない場合、リソースの存在有無にかかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーとし **なければならない**（must）。権限チェックは、リソースの存在確認より先に行われ **なければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合（一度も作成されていないか、すでに expunge されている）、サービスは `NOT_FOUND`（HTTP 404）でエラーとし **なければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功し **なければならず**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーとす **べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答し **なければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（一度も作成されていないか、すでに expunge されている）を要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ **なければならない**（must）。

リソースが存在するが ready または soft-deleted 状態でない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ **なければならない**（must）。

標準の権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的な expunge 権限（例：`<service>.<resource>.expunge`）を要求し **なければならない**（must）。
