# ソフトデリート

クライアントがソフトデリートおよび復元機能を必要とする理由はいくつかあるが、
最も重要な理由は誤操作からの回復である。復元をサポートするサービスでは、
ユーザーが誤って削除したリソースを回復できるようになる。

## ガイダンス

API は、ユーザーが誤ってリソースを削除した場合に回復機能を提供するため、
「復元」機能をサポート**してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドはリソースを
削除済みとしてマークするだけで、システムから完全に削除してはならない。
このように動作する場合、`google.protobuf.Empty` ではなく更新されたリソースを
返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されている
`delete_time` および `purge_time` フィールドの両方を持つ**べきである**（should）。
さらに、リソースに `state` フィールドが含まれている場合（AIP-216）、
`DELETED` 状態値を含める**べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する
**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージはリソース自体で**なければならない**（must）。
  `UndeleteBookResponse` は存在しない。
  - 実行不可能でない限り、レスポンスには完全に設定されたリソースを含める
    **べきである**（should）。
  - Undelete RPC が[長時間実行](#long-running-undelete)の場合、レスポンス
    メッセージはリソース自体に解決される `google.longrunning.Operation` で
    **なければならない**（must）。

### Undelete リクエストメッセージ

Undelete メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。フィールド名は
  `name` とす**べきである**（should）。
  - フィールドは[必須としてアノテーション][aip-203]す**べきである**（should）。
  - フィールドは参照する[リソースタイプ][aip-123]を識別す**べきである**
    （should）。
  - フィールドのコメントにはリソースパターンを文書化す**べきである**
    （should）。
- リクエストメッセージには、この AIP または他の AIP で説明されている
  フィールドを除き、他の必須フィールドを含めてはならず、他のオプション
  フィールドを含める**べきではない**（should not）。

### 長時間実行 Undelete

一部のリソースでは、復元に通常の API リクエストとしては妥当でない時間が
かかる場合がある。そのような状況では、API は代わりに長時間実行オペレーション
（AIP-151）を使用する**べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソースに設定し**なければならない**（must）
  （RPC が長時間実行でない場合の戻り値の型）。
- `response_type` と `metadata_type` の両方のフィールドを指定し**なければ
  ならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除を
トリガーできるようにする `Expunge` カスタム メソッドを提供**してもよい**
（may）。このメソッドは、現在 `CREATING`、`READY`、または
`SOFT_DELETED` 状態（例：`delete_time` が設定されている）にあるリソースに
対して動作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタム メソッドを使用しなければならない。
- HTTP 動詞は `POST`、body 句は `"*"` でなければならない。
- レスポンスメッセージは `google.protobuf.Empty` または
  `google.longrunning.Operation` でなければならない。

### 長時間実行 Expunge

Expunge 処理にかなりの時間がかかる場合、メソッドは代わりに
`google.longrunning.Operation`（AIP-151）に**してもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expunge するリソースを名前で参照し**なければ
  ならない**（must）。
- 他のリクエストフィールドを設ける**べきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは（`bool show_deleted` が
`true` の場合を除き）`List`（AIP-132）レスポンスで返す**べきではない**
（should not）。ソフトデリートされたリソースに対する `Get`（AIP-131）
リクエストは、(`NOT_FOUND` エラーではなく) リソースを返す**べきである**
（should）。

ソフトデリートを行う API は、妥当な期間（30 日間など）経過後の自動完全
削除、ユーザーによる有効期限の設定（AIP-214）、リソースの無期限保持など、
妥当な完全削除戦略を選択**してもよい**（may）。どの戦略を選択するかに
かかわらず、API はソフトデリートされたリソースがいつ完全に削除されるかを
文書化す**べきである**（should）。

### 宣言的クライアント向けリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいてハードデリート
されたリソースよりも劣る体験となる。ソフトデリートされたリソースの ID は、
カスタムメソッド（undelete）が呼び出されない限り再利用できないため、命令的
クライアントを導入するか、カスタムメソッドを使用するための手書きコードが
必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持たない場合、リソースの存在有無に
かかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーとし
**なければならない**（must）。権限のチェックは、リソースの存在確認よりも
先に実行し**なければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない場合
（一度も作成されていないか、すでに完全削除されている場合）、サービスは
`NOT_FOUND`（HTTP 404）でエラーとし**なければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求された
リソースがすでに削除されている場合、`allow_missing` が `true` であれば
サービスは成功し**なければならず**（must）、`allow_missing` が `false` で
あれば `NOT_FOUND`（HTTP 404）でエラーとす**べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求された
リソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）
で応答し**なければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（一度も作成されていないか、
すでに完全削除されている）を要求した場合、メソッドは `NOT_FOUND`（HTTP 404）
を返さ**なければならない**（must）。

リソースが存在するが、ready 状態またはソフトデリート状態でない場合、
メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**
（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の
削除権限とは別の明示的な expunge 権限（例：`<service>.<resource>.expunge`）を
要求し**なければならない**（must）。
