# ソフトデリート

クライアントがソフトデリートおよび undelete 機能を求める理由はいくつかあるが、中でも際立った理由が一つある。それはミスからの復旧である。undelete をサポートするサービスは、ユーザーが誤って削除したリソースを復旧することを可能にする。

## ガイダンス

API は、ユーザーが誤ってリソースを削除し、復旧する必要がある状況に対応するため、「削除取り消し」（undelete）機能をサポート**してもよい**（may）。

リソースが undelete をサポートする必要がある場合、`Delete` メソッドは、単にリソースを削除済みとしてマーク**しなければならず**（must）、システムから完全に削除してはならない。メソッドがこのように動作する場合、`google.protobuf.Empty` ではなく、更新されたリソースを返す**べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されている `delete_time` と `purge_time` の両方のフィールドを持つ**べきである**（should）。さらに、リソースに `state` フィールドが含まれる場合（AIP-216）、`DELETED` 状態値を含める**べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する**べきである**（should）:

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージはリソースそのもので**なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - レスポンスには、実行不可能でない限り、完全にデータが埋め込まれたリソースを含める**べきである**（should）。
  - undelete RPC が[長時間実行](#long-running-undelete)の場合、レスポンスメッセージは `google.longrunning.Operation` で**なければならず**（must）、この Operation はリソースそのものに解決される。

### Undelete リクエストメッセージ

Undelete メソッドは、共通のリクエストメッセージパターンを実装する:

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。`name` という名前に**すべきである**（should）。
  - フィールドは [required としてアノテーション][aip-203]**すべきである**（should）。
  - フィールドは、参照する[リソースタイプ][aip-123]を識別**すべきである**（should）。
  - フィールドのコメントは、リソースパターンをドキュメント化**すべきである**（should）。
- リクエストメッセージには、他の必須フィールドを含め**てはならず**（must not）、この AIP または他の AIP で説明されているものを除き、他のオプションフィールドを含める**べきではない**（should not）。

### 長時間実行 Undelete

一部のリソースでは、undelete に通常の API リクエストとして合理的な時間よりも長くかかる場合がある。この状況では、API は代わりに長時間実行オペレーション（AIP-151）を使用する**べきである**（should）:

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプには、リソース（RPC が長時間実行でない場合の戻り値の型）を設定**しなければならない**（must）。
- `response_type` と `metadata_type` の両方のフィールドを指定**しなければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時完全削除をトリガーできるように、`Expunge` カスタムメソッドを提供**してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態（例: `delete_time` が設定されている）にあるリソースに対して動作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタム メソッドを使用しなければならない。
- HTTP 動詞は `POST` でなければならず、`body` 句は `"*"` でなければならない。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` でなければならない。

### 長時間実行 Expunge

expunge プロセスに大幅な時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）**であってもよい**（may）:

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは、共通のリクエストメッセージパターンを実装する:

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expunge 対象のリソースを名前で参照**しなければならない**（must）。
- 他のリクエストフィールドは存在**すべきではない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは `List`（AIP-132）のレスポンスに返さ**れるべきではない**（should not）（`bool show_deleted` が true の場合を除く）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、（`NOT_FOUND` エラーではなく）リソースを返す**べきである**（should）。

リソースをソフトデリートする API は、妥当な期間（30日など）経過後の自動完全削除、ユーザーによる有効期限の設定（AIP-214）、リソースの無期限保持など、それらのリソースを完全削除するための合理的な戦略を選択**してもよい**（may）。どの戦略を選択するかにかかわらず、API はソフトデリートされたリソースがいつ完全に削除されるかをドキュメント化**すべきである**（should）。

### 宣言的クライアント向けのリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいて、ハードデリートされたリソースよりも劣る体験となる。ソフトデリートされたリソースの ID は、カスタムメソッド（undelete）が呼び出されない限り再利用できないため、命令型クライアントを導入するか、カスタムメソッドの使用を組み込むために手書きのコードが必要となる。

### エラー

ユーザーがリソースへのアクセス権限を持たない場合、リソースが存在するかどうかに関わらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラー**しなければならない**（must）。権限は、リソースが存在するかどうかの確認よりも先にチェック**しなければならない**（must）。

ユーザーが適切な権限を持っているが、リクエストされたリソースが存在しない場合（作成されたことがないか、すでに expunge されている場合）、サービスは `NOT_FOUND`（HTTP 404）でエラー**しなければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、リクエストされたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功**しなければならず**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラー**すべきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、リクエストされたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答**しなければならない**（must）。

`Expunge` を呼び出すユーザーが、存在しないリソース（作成されたことがないか、すでに expunge されている）をリクエストした場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ**なければならない**（must）。

リソースが存在するが、ready または soft-deleted 状態にない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準的な権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の delete 権限とは別の、明示的な expunge 権限（例: `<service>.<resource>.expunge`）を要求**しなければならない**（must）。
