# ソフトデリート

クライアントがソフトデリートと復元の機能を求める理由はいくつかあるが、最も重要な理由は「ミスからの回復」である。復元をサポートするサービスでは、誤って削除されたリソースをユーザーが回復できる。

## ガイダンス

APIは、ユーザーが誤ってリソースを削除し回復する必要がある状況に対応するために、「復元」（undelete）機能をサポート **してもよい**（may）。

リソースが復元をサポートする必要がある場合、`Delete` メソッドはリソースを削除済みとしてマークするだけで、システムから完全に削除**してはならない**（must not）。メソッドがこのように動作する場合、`google.protobuf.Empty` ではなく更新されたリソースを返す **べきである**（should）。

ソフトデリートをサポートするリソースは、AIP-148 で説明されている `delete_time` および `purge_time` フィールドの両方を持つ **べきである**（should）。さらに、リソースが `state` フィールドを含む場合（AIP-216）、リソースは `DELETED` 状態の値を含む **べきである**（should）。

### Undelete

ソフトデリートをサポートするリソースは、`Undelete` メソッドを提供する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
}
```

- HTTP 動詞は `POST` で**なければならない**（must）。
- `body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージはリソース自体で**なければならない**（must）。`UndeleteBookResponse` は存在しない。
  - レスポンスには、不可能でない限り、完全に設定されたリソースを含める **べきである**（should）。
  - undelete RPC が [長時間実行](#long-running-undelete) の場合、レスポンスメッセージは `google.longrunning.Operation` で**なければならず**（must）、リソース自体に解決される。

### Undelete リクエストメッセージ

Undelete メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message UndeleteBookRequest {
  // The name of the deleted book.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"];
}
```

- `name` フィールドを含め**なければならない**（must）。これは `name` と呼ばれる **べきである**（should）。
  - フィールドは [必須としてアノテーション][aip-203] される **べきである**（should）。
  - フィールドは参照する [リソースタイプ][aip-123] を識別する **べきである**（should）。
  - フィールドのコメントはリソースパターンを文書化する **べきである**（should）。
- リクエストメッセージには他の必須フィールドを含め**てはならず**（must not）、この AIP または他の AIP で説明されているものを除き、他のオプショナルフィールドを含める **べきではない**（should not）。

### 長時間実行 Undelete

一部のリソースでは、通常の API リクエストに適切な時間よりも復元に時間がかかる。この状況では、API は代わりに long-running operation（AIP-151）を使用する **べきである**（should）：

```proto
rpc UndeleteBook(UndeleteBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:undelete"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンスタイプはリソース（RPC が long-running でない場合の戻り型）に設定**しなければならない**（must）。
- `response_type` と `metadata_type` の両方のフィールドを指定**しなければならない**（must）。

### Expunge

ソフトデリートをサポートするリソースは、ユーザーがリソースの即時永続削除をトリガーできるようにする `Expunge` カスタムメソッドを提供 **してもよい**（may）。このメソッドは、現在 `CREATING`、`READY`、または `SOFT_DELETED` 状態（例: `delete_time` が設定されている）にあるリソースに対して操作できる。

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.protobuf.Empty) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.api.method_signature) = "name";
}
```

- URI は `:expunge` サフィックスを持つカスタムメソッドを使用**しなければならない**（must）。
- HTTP 動詞は `POST` で**なければならず**（must）、`body` 句は `"*"` で**なければならない**（must）。
- レスポンスメッセージは `google.protobuf.Empty` または `google.longrunning.Operation` で**なければならない**（must）。

### 長時間実行 Expunge

expunge プロセスにかなりの時間がかかる場合、メソッドは代わりに `google.longrunning.Operation`（AIP-151）であっ**てもよい**（may）：

```proto
// Permanently deletes a soft-deleted Book.
rpc ExpungeBook(ExpungeBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{name=publishers/*/books/*}:expunge"
    body: "*"
  };
  option (google.longrunning.operation_info) = {
    response_type: "google.protobuf.Empty"
    metadata_type: "OperationMetadata"
  };
  option (google.api.method_signature) = "name";
}
```

### Expunge リクエストメッセージ

Expunge メソッドは共通のリクエストメッセージパターンを実装する：

```proto
message ExpungeBookRequest {
  // The name of the soft-deleted book to expunge.
  // Format: publishers/{publisher}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference).type = "library.googleapis.com/Book"
  ];
}
```

- リクエストメッセージは、expunge するリソースを名前で参照**しなければならない**（must）。
- 他のリクエストフィールドがあっ**てはならない**（should not）。

### List と Get

ソフトデリートされたリソースは、デフォルトでは `List`（AIP-132）レスポンスで返される **べきではない**（should not）（`bool show_deleted` が true の場合を除く）。ソフトデリートされたリソースに対する `Get`（AIP-131）リクエストは、リソースを返す **べきである**（should）（`NOT_FOUND` エラーではなく）。

リソースをソフトデリートする API は、それらのリソースをパージするための合理的な戦略を選択 **してもよい**（may）。これには、妥当な期間（例: 30 日）後の自動パージ、ユーザーが有効期限を設定できるようにすること（AIP-214）、またはリソースを無期限に保持することが含まれる。どの戦略を選択するかにかかわらず、API はソフトデリートされたリソースがいつ完全に削除されるかを文書化する **べきである**（should）。

### 宣言的フレンドリーなリソース

ソフトデリート可能なリソースは、宣言的クライアントにおいてハードデリートされたリソースよりも劣る体験となる。ソフトデリートされたリソースの ID は、カスタムメソッド（undelete）が呼び出されない限り再利用できないため、命令的クライアントを導入するか、カスタムメソッドの使用を組み込むために手書きのコードが必要になる。

### エラー

ユーザーがリソースにアクセスする権限を持っていない場合、リソースの存在の有無にかかわらず、サービスは `PERMISSION_DENIED`（HTTP 403）でエラーに**しなければならない**（must）。権限はリソースの存在を確認する前にチェック**しなければならない**（must）。

ユーザーが適切な権限を持っているが、要求されたリソースが存在しない（作成されたことがないか、すでに expunge されている）場合、サービスは `NOT_FOUND`（HTTP 404）でエラーに**しなければならない**（must）。

ソフト `Delete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースがすでに削除されている場合、`allow_missing` が `true` であればサービスは成功**しなければならず**（must）、`allow_missing` が `false` であれば `NOT_FOUND`（HTTP 404）でエラーにする **べきである**（should）。

`Undelete` を呼び出すユーザーが適切な権限を持っているが、要求されたリソースが削除されていない場合、サービスは `ALREADY_EXISTS`（HTTP 409）で応答**しなければならない**（must）。

`Expunge` を呼び出すユーザーが存在しないリソース（作成されたことがないか、すでに expunge されている）を要求した場合、メソッドは `NOT_FOUND`（HTTP 404）を返さ**なければならない**（must）。

リソースは存在するが ready またはソフトデリート状態でない場合、メソッドは `FAILED_PRECONDITION`（HTTP 400）を返さ**なければならない**（must）。

標準の権限エラー（`PERMISSION_DENIED`）が適用される。サービスは、標準の削除権限とは別の明示的な expunge 権限（例: `<service>.<resource>.expunge`）を要求**しなければならない**（must）。
