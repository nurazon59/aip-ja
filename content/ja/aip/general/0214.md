# リソースの有効期限

顧客は多くの場合、特定のリソースやリソース属性がもはや有用でなくなるか無効になる時間を指定したい（例えば、ローテーションするセキュリティキー）。現在は、`google.protobuf.Timestamp expire_time`フィールドに正確な「有効期限時刻」を指定することを推奨している。しかし、これはユーザーが有効期限までの相対的な時間オフセットを指定したい場合に追加の負担となる。

さらに、世界は「time-to-live」（TTLと略されることが多い）の概念を理解しているが、このフィールドの典型的な形式（秒単位の整数）は、自動生成されたクライアントライブラリを使用する際に劣ったエクスペリエンスをもたらす。

## ガイダンス

1.  有効期限を伝えたいAPIは、`expire_time`という名前の[`google.protobuf.Timestamp`][timestamp]フィールドに依存**しなければならない**（must）。
2.  相対的な有効期限を許可したいAPIは、`expire_time`フィールドと、別の[`google.protobuf.Duration`][duration]フィールドである`ttl`（後者はinput onlyとしてマーク）の両方を含む`expiration`（または`{something}_expiration`）という`oneof`を定義**しなければならない**（must）。
3.  APIは、リソース取得時には常に`expire_time`フィールドに有効期限を返し、`ttl`フィールドは空白のままに**しなければならない**（must）。
4.  「time to live」の特定のセマンティクスに依存するAPI（例えば、TTLを整数で表現しなければならないDNS）は、`int64 ttl`フィールドを使用**してもよい**（may）（この場合、[aip.dev/not-precedent](./0200.md)コメントを提供**すべきである**（should））。

### 例

```proto
message ExpiringResource {
  // google.api.resource and other annotations and fields

  oneof expiration {
    // Timestamp in UTC of when this resource is considered expired.
    // This is *always* provided on output, and is calculated using
    // the value of [ttl][] if set when created.
    google.protobuf.Timestamp expire_time = 2;

    // Input only. The TTL for this resource.
    google.protobuf.Duration ttl = 3 [(google.api.field_behavior) = INPUT_ONLY];
  }
}
```

## 理由

### 検討した代替案

#### `ttl`という新しい標準フィールド

`ttl`という標準フィールドを有効期限を定義する代替方法として許可することを検討した。しかし、そうするとAPIサービスが時計がカウントダウンするように常にフィールドを更新する必要が生じる。これにより、リソースがしばらく処理され、その処理時間の結果として実質的に寿命が延びるという、read-modify-writeライフサイクルで問題が発生する可能性がある。

#### 常に`expire_time`を使用

これはいくつかの例外を除いた現在の状態である。このシナリオでは、`now + ttl = expire_time`の計算をクライアントライブラリに押し込める可能性がある。しかし、これはコマンドラインやREST/JSONを使用する際にやや不便なエクスペリエンスをもたらす。現状維持がデフォルトであることが多いが、多くの顧客は相対的な有効期限を定義できることを望んでいる。それはかなり簡単で、タイムゾーン、古い時計、その他の愚かなミスの問題を排除するからである。
