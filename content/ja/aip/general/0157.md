# 部分レスポンス

リソースが大きい、または計算コストが高くなる場合があり、APIがユーザーにどのフィールドを返すかの制御を提供する必要がある。

## ガイダンス

APIは、次のいずれかの方法で部分レスポンスをサポート **してもよい**（may）：

### フィールドマスクパラメータ

フィールドマスク（`google.protobuf.FieldMask`）は、ユーザーに返されるフィールドを細かく制御するために使用できる。APIはサイドチャネルでマスクをサポート **すべきである**（should）。例えば、パラメータはHTTPクエリパラメータ、HTTPヘッダー、または[gRPCメタデータエントリ][1]を使用して指定できる。Google Cloud APIはフィールドマスクを[システムパラメータ][0]として指定する。

フィールドマスクは[リクエスト](./0157.md#read-masks-as-a-request-field)内で指定 **すべきではない**（should not）。

- フィールドマスクパラメータの値は `google.protobuf.FieldMask` で**なければならない**（must）。
- フィールドマスクパラメータはオプションで**なければならない**（must）：
  - 明示的な値 `"*"` はサポート**されるべきである**（should）、すべてのフィールドを返さ**なければならない**（must）。
  - フィールドマスクパラメータが省略された場合、特に文書化されていない限り、`"*"` をデフォルトと **しなければならない**（must）。
- APIは、非末端の繰り返しフィールドを持つ読み取りマスクを許可 **してもよい**（may）するが、その義務はない。

**注記:** フィールドマスクパラメータのデフォルト値を変更することは[破壊的変更](../0180.md#semantic-changes)である。

### View列挙型

別の方法として、APIはview enumを使用した部分レスポンスをサポート **してもよい**（may）。View enumは、APIがユーザーに少数の組み合わせのみを公開したい場合に有用である：

```proto
enum BookView {
  // デフォルト/未設定の値。
  // APIはBASICビューをデフォルトとする。
  BOOK_VIEW_UNSPECIFIED = 0;

  // 書籍に関する基本メタデータを含めるが、全文は含めない。
  // これがデフォルト値（ListBooksとGetBookの両方）。
  BOOK_VIEW_BASIC = 1;

  // すべてを含める。
  BOOK_VIEW_FULL = 2;
}
```

- enumはリクエストメッセージ上の `view` フィールドとして指定 **すべきである**（should）。
- enumの名前は `-View` で終わるように **すべきである**（should）。
- enumは最低でも `BASIC` と `FULL` という名前の値を持つ **すべきである**（should）（ただし、これら以外の値を持って **してもよい**（may））。
- `UNSPECIFIED` の値は有効（エラーではない）で**なければならない**（must）。APIは未指定の値が何を行うかを文書化 **しなければならない**（must）。
  - List RPCの場合、有効なデフォルト値は `BASIC` と **すべきである**（should）。
  - 以下のRPCタイプの場合、有効なデフォルト値は `BASIC` または `FULL` のいずれかと **すべきである**（should）：
    - Get
    - Create
    - Update
    - Soft Delete
    - カスタム メソッド
- enumはprotoファイルのトップレベルで定義 **すべきである**（should）（複数のリクエスト、例えば `Get` と `List` の両方で必要になる可能性が高いため）。トップレベルの列挙型の詳細は[AIP-126][]を参照。
- APIは、時間の経過とともに特定のビューにフィールドを追加 **してもよい**（may）。APIは、特定のビューからフィールドを削除しては **してはならない**（must not）（これは破壊的変更である）。

  **注記:** サービスが重複するが異なる値を持つ複数のビューを必要とする（または必要になる可能性がある）場合、名前空間の競合が発生する可能性がある。この状況では、サービスはview enumを個々のリソース内にネスト **すべきである**（should）。

  **注記:** 標準メソッドのデフォルトが部分レスポンスであると、宣言的クライアントの効果が低下する可能性がある。部分レスポンスが必要と判断される場合は、このViewパターンのように、レスポンスに完全なリソースを入力するよう要求するメカニズムを提供することが推奨される。

### リクエストフィールドとしての読み取りマスク

**警告:** リクエストメッセージ上の単一の明示的なフィールドとしての読み取りマスクは、Google APIでは**非推奨**である。代わりに[システムパラメータ](#field-masks-parameter)を使用 **しなければならない**（must）。以下のガイダンスは、既存のレガシーGoogleおよび外部の非Google使用法を対象とする。

APIは、リクエストメッセージ上の単一フィールドとして読み取りマスクをサポート **してもよい**（may）：`google.protobuf.FieldMask read_mask`。

- 読み取りマスクは `google.protobuf.FieldMask` で**なければならない**（must）。`read_mask` という名前に **すべきである**（should）。
- フィールドマスクはオプション**であるべきである**（should）：
  - 明示的な値 `"*"` はサポート**されるべきである**（should）、すべてのフィールドを返さ**なければならない**（must）。
  - フィールドマスクパラメータが提供されない場合、すべてのフィールドが返され**なければならない**（must）。
- APIは、非末端の繰り返しフィールドを持つ読み取りマスクを許可 **してもよい**（may）するが、その義務はない。

## 理由

### リクエストメッセージ内の `read_mask` の非推奨化

前述の通り、Google APIインフラストラクチャはサービス全体のレスポンスフィールドフィルタリングメカニズムを実装しているため、個々のAPIメソッドがリクエストスキーマに `read_mask` を指定する必要はない。そうすることは冗長であり、クライアントまたはサービスにとって競合の可能性がある。
