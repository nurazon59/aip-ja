# 時刻と期間

多くのサービスは、時刻に関する概念を表現する必要がある。時刻の表現は、暦やタイムゾーンの複雑さ、および一般的な交換フォーマット（JSONなど）に時刻のネイティブな概念が欠けていることにより、困難な場合がある。

## ガイダンス

時刻を表すフィールドは、時刻や期間の型を表すために一般的に使用される共通コンポーネント（[`google.protobuf.Timestamp`][timestamp]や[`google.type.Date`][date]など）を使用**すべきである**（should）。これらの型は[共通コンポーネント][aip-213]であり、一貫して使用することでインフラストラクチャとツールが時刻値との対話においてより良い体験を提供できるようになる。

### タイムスタンプ

（タイムゾーンや暦から独立した）絶対的な時点を表すフィールドは、[`google.protobuf.Timestamp`][timestamp]型を使用**すべきである**（should）（これは内部的にUNIXタイムスタンプを使用し、ナノ秒精度を持つ）。

これらのフィールドは`create_time`や`update_time`のように`_time`で終わる名前を持つ**べきである**（should）。Repeatedフィールドの場合、名前は代わりに`_times`で終わる**べきである**（should）。

多くのタイムスタンプフィールドはアクティビティを参照する（例：`create_time`は該当リソースが作成された時刻を参照する）。これらの場合、フィールドは`{imperative}_time`形式で命名される**べきである**（should）。例えば、ブックが公開される場合、これが発生する時刻を格納するフィールドは動詞"to publish"の命令形（"publish"）を使用し、`publish_time`というフィールドになる。フィールドは過去形（`published_time`、`created_time`、`last_updated_time`など）を使用して命名される**べきではない**（should not）。

### 期間

（タイムゾーンや暦から独立した）2つの時点間の間隔を表すフィールドは、[`google.protobuf.Duration`][duration]型を使用**すべきである**（should）。

タイムスタンプと期間の区別を説明するために、フライト記録を考えてみる：

```proto
// （非常に不完全な）フライトログの表現。
message FlightRecord {
  // 飛行機が離陸した絶対的な時点。
  google.protobuf.Timestamp takeoff_time = 1;

  // 離陸から着陸までのフライトの長さ（期間）。
  google.protobuf.Duration flight_duration = 2;
}
```

**注:** 注意深い読者は、[timestamp][]と[duration][]メッセージが同じ構造（`int64 seconds`と`int32 nanos`）を持つことに気づくかもしれない。しかし、これらは異なる意味論的意味を持つため、これらの区別は重要である。さらに、ツールはどのメッセージが使用されているかに基づいて動作を変更できる。例えば、Pythonベースのツールはタイムスタンプを[datetime][py_datetime]オブジェクトに、期間を[timedelta][py_timedelta]オブジェクトに変換できる。

### 相対時間セグメント

ストリーム内の時間セグメントを表現する必要がある場合がある。このような場合、[`google.protobuf.Duration`][duration]型を使用**すべきであり**（should）、フィールド名は`_offset`で終わる**べきである**（should）。意味が明確であることを保証するため、フィールドにはオフセットが相対する時点を記載したコメントが**なければならない**（must）。

これを説明するために、オーディオストリームのセグメントを表すリソースを考えてみる：

```proto
message AudioSegment {
  // セグメントの開始を表す、ストリームの開始からの相対的な期間。
  google.protobuf.Duration start_offset = 1;

  // セグメントの合計長さ。
  google.protobuf.Duration segment_duration = 2;
}
```

### 日付と時刻

カレンダー日付または壁時計時刻を表すフィールドは、適切な共通コンポーネントを使用**すべきである**（should）：

- 日付：[`google.type.Date`][date]
- 時刻：[`google.type.TimeOfDay`][time_of_day]

日付を表すフィールドは`_date`で終わる名前を持つ**べきであり**（should）、時刻または日時を表すフィールドは`_time`で終わる名前を持つ**べきである**（should）。

**注:** `Date`と`TimeOfDay`の両コンポーネントはタイムゾーン非対応である。タイムゾーン対応が必要なフィールドは`DateTime`（以下参照）を使用**すべきである**（should）。

### 日時

（日付と時刻、オプションでタイムゾーンを持つ）日時を表すフィールドは、[`google.type.DateTime`][datetime]コンポーネントを使用**すべきであり**（should）、フィールド名は`_time`で終わる**べきである**（should）。

### 互換性

レガシーまたは互換性の理由で共通構造を使用できないAPIがある。例えば、APIがタイムスタンプを整数または[ISO-8601][]文字列とすることを義務付ける別の仕様に準拠する場合がある。

このような状況では、フィールドは他の型を使用**してもよい**（may）。可能な場合、以下の命名規則が適用される：

- 整数の場合、意味（例：`time`、`duration`、`delay`、`latency`）**および**測定単位（有効な値：`seconds`、`millis`、`micros`、`nanos`）を最終サフィックスとして含める。例：`send_time_millis`。
- 文字列の場合、意味（例：`time`、`duration`、`delay`、`latency`）を含めるが、単位サフィックスは含めない。

すべての場合において、期待される形式とその使用理由を明確に文書化すること。
