# 数量

多くのサービスは、アイテムの離散的な数量（バイト数、マイル数、ノード数など）を表す必要がある。

## ガイダンス

明確な測定単位（バイト、マイルなど）を持つ数量には、サフィックスとして測定単位を含め**なければならない**（must）。適切な場合、単位は一般的に受け入れられている略語を使用**すべきであり**（should）、略語は複数形に**すべきではない**（should not）（例：`distance_kilometers`ではなく`distance_km`）。

```proto
// 無着陸の航空路線を表す。
message Route {
  // 路線が始まる空港。
  string origin = 1;

  // 目的地の空港。
  string destination = 2;

  // 出発地と目的地の空港間の距離。
  // この値は認定されたマイレージの計算にも使用される。
  int32 distance_miles = 3;
}
```

数量がアイテムの数（例：クラスター内のノード数）の場合、フィールドはサフィックス`_count`を使用**すべきである**（should）（接頭辞`num_`ではない）：

```proto
// 個々のノードのクラスター。
message Cluster {
  // クラスター内のノード数。
  int32 node_count = 1;
}
```

**注:** フィールドは符号なし整数型を使用しては**ならない**（must not）。多くのプログラミング言語とシステムがそれらを適切にサポートしていないためである。

### 複合単位

複合的な測定単位を持つ数量は、必要に応じて明確化のために単位間にアンダースコアを区切りとして使用**してもよい**（may）。略語化されていない単位は区切ら**なければならない**（must）。略語化された単位は、曖昧でない限り区切る**すべきではない**（should not）。複合単位は複数形**とするべきであり**、すべての構成単位は最後の構成単位を除いて単数形とし、最後の構成単位は略語化されていない限り複数形**とすべきである**（should）。

- `energy_kwh`（`energy_kw_h`ではない）
- `energy_kw_fortnights`（`energy_kwfortnight`や`energy_kw_fortnight`ではない）

**注:** メートル法接頭辞はその基本単位から区切っては**ならない**（must not）。

### 逆数単位

逆数単位を含む（または逆数単位である）測定単位を持つ数量は、"per"という単語で区切られた非逆数単位の複合体の後に、すべての逆数単位を複合単位として示す**すべきである**（should）。逆数複合単位は単数形**すべきである**（should）。

- `speed_miles_per_hour`（`speed_mph`ではない）
- `speed_meters_per_second`（`speed_meters_per_seconds`や`speed_meter_per_second`ではない）
- `event_count_per_hour`（`events_per_hour`、`event_counts_per_hour`、`hourly_events`ではない）
- `price_per_kwh`（[`google.type.Money`][money]を使用）

**注:** このガイダンスは、逆数量に対して特別な名前と記号を持つ一般的に受け入れられている派生単位が存在する場合には適用されない。例えば、派生単位「hertz」は、逆時間に適切な場合に使用**すべきである**（should）。

### 特化メッセージ

特定の数量を表すメッセージを作成することが有用な場合がある。これは特に2つの状況で価値がある：

- 2つ以上の個別の数量をグループ化する。例：[`google.protobuf.Duration`][duration]
- 測定単位自体が変動する可能性のある共通概念を表す。例：[`google.type.Money`][money]

APIは、適切な場合に数量を表すメッセージを作成**してもよい**（may）。これらのメッセージをフィールドとして使用する場合、APIは直感的に意味をなすのであれば、フィールド名のサフィックスとしてメッセージの名前を使用**すべきである**（should）。
