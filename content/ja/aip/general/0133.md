# 標準メソッド: Create

REST APIでは、新しいリソースを作成するためにコレクションのURI（例：`/v1/publishers/{publisher}/books`）に`POST`リクエストを行うのが慣例である。

リソース指向設計（[AIP-121][]）は、`Create`メソッドを通じてこのパターンを尊重する。これらのRPCは親コレクションと作成するリソース（および必要に応じて他のパラメータ）を受け入れ、作成されたリソースを返す。

## ガイダンス

APIは、ユーザーにとって価値がない場合を除き、一般的にリソースに対してcreateメソッドを提供**すべきである**（should）。createメソッドの目的は、既存のコレクションに新しいリソースを作成することである。

Createメソッドは以下のパターンを使用して指定される：

```proto
rpc CreateBook(CreateBookRequest) returns (Book) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
  };
  option (google.api.method_signature) = "parent,book";
}
```

- RPC名は`Create`という単語で始まら**なければならない**（must）。RPC名の残りの部分は、作成されるリソースの単数形**にすべきである**（should）。
- リクエスト メッセージはRPC名と一致し、`Request`サフィックスが付かな**ければならない**（must）。
- レスポンス メッセージはリソース自体**でなければならない**（must）。`CreateBookResponse`は存在しない。
  - レスポンスは完全に設定されたリソースを含む**べきであり**（should）、input only（[AIP-203][]参照）または部分的なレスポンスを返す理由がある場合（[AIP-157][]参照）を除き、提供されたすべてのフィールドを含め**なければならない**（must）。
  - create RPCが[long-running](#long-running-create)の場合、レスポンス メッセージはリソース自体に解決される`google.longrunning.Operation`**でなければならない**（must）。
- HTTP動詞は`POST`**でなければならない**（must）。
- リソースが追加されるコレクションはURIパスにマッピング**すべきである**（should）。
  - コレクションの親リソースは`parent`**とすべきである**（should）、URIパス内の唯一の変数**であるべきである**（should）。
  - コレクション識別子（上記の例では`books`）はリテラル文字列**でなければならない**（must）。
- `google.api.http`アノテーションに`body`キーを**含めなければならない**（must）、リクエスト メッセージのリソースフィールドにマッピング**しなければならない**（must）。
  - 残りのすべてのフィールドはURIクエリパラメータにマッピング**すべきである**（should）。
- `google.api.method_signature`アノテーションは1つだけ存在**すべきであり**（should）、値は`"parent,{resource},{resource}_id"`、またはリソースIDが必須でない場合は`"parent,{resource}"`とする。
- APIが[管理プレーン][management plane]で動作している場合、操作は[強い一貫性][strong consistency]を持つ**べきである**（should）：create操作の完了は、すべてのユーザー設定可能な値とリソースの存在が定常状態に達し、リソース状態の読み取りが一貫したレスポンスを返すことを意味**しなければならない**（must）。

### リクエスト メッセージ

Createメソッドは共通のリクエスト メッセージパターンを実装する：

```proto
message CreateBookRequest {
  // このブックが作成される親リソース。
  // 形式: publishers/{publisher}
  string parent = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      child_type: "library.googleapis.com/Book"
    }];

  // ブックに使用するID。ブックのリソース名の最終要素になる。
  //
  // この値は4〜63文字で、有効な文字は /[a-z][0-9]-/。
  string book_id = 2 [(google.api.field_behavior) = REQUIRED];

  // 作成するブック。
  Book book = 3 [(google.api.field_behavior) = REQUIRED];
}
```

- 作成するリソースがトップレベルリソースでない限り、`parent`フィールドを含め**なければならない**（must）。`parent`**とすべきである**（should）。
  - フィールドは [required としてアノテートされるべきである][aip-203]（should）。
  - フィールドは作成されるリソースの[リソース型][aip-123]を識別**しなければならない**（must）。
- `{resource}_id`フィールドは、管理プレーン リソースには含め**なければならず**（must）、データプレーン リソースには含める**べきである**（should）。
- リソースフィールドは含め**なければならず**（must）、POSTボディにマッピング**しなければならない**（must）。
- リクエスト メッセージに他の必須フィールドを含めては**ならない**（must not）、このAIPまたは別のAIPで説明されているものを除き、他のオプショナル フィールドを含める**べきではない**（should not）。

### Long-running create

一部のリソースは、通常のAPIリクエストとして合理的な時間よりも作成に時間がかかる。この場合、APIは代わりにlong-running operation（AIP-151）を使用**すべきである**（should）：

```proto
rpc CreateBook(CreateBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンス型はリソース（RPCがlong-runningでない場合の戻り値型）に設定**しなければならない**（must）。
- `response_type`と`metadata_type`の両方を指定**しなければならない**（must）。

**重要:** 宣言的フレンドリーなリソース（AIP-128）はlong-running operationsを使用**すべきである**（should）。リクエストが事実上即時である場合、サービスは既に完了に設定されたLROを返して**してもよい**（may）。

### ユーザー指定ID

APIが[管理プレーン][management plane]で動作している場合、APIは作成時にユーザーがリソースのIDコンポーネント（リソース名の最後のセグメント）を指定できるように**しなければならない**（must）。

[データプレーン][data plane]では、APIはユーザーがIDを指定できるように**すべきである**（should）。例外的なケースでは以下の動作とする：

- データプレーン リソースは、2つを区別する必要なく同一のレコードを許可する（例：主キーのないテーブルの行）。
- データプレーン リソースが[宣言的クライアント][Declarative clients]に公開されない。

APIは`{resource}_id`フィールドに[field_behavior][] `OPTIONAL`を許可**してもよく**（may）、指定されない場合はシステム生成IDを生成する。

例：

```
// ユーザー指定IDを使用。
publishers/lacroix/books/les-miserables

// システム生成IDを使用。
publishers/012345678-abcd-cdef/books/12341234-5678-abcd
```

- `{resource}_id`フィールドはリソース自体ではなく、リクエスト メッセージに存在**しなければならない**（must）。
  - フィールドはrequiredまたはoptional**してもよい**（may）。requiredの場合は対応するアノテーションを含める**べきである**（should）。
- リソースの`name`フィールドは無視**しなければならない**（must）。
- RPCには`google.api.method_signature`アノテーションが1つだけ存在**すべきであり**（should）、作成されるリソースがトップレベルでない場合は値`"parent,{resource},{resource}_id"`、トップレベルの場合は`"{resource},{resource}_id"`とする。
- ドキュメントは許容される形式を説明**すべきであり**（should）、形式は[AIP-122][]のリソース名フォーマットのガイダンスに従う**すべきである**（should）。
- ユーザーが重複するリソース名になるIDでリソースを作成しようとした場合、サービスは`ALREADY_EXISTS`でエラーに**しなければならない**（must）。
  - ただし、呼び出しを行うユーザーに重複リソースを表示する権限がない場合、サービスは代わりに`PERMISSION_DENIED`でエラーに**しなければならない**（must）。

**注:** REST APIの場合、ユーザー指定IDフィールド`{resource}_id`はリクエストURIのクエリパラメータとして提供される。

## 理由

### ユーザー指定IDの必要性

[宣言的クライアント][Declarative clients]は、リソースIDを更新の適用と競合解決のためにリソースを識別する方法として使用する。ユーザー指定IDがないと、クライアントは識別子をローカルに保存しない限りリソースを見つけることができず、リソースの再作成につながる可能性がある。これは、そのリソースを参照するすべてのリソースに下流への影響を与え、新しく作成されたリソースのIDに更新することを強制する。

ユーザー指定IDを持つことは、クライアントがリソース名を事前計算し、他のリソースからの参照で使用できることも意味する。
