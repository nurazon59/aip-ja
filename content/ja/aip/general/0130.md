# メソッド

APIは1つ以上のメソッドで構成され、メソッドはサービスがコンシューマーに代わって実行できる特定の操作を表す。

## ガイダンス

### メソッドのカテゴリ

以下に、存在するメソッドの複数のカテゴリを列挙する。これらは多くの場合、メソッドが操作するオブジェクト（例：コレクションやリソース）の下にグループ化される。

| カテゴリ名 | 関連AIP | [宣言的クライアント][Declarative client]連携 | CLI / UI連携 | SDK連携 |
| --- | --- | --- | --- | --- |
| *標準メソッド* | | | | |
| **標準コレクション メソッド**: リソースのコレクションに対して操作する（ListまたはCreate）。 | [AIP-121][], [AIP-132][], [AIP-133][] | 自動化可能 | 自動化可能 | 自動化可能 |
| **標準リソース メソッド**: 単一のリソースを取得または変更する（Get、Update、Delete）。 | [AIP-121][], [AIP-131][], [AIP-134][], [AIP-135][] | 自動化可能 | 自動化可能 | 自動化可能 |
| **バッチ リソース メソッド**: コレクション内の複数のリソースを名前で取得または変更する。 | [AIP-231][], [AIP-233][], [AIP-234][], [AIP-235][] | クエリ最適化に使用可能 | 自動化可能 | 自動化可能 |
| **集約リスト メソッド**: 複数のコレクションにわたって同じ型の複数のリソースを取得または変更する。 | [AIP-159][] | 有用でも自動化可能でもない | 自動化可能 | 自動化可能 |
| *カスタム取得メソッド* | | | | |
| **カスタム コレクション取得メソッド**: 標準メソッドでは表現できない情報をコレクションから取得する。 | [AIP-136][] | 手書き | 自動化可能 | 自動化可能 |
| **カスタム リソース取得メソッド**: 標準メソッドでは表現できない情報を単一のリソースから取得する。 | [AIP-136][] | 手書き | 自動化可能 | 自動化可能 |
| *カスタム変更メソッド* | | | | |
| **リソースのバックアップ**: 特定の時点におけるリソースのコピーを保存する。 | [AIP-162][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| **リソースの復元**: リソースを特定の時点のバージョンに設定する。 | [AIP-162][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| **リソースのリネーム**: 構成とデータを保持したままリソースの名前またはIDを変更する。 | [AIP-136][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| **カスタム コレクション変更メソッド**: コレクションを参照する命令的操作を実行し、標準メソッドでは容易に実現できない方法でそのコレクション内の1つ以上のリソースを変更する（例：状態遷移）。 | [AIP-136][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| **カスタム リソース変更メソッド**: リソースに対して命令的操作を実行し、標準メソッドではできない方法でリソースを変更する（例：状態遷移）。 | [AIP-136][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| *その他のカスタム メソッド* | | | | |
| **ステートレス メソッド**: API内のデータに永続的な影響を与えないメソッド（例：テキスト翻訳）。 | [AIP-136][] | 未使用または手書き | 自動化可能 | 自動化可能 |
| *上記以外* | | | | |
| **ストリーミング メソッド**: クライアント、サーバー、または双方向ストリームを介して通信するメソッド。 | | 手書き | 手書き | 自動化可能 |

### メソッドカテゴリの選択

メソッドを設計する際、API作者は定義されたカテゴリから以下の順序で選択すべきである（should）：

1. 標準メソッド（コレクションおよびリソース）
1. 標準バッチまたは集約メソッド
1. カスタム メソッド（コレクション、リソース、またはステートレス）
1. ストリーミング メソッド

## 理由

リソース指向の標準メソッドとカスタム メソッドは、最も多様なクライアント（宣言的クライアント、CLI、UIなど）で表現でき、あるAPIの知識を別のAPIに適用できる最も均一な体験を提供するため、最初に推奨される。

標準メソッドが不適切な場合でも、リソースまたはコレクションにマウントされるカスタム メソッドは、より劣るが依然として価値のある一貫性を提供し、ユーザーがアクションのスコープとそのアクションを通知するために読み取られる構成のオブジェクトについて推論するのに役立つ。変更系カスタム メソッドは、[宣言的クライアント][Declarative client]のようなリソース指向クライアントのみとの自動化された統合を行うには十分に均一ではないが、CLI、UI、およびSDKが容易に認識できるパターンである。

APIをリソース指向の方法でまったく表現できない場合、その操作は、SDK以外のクライアントが操作をモデル化することを困難にする均一性の欠如があるカテゴリに分類される。このカテゴリは、ユーザーが類似のAPIに関する知識に頼ることができず、多くのクライアントとの統合を手書きする必要がある可能性が高いため、最後に優先される。
