# Long-running operations

APIは、完了にかなりの時間がかかるメソッドを公開する必要がある場合がある。このような状況では、タスクの実行中に単にブロックするのはユーザーエクスペリエンスが悪い。むしろ、ユーザーに何らかの約束を返し、後で確認できるようにする方が良い。

Long-running operationsパターンは、[Python Future][python future]や[Node.js Promise][node.js promise]におおよそ類似している。基本的に、ユーザーは進捗を追跡し結果を取得するために使用できるトークンを受け取る。

## ガイダンス

完了にかなりの時間がかかる可能性がある個々のAPIメソッドは、最終的なレスポンスメッセージの代わりに[`google.longrunning.Operation`][lro]オブジェクトを返す**べきである**（should）。

```proto
// Create a book.
rpc CreateBook(CreateBookRequest) returns (google.longrunning.Operation) {
  option (google.api.http) = {
    post: "/v1/{parent=publishers/*}/books"
    body: "book"
  };
  option (google.longrunning.operation_info) = {
    response_type: "Book"
    metadata_type: "OperationMetadata"
  };
}
```

- レスポンス型は`google.longrunning.Operation`と**しなければならない**（must）。`Operation`のproto定義は個々のAPIにコピーしては**ならない**（must not）。
  - レスポンスはストリーミングレスポンスであっては**ならない**（must not）。
- メソッドは`google.longrunning.operation_info`アノテーションを含む**しなければならない**（must）。これはレスポンス型とメタデータ型の両方を定義する**しなければならない**（must）。
  - レスポンス型とメタデータ型は、RPCが出現するファイル、またはそのファイルがインポートするファイルで定義される**しなければならない**（must）。
  - レスポンス型とメタデータ型が別のパッケージで定義されている場合、完全修飾メッセージ名を使用**しなければならない**（must）。
  - レスポンス型は、レスポンスデータが決して必要にならないことが確実でない限り（[`Delete`][aip-135]メソッドを除く）、`google.protobuf.Empty`と**すべきではない**（should not）。将来レスポンスデータが追加される可能性がある場合は、RPCレスポンス用の空のメッセージを定義し、それを使用すること。
  - メタデータ型は、各`GetOperation`呼び出しで進捗、部分的な失敗、および類似の情報を提供するために使用される。メタデータ型は、メタデータが決して必要にならないことが確実でない限り、`google.protobuf.Empty`と**すべきではない**（should not）。将来メタデータが追加される可能性がある場合は、RPCメタデータ用の空のメッセージを定義し、それを使用すること。
- `Operation`を返すメッセージを持つAPIは、[`Operations`][lro]サービスを実装**しなければならない**（must）。個々のAPIは、不均一性を避けるために、long-running operations用の独自のインターフェースを定義しては**ならない**（must not）。
- RPCが[validate-onlyモード][aip-163]をサポートする場合、バリデーションリクエストへのレスポンスは以下のいずれかとする**しなければならない**（must）：
  - 既に完了している`Operation`を持つ成功レスポンス。`done`フィールドが`true`に設定され、有効な（ただし空の可能性がある）レスポンスメッセージが`google.protobuf.Any`メッセージでラップされて`response`フィールドに含まれる。`name`フィールドは、サービスが成功したバリデーションのために状態を維持する必要を避けるため、空でも**よい**（may）。
  - 即時のエラーレスポンス（通常は"bad request"）
  - `done`フィールドが`false`に設定された`Operation`。長時間実行されるバリデーションを示す。この場合、`name`フィールドは設定され**なければならない**（must）。これによりクライアントは長時間実行バリデーション操作をポーリングして完了を待つことができる。成功したバリデーションは最終的に`done=true`と有効な（ただし空の可能性がある）ラップされたレスポンスメッセージを`response`フィールドに持つ操作で表現され**なければならない**（must）。失敗したバリデーションは最終的に`done=true`と`error`フィールドにエラー詳細が提供された操作で表現され**なければならない**（must）。

**注:** 「かなりの時間」と見なされるものについてのユーザーの期待は、行われている作業によって異なる可能性がある。目安としては10秒が妥当である。

### 標準メソッド

APIは、適切であれば、[`Create`][aip-133]、[`Update`][aip-134]、または[`Delete`][aip-135]標準メソッドから`Operation`を返して**もよい**（may）。この場合、`operation_info`アノテーションのレスポンス型は、その標準メソッドの標準的かつ期待されるレスポンス型とする**しなければならない**（must）。

long-running operationでリソースを作成または削除する場合、リソースは[`List`][aip-132]および[`Get`][aip-131]呼び出しに含める**べきである**（should）。ただし、リソースは一般的に[state enum][aip-216]を使用して、使用可能でないことを示す**べきである**（should）。

### 並列操作

リソースは、並行して処理される複数の操作を受け入れても**してもよい**（may）が、その義務はない：

- 複数の並列操作を受け入れるリソースは、それらを同時に処理するのではなくキューに入れても**してもよい**（may）。
- 複数の操作を並列に許可しないリソース（進行中の操作が完了するまで新しい操作を拒否する）は、ユーザーが並列操作を試みた場合に`ABORTED`を返し、状況を説明するエラーメッセージを含め**なければならない**（must）。
- [宣言的フレンドリーなAPI][aip-128]を持つリソースは、後続の更新が既存の操作を先取りすることを許可しても**してもよい**（may）。この場合、最新の更新が処理を開始し、以前の操作は状況を説明するエラーメッセージとともに`ABORTED`としてマークされる。

### 有効期限

APIは、操作完了後十分な時間が経過した後に、操作リソースの有効期限が切れることを許可しても**してもよい**（may）。

**注:** 操作の有効期限の目安としては30日が妥当である。

### エラー

long-running operationの開始を妨げるエラーは、他のメソッドと同様に、エラーレスポンス（[AIP-193][]）を返す**しなければならない**（must）。

実行フェーズ中に失敗する操作は、`Operation.error` [google.rpc.Status][]フィールドに配置されたエラーレスポンス（[AIP-193][]）を返す**しなければならない**（must）。

操作の過程で発生する非終端エラーは、メタデータメッセージに配置しても**してもよく**（may）、フィールドは[AIP-193][]準拠の[google.rpc.Status][]とする**しなければならない**（must）。

### 後方互換性

long-running operationの`response_type`または`metadata_type`を変更することは破壊的変更である。

## 根拠

### Validate-only動作

validate-onlyレスポンスのガイダンスは、クライアント（統一的に扱える「完全に形成された」操作から利益を得る）とサーバー（些細な操作のために追加の状態を維持したくない）の間の緊張から生じている。リクエストを検証するだけでより多くの状態が生成されるのは直感に反するように思えるが、後で取得できる完全な操作レスポンスはそれを必要とするか、「特別な」シングルトン操作IDを必要とする。提供されたガイダンスは妥協案である：「完了した」操作を返すことで、クライアントは既存のロジックを使用して操作が正常に完了したことを確認でき（したがって更新されたステータスを取得する必要がない）、サーバーは追加の状態を維持する必要がない。
