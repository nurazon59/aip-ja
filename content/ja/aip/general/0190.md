# 命名規則

このトピックでは、Google APIで使用される命名規則について説明する。一般的に、これらの規則はすべてのGoogle管理サービスに適用される。

## ガイダンス

多くのAPIにわたって長期間にわたり一貫した開発者体験を提供するために、APIで使用されるすべての名前は以下の条件を満たす**べきである**（should）：

-   単純明快
-   直感的
-   一貫性がある

これには、インターフェース、リソース、コレクション、メソッド、メッセージの名前が含まれる。

多くの開発者にとって英語は第二言語であるため、これらの命名規則の目標の1つは、すべてのAPI名を大多数の開発者にとって理解可能にすることである。これは、メソッドやリソースに名前を付ける際に、シンプルで一貫性のある小さな語彙の使用を推奨することによって実現する。

-   APIで使用される名前は、正しいアメリカ英語を使用**すべきである**（should）。例えば、license（licenceではなく）、color（colourではなく）。
-   長い単語の一般的に受け入れられている短縮形や略語は、簡潔さのために使用**してもよい**（may）。例えば、Application Programming InterfaceよりもAPIが優先される。
-   特に指定がない限り、定義は[Google Java Style](https://google.github.io/styleguide/javaguide.html#s5.3-camel-case)で定義されているUpperCamelCase名を使用**しなければならない**（must）。
-   可能な場合は直感的で親しみやすい用語を使用する。例えば、リソースの削除（および破棄）を説明する場合、eraseよりもdeleteが優先される。
-   同じ概念には同じ名前または用語を使用する（API間で共有される概念も含む）。
-   名前の過負荷を避ける。異なる概念には異なる名前を使用する。
-   APIおよびより大きなGoogle APIエコシステムのコンテキスト内で曖昧な、過度に一般的な名前を避ける。これらはAPI概念の誤解を招く可能性がある。むしろ、API概念を正確に説明する具体的な名前を選択する。これは、リソースなどの一次的なAPI要素を定義する名前にとって特に重要である。避けるべき名前の決定的なリストは存在しない。すべての名前は他の名前とのコンテキストで評価されなければならない。Instance、Info、Serviceは過去に問題となった名前の例である。選択する名前はAPI概念を明確に説明し（例：何のインスタンス？）、他の関連概念と区別する（例：「alert」はルール、シグナル、通知のどれを意味するのか？）必要がある。
-   一般的なプログラミング言語のキーワードと競合する可能性のある名前の使用は慎重に検討する。そのような名前は使用**してもよい**（may）が、APIレビューでさらなる精査を引き起こす可能性が高い。賢明かつ控えめに使用すること。

### インターフェース名

`pubsub.googleapis.com`のような[サービス名](./0009.md#api-service-name)との混同を避けるため、*インターフェース名*という用語は.protoファイルで`service`を定義する際に使用される名前を指す：

```proto
// Library がインターフェース名である。
service Library {
  rpc ListBooks(...) returns (...);
  rpc ...
}
```

*サービス名*はAPIセットの実際の実装への参照と考え、*インターフェース名*はAPIの抽象的な定義を指すと考えることができる。

インターフェース名は、CalendarやBlobStoreのような直感的な名詞を使用**すべきである**（should）。名前は、プログラミング言語やそのランタイムライブラリにおける確立された概念（Fileなど）と競合**すべきではない**（should not）。

まれに、*インターフェース名*がAPI内の別の名前と競合する場合、サフィックス（`Api`や`Service`など）を使用して曖昧さを排除**すべきである**（should）。

### メソッド名

サービスは、そのIDL仕様において、コレクションやリソース上のメソッドに対応する1つ以上のAPIメソッドを定義**してもよい**（may）。メソッド名は、UpperCamelCaseの`VerbNoun`の命名規則に従う**べきである**（should）。ここで名詞は通常リソースタイプである。

標準メソッドとそのバッチバリアントは、以下のドキュメントで命名ガイダンスを定義している：

Method | Standard | Batch
------ | -------- | -----
`Get`  | [AIP-131][] | [AIP-231][]
`List` | [AIP-132][] | N/A
`Create` | [AIP-133][] | [AIP-233][]
`Update` | [AIP-134][] | [AIP-234][]
`Delete` | [AIP-135][] | [AIP-235][]

その他すべてのメソッドはカスタム メソッドと見なされ、AIP-136の命名ガイダンスに従う。

### メッセージ名

メッセージ名は短く簡潔に**すべきである**（should）。不必要または冗長な単語を避ける。形容詞のない対応するメッセージが存在しない場合、形容詞は省略できることが多い。例えば、_共有されていない_プロキシ設定が存在しない場合、`SharedProxySettings`の`Shared`は不要である。

メッセージ名には前置詞（例："With"、"For"）を含める**べきではない**（should not）。一般に、前置詞を含むメッセージ名は、メッセージ上のオプショナルフィールドで表現する方が適切である。

#### リクエストメッセージとレスポンスメッセージ

リクエストおよびレスポンスメッセージ名については、カスタム メソッドの場合はAIP-136、[標準メソッド](https://google.aip.dev/general#operations)の場合は適切なAIPを参照すること。
