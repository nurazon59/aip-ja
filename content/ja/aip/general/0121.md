# リソース指向設計

リソース指向設計は、[RPC][] APIを指定するためのパターンであり、いくつかの高レベルの設計原則（そのほとんどは最近のパブリックHTTP APIに共通する）に基づいている：

- APIの基本的な構成要素は、個別に名前が付けられた_リソース_（名詞）と、それらの間に存在する関係および階層である。
- 少数の標準_メソッド_（動詞）が、最も一般的な操作のセマンティクスを提供する。ただし、標準メソッドが適合しない状況では、カスタム メソッドが利用可能である。
- ステートレスプロトコル：クライアントとサーバー間の各インタラクションは独立しており、クライアントとサーバーの両方が明確な役割を持つ。

読者は、これらの原則と[REST][]のいくつかの原則との類似性に気付くかもしれない。リソース指向設計はRESTから多くの原則を借用しつつ、適切な場所では独自のパターンを定義している。

## ガイダンス

APIを設計する際は、以下の点を（おおよそ論理的な順序で）検討する：

- APIが提供するリソース（名詞）
- それらのリソース間の関係と階層
- 各リソースのスキーマ
- 各リソースが提供するメソッド（動詞）。可能な限り標準動詞に依存する。

### リソース

リソース指向APIは、一般的にリソース階層としてモデル化**すべきである**（should）。各ノードは単一のリソースまたはリソースのコレクションである。

_コレクション_は_同じタイプ_のリソースを含む。例えば、出版社は出版する書籍のコレクションを持つ。リソースは通常フィールドを持ち、任意の数のサブリソース（通常はコレクション）を持つことができる。

**注:** ストレージシステムとAPIの間には概念的な類似性があるが、リソース指向APIを持つサービスは必ずしもデータベースではなく、リソースとメソッドの解釈において非常に柔軟性がある。API設計者は、APIがデータベーススキーマを反映することを期待**すべきではない**（should not）。実際、APIが基盤となるデータベーススキーマと同一であることはアンチパターンであり、サーフェスを基盤システムに強く結合させることになる。

### メソッド

リソース指向APIは、リソースに対して実行されるメソッド（機能）よりも、リソース（データモデル）を重視する。典型的なリソース指向APIは、多数のリソースに対して、各リソースに少数のメソッドを公開する。メソッドは、標準メソッド（[Get][]、[List][]、[Create][]、[Update][]、[Delete][]）または[カスタム メソッド][custom methods]のいずれかである。

標準メソッド（または同じ*サービス*内のカスタム メソッド）へのリクエストまたはレスポンスが、リソース**である**かリソースを**含む**場合、そのリソースのリソーススキーマはすべてのメソッド間で同一**でなければならない**（must）。

| 標準メソッド | リクエスト               | レスポンス               |
| ------------ | ------------------------ | ------------------------ |
| Create       | リソースを含む           | リソースである           |
| Get          | なし                     | リソースである           |
| Update       | リソースを含む           | リソースである           |
| Delete       | なし                     | なし                     |
| List         | なし                     | リソースを含む           |

*上の表は、各標準メソッドとリソースの関係を示している。「なし」は、リソース**ではない**ことも**含んでいない**こともないことを示す*

リソースは最低でも[Get][]をサポート**しなければならない**（must）。クライアントは、[Create][]、[Update][]、[Delete][]などの変更を実行した後、リソースの状態を検証できる必要がある。

リソースは[List][]もサポート**しなければならない**（must）。ただし、複数のリソースが存在できない[シングルトン リソース][singleton resources]は除く。

**注:** リソース指向設計におけるカスタム メソッドは、新しいまたはカスタムのHTTP動詞を定義することを意味するものではない。カスタム メソッドは従来のHTTP動詞（通常は`POST`）を使用し、カスタム動詞をURIで定義する。

APIは、標準メソッドをカスタム メソッドよりも優先**すべきである**（should）。カスタム メソッドの目的は、標準メソッドのいずれにもきれいにマッピングできない機能を定義することである。カスタム メソッドは、従来のRPC APIと同じ設計の自由を提供し、データベーストランザクション、インポートとエクスポート、データ分析などの一般的なプログラミングパターンを実装するために使用できる。

### 強い整合性

[管理プレーン][management plane]で操作するメソッドの場合、それらの操作の完了（成功またはエラー、long-running operation、同期のいずれか）は、リソースの存在状態とすべてのユーザー設定可能な値が定常状態に達したことを意味**しなければならない**（must）。

リソース[状態][state]に関連しない値については、[Output only][]の値も定常状態に達**すべきである**（should）。

例としては以下が含まれる：

- リソースに対する最新の変更として正常な作成が行われた後、そのリソースのgetリクエストはリソースを返却**しなければならない**（must）。
- リソースに対する最新の変更として正常な更新が行われた後、そのリソースのgetリクエストは、更新リクエストの最終値を返却**しなければならない**（must）。
- リソースに対する最新の変更として正常な削除が行われた後、そのリソースのgetリクエストは`NOT_FOUND`を返却**しなければならない**（must）（[soft delete][]の場合は`DELETED`状態値を持つリソースを返す）。

リソース指向APIのクライアントは、複数の操作を順次調整する必要があることが多く（例：リソースAを作成し、Aに依存するリソースBを作成する）、操作完了後にリソースがすぐに安定したユーザー状態を反映することを保証することで、クライアントはメソッドの完了を次の操作を開始する信号として信頼できる。

[Output only][]フィールドは理想的には同じガイドラインに従うべきであるが、これらのフィールドはリソースのライブ状態を表すことが多いため、状態変更を反映するために正常な変更操作の後にこれらの値が変更される必要が時として生じる。

### ステートレスプロトコル

現在利用可能なほとんどのパブリックAPIと同様に、リソース指向APIは[ステートレスプロトコル][stateless protocol]上で動作**しなければならない**（must）：個々のリクエストの基本的な動作は、呼び出し元による他のリクエストから独立している。つまり、各リクエストはそのクライアントまたは他のクライアントによる他のリクエストから分離して発生し、APIによって公開されるリソースは、一連の特定のリクエストを適用して目的のリソースに「到達」する必要なく、直接アドレス指定可能である。

ステートレスプロトコルを持つAPIでは、サーバーは複数のクライアント間で共有される可能性のあるデータの永続化を担当し、クライアントはアプリケーション状態の維持について単独の責任と権限を持つ。

### 循環参照

[リソース参照][resource references]など、リソース間の関係は[有向非巡回グラフ][directed acyclic graph]を介して表現可能**でなければならない**（must）。親子関係も非巡回**でなければならない**（must）、[AIP-124][]に従い、特定のリソースインスタンスは1つの正規の親リソースのみを持つ。

リソース間の循環関係は、リソース管理の複雑さを増大させる。相互に参照するリソースAとBを考えてみよう。これらのリソースを作成するプロセスは以下のとおり：

1. Bへの参照なしでリソースAを作成する。リソースAのIDを取得する。
2. Aへの参照付きでリソースBを作成する。リソースBのIDを取得する。
3. Bへの参照付きでリソースAを更新する。

削除操作も、正常な削除のためにどのリソースの参照を最初に解除する必要があるかの判断が必要となり、より複雑になる可能性がある。

この要件は、[output only][]フィールドを介して表現される関係には適用されない。これらのフィールドはユーザーが値を指定する必要がなく、結果としてリソース管理の複雑さを増大させないためである。
