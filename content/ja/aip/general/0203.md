# フィールド動作の文書化

Protocol Buffersでフィールドを定義する際には、フィールドの動作の特定の側面（requiredかoptionalかなど）をユーザーに説明するのが慣習である。さらに、他のツールがこの動作を理解することも有用である（例えば、クライアントライブラリのシグネチャを最適化するため）。

## ガイダンス

APIは`google.api.field_behavior`アノテーションを使用して、フィールドがrequiredやimmutableであるといった、よく理解されたフィールド動作を記述する。

```proto
// The audio data to be recognized.
RecognitionAudio audio = 2 [(google.api.field_behavior) = REQUIRED];
```

- APIは、リクエストで使用されるメッセージまたはサブメッセージ上の全てのフィールドに`google.api.field_behavior`アノテーションを適用**しなければならない**（must）。
- アノテーションは、フィールドの動作を正確に記述する[google.api.FieldBehavior][]値を含ま**なければならない**（must）。
  - `FIELD_BEHAVIOR_UNSPECIFIED`は使用**してはならない**（must not）。
- APIは最低でも`REQUIRED`、`OPTIONAL`、`OUTPUT_ONLY`のいずれかを使用**しなければならない**（must）。

**Warning:** `field_behavior`はprotoレベルの動作に影響を与えないが、多くのクライアント（CLIやSDKなど）はこれらに依存してコードを生成する。新しいフィールドを追加する際は、どの値が関連するかを徹底的にレビューし検討すること。

アノテーションがないフィールドは、後方互換性のために`OPTIONAL`として解釈される。それにもかかわらず、このアノテーションを省略**してはならない**（must not）。

**Note:** このドキュメントで与えられる語彙は*記述*目的のみであり、それ自体は検証を追加するものではない。目的はクライアントのためにこの動作を一貫して文書化することである。

### ネストされたメッセージのfield behavior

ネストされたメッセージ上の`google.api.field_behavior`アノテーションは、親のアノテーションから独立している。

例えば、ネストされたメッセージが`REQUIRED`のフィールド動作を持ち、親フィールドが`OPTIONAL`であることができる：

```proto
message Title {
  string text = 1 [(google.api.field_behavior) = REQUIRED];
}

message Slide {
  Title title = 1 [(google.api.field_behavior) = OPTIONAL];
}
```

上記のケースでは、`title`が指定された場合、`text`フィールドはrequiredである。

## 語彙

### Identifier

`IDENTIFIER`の使用は、リソースメッセージ内のフィールドがリソースを識別するために使用されることを示す。これは`name`フィールドに付与**しなければならず**（must）、他のフィールドに付与**してはならない**（must not）（[fields representing resource names]を参照）。

`IDENTIFIER`値は、そのフィールドがcreateメソッドのコンテキストでは入力として受け付けられない（すなわち`OUTPUT_ONLY`）一方で、`IMMUTABLE`と見なされ、リソースを主要な入力として受け付けるミューテーションメソッド（例：[Standard Update][aip-134]）では入力として受け付けられることを伝える。

このアノテーションは、メッセージ内の他のリソースへの参照には適用**してはならない**（must not）。

### Immutable

`IMMUTABLE`の使用は、リソース上のフィールドが作成後に変更できないことを示す。これは入力または出力、requiredまたはoptionalのフィールドに適用できる。

サービスが更新リクエスト（または類似のもの）でimmutableフィールドを受け取った場合、更新マスクに含まれていても、値が一致する場合はサービスはそのフィールドを無視**すべきである**（should）が、変更が要求された場合は`INVALID_ARGUMENT`でエラーに**すべきである**（should）。

immutableフィールドの潜在的なユースケース（網羅的なリストではない）は以下の通りである：

- アプリケーションのライフタイム中に変更できないリソースの属性（例：ディスクタイプ）。

**Note:** 「条件付きでimmutable」なフィールドには、immutableアノテーションを付与**してはならない**（must not）。

### Input only

`INPUT_ONLY`の使用は、そのフィールドがリクエストで提供され、対応するフィールドが出力に含まれないことを示す。

さらに、フィールドはリソースメッセージ内のフィールド、またはリソースメッセージに含まれるメッセージのフィールドである場合にのみ、input onlyとして記述**すべきである**（should）。特に、リクエストメッセージ（RPCの引数としてのみ機能し、名前が通常`Request`で終わるメッセージ）のフィールドは、これが既に暗黙的であるため、input onlyとして記述**すべきではない**（should not）。

input onlyフィールドの潜在的なユースケース（網羅的なリストではない）は以下の通りである：

- AIP-214で説明されている`ttl`フィールド。

**Warning:** Input onlyフィールドは稀であり、使用前に慎重に検討すべきである。

### Optional

`OPTIONAL`の使用は、フィールドが必須ではないことを示す。

フィールドは、リクエストメッセージ（RPCの引数であり、通常`Request`で終わるメッセージ）のフィールド、またはサブメッセージのフィールドである場合にoptionalとして記述**してもよい**（may）。

### Output only

`OUTPUT_ONLY`の使用は、そのフィールドがレスポンスで提供されるが、リクエスト内のメッセージにそのフィールドを含めても何も起こらない（サーバーはこのフィールドの値を全てクリア**しなければならず**（must）、入力時にこのフィールドに値が存在することを理由にエラーをスロー**してはならない**（must not））ことを示す。同様に、サービスは更新フィールドマスク内のoutput onlyフィールドの存在を無視**しなければならない**（must）（AIP-161参照）。

さらに、フィールドはリソースメッセージ内のフィールド、またはツリーのさらに下位のメッセージのフィールドである場合にのみ、output onlyとして記述**すべきである**（should）。特に、レスポンスメッセージ（RPCの戻り値としてのみ機能し、通常`Response`で終わるメッセージ）のフィールドは、これが既に暗黙的であるため、output onlyとして記述**すべきではない**（should not）。

Output onlyフィールドは、APIに適している場合は空の値に設定**してもよい**（may）。

output onlyフィールドの潜在的なユースケース（網羅的なリストではない）は以下の通りである：

- 作成または更新のタイムスタンプ。
- 元のユーザー入力に基づく派生または構造化情報。
- サービスによって割り当てられ、変更できないリソースのプロパティ。

### Required

`REQUIRED`の使用は、そのフィールドがリクエストまたはリソース上に存在**しなければならない**（must）（かつ空でない値に設定**しなければならない**）ことを示す。

フィールドは、*以下のいずれか*の場合にのみrequiredとして記述**すべきである**（should）：

- ユーザーが入力としてどこかで提供するリソース上のフィールドである。この場合、リソースは「truthy」な値が*保存*されている場合にのみ有効である。
  - リソースを[作成][aip-133]する際、createリクエストでフィールドに値を提供**しなければならない**（must）。
  - リソースを[更新][aip-134]する際、ユーザーはフィールドがフィールドマスクからも省略されている場合（フィールドに変更がないことを示す）、そのフィールドを省略**してもよい**（may）（そうでない場合は提供**しなければならない**（must））。
- リクエストメッセージ（RPCの引数であり、名前が通常`Request`で終わるメッセージ）のフィールドである。この場合、リクエストの一部として値を提供**しなければならず**（must）、提供しない場合はエラー（通常`INVALID_ARGUMENT`）が発生**しなければならない**（must）。

上記の「truthy」の定義は以下の通りである：

- プリミティブの場合、`0`、`0.0`、空文字列/バイト、`false`以外の値
- 繰り返しフィールドやマップの場合、少なくとも1つのエントリを持つ値
- メッセージの場合、少なくとも1つの「truthy」なフィールドを持つメッセージ。

フィールドは以下のことを示すためにrequiredとして記述**すべきではない**（should not）：

- レスポンスに常に存在するフィールド。
- 状況に応じて条件付きでrequiredとなるフィールド。
- ユーザー入力として使用されないメッセージ（リソースであるメッセージを含む）上のフィールド。

**Note:** ほとんどの場合、空の値（ブール値の`false`、整数の`0`、enumの未指定値など）は未設定の値と区別がつかず、したがってrequiredフィールドをfalsyな値に設定するとエラーになる。この帰結として、requiredなブール値は`true`に設定されなければならない。

### Unordered List

`UNORDERED_LIST`の使用は、リソースの繰り返しフィールドにおいて、サービスがリスト内のアイテムの順序を保証しないことを示す。

フィールドは、サービスがリスト内の要素の順序がユーザーが送信した順序と一致することを保証しない場合（サービスがユーザーに代わってリストをソートする場合を含む）、unordered listとして記述**すべきである**（should）。

unordered listを持つリソースは、リストを安定した順序で返して**もよい**（may）し、ランダム化された不安定な順序で返して**もよい**（may）。

## 後方互換性

`google.api.field_behavior`値を追加または変更することは、既存のクライアントにとって互換性がないと認識されるAPIの意味的変更を表す可能性がある。以下は`google.api.field_behavior`における後方互換性のない変更の例である：

* 既存の（暗黙的かどうかにかかわらず）`OPTIONAL`と見なされていたフィールドに`REQUIRED`を追加する
* 既存のリクエストメッセージに`REQUIRED`と注釈された新しいフィールドを追加する
* 以前は入力として受け付けられていた既存のフィールドに`OUTPUT_ONLY`を追加する
* 以前は出力として出力されていた既存のフィールドに`INPUT_ONLY`を追加する
* 以前は可変と見なされていた既存のフィールドに`IMMUTABLE`を追加する
* 以前は入力として無視されていた既存のフィールドから`OUTPUT_ONLY`を削除する
* 既存のフィールドから`IDENTIFIER`を削除する

後方互換性のある変更は以下の通りである：

* 既存のフィールドに`OPTIONAL`を追加する
* 既存の`name`フィールドに`IDENTIFIER`を追加する
* 既存のフィールドで`REQUIRED`から`OPTIONAL`に変更する
* 既存のフィールドで`OUTPUT_ONLY`や`IMMUTABLE`から`IDENTIFIER`に変更する
* 既存のフィールドから`REQUIRED`を削除する
* レスポンスから以前除外されていた既存のフィールドから`INPUT_ONLY`を削除する
* 以前はimmutableと見なされていた既存のフィールドから`IMMUTABLE`を削除する

## 理由

### Identifierフィールド動作

リソース名は、準拠するリソースの主要な識別子であり、作成時にユーザーが完全に構築することは決してない。そのようなフィールドは通常`OUTPUT_ONLY`フィールド動作が割り当てられる。しかし、リソース自体が主要なリクエストペイロードであるシナリオでは、主要な識別子として消費されることも多い。そのようなフィールドは`OUTPUT_ONLY`とは見なせ*ない*。さらに、Standard Updateのようなミューテーションリクエストでは、主要な識別子としてのリソース名はその場で変更できない。そのようなフィールドは通常`IMMUTABLE`フィールド動作が割り当てられる。これらの相反するコンテキスト依存のフィールド動作は、リソース名フィールドの動作を特定して伝えるために新しい値が必要であることを意味していた。

### 必須のアノテーションセット

リクエストメッセージで使用されるフィールドは、入力または出力のいずれかでなければならない。

出力の場合、`OUTPUT_ONLY`アノテーションで十分である。

入力の場合、フィールドはrequiredまたはoptionalのいずれかであり、したがって少なくとも`REQUIRED`または`OPTIONAL`のアノテーションを持つべきである。`INPUT_ONLY`のみを提供してもフィールドの必要性は伝わらないため、`REQUIRED`または`OPTIONAL`のいずれかを指定することが依然として必要である。

### フィールド動作の要求

各フィールドにフィールド動作アノテーションを含めることで、リソースが示す全体的な動作がより明確に定義される。明確に定義されたフィールド動作は、プログラムによるクライアントとユーザーの理解を向上させる。

アノテーションを要求することは、API作成者にAPIの初期作成時に動作を明示的に検討させることにもなる。

初期作成後にフィールド動作を変更すると、クライアントに後方互換性のない変更が生じる可能性がある。例えば、optionalフィールドをrequiredにすると、RPCのメソッドシグネチャや[宣言的クライアント][Declarative client]内のリソースに後方互換性のない変更が生じる。より詳細な互換性ガイダンスについては、[後方互換性](#backwards-compatibility)のセクションを参照のこと。

## 歴史

2023年5月にfield_behaviorが必須となった。この変更以前は、アノテーションはしばしば省略されていた。その値は高品質なクライアントを生成するために信頼されている。さらに、メジャーバージョン内で一部のfield_behavior値を後から追加または変更することは、後方互換性を損なう可能性がある。より詳細な互換性ガイダンスについては、[後方互換性](#backwards-compatibility)のセクションを参照のこと。

API作成時にfield_behaviorを要求することの利点は、そうしない場合のクライアントやAPIユーザーへのコストを上回る。
