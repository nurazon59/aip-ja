# プロジェクト識別子

歴史的に、GoogleにはAPIプロジェクトとApp Engineプロジェクトの2種類のプロジェクトがあった。APIプロジェクトはプロジェクト番号（例：`12345`）を識別子として使用し、App EngineプロジェクトはプロジェクトID（例：`happy-armadillo-789`）を識別子として使用していた。その後、GoogleはAPIプロジェクトとApp Engineプロジェクトを統合したため、現在は各プロジェクトが_両方_の一意かつimmutableな識別子を持つ。

2種類の識別子は異なるコンテキストで異なる方法で使用され、アプリケーション開発に多くの複雑さをもたらしている。重要な問題の1つは、異なるサービスが異なるプロジェクト識別子を使用するため、アプリケーションが異なるサービスのデータを確実に結合できないことである。

## ガイダンス

**要約：** プロジェクト番号が正規の識別子であり、プロジェクトIDは[エイリアス][alias]である。ただし、通常のエイリアスとは異なり、ユーザーが送信したものである場合は返す**すべきである**（should）。さらに、サードパーティのサービスはプロジェクトIDを_受け付ける_ことができない。

その理由は以下の通りである：

- 各リソースは常に1つの正規の識別子を持つべきである。
- GoogleのプライバシーポリシーがプロジェクトIDの使用を、内部的にもパートナーとの間でも制限しているため、プロジェクト番号のみが正規の識別子になり得る。
- しかし、プロジェクト番号が正規の識別子であっても、ユーザーがプロジェクトIDを送信した場合でもそれを返すというポリシーは、人間と宣言的ツールの両方にとって使い勝手が悪いことが証明されている。

### Google API

外部向けGoogle APIは、受信APIリクエストに対してプロジェクトIDとプロジェクト番号の両方を受け付ける**すべきである**（should）。

しかし、AIP-122で説明されているようにプロジェクト番号が正規の識別子であっても、サービスはユーザーが送信した方のIDを返す**すべきである**（should）。その理由は、ユーザーフレンドリーなプロジェクトIDとユーザーフレンドリーでないプロジェクト番号の間の自動変換が、ユーザーや宣言的ツールに実際の困難を引き起こすことが証明されているためである（宣言的フレンドリー性の詳細についてはAIP-128を参照）。

追加のポイント：

- エラーレスポンスは、元の提供値を変更せずに返さ**なければならない**（must）。エラーレスポンスは、プロジェクトIDとプロジェクト番号の間の変換を実行**してはならない**（must not）。
- サービスが自身の所有しないリソースのリソース名を受け取った場合、それらのリソース名に対してプロジェクトIDとプロジェクト番号の間の変換を実行**すべきではない**（should not）。

### 内部Googleサービス

内部Googleサービスは、内部データストレージおよび出力にプロジェクト番号を使用**しなければならない**（must）。プロジェクト識別子はストレージキーとして広く使用されており、ログやメトリクスに頻繁に現れる。プロジェクトIDはユーザーが設定可能であり、したがってPIIおよびユーザーデータと見なされるが、プロジェクト番号はそうではない。

したがって、内部サービスが外部のGoogle APIを呼び出す場合、APIリクエストにはプロジェクト番号を使用**すべきである**（should）。

### リソース参照

プロジェクト識別子は[リソース名][resource names]にも現れる。これらのリソース名はリソース自体を識別するために使用され、他のリソースを参照することもできる（[例][example]）。

プロジェクト識別子が提供された場合、レスポンスはリクエストで出現したままの識別子を含む**すべきである**（should）。プロジェクトIDが提供された場合はそれが返されるべきであり、プロジェクト番号が提供された場合はそれがレスポンスに含まれるべきである。

例えば、`Book` リソースを考える。

```proto
message Book {
  option (google.api.resource) = {
    type: "pubsub.googleapis.com/Book"
    pattern: "projects/{project}/books/{book}"
  };

  // The resource name of the Book.
  string name = 1 [(google.api.field_behavior) = IDENTIFIER];

  // A reference to another resource, a Shelf.
  string shelf = 2 [(google.api.resource_reference) = {
    type: "library.googleapis.com/Shelf"
  }];

  // Other fields...
}
```

以前の作成リクエストから送信された以下のブックを（この例ではJSONで表現）、

```json
{
  "name": "projects/my-project/books/les-miserables",
  "shelf": "projects/12345/shelves/top-shelf"
}
```

そして `GetBookRequest` を考える。

```proto
message GetBookRequest {
  // The name of the Book to retrieve.
  // Format: projects/{project}/books/{book}
  string name = 1 [
    (google.api.field_behavior) = REQUIRED,
    (google.api.resource_reference) = {
      type: "library.googleapis.com/Book"
    }];
}

```

そのようなリクエストの `name` の値が以下の場合、

```
projects/my-project/books/les-miserables
```

その場合、`Book` について返されるフィールド `name` の値は一致**すべきである**（should）。プロジェクトIDが返されるべきであり、番号ではない。しかし、フィールド `shelf` の値はプロジェクト番号を使用すべきである。なぜなら作成リクエストがリソース名にプロジェクト番号を含むshelfを送信したからである。

言い換えると、以下の値が返されるべきである：

```
book: projects/my-project/books/les-miserables
shelf: projects/12345/shelves/top-shelf
```

このガイダンスは、ユーザーの入力と出力の間の差分を最小限にすることを保証するためのものである。

### サードパーティサービス

Google Cloud Platformと統合されている[サードパーティサービス](https://cloud.google.com/marketplace)は、プロジェクト番号のみを保存または提供**しなければならない**（must）。Googleのプライバシーポリシーは、サードパーティサービスとのプロジェクトIDの共有、または実行時にプロジェクトIDとプロジェクト番号を変換するサービスのサードパーティサービスへの提供を禁止している。

### プロジェクト識別子の形式

サービスは、プロジェクトを参照するために[Resource Manager API][]で定義されているプロジェクトリソース名（例：`projects/123456`）を使用**しなければならない**（must）。これにより、同じAPIが組織やフォルダなど、プロジェクトに類似した他のリソースでも機能する。
