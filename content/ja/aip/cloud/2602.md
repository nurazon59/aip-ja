# パラメータ依存の引数

CLIコマンドを設計する際、ある概念的なアクションに対応するコマンドが、特定のパラメータの値に応じて、大きく異なる、ほとんど重複しない引数を取るというのは一般的なシナリオである。

例えば、画像を管理する新しいgcloudサーフェスを設計していて、画像をJPEG、GIF、PNGにフォーマットするコマンドが必要だとしよう。画像フォーマットに関わらず、コマンドはソースファイルと出力先ファイルの引数を必要とするが、それ以外はフォーマットに応じてまったく異なるオプションセットが適用される。例えばJPEGは、スムージング、サブサンプリング、DCT方式の引数を取るかもしれない。これらはJPEG画像にのみ関連する。一方GIFは、アニメーションgifを制御する引数（ループを永遠に続けるかどうか、フレーム間の遅延時間など）を取るかもしれない。最後にPNGは、カラータイプとチャンネルあたりのビット数の引数を取るかもしれない。

## ガイダンス

パラメータ固有の引数が他の引数と比較して多い場合、アクションのコマンドグループを作成し、各パラメータ値にちなんだ個別のサブコマンドを作成する。上記の例では、`gcloud images format` コマンドグループを作成し、`jpeg`、`gif`、`png` というサブコマンドを用意する。これらのサブコマンドは、それぞれフォーマット固有のフラグを取ることができる。

使用例：

```
$ gcloud images format jpeg --help

$ gcloud images format jpeg --source-file=foo --destination-file=bar \
    --dct-method=integer --smoothing=0.1 --subsampling=4:4:4

$ gcloud images format png --source-file=foo --destination-file=bar \
    --color-type=0 --bits-per-channel=16

$ gcloud images format gif --source-file=foo --destination-file=bar \
    --loop-forever --frame-delay=1ms
```

時間の経過とともに、すべてのパラメータに共通する引数が増えていく可能性がある。その場合、代わりにパラメータの引数を取る単一のコマンドを使用することが適切かもしれない。これは、パラメータを位置引数にすることで、後方互換性を壊さずに行える。上記の例では、`format` をコマンドグループからコマンドに変更し、`jpeg`、`gif`、`png` のいずれかである画像フォーマットの位置引数を取るようにする。上記のすべてのコマンド例は、同一の機能を維持することに注意（最初のコマンドはヘルプテキストのみに影響するため、破壊的変更とは見なされない）。

## 検討された代替案

このようなコマンドの設計には、他にもいくつかの可能性がある。以下に概説する：

### パラメータに対応する明示的なフラグを持つ単一のコマンド

例では、画像フォーマットを指定する `--type` フラグを使用する：

```
$ gcloud images format --source-file=source --destination-file=dest \
    --type=JPEG --dct-method=integer --smoothing=0.1 --subsampling=4:4:4

$ gcloud images format --source-file=source --destination-file=dest \
    --type=GIF --loop-forever --frame-delay=1ms

$ gcloud images format --source-file=source --destination-file=dest \
    --type=PNG --color-type=0 --bits-per-channel=16
```

概念的には単一のフォーマットコマンドを持つことが最も理にかなっている。しかし、このアプローチにはいくつかの欠点がある：

- 不必要なヘルプテキスト。ユーザーにはフォーマット固有のオプションがすべて表示され、そのほとんどは異なるフォーマットに適用されるため無関係である。PhotoshopやGIMPのようなグラフィカルな画像編集プログラムでは、ユーザーがドロップダウンから目的のフォーマットを選択すると、UIがこれらのフォーマット固有のオプションを選択的に表示できる。しかしCLIでは、ヘルプテキストが静的に生成されるため、そのような機能はない。
- バリデーションに追加のロジックが必要。フォーマットによって無効な引数があるため、コマンド作成者は無効な組み合わせの指定が適切なエラーを返すようにする必要がある。これは適切にネストされたミューテックスグループで実現できるが、ネストが過度に深く複雑になる可能性がある。

### アクション名とパラメータ値をハイフンでつなげた複数のコマンド

例では、以下のようになる：

```
$ gcloud images format-jpeg ...
$ gcloud images format-gif ...
$ gcloud images format-png ...
```

これは、各パラメータ値が独自のコマンドを持つという点で推奨される設計と類似している。しかし、欠点もある：

- 後方互換性。将来的にパラメータ値を単一のコマンドの引数にすることが望ましくなった場合、破壊的変更が必要になる。
- コマンドツリーのレイアウトの観点からエレガントでない。パラメータ固有のコマンドをコマンドグループにグループ化することで、コマンド空間を自然に分解でき、gcloudのCLI設計思想に沿ったものになり、ヘルプテキストとオートコンプリートにおける[プログレッシブ・ディスクロージャー](https://en.wikipedia.org/wiki/Progressive_disclosure)が可能になる。
