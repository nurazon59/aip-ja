# 数値引数

一部のAPIフィールドは、パーセンテージまたは固定数のいずれかを参照する。例えば、GCEではグループ内で同時にメンテナンスを受けられるインスタンスの割合または数の構成可能な制限を持つグループを作成できる：

```proto
message ConcurrencyControl {
  enum LimitType {
    INVALID = 0;
    PERCENT = 1;
    FIXED = 2;
  }
  optional int32 concurrency_limit = 1;
  optional LimitType limit_type = 2;
}
```

## ガイダンス

### フラグレイアウト

gcloudでは、そのようなAPIフィールドは、数値を指定するフラグとパーセンテージを指定するフラグの2つからなる相互排他グループに対応すべきである。上記の例のAPIフィールドは、gcloudでは以下のサーフェス仕様に対応する：

```yaml
- group:
    mutex: true
    arguments:
      - name: concurrency-limit
        help_text: |
          Maximum number of instances in the group that can undergo maintenance
          simultaneously.
      - name: concurrency-limit-percent
        help_text: |
          Integer from 0 to 100 representing the maximum percentage of instances
          in the group that can undergo maintenance simultaneously.
```

### フラグ命名

パーセンテージを取るフラグは、`-percent` で終わるべきである（`-percentage` ではない）。

### フラグタイプ

パーセンテージを取るフラグは、0から100の整数を取るべきである。より高い精度が必要な場合は、0から100の浮動小数点数を取ることも許容される。

## 検討された代替案

### 1つのフラグ、フラグ値が「%」で終わる場合はパーセンテージ、それ以外は数値と見なす

言い換えると、10%を指定するには `--concurrency-limit=10%` を使用し、10インスタンスを指定するには `--concurrency-limit=10` を使用する。

これは、単一の概念（制限）を単一のフラグ（`--concurrency-limit`）に自然にマッピングするため、よりクリーンな設計であり、ヘルプテキストを乱雑にする追加のフラグを不要にする。

しかし、このアプローチを推奨しない主な理由は、ユーザーエラーのリスクが大幅に増加することである。例えば、グループに現在20のインスタンスがあり、同時実行制限が10%で、ユーザーが制限を20%に更新したいとする。ユーザーがdescribeコマンドを発行して（またはGETリクエストを実行して）既存の制限を確認すると、APIレスポンスには以下のような内容が含まれる：

```yaml
concurrencyControl:
  concurrencyLimit: 10
  limitType: PERCENT
```

ユーザーはlimitTypeを見落とし、`--concurrency-limit=20` で制限が20%に更新されると単純に想定しやすい。しかし、実際にはこれは制限を20インスタンス（グループの100%）に設定することになり、壊滅的な結果を招く可能性がある。

推奨されるアプローチは、この曖昧さをエレガントさを犠牲にして回避するものであるが、このトレードオフは必要であると判断する。

### 1つのフラグ、フラグ値が「%」で終わる場合はパーセンテージ、「n」で終わる場合は数値と見なす

言い換えると、10%を指定するには `--concurrency-limit=10%` を使用し、10インスタンスを指定するには `--concurrency-limit=10n` を使用する。`%` または `n` で終わらない値を提供するとエラーになる。

これはパーセンテージと数値をユーザーが混同する可能性を回避するが、数値の末尾に 'n' を要求するのはエレガントでないUXである。数値は当然数値のように見えるべきであり、'n' の選択もかなり恣意的である。

## 例外

このCIPは、タイムスタンプのようなAPIフィールドには適用されない。ここでは、ユーザーが同じフラグに絶対時刻または期間のいずれかを入力できるようにする。これは、絶対時刻と期間には異なる自然な形式があり、したがってユーザーがこれらを混同する可能性がないためである。一般にこのような場合、UXのシンプルさのために両方の形式を受け付ける1つのフラグを優先すべきである。
