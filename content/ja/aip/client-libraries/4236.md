# Version-aware clients

APIは[APIインターフェース][interface def]に [`google.api.api_version`][annotation] でアノテーションできる。`google.api.api_version` が指定されている場合、バージョン認識クライアントは**しなければならない**（must）`google.api.api_version` の値をAPIへのリクエストおよびAPIインターフェース単位のクライアントのドキュメントに含める。

## Generator and client library behavior

APIインターフェースに `google.api.api_version` がアノテーションされている場合、client libraryジェネレータは**しなければならない**（must）HTTPクエリパラメータ `$apiVersion` またはHTTPヘッダー/gRPCメタデータキー `X-Goog-Api-Version` のいずれかを含め、リクエストには両方を含めては**ならない**（must not）。

### Generated client documentation

特定のAPIインターフェースに対する生成クライアントドキュメントは**しなければならない**（must）ソースprotoに存在する場合、`google.api.api_version` の値を含める。コメントは**しなければならない**（must）元のAPIインターフェース名とアノテーションの正確な内容を含める。

**Note:** テキストは以下の例に正確に一致する必要はないが、ここで説明する要件を満たす必要がある。

例えば、以下のAPIインターフェース定義は、次のGoクライアントドキュメントを生成する：

```proto
service LibraryService {
    option (google.api.api_version) = "2026-01-01";
}
```

```go
// The LibraryClient is a client for interacting with the Cloud Library...
//
// This client uses LibraryService version 2026-01-01.
type LibraryClient struct {}
```

[APIサービス][service def]全体のクライアントパッケージに対する生成ドキュメントは**しなければならない**（must）パッケージ内に存在するクライアント-インターフェース-バージョンのタプルをリストするセクションを含める。例えば、以下のAPIインターフェース定義は、次のクライアントパッケージドキュメントセクションを生成する：

```proto
service LibraryService {
    option (google.api.api_version) = "2026-01-01";
}
service BookService {
    option (google.api.api_version) = "2026-05-15";
}
service ShelfService {
    option (google.api.api_version) = "2026-02-05";
}
```

```md
## API Versions

* LibraryClient uses LibraryService version 2026-01-01
* BookClient uses BookService version 2026-05-15
* ShelfClient uses ShelfService version 2026-02-05

```

すべてのAPIインターフェースが同じAPIバージョンを共有する場合、このリストは**すべきである**（should）簡潔さのために単一の文に縮約される。例えば、上記の定義で全てのバージョンが同じだった場合、生成されるクライアントパッケージドキュメントは以下のようになる：

```md
## API Versions

All clients use API version 2026-01-01.
```

### Version opacity

クライアントとジェネレータは**しなければならない**（must）`google.api.api_version` の値を不透明として扱い、堅牢な互換性を確保する。これは、バージョン文字列の特定の形式や構造を、意図されたAPIバージョンを識別する目的以外で解析または解釈しては**ならない**（must not）ことを意味する。

## Rationale

### Necessity for Versioning

`google.api.api_version` アノテーションを使用した明示的なAPIバージョニングは、時間の経過とともにクライアントとサービスの間の互換性を維持するために不可欠である。サービスが進化するにつれて、そのスキーマや振る舞いが変更される可能性がある。APIバージョンを指定することで、クライアントはその期待をAPIサービスに伝える。これにより、APIサービスはクライアントの意図したセマンティクスに一貫した方法で応答し、互換性のない変更によるエラーや予期しない結果を防ぐことができる。

### Importance of Opaque Treatment

`google.api.api_version` の値を不透明として扱うことは、堅牢な互換性保証を確保するために重要である。この識別子を不透明に使用することで、クライアントはAPI自体とは独立して変更される可能性のある基盤となるバージョニングスキームについての仮定を行うことを避けられる。この柔軟性により、APIサービスのプロバイダーはクライアントの互換性に影響を与えることなく、バージョニング戦略を進化させることができる。

### Mutual Exclusivity of Query and Header

クエリパラメータとヘッダーの両方のメカニズムは、異なるクライアント環境に柔軟性を提供するために存在する。しかし、両方を同時に許可すると、値が異なる場合に曖昧さが生じる可能性がある。一貫したバージョン識別を確保し、潜在的な競合を防ぐために、一度に1つのメカニズムのみを使用すべきである。

### Inclusion in documentation

APIバージョンは、クライアントひいてはエンドユーザーが消費しているAPIコントラクトのイテレーションを識別する。エンドユーザーは、クライアントが使用しているAPIバージョンを他のAPIアーティファクト（プロダクトドキュメントや他のクライアントサーフェスなど）と関連付ける手段を必要とし、その逆も同様である。
