# Parsing resource names

リソース指向設計（[AIP-121][]）では、_リソース_ はAPI内の主要な名詞を表し、多くの場合 _リソース名_（[AIP-122][]）を持つ。これらのリソース名は、そのAPIにおけるリソース構造の階層と構造に関する情報を伝える。APIはリソースの取得や変更時、および他のオブジェクト内で参照する際に、これらのリソース名をパラメータとして受け付ける。しかし、ユーザーは送信用のリソース名を組み立てるのに苦労することがあり、client libraryはヘルパーコンポーネントを使用することでこれを容易にする機会がある。

## Guidance

client libraryは**してもよい**（may）リソース名の構築をより簡単にするためのヘルパークラスや関数を提供する。ただし、この機能を実装することを選択したclient libraryは**しなければならない**（must）プレーンな文字列も常に受け入れる。その理由は2つある：

1. リソースディスクリプタアノテーションを追加する既存のAPIは、破壊的変更を引き起こさずにそれを実行できる必要がある。
2. リソース名パターンは時々進化するため、ユーザーは静的にコンパイルされたclient libraryがまだ認識していない可能性のあるリソース名を送受信できる必要がある。

### Resource messages

API内のリソースは常に、そのAPIでのリソースの表現を記述するメッセージを持つ。client libraryジェネレータは、これらのメッセージが `google.api.resource` アノテーションでアノテーションされている場合に認識できる：

```proto
// A representation of a Topic in Pub/Sub.
message Topic {
  option (google.api.resource) = {
    type: "pubsub.googleapis.com/Topic"
    pattern: "projects/{project}/topics/{topic}"
  };

  // name and so on...
}
```

- `type` フィールドは[統一リソース型][aip-123]名を提供する。
  - client libraryは**すべきである**（should）この値に基づいてヘルパーコンポーネントを命名する。
  - ユニバーサルリソース型は通常（常にではないが）メッセージ名と一致する名前で終わる。
- `pattern` フィールドはこのリソース型の名前の構造を提供する。
  - 中括弧内のコンポーネントは変数置換を表す。この機能を実装するclient libraryは**しなければならない**（must）リソース名文字列を構築する際にこれらの名前に基づく変数を受け入れる。
  - 変数置換名は通常 `snake_case` で指定されるが、これは保証されない。client libraryは**すべきである**（should）一貫性のあるケースシステムを使用する任意のアノテーションを受け入れられる。
  - パターンは通常スラッシュ区切りであるが、これは保証されない。client libraryは**すべきである**（should）文字列補間を使用してリソース名を組み立てる。
- 定義メッセージは `name` というフィールド（リソース名を保持するフィールド）を含むことが期待される。
  - APIは `google.api.resource` アノテーションの `name_field` プロパティを設定することで、nameフィールドの名前を上書きできる。
  - コードジェネレータは**すべきである**（should）メッセージがリソースとしてアノテーションされており、nameフィールドがない（デフォルトの `name` またはアノテーションの `name_field` プロパティで提供されたフィールドのいずれも）場合、エラーで失敗する。コードジェネレータは**すべきである**（should）フィールドがstring型でない場合もエラーで失敗する。

### Multi-pattern resources

リソースが複数のパターンを持つことがある。一般的なケースは、リソースが複数の親タイプの下に存在できる場合である。この状況では、アノテーションの `pattern` フィールドを複数回指定できる：

```proto
message LogEntry {
  option (google.api.resource) = {
    type: "logging.googleapis.com/Log"
    pattern: "projects/{project}/logs/{log}"
    pattern: "organizations/{organization}/logs/{log}"
    pattern: "folders/{folder}/logs/{log}"
    pattern: "billingAccounts/{billing_account}/logs/{log}"
  };

  // name and so on...
}
```

必要に応じて、client libraryは**してもよい**（may）パターンごとに個別のヘルパーコンポーネントを作成し、**してもよい**（may）統合コンポーネントを提供してもよい。

### Resources without messages

リソースがAPIサービスによって暗黙的に定義されているが、そのリソースを表す明示的なメッセージがない場合がある。（例えば、[Firestore API][] はデータベースをそのリソースの共通祖先として定義しているが、データベースメッセージを定義していない。）

この状況では、APIは _メッセージ_ ではなく _ファイル_ に `google.api.resource_definition` アノテーションを使用してリソースをアノテーションする：

```proto
option (google.api.resource_definition) = {
  type: "firestore.googleapis.com/Database"
  pattern: "projects/{project}/databases/{database}"
};
```

この機能を実装するclient libraryジェネレータは**しなければならない**（must）[リソースメッセージ](#resource-messages)に遭遇した場合に生成されるものと同じユーティリティコンポーネントを生成する。

### Referencing other resources

APIは、他の場所で定義されたフィールドを参照するためにリソース名をよく使用する。これは特に[Get][aip-131]や[Update][aip-134]などの標準メソッドのリクエストメッセージで一般的であるが、リソースや他の構造もリソース名文字列を参照として使用する。

この機能を実装するclient libraryは**すべきである**（should）リソースが参照されている場合にもヘルパーコンポーネントを提供する。client libraryは、`google.api.resource_reference` アノテーションでアノテーションされている場合にこれらのフィールドを認識できる：

```proto
message GetTopicRequest {
  // The name of the topic to retrieve.
  string name = 1 [(google.api.resource_reference) = {
    type: "pubsub.googleapis.com/Topic"
  }];
}
```

リソース参照は[統一リソース型][aip-123]名を参照する。

一部のメソッドは型の _親_ も参照し、複数の親がある状況では個々の親型をすべて参照するのは冗長でエラーが発生しやすい。このような状況では、APIプロデューサーは `type` ではなく `child_type` を指定する：

```proto
message ListLogEntriesRequest {
  // The collection of log entries to list.
  string parent = 1 [(google.api.resource_reference) = {
    child_type: "logging.googleapis.com/Log"
  }];
}
```

この状況では、この機能を実装するclient libraryジェネレータは**しなければならない**（must）子タイプから親リソースのセットを導出する。client libraryジェネレータは**しなければならない**（must）`type` と `child_type` の両方が提供された場合はエラーで失敗する。

### Referencing an arbitrary resource

フィールドが任意のリソースを参照する場合がある。この場合、APIはリソース参照に特別な値 `*` を使用する。

```proto
message GetIamPolicyRequest {
  string resource = 1 [(google.api.resource_reference) = {
    type: "*"
  }];
}
```

この状況では、この機能を実装するclient libraryジェネレータは**してもよい**（may）そのリソース名を扱うための汎用ユーティリティクラスまたは関数を提供する。

### Complex resource ID path segments

**Warning:** 複合リソースIDパスセグメントは、新しいAPIでは一般的に使用**すべきではない**（should not）。[AIP-124][] には多対多の関連付けの扱いに関するアドバイスが含まれている。

リソースパターンは**してもよい**（may）変数セパレータで区切られた複数のパターン変数を含むリソースIDパスセグメントを含む：

```proto
message FeedItemTarget {
  option (google.api.resource) = {
    type: "googleads.googleapis.com/FeedItemTarget"
    pattern: "customers/{customer}/feedItemTargets/{feed}~{feed_item}"
  };

  // name and other fields...
}
```

これは、リソースIDが自然に複数の部分から構成され、ユーザーが個別のID部分を操作できると便利な場合にのみ使用される。

- 変数セパレータは1文字の長さであり、**しなければならない**（must）`_`、`-`、`.`、`~`（アンダースコア、ハイフン、ピリオド、チルダ）のいずれかのみである。
- 変数セパレータは**してはならない**（must not）最初のパターン変数の前または最後のパターン変数の後に出現する。

### Backwards compatibility

リソースのヘルパーコンポーネントを実装するclient libraryは**しなければならない**（must）以下の後方互換性の期待に従う：

- 既存のメッセージへの `google.api.resource` アノテーションの追加は**しなければならない**（must）後方互換性のある変更である。
- 既存のリソースは、以下の条件が満たされる限り、破壊的変更なしに新しいパターン（"*" ワイルドカードパターンを含む）を追加でき**なければならない**（must）：
  - 新しいパターンは常にリストの末尾に追加されなければならない。
  - 新しいパターンは、このリソース内の既存のすべてのパターンと比較して、異なるコレクション識別子のシーケンスを使用しなければならない（[AIP-122][] 参照）。
- パターン変数内の識別子は最終的なものであり、変更**してはならない**（must not）。これらの識別子は生成されたclient libraryのサーフェスで使用される。
- 既存のフィールドへの `google.api.resource_reference` アノテーションの追加は**しなければならない**（must）後方互換性のある変更である。
- `google.api.resource_reference` を `child_type` から `type` に変更することは、参照されていた `child_type` が単一の `pattern` を持ち、新しく参照される `type` が一致する親リソース `pattern` である場合、後方互換性のある変更で**しなければならない**（must）（詳細は[他のリソースの参照](#referencing-other-resources)セクションを参照）。
- リクエスト内の参照について、`type` から `child_type` に変更することは、元々 `type` を介して参照されていたリソース `pattern` が、`child_type` の解決を通じて導出される可能なリソース `pattern` のセットに含まれている場合、後方互換性のある変更で**しなければならない**（must）（詳細な例については上記の[マルチパターンリソース](#multi-pattern-resources)および[他のリソースの参照](#referencing-other-resources)セクションを参照）。

Note: `ORIGINALLY_SINGLE_PATTERN` および `FUTURE_MULTI_PATTERN` フラグは非推奨であり、使用してはならない。
