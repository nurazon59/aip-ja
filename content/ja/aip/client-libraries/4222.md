# Routing headers

状況によっては、gRPC APIバックエンドはリクエスト内のいくつかの値を把握していればトラフィックをより効率的にルーティングできるが、リクエストペイロードをワイヤ上で合理的に分解してそのルーティングを実行することはできない。

## Guidance

コードジェネレータは**しなければならない**（must）アノテーションを使用して、RPC単位でリクエストペイロードからルーティング情報を抽出し、その情報をルーティングヘッダーに追加するclient libraryを生成する。

リクエストペイロードからルーティング情報を抽出する方法を指定するアノテーションは2つある：

*  ルーティングヘッダーを明示的に構築する方法を指定する [`google.api.routing`][routing] アノテーション
*  ルーティングヘッダーを暗黙的に構築する方法を指定する可能性のある [`google.api.http`][http] アノテーション

任意のRPCについて、明示的なルーティングヘッダーアノテーションが存在する場合、コードジェネレータは**しなければならない**（must）それを使用し、[`google.api.http`][http] アノテーションで暗黙的に指定されている可能性のあるルーティングヘッダーを無視する。明示的なルーティングヘッダーアノテーションがない場合、コードジェネレータは**しなければならない**（must）[`google.api.http`][http] アノテーションを解析してルーティングヘッダーが暗黙的に指定されているかどうかを確認し、その指定を使用する。

## Explicit Routing Headers（`google.api.routing`）

unaryまたはserver-streaming RPCの場合、コードジェネレータは**しなければならない**（must）[`google.api.routing`][routing] アノテーションで指定されたルーティングパラメータを（存在する場合）確認する。任意のルーティングパラメータは、フィールド名と、正確に1つの名前付きリソースIDパスセグメントを持つパターンを指定する。例：

```proto
rpc CreateTopic(CreateTopicRequest) {
  option (google.api.routing) = {
    routing_parameters {
      field: "parent"
      path_template: "{project=projects/*}/**"
    }
  }
}
```

`field` フィールドの値は**しなければならない**（must）以下のいずれかである：

1. リクエストメッセージのトップレベルにあるフィールドの名前
1. リクエストメッセージのサブメッセージ内のフィールドに至るドット区切りのフィールド名パス（例： `"book.author.name"`。ここで `book` はリクエストメッセージ内のメッセージフィールド、`author` は `book` メッセージ内のメッセージフィールド、`name` は `author` メッセージ内の `string` フィールド）

`field` フィールドで指定される _実際のフィールド_ は**しなければならない**（must）以下の特性を持つ：
- 型が `string` である
- リソース名に類似したパス形式の値を持つか、`project_id` などの個別のパスセグメントとして適切な非構造化値を含む

**Note:** 空の `google.api.routing` アノテーションは許容される。これは、そうでなければ（例えば `google.api.http` アノテーションから暗黙的に）生成されるであろうルーティングヘッダーを、そのRPCに対して生成しないことを意味する。

**Note:** リソースIDセグメントのパターンを省略することは許容される。例えば `{parent}` は `{parent=*}` と同等であり、解析**しなければならない**（must）。例：
```proto
routing_parameters {
  field: "parent"
  path_template: "projects/{parent}"
}
```

以下と同じである：

```proto
routing_parameters {
  field: "parent"
  path_template: "projects/{parent=*}"
}
```

**Note:** `path_template` フィールド自体を完全に省略することは許容される。`path_template` の省略は、フィールドと同じリソースID名とパターン `**` を持つ `path_template` と同等であり、解析**しなければならない**（must）。例：
```proto
routing_parameters {
  field: "parent"
}
```

以下と同じである：

```proto
routing_parameters {
  field: "parent"
  path_template: "{parent=**}"
}
```

**Note:** `path_template` フィールドが省略されていることは、値が空のキーと値のペアを送信できることを示すものではない。これは単なる省略記法である。

ユーザーが `CreateTopicRequest` のインスタンスをメソッドに提供すると、client libraryは**しなければならない**（must）すべてのルーティングパラメータを指定された順序でそのインスタンスのフィールドにマッチさせる。各ルーティングパラメータについて、`path_template` のパターンは**しなければならない**（must）ルーティングパラメータの `field` フィールドで指定された入力メッセージフィールドにマッチされる。マッチした場合、リソースIDパスセグメントの名前をキーとして使用し、リソースIDパスセグメントのマッチ値を `x-goog-request-params` ヘッダーに追加するキーと値のペアの値として使用する。

キーと値の両方は**しなければならない**（must）[RFC 6570 §3.2.2][] に従ってURLエンコードされる。これは標準ライブラリのURLエンコーディングで実行できる。例えば、RubyでこのヘッダーをgRPCリクエストに追加する場合：

```ruby
header_params = {}
if (pattern_matches("{project=projects/*}/**", request.parent))
  header_params["project"] = extract_match_value("{project=projects/*}/**", request.parent)
end
request_params_header = URI.encode_www_form header_params
metadata[:"x-goog-request-params"] = request_params_header
```

複数のルーティングパラメータが同じリソースIDパスセグメント名を持ち、結果として同じヘッダーキーを参照する場合、「最後が勝つ」ルールを使用して送信する値を決定する。ここでの「最後」とは、アノテーションで指定された順序を意味する。同じリソースIDセグメント名を持つルーティングパラメータの一部がフィールドとのマッチに失敗した場合、フィールドが未設定の場合、または抽出されたマッチ値が空文字列の場合は、これらのパラメータは送信する値を決定する際に考慮されない。

例：

```proto
option (google.api.routing) = {
  routing_parameters {
    field: "parent"
    path_template: "{project=projects/*}/**"
  }
  routing_parameters {
    field: "parent"
    path_template: "{project=projects/*/subprojects/*}/**"
  }
  routing_parameters {
    field: "billing_project"
    path_template: "{project=**}"
  }
}
```

この場合、特定のリクエストで `billing_project` フィールドが空でない値に設定されている場合、`billing_project` フィールドを参照するルーティングパラメータが最後に指定されているため、その値が `project` キーで送信される。`billing_project` フィールドが設定されていない場合は、`parent` フィールドが考慮され、最初にサブプロジェクトが指定されたプロジェクトを送信しようとし、次にサブプロジェクトなしを試みる。特定のリクエストの `parent` フィールドが例えば `projects/100/subprojects/200/foo` のような値を持つ場合、1番目と2番目の `routing_parameters` の両方のパターンがマッチするが、2番目のパターンが「より後に」指定されているため「勝つ」。

同じリソースIDセグメント名を持つすべてのルーティングパラメータがフィールドとのマッチに失敗した場合、それらのルーティングパラメータのリソースIDパスセグメント名に対応するキーと値のペアは送信**してはならない**（must not）。

ルーティングパラメータのいずれもそれぞれのフィールドにマッチしなかった場合、ルーティングヘッダーは送信**してはならない**（must not）。

URLパラメータと同様に、送信するキーと値のペアが複数ある場合は、`&` 文字をセパレータとして使用する。

### `path_template` 構文

上記の例で見られるように、`path_template` は正規表現や非正規表現マッチャーの実装への変換時にコードジェネレータによって解釈されるさまざまな記号を使用できる。`path_template` はセグメントデリミタで区切られたセグメントで構成される。`path_template` の構文は以下の通りである：

- 許容されるセグメントデリミタは `/` のみである。
  - `path_template` の末尾のデリミタは**してもよい**（may）無視する。
- セグメントは**しなければならない**（must）以下のタイプのいずれかである：
  - `*`：単一セグメントのワイルドカード。1つ以上の非 `/` 記号に対応する。これを記述する正規表現は `[^/]+` である。
    - 単一セグメントワイルドカードは通常、リソースIDを表す。
  - `**`：複数セグメントのワイルドカード。0個以上のセグメントに対応する。
    - 複数セグメントワイルドカードは**しなければならない**（must）最終セグメントとしてのみ出現するか、`path_template` 全体を構成する。
    - 複数セグメントの `path_template` では、複数セグメントワイルドカードは**しなければならない**（must）セグメントデリミタの直後に出現する。このデリミタはマッチング時に消費されるため、`foo/**` のような `path_template` は以下すべてにマッチする：`foo`、`foo/`、`foo/bar/baz`。
    - 複数セグメントの `path_template` で最終セグメントとして使用される場合、それを記述する正規表現は `([:/].*)?` である。
    - `path_template` 全体として使用される場合、それを記述する正規表現は `.*` である。
    - セグメントデリミタはマッチング時に消費される（先行するデリミタを含む）。
  - `LITERAL`：リテラルセグメント。リテラルセグメントは任意の英数字を含むことができる。
    - リテラルセグメントは**してはならない**（must not）この構文で予約された記号を含む。
    - リテラルセグメントは通常、リソースコレクションIDまたはベースパスを表す。
  - `{}`：可変セグメント。テンプレートで指定されたパスの一部にマッチする。
    - 可変セグメントは以下のいずれかである：
      - `{key}`。`key` はヘッダーのキーと値のペアで使用される名前である
      - `{key=template}`。`template` は `key` とペアになる値として抽出するセグメント（この `path_template` 構文で表現される）である
    - `{key}` のみの可変セグメントは、デフォルトで `*` のテンプレートになり、1つ以上の非 `/` 記号にマッチする。
      - `{key=*}` は技術的に有効な構文だが、よりシンプルな `{key}` 構文を**すべきである**（should）使用する。
    - 可変セグメントは**してはならない**（must not）他の可変セグメントを含む。この構文は再帰的ではない。
- セグメントは**してはならない**（must not）[AIP-4231][] に記載されている複合リソースIDを表す。ジェネレータはこの場合、エラーを出力**すべきである**（should）。

## Implicit Routing Headers（`google.api.http`）

**Note:** [`google.api.routing`][routing] アノテーションが付与されたRPCの場合、ルーティングヘッダーを追加する目的では [`google.api.http`][http] アノテーションは無視されなければならない。

unaryまたはserver-streaming RPCに [`google.api.routing`][routing] アノテーションが付与されていない場合、コードジェネレータは**しなければならない**（must）[`google.api.http`][http] アノテーションで宣言されたURIベースの変数を確認し、これらをunaryコールの `x-goog-request-params` ヘッダーに転記する。URIベースの変数とは、URI文字列中で中括弧内のキーとして宣言された変数である。例えば：

```proto
rpc CreateTopic(CreateTopicRequest) {
  option (google.api.http).post = "{parent=projects/*}/topics";
}
```

**Note:** リソースIDセグメントのパターンを省略することは許容される。例えば `{parent}` は `{parent=*}` と同等であり、解析**しなければならない**（must）。

この場合、該当する変数は `parent` であり、`CreateTopicRequest` の `parent` フィールドを参照する。ユーザーが `CreateTopicRequest` のインスタンスをメソッドに提供すると（またはclient libraryがそれを構築した後、メソッドオーバーロードの場合）、client libraryはキーと値を抽出し、それらを `x-goog-request-params` ヘッダーに追加**しなければならない**（must）。キーと値の両方は**しなければならない**（must）[RFC 6570 §3.2.2][rfc 6570 §3.2.2] に従ってURLエンコードする。これは標準ライブラリのURLエンコーディングで実行できる。例えば、GoでこのヘッダーをgRPCリクエストに追加する場合：

```go
md := metadata.Pairs("x-goog-request-params",
  url.QueryEscape("parent") + "=" + url.QueryEscape(req.GetParent()))
```

実行時に、名前付きパラメータと同じ名前のフィールドがリクエストメッセージで未設定の場合、そのパラメータに対応するキーと値のペアはルーティングヘッダーに含めては**ならない**（must not）。パラメータのいずれもルーティングヘッダーに含める必要がない場合、ルーティングヘッダーは送信**してはならない**（must not）。

[`google.api.http`][http] アノテーションに `additional_bindings` が含まれている場合、これらのパターンは**しなければならない**（must）追加のリクエストパラメータとして解析する。トップレベル（または `additional_bindings`）パターンで重複していないフィールドは**しなければならない**（must）リクエストパラメータに含め、同じ方法でエンコードする。

URLパラメータと同様に、複数のキーと値のペアがある場合は、`&` 文字をセパレータとして使用する。
