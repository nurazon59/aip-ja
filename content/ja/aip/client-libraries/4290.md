# Docker interface

API client libraryの生成に個別のジェネレータを使用することの結果として、各ジェネレータは実行するために独自の依存関係と要件のセットを持つことになる。

これは単一の環境で多くのライブラリを生成したいユーザーにとっては合理的であるが、単一のAPIを多くの言語や環境に対して生成したいユーザーにとっては課題を提示する。ユーザーは、言語ごとの準備作業を最小限に抑えつつ、簡単かつ迅速にライブラリを生成する方法を必要としている。

## Guidance

client libraryジェネレータは**すべきである**（should）ジェネレータを提供し共通インターフェースを公開する[Docker][]イメージを配布する。これにより、同じAPIを複数の言語で生成する場合、通常は適切なDockerイメージを置き換えるだけで済むようになる。Google作成のジェネレータのDockerイメージは一貫したスキームに従う。

### CLI usage

Dockerイメージ内のコードジェネレータを呼び出す際の期待されるユーザーコマンド（POSIXマシン上のprotoインポートルートから）：

```bash
$ docker run --rm --user $UID \
  --mount type=bind,source=`pwd`/a/b/c/v1/,destination=/in/a/b/c/v1/,readonly \
  --mount type=bind,source=/path/to/dest/,destination=/out/ \
  gcr.io/gapic-images/{GENERATOR} \
  [-- additional options...]
```

**Note:** これらの各コンポーネントはDockerエコシステムで標準的であるが（宛先パスの問題を除く。これはprotocがインポートを処理する方法に起因する）、これは依然としてかなり長いコマンドである。これをさらに簡略化するための[ショートカットスクリプト][shortcut script]を提供できるが、そのようなスクリプトは利便性のためであり、このインターフェースの代替ではない。

### Container composition

コンテナは**しなければならない**（must）以下を含める：

- プロトコルバッファコンパイラであるprotocの最新バージョン。
- すべてのAPIで使用が許可されている共通proto（[googleapis][]）。
- 該当するコードジェネレータプラグイン、およびそれが要求する依存関係。
  - コードジェネレータプラグイン自体は**すべきである**（should）ビルド時にホストマシンから `ADD` または `COPY` ステートメントを使用して追加され、ローカルにインストールされる。パッケージマネージャからプルする**すべきではない**（should not）。（これはリリースをカットする際に Catch-22 の状況を引き起こす。）
  - イメージは**してもよい**（may）該当するエコシステムのニーズに応じて、プラグインのプリコンパイル済みバイナリまたはインストール済みソースコードのいずれかを含めてもよい。
  - 依存関係のインストールは**すべきである**（should）適切なパッケージマネージャを使用する。

共通protoとprotocコンパイラは、独立したイメージ（`gcr.io/gapic-images/api-common-protos`）によって提供される。`protoc` と共通protoの両方は、`COPY --from` 構文（[マルチステージビルド][multi-stage builds]参照）を使用して、このイメージからジェネレータのイメージに取得できる。これはDockerの慣習に従う好ましいアプローチであり、protoを独立してバージョニングできる。

### Base images

**TL;DR:** 各言語はおそらく `language:x.y-alpine` または `language:x.y-slim` を望んでいる。例えば、`ruby:2.5-alpine` や `python:3.7-slim` など。（Alpineイメージは小さいが独特の特性がある。）

ベースイメージの選択には以下のガイドラインが適用される（おおよそ最も重要なものから順に）：

- イメージは**すべきである**（should）一般的に、ジェネレータが実装されている言語の最新安定バージョンの公式イメージをベースとする。
- イメージは**すべきである**（should）よく理解されたパッケージマネージャから必要なシステム依存関係をインストールできる。
- イメージは**すべきである**（should）最終的にAlpine、Debian、またはUbuntuをベースとする。これはGCRの[脆弱性スキャン][vulnerability scanning]の恩恵を受けるためである。
- イメージはDockerコミュニティの一般的な期待に沿って、可能な限り小さくするよう努めるべきである：
  - 可能な限り最小のベースイメージを使用する。Alpineベースのイメージは可能であれば素晴らしいが、常に合理的であるとは限らない。「Slim」Debianイメージは通常、次善の策である（そしておそらく多くの状況ではるかに実現可能である）。

### Mount points

`protoc` はディスクからproto（生成するAPIを表す）を読み取り、最終出力（client library）をディスクに書き出さなければならない。ユーザーはホストマシン上にAPI protoを持ち、最終的に出力をそのホストマシンに出力する必要があるため、Dockerイメージは**すべきである**（should）2つのマウントポイントを使用する。これにより抽象化レイヤーに穴が開く：ユーザーはホストマシン上の適切な場所をコンテナ内の適切な場所にマウントしなければならない。

コンテナ内の期待される場所は**しなければならない**（must）一定であり、すべてのジェネレータコンテナイメージ間で一貫している：

- `/in/`：生成するprotoの場所。これは**しなければならない**（must）インポートルートである。
  - 例：Language APIのprotoを生成する場合、Dockerイメージ内のprotoは `/in/google/cloud/language/v1/` に存在しなければならない。
- `/out/`：client libraryが書き出される場所。

### Plugin options

一部のマイクロジェネレータは、protocプラグインオプションを介して提供される設定をサポートしている。そのような場合、オプションはCLI入力からprotocコマンドにルーティングされなければならない。

最終的なprotoc呼び出しは以下のようになる：

```bash
protoc --proto_path {path/to/common/protos} --proto_path /in/ \
       --{LANG}_gapic_out /out/ \
       --{LANG}_gapic_opt "go-gapic-package=GO_PACKAGE_VALUE" \
       `find /in/ -name *.proto`
```

Dockerイメージの結果的な呼び出しは以下のようになる：

```bash
$ docker run --rm --user $UID \
      --mount type=bind,source=`pwd`/a/b/c/v1/,destination=/in/a/b/c/v1/,readonly \
      --mount type=bind,source=/path/to/dest/,destination=/out/ \
      gcr.io/gapic-images/{GENERATOR} \
      --go-gapic-package GO_PACKAGE_VALUE
```

したがって、Dockerイメージの呼び出しをラップするために書かれるショートカットスクリプトは**しなければならない**（must）`--` の後に発生するすべてのオプションを基盤となるdocker runコマンドに渡す。内部のDockerイメージは**しなければならない**（must）通常のシェル構文からprotocオプション構文への変換を提供しなければならない。

プラグインオプションを利用するclient libraryジェネレータは**しなければならない**（must）それらのオプションをフラグまたは `key=value` ペアのいずれかとして受け入れる。（ジェネレータが `=` 文字なしで文字列を受け取った場合、それはフラグであり、暗黙の値は `true` である。）複数のオプションが提供された場合、それらはカンマ区切りとなる。これは、複数の `--opt` フラグが指定された場合のprotocの動作に準拠するためである。

さらに、ジェネレータは**すべきである**（should）理解するすべてのオプションキーにそのジェネレータのターゲット言語をプレフィックスとして付け（例：`go-gapic-package`、`java-gapic-package`）、キーには**すべきである**（should）`kebab-case` を使用する（protocが一貫していないため、Dockerに合わせるため）。

マイクロジェネレータは**してはならない**（must not）認識しないオプションキーでエラーを出す。ただし、警告を発しても**してもよい**（may）。

### Publishing images

Google作成のジェネレータのイメージは**すべきである**（should）[Google Container Registry][] の専用プロジェクトである `gcr.io/gapic-images` に公開される。

イメージは**すべきである**（should）以下の命名スキームに従う：

    gcr.io/gapic-images/gapic-generator-{lang}

CIは**すべきである**（should）リリースが作成されたときに新しいDockerイメージをレジストリにプッシュするように構成される。GitHubでリリースがタグ付けされた場合（バージョン番号、例えば 1.0.3）、CIサービスはそのタグのコードに基づいてイメージをビルドすべきである。

結果として得られるイメージは**すべきである**（should）バージョン番号の各コンポーネントと `latest` でタグ付けされ、結果のタグがレジストリにプッシュされる。（これは適切であればパッケージマネージャへのプッシュに追加されるものであり、このAIPの範囲外である。）

つまり、GitHubでのリリースタグ `1.0.3` は、以下の4つのタグをGCRにプッシュすることになる：

- `gcr.io/gapic-images/gapic-generator-{lang}:1`
- `gcr.io/gapic-images/gapic-generator-{lang}:1.0`
- `gcr.io/gapic-images/gapic-generator-{lang}:1.0.3`
- `gcr.io/gapic-images/gapic-generator-{lang}:latest`

**Note:** これらのルールは、リリースが常に増加するバージョン番号を持つことを前提としている。ジェネレータが複数のバージョンストリームを同時に維持する必要がある場合、このプロセスは若干修正される必要がある。
