# Common service client mixins

多くの場合、APIはリソース管理のための共通機能を持ち、これらはユーティリティサービスに分割される。これらの共通ユーティリティサービスは、個別のプロダクトサービス間で共有される。これらの共通サービスはそれぞれ中央で定義されたサーフェスを持つが、個々のインスタンスは各プロダクトサービスと一緒にホストされる。したがって、プロダクトサービスによってホストされる各共通サービスのサーフェスは、実行時に「mixin」される。

## Guidance

client libraryは**してもよい**（may）自身のAPIがmixinとして宣言する共通サービスのクライアントメソッドを提供する。そのようなメソッドは、mixinサービスをホストするプロダクトサービスと通信するように構成されたクライアントからmixinメソッドを提示することで、ユーザーエクスペリエンスを向上させる。

client libraryがmixinサービスをサポートする場合、以下の共通サービスをサポート**しなければならず**（must）（その他は**してもよい**（may）サポートする）：

- `google.cloud.location.Locations`
- `google.iam.v1.IAMPolicy`
- `google.longrunning.Operations`

**Note:** サポートされるmixinサービスのリストとmixinサービス自体は、どちらも頻繁には変更されない。サポートされる新しい共通サービスは**しなければならない**（must）ここにリストに追加され、ジェネレータは**しなければならない**（must）更新される。

client libraryに生成されるためには、mixinサービスは [google.api.Service] の [apis] の下で宣言され**なければならない**（must）。さらに、同じ [google.api.Service] の [http] 設定で宣言された [google.api.http] バインディングを持つRPCのみを生成できる（これらのバインディングの詳細はAIP-127を参照）。mixinサービスのRPCが [google.api.Service] で宣言された [google.api.http] ルールを持たない場合、生成**してはならない**（must not）。

## Implementing mixin support

### Generator configuration

client libraryジェネレータは**しなければならない**（must）YAML形式の [google.api.Service] のファイルパスを受け入れる。このファイルは**しなければならない**（must）optionalであるフラグを介して指定する。言い換えれば、mixinメソッドを省略した基本的なクライアント生成では、このフラグは**requiredであってはならない**（must not）。

### Mixin API client configuration

mixin APIがclient libraryの設定（アノテーション、デフォルトのリトライ設定など）（例：`google.api.method_signature` やリクエストヘッダーインジェクション）を利用する場合、設定がアクセス可能であれば、ジェネレータは**してもよい**（may）生成されたmixinメソッドでそれをサポートしてもよい。

client libraryの設定が完全修飾名を要求する場合、_mixinの_ 完全修飾名が使用され**なければならない**（must）。つまり、ホストサービスはmixin要素を自身のパッケージ内にあるかのように参照**してはならない**（must not）。

例えば、デフォルトのリトライとタイムアウト設定で `google.iam.v1.IAMPolicy.GetIAMPolicy` を構成する場合は以下のようになる：

```json
{
  "name": [{ "service": "google.iam.v1.IAMPolicy", "method": "GetIAMPolicy" }],
  "timeout": "60s",
  "retryPolicy": {
    "initialBackoff": "0.1s",
    "maxBackoff": "5s",
    "backoffMultiplier": 1.3,
    "retryableStatusCodes": ["UNKNOWN"]
  }
}
```

### Generating mixin methods

mixin APIのRPCは**すべきである**（should）ホストAPIのサービスclient libraryのサーフェス上に、ホストサービスのRPCと並んでメソッドとして生成される。これにより、クライアント上のトップレベルメソッドとして提示されるが、内部的には適切なmixin APIスタブまたはリクエストが呼び出される。ただし、ジェネレータは**してもよい**（may）より言語慣用的であれば、mixin APIのRPCを異なる方法で提示してもよい。

**Note:** gRPCクライアントの場合、gRPCリクエストを適切に構築するためにmixin APIのgRPCスタブを使用**しなければならない**（must）。例えば、生成されたGoのgRPCスタブ [LocationsClient] を使用して `google.cloud.location.Locations` mixinのRPCを呼び出さなければならず、ホストサービスのgRPCスタブを使用してはならない。

### Multiple host service clients

ホストAPIによって複数のサービスが定義され、その結果複数のクライアントが生成される場合、ジェネレータは**しなければならない**（must）該当するすべての生成サービス クライアントにmixinメソッドを含める。

### Overriding a duplicate RPC

client libraryジェネレータは**してはならない**（must not）ホストサービスが同じ名前のRPCを既に定義している場合、そのホストサービス クライアントにmixinメソッドを生成する。例えば、以下のサービス定義を考えよう：

```proto
service LibraryService {
  rpc ListBooks(ListBooksRequest) returns (ListBooksResponse);

  // Other host API RPCs...

  // Redefinitions of the google.iam.v1.IAMPolicy mixin service.
  rpc GetIamPolicy(google.iam.v1.GetIamPolicyRequest)
      returns (google.iam.v1.Policy);

  rpc SetIamPolicy(google.iam.v1.SetIamPolicyRequest)
      returns (google.iam.v1.Policy);

  rpc TestIamPermissions(google.iam.v1.TestIamPermissionsRequest)
      returns (google.iam.v1.TestIamPermissionsResponse);
}
```

ホストAPIが `google.iam.v1.IAMPolicy` をmixinサービスとして宣言した場合、client libraryジェネレータは**してはならない**（must not）ホストprotoパッケージ内の _任意の_ サービス クライアントについて、上記で明示的に宣言された名前と一致する結果のmixinメソッドを生成するが、このmixinの他のメソッドは生成**しなければならない**（must）。この場合、ジェネレータは**してもよい**（may）競合が回避されたことを示す警告をログに記録するか、該当するmixinメソッドをサイレントにスキップする。
