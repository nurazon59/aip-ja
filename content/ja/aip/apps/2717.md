# ジェネリックフィールドのパターン

開発者は、protoメッセージ内でジェネリックな値を表現する方法として複数の選択肢を持つ。どれを選ぶかには理由があり、それらを理解することで、より良い一貫性のあるAPIを実現できる。

## ガイダンス

APIは、`oneof`、`map<string, Foo>`、`Any`、`Struct` の一貫した適用に従う**べきである**（should）。

### `oneof`

`oneof` は、オプショナルなフィールドのセットに制約を設け、そのうちの1つだけが設定可能であることを強制するために使用される（これらのフィールドは依然として個別のフィールドである）。一般的なパターンとして、複数のメッセージタイプからなる単一の `oneof` コレクションを含むメッセージがある。このような `oneof` メッセージは、概念的にCの `union` やC++の `std::variant` に類似している。これらは、ジェネリックなメッセージタイプが必要なほとんどの場所で、他のアプローチよりも優先して使用される**べきである**（should）。

**注:** 既存の `oneof` に追加の値を加えることは非破壊的変更であるが、既存のフィールドを `oneof` に出入りさせることは破壊的変更である（Go protobufスタブに後方互換性のない変更が生じる）。

### `map<string, Foo>`

よりジェネリックな構造が必要な場合、文字列からオブジェクトへのマップが使用されて**もよい**（may）。このようなマップは、通常のJSONオブジェクト（`{"a": "foo", "b": "bar"}` など）で表現される。このようなマップの欠点は、フラットな構造に限定されること、および多くの文字列定数が必要になる可能性があるため扱いが難しいことである。

### `Any`

[`Any`][any]は、任意のメッセージをフィールドにパックすることを可能にする。これは概念的に、シリアライズされたメッセージを含む「bytes」フィールドに類似している。`Any` の利点は、ユーザー定義のprotoメッセージを型情報とともに格納できることである。ユーザーは `Any` 内にどの種類のオブジェクトがあるかを認識できる必要があり、これが設計を複雑にする。欠点は、コードがパックされたメッセージを逆シリアライズする方法を知っている場合もあれば知らない場合もあるため、any protosの操作とデバッグがはるかに複雑になることである。また、開発者は `Any` 内の予期しないタイプのメッセージに対処する必要がある。

`Any` はリクエストパラメータとして使用される**べきではない**（should not）。リクエストパラメータは、固定されたタイプのセットになる（その場合は `oneof` を使用する**べきである**（should））、または型記述子を一緒に送信する必要がある（その場合は struct を使用する**べきである**（should））かのいずれかであり、struct でははるかに単純なセマンティクスで本質的に同じことを実現できる。

### `Struct`

[`Struct`][struct]メッセージは、任意のネストされたJSONを表現するために使用できる。例えば、以下のコードを考える：

```java
Struct.Builder builder = Struct.newBuilder();
Value town = Value.newBuilder().setStringValue("Springfield").build();
Value population = Value.newBuilder().setNumberValue(273).build();
builder.putFields("town", town);
builder.putFields("population", population);
Struct survey = builder.build();
```

`survey` は、実際のJSON `{"town": "Springfield": "population": 273}` としてシリアライズされる。このメッセージタイプは比較的稀であり、めったに使用される**べきではない**（should）。
